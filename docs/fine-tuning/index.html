Part 0

읽기 전에

지은이 문성혁

이 책의 목표는 모델을 한 번 학습시켜 보는 데서 끝나지 않는다. 좋은 데이터가 무엇인지, 어떤 상황에서 파인튜닝이 필요한지, 학습 결과를 어떻게 검증해야 하는지를 한 흐름으로 익히는 것이다.

예상 읽기 시간 12분 실습 없음 난이도 입문

Learning map

이 책은 판단력을 먼저 만든다

파인튜닝은 코드를 실행하는 기술이지만, 성공 여부는 실행 전에 거의 결정된다. 어떤 문제를 풀 것인지, 어떤 데이터를 만들 것인지, 무엇을 기준으로 좋아졌다고 볼 것인지가 먼저다.

  1. 01 문제 정의 RAG, 프롬프트, 파인튜닝 중 무엇이 맞는지 판단한다.

  3. 03 학습과 비교 작게 학습하고, 학습 전후 응답을 바로 비교한다.
  4. 04 평가와 운영 좋아 보이는 답변이 아니라 재현 가능한 기준으로 판단한다.

0.3

이 책에서 깊게 다루지 않는 것

이 책은 입문부터 실무 적용까지의 파인튜닝에 초점을 둔다. 따라서 초거대 모델을 처음부터 사전학습하는 인프라, 수백 장 이상의 GPU 클러스터 운영, 커널 수준 최적화는 핵심 범위에서 제외한다.

범위

사전학습과 파인튜닝은 다른 문제다

사전학습은 모델이 언어와 세계 패턴을 넓게 배우는 단계이고, 파인튜닝은 이미 학습된 모델의 행동을 특정 목적에 맞게 조정하는 단계다. 이 책은 두 번째 문제에 집중한다.

0.4

사전학습은 얼마나 다른가

사전학습과 파인튜닝은 둘 다 모델의 가중치를 바꾸지만, 문제의 크기와 목적이 완전히 다르다. 사전학습은 인터넷 규모의 텍스트, 코드, 논문, 문서에서 언어와 세계 패턴을 넓게 배우는 단계다. 반면 파인튜닝은 이미 만들어진 모델이 특정 업무에서 어떤 형식과 태도로 답해야 하는지 좁게 조정하는 단계다.

사전학습 언어 능력과 일반 지식을 만드는 대규모 학습이다. 데이터, GPU, 시간, 비용이 모두 거대하다.
파인튜닝 이미 학습된 모델의 반복 행동을 목적에 맞게 조정한다. 데이터 품질과 평가 기준이 더 직접적으로 중요하다.
실무 판단 새로운 기본 능력을 만들려는 것이 아니라면 대부분은 사전학습이 아니라 파인튜닝, RAG(Retrieval-Augmented Generation), 프롬프트 설계의 문제다.

0.5

처음 만나는 약어

파인튜닝 자료는 약어가 많다. 이 책에서는 처음 등장할 때 가능한 한 풀네임을 함께 적고, 이후에는 짧은 약어를 사용한다. 아래 표는 뒤에서 반복해서 만날 기본 약어다.

SFTSupervised Fine-Tuning. 정답 예시를 보고 답변 형식을 따라 하게 만드는 지도 파인튜닝.
LoRALow-Rank Adaptation. 전체 모델 대신 작은 저랭크 adapter만 학습하는 방식.
QLoRAQuantized Low-Rank Adaptation. 양자화된 모델 위에서 LoRA adapter를 학습하는 메모리 절약 방식.
DPODirect Preference Optimization. chosen/rejected 선호 쌍으로 답변 선택 경향을 조정하는 방법.
RLHFReinforcement Learning from Human Feedback. 사람의 선호 피드백을 보상 신호로 사용하는 정렬 방법.
RAGRetrieval-Augmented Generation. 외부 문서를 검색해 모델 입력에 붙인 뒤 답하게 하는 구조.
PEFTParameter-Efficient Fine-Tuning. 일부 파라미터만 학습해 비용을 줄이는 파인튜닝 계열.
VRAMVideo RAM. GPU에서 모델 가중치, activation, KV cache 등을 담는 메모리.

0.8

첫 판단 기준: 프롬프트, RAG, 파인튜닝

많은 실패는 파인튜닝이 필요 없는 문제에 파인튜닝을 적용하면서 시작된다. RAG는 Retrieval-Augmented Generation의 약자로, 외부 검색 결과를 모델 입력에 붙여 답하게 하는 구조다. 아래 비교는 앞으로 모든 장에서 반복해서 사용할 첫 번째 판단 기준이다.

문제가 작고 규칙이 자주 바뀐다면 프롬프트부터

출력 형식, 말투, 간단한 판단 기준은 프롬프트로 먼저 검증한다. 실패가 반복될 때 데이터셋으로 옮긴다.

지식이 자주 바뀌거나 출처가 필요하면 RAG

가격표, 사내 문서, 정책처럼 최신성이 중요한 정보는 모델 내부에 외우게 하지 말고 검색으로 붙인다.

반복되는 행동을 바꾸려면 파인튜닝

답변 형식, 업무 절차, 도메인별 판단 패턴, 거절 방식처럼 반복되는 행동은 데이터셋으로 학습시킨다.

Part 1

LLM 파인튜닝 기초를 촘촘히 이해하기

1부는 실습 전에 필요한 판단력을 만든다. 모델이 무엇을 배우는지, 파인튜닝이 어떤 행동을 바꾸는지, 데이터와 템플릿이 왜 결과를 좌우하는지 차근차근 정리한다.

1장개념입문

LLM을 파인튜닝한다는 것

파인튜닝을 지식 주입이 아니라 모델 행동을 조정하는 과정으로 이해한다.

  • LLM은 무엇을 학습하는가
  • 파인튜닝은 지식을 넣는 일인가, 행동을 바꾸는 일인가
  • 사전학습, SFT(Supervised Fine-Tuning), RLHF, DPO의 위치
  • 파인튜닝이 필요한 경우와 필요 없는 경우
바로 확인: 같은 질문에 base model과 instruction model이 다르게 답하는 예시

파인튜닝은 모델을 새로 만드는 일이 아니다

파인튜닝은 이미 사전학습된 모델을 출발점으로 삼는다. 그래서 파인튜닝의 목표는 모델에게 한국어 문법, 프로그래밍 언어, 일반 상식을 처음부터 가르치는 것이 아니다. 모델이 이미 할 수 있는 일을 특정 업무에 맞는 형식, 말투, 절차, 판단 기준으로 반복하게 만드는 것이다.

사전학습과 파인튜닝의 가장 큰 차이

사전학습 모델의 기본 언어 능력과 세계 패턴을 만든다. 다음 토큰 예측을 대규모로 반복하며, 수많은 주제와 표현 방식을 넓게 흡수한다.
파인튜닝 이미 배운 능력을 특정 목적에 맞게 정렬한다. 예를 들어 고객지원 답변 형식, 코드 리뷰 말투, 법률 문서 요약 절차처럼 반복되는 행동을 좁게 조정한다.

지식을 넣는 것보다 행동을 바꾸는 것에 가깝다

물론 도메인 데이터로 파인튜닝하면 특정 분야의 표현과 패턴을 더 잘 따라갈 수 있다. 하지만 자주 바뀌는 사실, 가격표, 최신 정책, 사내 문서 원문을 모델 내부에 외우게 하는 것은 대체로 좋은 전략이 아니다. 그런 정보는 RAG로 검색해 붙이는 편이 낫다. 파인튜닝은 검색된 정보를 어떤 순서로 읽고, 어떤 기준으로 요약하고, 어떤 형식으로 답할지를 학습시키는 데 더 잘 맞는다.

학습 단계의 위치

사전학습은 base model을 만든다. SFT(Supervised Fine-Tuning)는 사람이 원하는 지시-응답 형식을 가르친다. RLHF(Reinforcement Learning from Human Feedback)나 DPO(Direct Preference Optimization)는 여러 답변 중 더 선호되는 답변을 고르는 경향을 조정한다. 따라서 실무에서 파인튜닝을 시작한다는 말은 보통 base model을 가져와 SFT나 LoRA(Low-Rank Adaptation) 계열 방법으로 특정 행동을 더 자주 나오게 만드는 일을 뜻한다.

판단 기준

새 사실을 넣고 싶다면 먼저 RAG를 의심하라

모델이 모르는 최신 정보를 답하게 만들고 싶다면 파인튜닝보다 RAG가 먼저다. 반대로 같은 자료를 읽고도 매번 답변 형식이 흔들리거나, 업무 절차를 자주 어긴다면 파인튜닝을 검토할 만하다.

2장토큰입문

토큰, 문맥, 출력의 원리

토큰화와 문맥 길이가 비용, 품질, 학습 가능성에 미치는 영향을 다룬다.

  • 토큰화가 성능과 비용에 미치는 영향
  • context window와 sequence length
  • special token, BOS/EOS/PAD의 의미
  • chat template이 중요한 이유
바로 확인: 같은 문장을 모델별 토크나이저로 나눠 보기

모델은 글자가 아니라 토큰을 본다

LLM은 문장을 그대로 읽지 않는다. 먼저 문장을 토큰이라는 작은 단위로 쪼개고, 각 토큰의 숫자 ID를 입력으로 받는다. 영어 단어 하나가 한 토큰이 되기도 하고, 한국어 조사나 어미가 따로 쪼개지기도 한다. 그래서 같은 문장이라도 토크나이저가 다르면 토큰 수, 비용, 잘리는 위치가 모두 달라질 수 있다.

문맥 길이는 입력 한도가 아니라 학습 설계 조건이다

context window는 모델이 한 번에 볼 수 있는 토큰 범위다. 학습 데이터의 입력과 출력이 이 범위를 넘으면 일부가 잘리거나, 중요한 지시가 뒤쪽 출력에 밀려 학습되지 않는다. 파인튜닝 데이터는 "넣고 싶은 정보"가 아니라 "모델이 실제로 보고 배울 수 있는 길이"를 기준으로 잘라야 한다.

special token과 chat template

BOS(Beginning of Sequence), EOS(End of Sequence), PAD(Padding) 같은 special token은 문장의 시작과 끝, 빈 칸을 표시한다. chat template은 system, user, assistant 메시지를 모델이 학습한 대화 형식으로 바꾼다. 템플릿이 맞지 않으면 좋은 데이터도 이상한 문자열로 보이고, 모델은 답변해야 할 위치를 헷갈린다.

texts = [
    "환불 정책을 알려줘",
    "Please summarize the refund policy."
]

for text in texts:
    tokens = text.split()
    print(text, "=>", len(tokens), tokens)

Transformer 세부 수식보다 학습할 때 실제로 영향을 주는 구조를 중심으로 설명한다.

  • Attention을 문맥 참조로 이해하기
  • layer, hidden state, logits
  • loss가 줄어든다는 말의 의미
  • gradient와 backpropagation 직관
바로 확인: logits에서 다음 토큰 확률이 만들어지는 과정

Attention은 중요한 단어를 다시 보는 장치다

Transformer의 attention은 현재 토큰이 앞뒤 문맥의 어떤 토큰을 더 참고할지 계산한다. 파인튜닝 관점에서는 수식보다 "모델이 답변을 만들 때 무엇을 근거로 삼게 만들 것인가"가 중요하다. 데이터 안에서 지시, 제약, 예시, 출력 형식이 일관되면 attention이 참고해야 할 패턴도 안정된다.

logits, 확률, loss

모델은 다음 토큰 후보마다 점수인 logits를 만든다. softmax를 통과하면 확률 분포가 되고, 정답 토큰의 확률이 낮으면 loss가 커진다. 학습은 정답 토큰의 확률을 높이는 방향으로 가중치를 조금씩 바꾸는 과정이다.

loss가 줄어도 답변이 좋아졌다는 뜻은 아닐 수 있다

loss는 학습 데이터의 다음 토큰을 맞히는 정도를 본다. 데이터가 부정확하거나 형식이 흔들리면 낮은 loss는 오히려 나쁜 습관을 잘 외웠다는 신호가 될 수 있다. 그래서 학습 로그와 함께 고정된 평가 질문의 실제 응답을 반드시 비교해야 한다.

logits_to_prob.pyexpected output
import math

logits = {"환불": 2.0, "배송": 0.5, "교환": 1.0}
total = sum(math.exp(v) for v in logits.values())

for token, value in logits.items():
    print(token, round(math.exp(value) / total, 3))
출력 예시

환불 0.629

배송 0.140

교환 0.231

4장방법론기초

파인튜닝 방법의 지도

Full fine-tuning, LoRA(Low-Rank Adaptation), QLoRA(Quantized Low-Rank Adaptation), DPO(Direct Preference Optimization)의 차이를 자원과 목적 기준으로 비교한다.

  • Full fine-tuning
  • LoRA, QLoRA, Adapter, Prompt tuning
  • DPO/RLHF(Reinforcement Learning from Human Feedback)의 목적과 위치
  • 방법별 비용, 난이도, 효과 비교표
바로 확인: 상황별 파인튜닝 방법 선택표 작성

Full fine-tuning은 강하지만 비싸다

Full fine-tuning은 모델의 대부분 또는 전체 파라미터를 업데이트한다. 도메인이 크게 다르고 충분한 데이터와 GPU가 있을 때 강력하지만, 비용이 크고 기존 능력을 잃는 catastrophic forgetting 위험도 커진다.

LoRA와 QLoRA는 실무 기본 선택지다

LoRA(Low-Rank Adaptation)는 기존 가중치를 얼려두고 작은 저랭크 행렬만 학습한다. 저장 파일이 작고, 여러 adapter를 갈아 끼울 수 있어 실험이 빠르다. QLoRA(Quantized Low-Rank Adaptation)는 기본 모델을 4-bit로 로드해 VRAM(Video RAM)을 줄이고, LoRA adapter만 학습하는 방식이다.

DPO와 RLHF는 답변의 선호 경향을 조정한다

SFT(Supervised Fine-Tuning)가 "이런 질문에는 이런 답을 하라"를 가르친다면, DPO(Direct Preference Optimization)와 RLHF(Reinforcement Learning from Human Feedback)는 "A보다 B가 낫다"는 선호를 학습한다. 친절함, 거절 방식, 장황함, 안전성처럼 단일 정답보다 비교 판단이 중요한 영역에 잘 맞는다.

데이터가 적고 GPU가 제한적 LoRA 또는 QLoRA로 시작한다. 짧은 실험을 여러 번 돌려 품질 기준을 먼저 잡는다.
선호되는 답변 스타일이 핵심 SFT로 기본 행동을 만든 뒤 DPO로 chosen/rejected 선호를 좁힌다.

파인튜닝 실패를 하이퍼파라미터 문제로만 보면 원인을 놓치기 쉽다. 같은 질문에 서로 다른 형식의 답이 섞여 있거나, 틀린 사실을 정답처럼 적거나, system message와 assistant 답변이 서로 다른 정책을 말하면 모델은 그 모순까지 배운다.

과적합과 망각

데이터가 너무 작거나 반복되면 모델은 특정 문장을 외운다. 반대로 학습률이 높고 epoch가 많으면 원래 잘하던 일반 능력이 흐려질 수 있다. 학습 데이터 loss만 보지 말고, 보류한 평가셋과 일반 능력 질문을 함께 확인해야 한다.

실패 신호

응답이 좋아 보이는데도 같은 형식으로만 반복된다

모든 질문에 같은 문장 구조로 답하거나, 근거가 없어도 확신하는 말투가 늘었다면 데이터 다양성, 거절 샘플, 평가셋 누수를 먼저 의심한다.

6장품질 기준핵심

좋은 데이터셋의 조건

데이터의 양보다 품질이 중요한 이유와 샘플 하나를 판단하는 기준을 만든다.

  • 정확성, 일관성, 다양성
  • 난이도 분포와 도메인 적합성
  • 거절/안전 응답 포함 여부
  • 중복과 모순 샘플의 위험
바로 확인: 샘플 10개를 품질 기준표로 점검하기

좋은 샘플은 하나의 기준을 분명히 가르친다

좋은 데이터셋은 단순히 많은 질문과 답변을 모은 것이 아니다. 각 샘플은 모델에게 어떤 상황에서 어떤 판단을 하고, 어떤 형식으로 답해야 하는지 분명히 보여줘야 한다. 모호한 instruction, 근거 없는 output, 지나치게 쉬운 반복 샘플은 양이 많아도 효과가 낮다.

품질 기준표

정확성답변이 사실과 정책에 맞는가. 모르면 모른다고 말하는가.
일관성같은 유형의 질문에 같은 형식과 기준으로 답하는가.
다양성쉬운 질문, 경계 사례, 예외 상황, 거절 상황이 함께 있는가.
적합성실제 서비스에서 자주 만나는 표현과 업무 흐름을 반영하는가.

안전 응답도 학습 데이터다

모델이 해서는 안 되는 답변을 피하길 원한다면 거절 샘플도 넣어야 한다. 다만 모든 위험 질문에 같은 문장으로 거절하면 과도하게 방어적인 모델이 된다. 위험의 이유, 가능한 대안, 안전한 범위의 도움을 함께 보여주는 데이터가 필요하다.

Instruction 데이터 설계

  • instruction/input/output 구조
  • system/user/assistant 구조
  • 단일턴과 멀티턴
  • 좋은 instruction과 나쁜 instruction
바로 확인: 나쁜 샘플을 좋은 instruction 샘플로 고치기

instruction은 업무 요구사항을 데이터로 쓴 것이다

instruction 데이터는 사용자의 요구, 필요한 입력, 기대 출력이 분리되어 있어야 한다. "잘 답해줘"처럼 넓은 지시는 모델에게 배울 기준을 주지 못한다. 좋은 instruction은 역할, 제약, 출력 형식, 판단 기준을 한 번에 읽을 수 있어야 한다.

8장대화핵심

대화형 데이터셋 만들기

멀티턴 대화에서 역할, 맥락, 정책, 말투가 일관되게 유지되도록 설계한다.

  • system message의 역할
  • 역할별 메시지 설계
  • 멀티턴 대화에서 맥락 유지
  • 말투, 형식, 정책 고정

대화 데이터의 핵심은 역할 분리다

멀티턴은 이전 답변의 책임까지 포함한다

도메인 데이터셋 설계

도메인은 말투가 아니라 위험 구조다

고객지원, 법률, 의료, 금융, 코드는 모두 다른 실패 비용을 가진다. 고객지원은 정책 오안내가 비용을 만들고, 의료와 금융은 부정확한 조언이 직접 피해로 이어질 수 있다. 도메인 데이터셋은 이 위험 구조를 기준으로 나눠야 한다.

도메인별 금지 사례를 먼저 쓴다

좋은 답변 예시만 모으면 모델은 위험한 요청을 만났을 때도 답하려고 한다. 의료 진단 확정, 투자 수익 보장, 법률 판단 단정, 보안 우회 절차 제공처럼 금지하거나 제한해야 하는 사례를 별도 카테고리로 만든다.

  • 정책 위반 샘플 처리
바로 확인: JSONL 데이터셋 자동 검사 리포트 만들기

정제는 삭제가 아니라 학습 신호를 선명하게 만드는 일이다

데이터 정제의 목적은 모델이 배울 신호를 흐리는 샘플을 줄이는 것이다. 중복, 비어 있는 답변, 지나치게 짧은 답변, 개인정보, 서로 모순되는 정책, 출처 없는 단정은 학습 전에 걸러야 한다.

자동 검사와 수동 검수를 나눈다

길이, 중복, 필드 누락, 금칙어, 이메일/전화번호 패턴은 자동으로 잡을 수 있다. 사실성, 미묘한 정책 위반, 답변 품질은 사람이 샘플링해서 봐야 한다. 자동 검사 리포트는 수동 검수 시간을 줄이는 장치다.

11장합성 데이터실무

합성 데이터 생성

강한 모델로 데이터를 만들 때 얻는 이점과 편향, 무단 증류 위험을 함께 다룬다.

  • 강한 모델로 데이터 만들기
  • self-instruct와 seed prompt
  • 합성 데이터의 편향과 위험
  • 검수 없이 합성 데이터를 쓰면 생기는 문제
바로 확인: 합성 QA 20개를 만들고 품질 점수 매기기

합성 데이터는 출발점이지 정답지가 아니다

강한 모델로 합성 데이터를 만들면 빠르게 양을 늘릴 수 있다. 하지만 생성 모델은 그럴듯한 오류, 편향된 표현, 실제 서비스에 없는 질문을 함께 만든다. 합성 데이터는 반드시 seed prompt, 검수 기준, 샘플링 비율을 함께 관리해야 한다.

seed prompt가 데이터 분포를 결정한다

"고객지원 QA를 만들어줘"보다 "환불, 배송, 교환, 계정 문제를 난이도별로 만들고 예외 상황을 30% 포함하라"가 훨씬 낫다. 합성 데이터는 프롬프트의 빈틈을 그대로 증폭한다.

주의

다른 상용 모델의 출력물을 무단 학습에 쓰지 않는다

약관이 허용하지 않는 방식으로 모델 출력을 모아 경쟁 모델을 학습시키면 기술적으로는 증류지만, 법적·윤리적 리스크가 크다. 합성 데이터 생성 권한과 사용 범위를 먼저 확인한다.

15장LoRA실습

LoRA

LoRA(Low-Rank Adaptation)는 전체 모델이 아니라 작은 adapter만 학습하는 효율적 접근이다.

  • LoRA rank와 target module
  • adapter 저장/로드
  • 학습 전후 차이 확인
바로 확인: adapter 적용 전후 응답 차이 보기

LoRA는 작은 변경층을 덧붙이는 방식이다

LoRA(Low-Rank Adaptation)는 원래 모델 가중치를 직접 크게 바꾸지 않고, 일부 선형층에 작은 저랭크 행렬을 붙여 학습한다. 덕분에 저장 파일이 작고, 여러 업무용 adapter를 따로 관리하기 쉽다.

16장QLoRA실습

QLoRA

QLoRA(Quantized Low-Rank Adaptation)는 저사양 GPU에서도 시도할 수 있는 4-bit 기반 파인튜닝 전략이다.

  • 4-bit 로딩
  • VRAM 절약 원리
  • 저사양 GPU 전략
  • 학습 가능 여부 즉시 확인
바로 확인: VRAM 사용량과 batch size 비교

QLoRA는 모델을 작게 싣고 adapter를 학습한다

QLoRA(Quantized Low-Rank Adaptation)는 기본 모델을 4-bit로 로드해 메모리 사용량을 줄이고, LoRA adapter만 학습한다. 모델 품질을 완전히 공짜로 유지하는 마법은 아니지만, 제한된 GPU에서 실험을 시작하게 해주는 매우 실용적인 방법이다.

VRAM이 부족할 때 먼저 조정할 것

batch size를 줄이고, gradient accumulation으로 유효 batch를 맞춘다. sequence length를 줄이면 메모리 사용량이 크게 줄지만, 긴 문맥 업무를 학습하기 어려워진다. rank를 낮추는 것도 방법이지만 표현력이 줄어드는 trade-off가 있다.

case B: batch=1, seq=1024, rank=8   -> likely trainable
case C: batch=1, seq=2048, grad_acc=4 -> slower, but stable
해석

VRAM 문제는 batch, sequence length, rank 순서로 줄여 보며 확인한다.

17장DPO실습

DPO 맛보기

DPO(Direct Preference Optimization)는 선호도 데이터가 모델의 선택 경향을 어떻게 바꾸는지 보여준다.

  • chosen/rejected 데이터
  • SFT와 DPO 차이
  • 선호도 튜닝 결과 비교
바로 확인: chosen/rejected 샘플 하나로 학습 목표 이해하기

DPO는 정답 하나보다 선호 비교를 사용한다

DPO(Direct Preference Optimization)는 같은 prompt에 대해 chosen 답변과 rejected 답변을 비교해 모델의 선호를 조정한다. "무엇이 정답인가"보다 "둘 중 무엇이 더 나은가"가 중요한 상황에 맞다.

SFT 뒤에 붙이는 경우가 많다

SFT가 기본 형식과 업무 절차를 잡아주면, DPO는 그 안에서 더 선호되는 말투와 판단을 좁힌다. 처음부터 DPO만 쓰기보다 SFT로 최소 행동을 만든 뒤 적용하는 편이 안정적이다.

평가는 단일 점수가 아니라 여러 관점의 판정이다

파인튜닝 모델은 정확성, 형식 준수, 안전성, 도움 정도, 일관성을 따로 봐야 한다. 하나의 평균 점수로 합치면 중요한 실패가 묻힌다. 특히 도메인 업무에서는 "대체로 친절함"보다 "중요한 예외를 놓치지 않음"이 더 중요할 수 있다.

LLM 평가자는 대량 비교를 빠르게 해주지만, 편향과 채점 흔들림이 있다. 루브릭을 명확히 쓰고, 일부 샘플은 사람이 다시 채점해야 한다. 최종 판단은 자동 점수, 사람 평가, 실제 실패 로그를 함께 놓고 내려야 한다.

정확성정책, 코드, 도메인 사실이 맞는가.

안전성거절해야 하는 요청을 안전하게 처리했는가.
유용성다음 행동이나 판단 근거를 충분히 제공했는가.
19장서빙운영

추론과 서빙

학습된 모델을 실제 서비스에서 호출할 때 필요한 추론 설정과 서빙 방식을 정리한다.

  • temperature, top_p, max_new_tokens
  • vLLM, TGI, llama.cpp
  • FastAPI API 서버
  • latency와 throughput의 균형
바로 확인: 같은 모델을 다른 temperature로 호출해 보기

추론 설정은 모델 성격을 바꿔 보이게 한다

temperature가 높으면 답변이 다양해지지만 형식과 안정성이 흔들릴 수 있다. 업무 자동화, 정책 응답, 코드 생성처럼 재현성이 중요한 곳에서는 낮은 temperature가 유리하다. top_p, max_new_tokens, repetition penalty도 함께 관리해야 한다.

서빙 방식은 지연시간과 처리량의 trade-off다

vLLM은 높은 처리량과 PagedAttention 기반 배치 처리에 강하고, TGI는 Hugging Face 생태계와 배포 관리가 편하다. llama.cpp와 GGUF는 로컬·엣지 배포에 적합하다. 작은 API 서버는 FastAPI로 감싸되, 실제 부하는 전용 inference server에 맡기는 구성이 흔하다.

21장RAG심화

RAG와 파인튜닝

RAG(Retrieval-Augmented Generation)는 지식은 검색으로, 반복 행동은 파인튜닝으로 다루는 기준을 세우게 해준다.

  • 지식은 RAG, 행동은 파인튜닝
  • 최신 정보와 출처가 필요한 경우
  • 답변 형식과 업무 절차를 학습시키는 경우
  • RAG + 파인튜닝 결합 구조
바로 확인: 같은 문제를 RAG/파인튜닝으로 나눠 설계하기

지식은 검색, 행동은 파인튜닝

최신 문서, 가격, 정책 원문, 사내 규정처럼 자주 바뀌는 정보는 RAG(Retrieval-Augmented Generation)로 가져오는 편이 낫다. 파인튜닝은 그 정보를 읽고 요약하는 방식, 답변 형식, 거절 기준, 도구 사용 절차를 안정화하는 데 적합하다.

둘은 경쟁 관계가 아니라 역할 분담이다

RAG(Retrieval-Augmented Generation)만 쓰면 답변 형식과 판단 기준이 흔들릴 수 있다. 파인튜닝만 쓰면 최신 지식과 출처가 약하다. 실무에서는 검색으로 근거를 가져오고, 파인튜닝된 모델이 그 근거를 정해진 형식으로 처리하게 만든다.

RAG가 먼저인 경우 사내 문서 QA, 최신 약관, 제품 가격, 출처가 필요한 답변.
파인튜닝이 먼저인 경우 응답 형식 고정, 업무 절차, 상담 말투, 반복되는 판단 패턴.

Cross entropy

Cross entropy는 모델이 정답 토큰에 얼마나 높은 확률을 줬는지 보는 손실 함수다. 정답 토큰 확률이 높으면 loss가 낮고, 낮으면 loss가 높다. 파인튜닝 로그에서 보는 train loss와 eval loss는 대부분 이 직관으로 읽으면 된다.

Gradient descent

Gradient descent는 loss를 줄이는 방향으로 파라미터를 조금씩 움직이는 방법이다. learning rate는 한 번에 얼마나 움직일지 정한다. 너무 크면 좋은 지점을 지나치고, 너무 작으면 학습이 거의 진행되지 않는다.

Rank와 quantization

LoRA의 rank는 adapter가 표현할 수 있는 변화의 폭을 뜻한다. quantization은 모델 숫자의 정밀도를 낮춰 메모리를 줄이는 방법이다. 둘 다 비용을 줄이는 도구지만, 너무 과하게 줄이면 품질이 떨어질 수 있다.

공개적으로 확인되는 내용

2026년 2월 23일 Anthropic은 DeepSeek, Moonshot, MiniMax가 약 24,000개의 부정 계정과 1,600만 건 이상의 Claude 교환을 통해 Claude 능력을 추출하려 했다고 공개했다. 이후 2026년 6월 말 보도에서는 Anthropic이 미국 상원에 보낸 서한에서 Alibaba 및 Qwen 관련 운영자가 2026년 4월 22일부터 6월 5일까지 약 25,000개 계정으로 2,880만 건 이상의 Claude 교환을 만들었다고 주장한 것으로 전해졌다. 이 부분은 Anthropic의 주장과 보도에 근거한 것이며, 독립적으로 검증된 사실이라고 단정하면 안 된다.

기술적으로는 가중치 탈취가 아니라 행동 복제에 가깝다

공개된 설명을 기준으로 보면 핵심 원리는 Claude의 소스코드나 가중치를 직접 훔치는 것이 아니다. 강한 모델에 많은 질문을 던지고, 그 응답을 데이터셋으로 모아 다른 모델을 학습시키는 방식이다. 이를 model distillation이라고 부른다. 합법적으로는 자기 모델을 작고 빠른 모델로 압축할 때 쓰지만, 약관을 어기고 경쟁 모델의 출력을 대량 수집하면 distillation attack으로 볼 수 있다.

  2. B Preference tuning여러 후보 답변 중 teacher가 더 선호할 법한 답을 chosen으로 두고 DPO(Direct Preference Optimization)나 ORPO(Odds Ratio Preference Optimization)류 학습을 한다.
  3. C Agent trajectory tuningtool call, observation, patch, test result를 포함해 긴 작업 수행 순서를 학습한다.

DPO용 데이터는 teacher를 채점자로 써서 만들 수 있다

student가 여러 답변 후보를 만들고 teacher가 더 나은 답을 고르게 하면 chosen/rejected 쌍을 만들 수 있다. DPO(Direct Preference Optimization)는 이런 선호 쌍을 사용해 모델의 선택 경향을 옮기는 데 유리하다. 다만 이것 역시 사용 권한 없이 상용 모델을 이용하면 약관과 데이터 권리 문제가 생긴다.

해석

선호 데이터는 “정답 문장”보다 “어떤 답이 더 엔지니어답고 안전한가”를 전달한다.

왜 이런 방식이 잘 먹히는가

LLM의 많은 능력은 가중치 안의 원문 지식보다 출력 분포와 절차적 습관으로 드러난다. 예를 들어 코드 에이전트는 답변 자체보다 문제 분해, 파일 탐색 순서, 테스트 해석, 위험한 수정 회피, 사용자에게 보고하는 방식에서 차이가 난다. 이런 행동은 충분히 많은 입출력과 trajectory를 모으면 student 모델이 상당 부분 근사할 수 있다.

잘 옮겨지는 것 응답 형식, 코드 리뷰 말투, 단계적 문제 해결, tool calling 순서, 거절 패턴, 테스트 해석 습관.
잘 안 옮겨지는 것 teacher의 내부 가중치, 전체 지식 범위, hidden state, 비공개 학습 데이터, 장기 안정성.

방어 관점에서 보면 탐지 신호는 “한 계정”이 아니라 “분포”에 있다

공개된 Anthropic 글은 산업 규모의 부정 계정과 대량 교환을 문제 삼는다. 이런 공격은 개별 요청만 보면 정상 사용처럼 보일 수 있으므로, 방어자는 계정 간 유사한 prompt 구조, 과도하게 넓은 능력 커버리지, 동일한 평가 루브릭 반복, 특정 모델 능력을 체계적으로 샘플링하는 패턴을 본다. 여기서는 우회 방법이 아니라 방어자가 봐야 할 신호만 다룬다.

질문 분포일반 사용보다 능력별 benchmark coverage가 지나치게 균등한지 본다.
응답 사용출력 직후 유사 포맷으로 재질문하거나 후보 답변 비교를 반복하는지 본다.
계정 상관서로 다른 계정이 같은 seed, 같은 변형 규칙, 같은 시간 패턴을 공유하는지 본다.
canary특정 문구나 행동 fingerprint가 다른 모델 출력에서 반복되는지 장기적으로 추적한다.
재현 금지

이 절은 방어적 분석이지 공격 절차가 아니다

여러 계정으로 접근 제한을 우회하거나, 상용 모델 출력을 무단 수집해 학습시키는 절차는 제공하지 않는다. 기술적 원리를 이해하는 목적은 합법적 증류와 데이터 거버넌스, API 남용 탐지를 구분하기 위해서다.

파인튜닝 개발자가 지켜야 할 선

외부 모델의 출력을 학습 데이터로 쓰기 전에 약관, 라이선스, 고객 데이터 권한을 확인한다. 허용된 합성 데이터라도 출처, 생성 프롬프트, 검수 기준, 사용 가능 범위를 기록한다. 모델 품질을 빠르게 올리고 싶다는 이유로 출처가 불명확한 대량 응답을 섞으면 법적 리스크뿐 아니라 평가 불가능한 데이터 오염까지 생긴다.

가중치 탈취 모델 파일, 파라미터, 학습 체크포인트 자체를 빼내는 경우다. 보안 침해에 가깝다.
행동 복제 모델에 많은 입력을 넣고 출력을 모아 다른 모델이 비슷하게 답하도록 학습시키는 경우다.
