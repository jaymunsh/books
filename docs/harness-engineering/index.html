하네스 엔지니어링 (Harness Engineering)

LLM은 엔진이고, 하네스(harness)는 그 엔진을 실제로 일하게 만드는 모든 것이다.

이 책은 모델을 둘러싼 엔지니어링 — 에이전트 루프, 툴, 컨텍스트, 오케스트레이션,

검증, 안전 — 을 입문부터 고급까지 다룬다.

이 책은 무엇인가

요즘 가장 흔한 오해 하나: "좋은 에이전트 = 좋은 모델". 틀렸다. 같은 모델이라도 하네스가 그 성능의 대부분을 결정한다. 모델이 추론 엔진이라면, 하네스는 그 엔진에 연료(컨텍스트)를 넣고, 손발(툴)을 달아주고, 여러 대를 묶어 (오케스트레이션) 일을 시키고, 결과가 맞는지 확인(검증)하고, 사고를 막는(안전) 차체다.

하네스 엔지니어링(harness engineering) 은 바로 이 차체를 설계하는 분야다. 이 책은 추상론을 피하고, 실제로 동작하는 production급 하네스(당신이 지금 쓰고 있는 Claude Code 같은 에이전트 런타임)에서 뽑은 구체적인 예시로 모든 개념을 설명한다.

누구를 위한 책인가

  • LLM API는 호출해봤지만 "에이전트"를 어떻게 구조화하는지 막막한 입문자
  • 단일 에이전트는 만들어봤고 멀티에이전트·검증·비용 문제에 부딪힌 실무 엔지니어
  • 하네스의 설계 철학·실패 모드·대규모 운영을 고민하는 아키텍트

이 책이 다루지 않는 것

  • 모델 학습/파인튜닝 (이 책은 모델을 둘러싼 엔지니어링이 주제다)
  • 특정 프레임워크 사용법 (개념을 전달하되 예제는 언어 무관 의사코드로 쓴다)

1.1 왜 중요한가

LLM을 처음 다루는 사람은 보통 이렇게 생각한다.

"더 똑똑한 모델을 쓰면 더 좋은 에이전트가 된다."

부분적으로만 맞다. 똑같은 모델, 똑같은 프롬프트라도 그것을 어떻게 감싸는지에 따라 결과는 극적으로 달라진다. 예를 들어 같은 모델에게 "이 버그를 고쳐줘"라고 시켜도:

  • 파일을 읽을 툴이 없으면 → 추측만 하다 끝난다.
  • 테스트를 돌릴 루프가 없으면 → 고쳤는지 확인하지 못한다.
  • 변경을 되돌릴 안전장치가 없으면 → 멀쩡한 코드를 날린다.
  • 결과를 검증하지 않으면 → 그럴듯하지만 틀린 수정을 자신 있게 내놓는다.

이 모든 "주변 장치"가 바로 하네스다. 모델은 추론 능력을 제공하지만, 그 능력을 실제 작업으로 바꾸는 것은 하네스의 몫이다. 그래서 하네스 엔지니어링은 모델 선택만큼 중요하다 — 때로는 더.

1.3 하네스가 하는 일 — 다섯 가지 책임

좋은 하네스는 다섯 가지를 한다. 이 다섯 가지가 이 책 Part 1~3의 뼈대다.

  1. 연결 (Wiring) — 모델을 툴·데이터·외부 시스템에 연결한다. (4·15장)
  1. 공급 (Feeding) — 매 호출마다 적절한 컨텍스트를 골라 넣는다. (5·8장)
  1. 제어 (Controlling) — 루프를 돌리고, 일을 분배하고, 멈출 때를 안다. (3·9~12장)
  1. 보호 (Guarding) — 위험한 작업을 막고 권한을 통제한다. (7장)
  1. 확인 (Verifying) — 결과가 실제로 맞는지 검증한다. (11·19장)

입문자라면 1~3번을 먼저 익히면 동작하는 에이전트를 만들 수 있다. 4~5번은 그것을 신뢰할 수 있게 만든다.

  • 제어: 큰 작업은 서브에이전트에게 위임하고(fan-out), 여러 단계를 Workflow로

    • 파이프라인화한다. → 9·10장

  • 보호: 되돌리기 어려운 작업은 사전 확인을 요구하고, 명령은 권한 모드 아래서

    • 실행된다. → 7장

  • 확인: 중요한 결과는 적대적 검증(adversarial verify) 으로 반증을 시도해 거짓

    • 양성을 걸러낸다. → 11장

이 책의 모든 장은 이렇게 실제로 동작하는 하네스에서 뽑은 예시로 개념을 설명한다. 그래서 "이론적으로는 이렇다"가 아니라 "실제로는 이렇게 한다"를 보여줄 수 있다.

1.5 흔한 오해 / 안티패턴

  • ❌ "프롬프트만 잘 쓰면 된다" — 프롬프트는 하네스의 한 부분(시스템 프롬프트)일 뿐이다.

    • 툴·루프·검증이 없으면 좋은 프롬프트도 무력하다.

  • ❌ "모델을 업그레이드하면 해결된다" — 하네스가 병목이면 모델을 바꿔도 그대로다.

    • 먼저 어디가 병목인지(연결? 공급? 검증?) 진단하라.

  • ❌ "에이전트는 그냥 모델 호출 한 번" — 진짜 에이전트는 반복 루프 다. 한 번의

    • 호출로 끝나는 건 에이전트가 아니라 텍스트 생성이다. (→ 2장)

  • ❌ "많이 자동화할수록 좋다" — 보호(4번 책임) 없는 자동화는 사고를 가속할 뿐이다.

1.6 핵심 정리

  • 하네스 = 모델을 감싸 실제 작업을 수행시키는 런타임 전체.
  • 에이전트의 체감 성능은 대부분 하네스가 결정한다 — 모델은 엔진일 뿐.
  • 하네스의 다섯 책임: 연결·공급·제어·보호·확인.
  • 이 책은 동작하는 하네스의 구체 예시로 이 다섯 가지를 입문→고급으로 풀어낸다.

2.1 왜 중요한가

부품 1 — 루프 (Loop) · 제어

에이전트의 심장. "모델 호출 → (툴 호출이면) 툴 실행 → 결과 주입 → 다시 호출"을 모델이 '끝'이라고 할 때까지 반복한다. 이 반복이 있어야 에이전트다. → 3장에서 직접 만든다.

부품 2 — 툴 (Tools) · 연결

모델이 바깥에 작용하는 통로. 각 툴은 이름·설명·파라미터 스키마 로 정의된다. 모델은 스키마를 보고 "어떤 툴을 어떤 인자로 부를지"를 텍스트가 아니라 구조화된 호출로 표현한다. → 4장.

부품 3 — 컨텍스트 (Context) · 공급

매 모델 호출에 들어가는 토큰 전체. 시스템 프롬프트 + 지금까지의 대화 + 툴 결과 + 불러온 메모리. 유한한 윈도우 안에 무엇을 넣고 뺄지가 핵심 설계 문제다. → 5장.

부품 4 — 시스템 프롬프트 (System prompt)

컨텍스트의 가장 앞, 거의 고정된 부분. 모델에게 너는 누구이고, 무엇을 할 수 있고, 무엇을 하면 안 되고, 어떤 환경에 있는지 를 알려준다. → 6장.

부품 5 — 제어/안전 (Control & Safety) · 보호·확인

루프를 언제 멈출지, 어떤 툴 호출을 허용할지, 결과가 맞는지 확인할지를 결정하는 정책층. 권한 모드·검증 루프가 여기 산다. → 7·11장.

  1. 루프 반복 — 다시 모델 호출. 이제 모델은 파일을 "본다". 오타를 찾아

  1. 툴 실행 — 수정 적용, 결과 주입.
  1. 종료 판단 — 모델이 더 부를 툴이 없다고 판단 → 최종 텍스트("오타를 고쳤습니다")를

3장 — 첫 하네스 만들기

한 줄 요약: 50줄 남짓의 의사코드로 동작하는 ReAct 루프를 직접 만든다.

"에이전트 = while 루프 + 툴"이라는 직관을 손에 새긴다.

3.1 왜 중요한가

에이전트가 마법처럼 보이는 이유는 루프가 숨어 있기 때문이다. 그 루프를 한 번 직접 써보면 마법이 공학 으로 바뀐다. 이 장이 끝나면 어떤 프레임워크를 보든 "아, 결국 이 루프를 포장한 거구나"를 알아볼 수 있다.

이 장의 예제는 언어 무관 의사코드다. 전체 흐름은 이 문서의 Minimal ReAct Loop 예제에 코드블록으로 들어 있다.

3.2 개념: ReAct 루프

ReAct = Reason + Act. 모델이 추론(다음에 뭘 할지)하고 행동(툴 호출)하기를 번갈아 반복한다. 골격은 1장의 다섯 부품 중 루프·툴·컨텍스트 만으로 충분하다.

flowchart TB
    Start(["user_input"]) --> Init["messages = [system, user]"]
    Init --> Call["① 모델 호출<br/>model.call(messages, tools)"]
    Call --> Q{"툴 호출이<br/>있는가?"}
    Q -->|"없음"| End(["③ 최종 답변 반환"])
    Q -->|"있음"| Run["② 툴 실행<br/>result = tool.run(args)"]
    Run --> Inject["★ 결과 주입<br/>messages.append(result)"]
    Inject --> Call
초기화: messages = [system_prompt, user_input]

반복(loop):
    response = model.call(messages, tools=TOOLS)   # ① 모델 호출 (추론)

    if response 가 툴 호출을 요청하면:               # ② 행동
        for each tool_call in response.tool_calls:
            result = run_tool(tool_call)            #   툴 실행
            messages.append(tool_call)              #   호출 기록
            messages.append(result)                 #   결과 주입 ★핵심
        continue                                    #   다시 ①로
    else:
        return response.text                        # ③ 종료: 최종 답변

★ 핵심은 결과 주입(result injection). 툴 결과를 messages에 다시 넣고 모델을 또 부르기 때문에, 모델이 다음 호출에서 그 결과를 "보고" 추론을 이어간다. 이 한 줄이 빠지면 에이전트가 아니라 일회성 생성이 된다.

3.3 단계별로 만들기

단계 1 — 툴을 정의한다

툴은 스키마(모델이 보는 설명) 와 구현(실제 코드) 의 쌍이다. (자세한 설계는 4장)

TOOLS = [
  {
    name: "read_file",
    description: "주어진 경로의 파일 내용을 반환한다.",
    parameters: { path: { type: "string", desc: "읽을 파일 경로" } },
    run: (args) => filesystem.read(args.path),       # 실제 구현
  },
  {
    name: "list_dir",
    description: "디렉토리의 파일 목록을 반환한다.",
    parameters: { path: { type: "string" } },
    run: (args) => filesystem.list(args.path),
  },
]

단계 2 — 루프를 작성한다

function run_agent(user_input):
    messages = [
       { role: "system", content: SYSTEM_PROMPT },
       { role: "user",   content: user_input },
    ]

    for step in 1..MAX_STEPS:                 # 안전장치: 무한 루프 방지 (2장)
        response = model.call(messages, tools=TOOLS)
        messages.append(response)             # 모델의 발화를 기록

        if not response.tool_calls:           # 툴 호출이 없으면 = 끝
            return response.text

        for call in response.tool_calls:      # 툴 실행 + 결과 주입
            tool = find_tool(call.name)
            try:
                result = tool.run(call.args)
            catch error:
                result = "ERROR: " + error.message   # 에러도 결과로 주입 (4장)
            messages.append({ role: "tool", id: call.id, content: result })

    return "중단: 최대 단계 수 초과"            # 안전장치 발동

이게 전부다. 30줄 남짓의 이 루프가 "에이전트"의 90%다. 나머지는 이것을 신뢰할 수 있고, 확장 가능하고, 안전하게 만드는 일 — 그게 이 책의 나머지다.

단계 3 — 시스템 프롬프트를 준다

SYSTEM_PROMPT = """
너는 파일 시스템을 탐색하는 보조 에이전트다.
- 사용자의 질문에 답하려면 read_file, list_dir 툴을 사용하라.
- 추측하지 말고, 필요한 파일을 직접 읽어 근거를 확보한 뒤 답하라.
- 충분한 정보를 얻으면 간결하게 최종 답을 한국어로 제시하라.
"""

3.4 실행해보기 (추적)

"src 폴더에 테스트 파일이 몇 개 있어?"라고 물으면:

step 1: model → list_dir("src")              → ["app.py","test_app.py","util.py","test_util.py"]
        (결과 주입)
step 2: model → "test_ 로 시작하는 파일이 2개다" (툴 호출 없음 → 종료)
return: "src 폴더에 테스트 파일은 2개입니다 (test_app.py, test_util.py)."

모델이 스스로 "목록을 봐야겠다 → 봤으니 답할 수 있다"를 판단했다. 우리는 루프와 툴만 줬을 뿐이다. 이것이 하네스의 힘이다.

3.5 흔한 실수 / 안티패턴

  • ❌ 결과 주입을 빼먹기 — 툴을 실행하고 결과를 messages에 안 넣으면 모델이

    • 다음 호출에서 결과를 못 본다. 가장 흔한 입문 버그.

  • ❌ MAX_STEPS 안전장치 생략 — 모델이 종료를 못 하면 무한 루프·비용 폭주. 반드시 둬라.
  • ❌ 에러를 던져서 루프를 죽이기 — 툴 에러는 throw가 아니라 결과로 주입 해야

    • 모델이 보고 복구한다(다른 경로 시도 등). (→ 4장)

  • ❌ 모델 발화를 기록 안 하기 — response를 messages에 안 넣으면 모델이 자기가

    • 방금 뭘 하려 했는지 잊는다.

3.6 핵심 정리

  • 에이전트 = while 루프 + 툴 + 결과 주입.
  • 종료 = 모델이 더 부를 툴이 없을 때. 안전장치(MAX_STEPS)는 하네스가 따로 둔다.
  • 툴 에러조차 결과로 주입 해 모델이 복구하게 한다.
  • 이 30줄이 골격이고, 책의 나머지는 이걸 신뢰·확장·안전하게 만드는 일이다.

4장 — 툴 설계 (Tool Design)

한 줄 요약: 툴은 모델이 세계에 작용하는 통로다. 좋은 툴 설계 — 명확한 이름,

잘 쓴 설명문, 단단한 스키마, 피드백이 되는 에러 — 가 에이전트 성능을 좌우한다.

4.1 왜 중요한가

3장에서 본 루프는 TOOLS 배열의 품질만큼만 똑똑하다. 모델은 툴 설명문을 읽고 "언제 어떤 툴을 어떤 인자로 부를지"를 결정한다. 즉 툴 스키마는 모델에게 주는 또 하나의 프롬프트 다. 설명이 모호하면 모델은 엉뚱한 툴을 부르거나, 인자를 틀리거나, 아예 안 부른다.

핵심 명제: 툴 스키마는 코드가 아니라 모델을 향한 인터페이스 문서 다.

사람 개발자에게 쓰는 API 문서처럼 — 아니, 그보다 더 명확하게 — 써야 한다.

4.2 툴의 해부

하나의 툴은 네 부분으로 이뤄진다.

flowchart LR
    subgraph Tool["하나의 툴"]
        Name["name<br/>모델이 호출할 식별자"]
        Desc["description<br/>언제·왜 쓰는지 (모델이 읽음)"]
        Params["parameters (schema)<br/>인자의 타입·제약·설명"]
        Impl["구현(run)<br/>실제 부작용 — 모델은 못 봄"]
    end
    Name --- Desc --- Params -.->|"모델에 노출"| LLM["모델"]
    Impl -.->|"하네스만 실행"| World["바깥 세계"]

모델에 노출되는 것은 name·description·parameters 셋뿐이다. run 구현은 모델이 보지 못한다. 그래서 "모델이 올바르게 행동하게 만드는 것"은 전적으로 앞의 세 가지에 달렸다.

4.3 좋은 설명문 쓰는 법

설명문(description)은 툴 설계의 90%다. 원칙:

  1. 언제 쓰는지를 먼저 — "무엇을 하는지"보다 "어떤 상황에서 이걸 골라야 하는지"가

    2. 모델에게 더 중요하다.

  1. 다른 툴과의 경계를 명시 — 비슷한 툴이 여럿이면 "X가 아니라 Y일 때 이걸 써라"를

    2. 적어 혼동을 막는다.

  1. 반환값의 형태를 알려라 — 모델은 결과를 받아 다음 행동을 정한다. 무엇이 돌아오는지

    2. 알면 계획이 정확해진다.

  1. 제약·부작용을 경고 — "되돌릴 수 없음", "비쌈", "한 번에 N개까지" 등.

나쁜 예 vs 좋은 예

// ❌ 나쁨: 모델이 언제 쓸지 모른다
{
  name: "search",
  description: "검색한다.",
  parameters: { q: { type: "string" } }
}

// ✅ 좋음: 선택 기준·경계·반환·제약이 분명
{
  name: "search_codebase",
  description: "키워드나 정규식으로 코드베이스 전체를 검색해 일치하는 파일 경로와
    줄을 반환한다. 특정 파일의 '내용'을 읽고 싶을 때는 read_file을 써라 — 이 툴은
    '어디에 있는지' 찾는 용도다. 결과는 최대 50건으로 잘리며, 50건이면 더 좁은
    검색어를 권한다.",
  parameters: {
    pattern: { type: "string", description: "검색할 키워드 또는 정규식" },
    is_regex: { type: "boolean", description: "pattern을 정규식으로 해석할지 (기본 false)" }
  }
}

📦 심화 박스 — 이 환경의 실제 사례: 지금 이 하네스의 Read 툴 설명을 보면

"Reading a directory, a missing file, or an empty file returns an error or system

reminder rather than content", "Do NOT re-read a file you just edited to verify" 처럼

반환 형태 와 하지 말아야 할 행동 까지 못 박는다. Agent/Explore 설명은 "언제

직접 검색하고 언제 위임하는지"의 경계를 길게 설명한다 — 모두 4.3의 원칙을 따른 것이다.

4.4 스키마 설계 — 모델이 인자를 틀리지 않게

파라미터 스키마(JSON Schema)는 모델의 출력을 강제 하는 장치다. 활용 포인트:

  • enum으로 선택지를 닫아라 — 자유 문자열 대신 enum: ["low","high"]를 주면 모델이

    • 오타·환각을 못 낸다.

  • required를 명시 — 빠지면 안 되는 인자를 표시. 다만 너무 많으면 모델이 호출을

    • 주저하므로 꼭 필요한 것만.

  • 설명을 각 필드에도 — 파라미터마다 짧은 description. 특히 형식이 중요한 것(경로,

    • 날짜, 단위)은 예시를 넣어라.

  • 기본값을 문서화 — "생략하면 false" 같은 동작을 description에 적는다.
{
  name: "create_task",
  description: "할 일 항목을 만든다.",
  parameters: {
    title:    { type: "string", description: "명령형 한 줄 제목, 예: '로그인 버그 수정'" },
    priority: { type: "string", enum: ["low","medium","high"],
                description: "긴급도. 불명확하면 medium." },
    due:      { type: "string", description: "마감일 ISO 8601 (예: 2026-07-01). 없으면 생략." }
  },
  required: ["title"]
}

4.5 에러를 피드백으로 (Errors as feedback)

3장에서 강조한 원칙을 심화한다. 툴 에러는 루프를 죽이는 예외가 아니라, 모델이 읽고 복구하는 또 하나의 결과 다. 좋은 에러 메시지는 모델을 올바른 다음 행동으로 안내한다.

// ❌ 나쁨: 모델이 뭘 해야 할지 모름
"Error: ENOENT"

// ✅ 좋음: 원인 + 복구 방향
"ERROR: 파일 'src/app.py'가 없습니다. list_dir('src')로 정확한 경로를 먼저 확인하세요.
 (오타나 잘못된 디렉토리일 수 있습니다.)"

에러 메시지를 "모델에게 보내는 짧은 코칭"이라고 생각하라. 이것만 잘해도 에이전트의

자가 복구율이 눈에 띄게 오른다.

4.6 툴이 너무 많을 때 — 지연 툴(Deferred tools)

툴이 수십~수백 개면 모든 스키마를 매 호출 컨텍스트에 싣는 비용이 커지고, 선택지가 많아 모델의 정확도도 떨어진다. 해법은 지연 로딩이다.

flowchart TB
    A["수백 개 툴<br/>이름만 노출 (스키마 X)"] --> B{"모델: 이 작업에<br/>어떤 툴이 필요?"}
    B --> C["ToolSearch('파일 편집')<br/>필요한 툴만 검색"]
    C --> D["해당 툴의 전체 스키마 로드"]
    D --> E["이제 그 툴을 호출 가능"]

핵심 아이디어: 이름은 싸고 스키마는 비싸다. 평소엔 이름만 노출하고, 모델이 필요하다고 판단할 때만 전체 스키마를 불러온다.

📦 심화 박스 — 이 환경의 실제 사례: 이 하네스에는 CronCreate, WebFetch,

NotebookEdit 등 다수의 툴이 deferred 상태로, 이름만 보이고 스키마는 빠져 있다.

ToolSearch로 "select:WebFetch" 같이 검색해야 전체 스키마가 로드되고 호출 가능해진다.

자주 쓰는 핵심 툴(Read/Edit/Bash…)만 항상 노출하고 나머지는 지연시키는 전형적 설계다.

4.7 툴 설계 체크리스트

  • 이름이 동사+목적어로 행동을 분명히 나타내는가 (search_codebase ✓ / search ✗)
  • 설명문이 "언제 이걸 고르는지"를 먼저 말하는가
  • 비슷한 툴과의 경계("X 말고 Y일 때")를 적었는가
  • 반환값의 형태/크기 제한을 알렸는가
  • 위험한 부작용("되돌릴 수 없음")을 경고했는가
  • enum·required·필드 설명으로 인자 오류를 막았는가
  • 에러 메시지가 복구 방향을 제시하는가

4.8 흔한 실수 / 안티패턴

  • ❌ 만능 툴 — do_everything(action, params) 한 개보다, 좁고 명확한 툴 여러 개가

    • 모델에게 더 쉽다. 단, 너무 잘게 쪼개도 선택 피로가 생기니 균형을.

  • ❌ 사람용 문서를 그대로 복붙 — 모델에겐 "언제/경계/반환"이 더 중요하다.
  • ❌ 거대한 결과를 통째 반환 — 윈도우를 잡아먹는다. 페이징·요약·잘림 표시를 둬라.(→5장)
  • ❌ 에러를 throw — 루프가 죽고 모델은 복구 기회를 잃는다.
  • ❌ 부작용을 설명 안 함 — 모델이 위험을 모르고 파괴적 툴을 가볍게 부른다.(→7장)

4.9 핵심 정리

  • 툴 스키마는 모델을 향한 인터페이스 문서 — name·description·parameters만 모델이 본다.
  • 설명문은 "언제 고르는지 · 다른 툴과의 경계 · 반환 형태 · 제약"을 담아라.
  • 에러는 예외가 아니라 복구를 돕는 피드백 으로 주입한다.
  • 툴이 많으면 지연 툴 + 검색 으로 컨텍스트와 정확도를 지킨다.

5장 — 컨텍스트 엔지니어링 (Context Engineering)

한 줄 요약: 컨텍스트 윈도우는 유한한 자원이다. 무엇을 넣고, 무엇을 빼고, 언제

압축할지를 설계하는 것 — 그것이 컨텍스트 엔지니어링이고, 긴 작업의 성패를 가른다.

5.1 왜 중요한가

모델은 컨텍스트 윈도우 안에 든 것만 "안다". 윈도우 밖의 정보는 존재하지 않는 것과 같다. 그런데 윈도우는 유한하다. 작업이 길어지면 대화·툴 결과가 쌓여 윈도우를 가득 채우고, 그러면 비용이 오르고, 모델이 앞부분을 "잊고", 결국 한계에 부딪힌다.

컨텍스트 엔지니어링은 이 유한한 공간을 예산 으로 보고 관리하는 기술이다.

핵심 명제: 토큰은 돈이자 주의력(attention)이다. 윈도우에 쓰레기를 채우면 비용도

늘고 모델의 집중도 흩어진다. 적게, 정확하게 넣는 것이 항상 이긴다.

5.2 컨텍스트에는 무엇이 들어가는가

flowchart TB
    subgraph Window["컨텍스트 윈도우 (유한)"]
        direction TB
        Sys["시스템 프롬프트 — 거의 고정 (캐시 대상)"]
        Mem["불러온 메모리 / 지침 — 선택적 (8장)"]
        Hist["대화 이력 — 누적 증가"]
        Tool["툴 결과 — 가장 빨리 부푸는 부분"]
        Cur["현재 사용자 입력"]
    end
    Window --> Model["모델 호출"]

비중을 보면, 툴 결과 가 가장 빠르게 부푼다. 파일 하나 통째로 읽기, 긴 로그, 검색 결과 수십 건 — 이것들이 윈도우를 잡아먹는 주범이다. 컨텍스트 관리의 절반은 "툴 결과를 어떻게 다이어트시키느냐"다.

5.3 핵심 기법

(1) 들어가기 전에 줄여라 — 입력 측 절약

  • 필요한 부분만 읽기: 파일 전체 대신 관련 구간만. 검색은 매치 줄만.
  • 요약해서 주입: 거대한 출력은 툴 단계에서 요약·발췌해 넣는다.
  • 위임으로 격리: 방대한 탐색은 서브에이전트에게 시키고 결론만 돌려받는다. → 9장.

📦 이 환경의 사례: Explore/Agent 서브에이전트는 "파일 덤프가 아니라 결론만

가져온다"고 설계돼 있다. 수십 개 파일을 읽는 무거운 탐색을 메인 컨텍스트 밖에서

처리하고, 정제된 답만 주입해 윈도우를 지킨다.

(2) 쌓인 것을 줄여라 — 압축(Compaction)

대화가 길어지면 앞부분을 요약으로 대체 한다. 원본 수천 토큰을 핵심 수백 토큰으로 접는 것이다. 무엇을 보존할지가 관건 — 결정 사항·미해결 과제·핵심 사실은 남기고, 중간 과정의 잡음은 버린다.

flowchart LR
    A["긴 대화 이력<br/>(윈도우 80% 차지)"] -->|compaction| B["요약 1개<br/>+ 최근 메시지 일부"]
    B --> C["윈도우 여유 회복<br/>작업 계속 가능"]

📦 이 환경의 사례: 이 하네스는 "대화가 길어지면 일부 또는 전체 컨텍스트가

요약되고, 다음 컨텍스트 윈도우에 그 요약과 남은 컨텍스트가 함께 제공된다"고 동작한다.

그래서 에이전트는 작업을 일찍 마무리할 필요 없이 길게 이어갈 수 있다.

(3) 밖에 저장하고 불러와라 — 외부 메모리

세션을 넘는 사실은 윈도우가 아니라 파일 에 저장하고, 필요할 때만 불러온다(recall). → 8장에서 본격적으로.

(4) 신호를 적시에 주입 — 런타임 주입

하네스는 실행 중 필요한 정보를 컨텍스트에 끼워 넣을 수 있다(예: 현재 날짜, 상태 알림). 이 환경의 <system-reminder> 가 그 예다 — 모델이 아니라 하네스가 적시에 넣는 신호다.

5.4 프롬프트 캐시 — 비용을 좌우하는 숨은 변수

대부분의 LLM API는 프롬프트 접두부(prefix)를 캐싱 한다. 시스템 프롬프트처럼 매 호출 동일한 앞부분은 캐시에서 재사용돼 비용·지연이 크게 준다. 단, 캐시에는 TTL(수명) 이 있다.

설계 함의:

  • 고정부를 앞에, 가변부를 뒤에 두어 캐시 적중률을 높여라. 앞부분이 한 글자라도

    • 바뀌면 그 뒤 캐시가 전부 깨진다.

  • 시스템 프롬프트에 매 턴 바뀌는 값(타임스탬프 등)을 넣지 마라 — 캐시를 깬다.
  • 오래 쉬었다 재개하면 캐시 TTL이 만료돼 다음 호출이 캐시 미스가 된다.

📦 이 환경의 사례: 이 하네스의 백그라운드 작업 가이드는 "Anthropic 프롬프트 캐시는

5분 TTL을 가지며, 300초를 넘겨 자면 다음 깨어남이 캐시 미스가 돼 더 느리고 비싸다"고

명시한다. 그래서 짧은 폴링은 270초 이내로, 긴 대기는 차라리 1200초+로 묶으라고 권한다.

토큰 경제학은 21장에서 더 다룬다.

5.5 컨텍스트 예산 사고법

윈도우를 한정된 예산으로 보고, 각 항목에 "이게 그만한 토큰을 쓸 값어치가 있나?"를 물어라.

항목질문
거대한 파일 전체정말 다 필요한가? 관련 함수만으로 충분한가?
긴 툴 결과요약하면 충분한가? 핵심 N줄만?
오래된 대화결정·미해결 과제만 남기고 압축 가능한가?
장황한 시스템 프롬프트모든 규칙이 매 턴 필요한가, 아니면 스킬/메모리로 뺄 수 있나?

5.6 흔한 실수 / 안티패턴

  • ❌ "일단 다 넣고 보자" — 윈도우가 클수록 좋은 게 아니다. 잡음은 정확도를 떨어뜨린다

    • (관련 없는 맥락이 모델을 헷갈리게 한다).

  • ❌ 거대 파일 통째 읽기 — 관련 구간만 읽어라. (→ 4장 결과 크기 제한)
  • ❌ 시스템 프롬프트에 타임스탬프 — 캐시를 매번 깬다.
  • ❌ 압축으로 핵심 결정까지 날리기 — 무엇을 보존할지 명시적으로 설계하라.
  • ❌ 세션 지속 사실을 대화에만 의존 — 압축되면 사라진다. 메모리에 저장하라.(→8장)

5.7 핵심 정리

  • 컨텍스트 윈도우 = 유한한 예산. 토큰은 돈이자 주의력.
  • 네 가지 도구: 입력 절약 · 압축 · 외부 메모리 · 런타임 주입.
  • 툴 결과가 가장 빨리 부푼다 — 거기서부터 다이어트하라.
  • 프롬프트 캐시를 의식해 고정부/가변부를 분리하라 (비용 직결).

5.8 더 읽기

  • 다음: 6장 — 시스템 프롬프트 설계 — 고정부를 어떻게 쓰나.
  • 외부화: 8장 — 메모리 시스템.
  • 위임으로 컨텍스트 격리: 9장 — 서브에이전트.
  • 비용 심화: 21장 — 비용·토큰·캐시.

6장 — 시스템 프롬프트 설계 (System Prompt Design)

한 줄 요약: 시스템 프롬프트는 에이전트의 헌법이다. 역할·능력·제약·환경·안전을

정의해 모델의 모든 판단에 영향을 준다. 짧고 명확하고 우선순위가 분명해야 한다.

6.1 왜 중요한가

시스템 프롬프트는 컨텍스트의 맨 앞, 거의 고정된 부분으로(5장) 매 호출 모델이 가장 먼저 읽는 지시문 이다. 여기서 정한 정체성과 규칙이 이후 모든 행동의 기준이 된다. 잘못 쓰면 — 모순된 지시, 모호한 우선순위, 과도한 분량 — 에이전트가 갈팡질팡한다.

핵심 명제: 시스템 프롬프트는 "모델이 매번 따를 규칙"이다. 사람에게 주는 한 번의

지시가 아니라, 수백 번 반복 적용될 정책이라고 생각하고 써라.

6.2 시스템 프롬프트의 구성 요소

좋은 시스템 프롬프트는 보통 이런 블록으로 이뤄진다.

flowchart TB
    Role["① 역할/정체성<br/>'너는 ~하는 에이전트다'"]
    Cap["② 능력/도구<br/>무엇을 할 수 있는가"]
    Const["③ 제약/규칙<br/>무엇을 하면 안 되는가, 어떻게 행동하라"]
    Env["④ 환경 컨텍스트<br/>OS·경로·날짜·도구 가용성"]
    Safety["⑤ 안전/우선순위<br/>충돌 시 무엇이 우선인가"]
    Style["⑥ 출력 스타일<br/>형식·톤·길이"]

    Role --> Cap --> Const --> Env --> Safety --> Style

① 역할/정체성

"너는 누구인가". 한 문장으로 명확히. 예: "너는 코드베이스를 수정하는 소프트웨어 엔지니어링 에이전트다." 모호한 정체성("도움이 되는 비서")보다 구체적 역할이 낫다.

② 능력/도구

어떤 툴을 가졌고 무엇을 위해 쓰는지. 툴 자체의 상세는 스키마(4장)에 있으니, 여기선 전략 을 준다 — "추측 말고 직접 읽어 확인하라" 같은.

③ 제약/규칙

가장 길어지는 부분. "~하라 / ~하지 마라". 핵심은 우선순위와 이유 를 함께 주는 것. 규칙만 나열하면 충돌 시 모델이 임의로 고른다.

④ 환경 컨텍스트

OS, 작업 디렉토리, 현재 날짜, 사용 가능한 도구 등 런타임 사실. 이게 없으면 모델이 환경을 추측한다(틀린 경로, 틀린 날짜).

⑤ 안전/우선순위

"규칙이 충돌하면 안전이 우선", "되돌리기 어려운 작업은 먼저 확인" 같은 메타 규칙. → 7장.

⑥ 출력 스타일

형식(마크다운), 톤, 길이. "간결하게", "코드는 ```로" 등.

6.3 좋은 시스템 프롬프트의 원칙

  1. **규칙에 이유 를 붙여라** — "왜"를 알면 모델이 새로운 상황에 규칙을 일반화 한다.

    2. "파일 삭제 전 확인하라(되돌릴 수 없으므로)"가 "파일 삭제 전 확인하라"보다 강하다.

  1. 우선순위를 명시 — 충돌하는 목표(빠름 vs 안전, 간결 vs 완전)의 서열을 정하라.
  1. 금지보다 지향을 — "X 하지 마라"보다 "대신 Y 하라"가 행동을 더 잘 유도한다.
  1. 짧게 — 모든 규칙은 매 턴 토큰을 쓰고 주의력을 나눈다(5장). 안 쓰는 규칙은 빼라.

    2. 상황별 규칙은 스킬/메모리로 분리(14·8장).

  1. 예시로 보여라 — 형식이 중요하면 한 줄 예시가 열 줄 설명보다 낫다.

📦 심화 박스 — 이 환경의 사례: 이 하네스의 시스템 프롬프트는 위 원칙을 그대로

보여준다. "되돌리기 어렵고 바깥에 영향을 주는 작업은 먼저 확인하라 — 왜냐하면 한

맥락의 승인이 다음 맥락으로 이어지지 않기 때문" 처럼 이유 를 붙이고, "안전이 다른

모든 지시에 우선한다(plan mode가 supersede)"로 우선순위 를 못 박는다. 또 OS·셸·

작업 디렉토리·현재 날짜를 환경 블록 으로 명시한다.

6.4 충돌과 우선순위 — 가장 흔한 실패점

규칙이 늘면 서로 부딪힌다. "항상 간결하라" vs "항상 모든 단계를 설명하라". 모델은 충돌을 만나면 둘 중 하나를 임의로 고르고, 그 선택은 불안정하다. 해법:

  • 명시적 서열: "안전 > 정확성 > 간결성" 같은 순서를 박아라.
  • supersede 언어: "이 지시는 다른 모든 지시에 우선한다"로 최상위 규칙을 표시.
  • 조건부 규칙: "보통은 간결하게; 단 사용자가 설명을 요청하면 자세히"처럼 발동 조건 을 줘라.
flowchart TB
    A["여러 규칙이 충돌"] --> B{"명시적 우선순위가<br/>있는가?"}
    B -->|있음| C["서열대로 결정 — 안정적"]
    B -->|없음| D["모델이 임의 선택 — 불안정"]

6.5 시스템 프롬프트 vs 다른 입력 — 무엇을 어디에

모든 걸 시스템 프롬프트에 넣으면 비대해진다. 정보의 수명 과 범위 로 위치를 정하라.

정보어디에왜
영구·전역 규칙시스템 프롬프트항상 적용
상황별 능력/절차스킬 (14장)필요할 때만 로드
세션 넘는 사실메모리 (8장)recall로 적시 주입
이번 작업 한정 지시사용자 메시지일회성
런타임 상태런타임 주입(system-reminder)적시성

6.6 흔한 실수 / 안티패턴

  • ❌ 규칙 백과사전 — 수백 줄 규칙은 서로 충돌하고 주의력을 분산시킨다. 핵심만, 나머진 분리.
  • ❌ 이유 없는 명령 — 모델이 새 상황에 일반화하지 못한다.
  • ❌ 우선순위 부재 — 충돌 시 행동이 불안정.
  • ❌ 가변 정보 삽입 — 타임스탬프 등은 캐시를 깬다(5장). 런타임 주입을 써라.
  • ❌ 부정형 일색 — "하지 마라"보다 "대신 ~하라"가 효과적.

6.7 핵심 정리

  • 시스템 프롬프트 = 에이전트의 헌법: 역할·능력·제약·환경·안전·스타일.
  • 규칙엔 이유 와 우선순위 를 붙여라 — 일반화와 안정성의 열쇠.
  • 짧게 유지하고, 상황별 내용은 스킬·메모리로 분리하라.
  • 가변 정보는 시스템 프롬프트가 아니라 런타임 주입으로.

6.8 더 읽기

  • 다음: 7장 — 권한과 안전 — ⑤ 안전 블록의 실제 구현.
  • 분리 대상: 8장 메모리 · 14장 스킬.
  • 캐시 영향: 5장 컨텍스트 · 21장 비용.

7장 — 권한과 안전 (Permissions & Safety)

한 줄 요약: 에이전트는 진짜로 파일을 지우고, 명령을 실행하고, 메시지를 보낸다.

권한 모델·샌드박스·사전 확인은 자율성과 안전 사이의 균형을 잡는 하네스의 보호층이다.

7.1 왜 중요한가

3장의 루프는 모델이 원하는 툴을 그냥 실행 했다. 읽기 전용 데모라면 괜찮다. 하지만 실제 에이전트는 rm, git push, 이메일 전송, 배포 같은 되돌리기 어렵고 바깥에 영향을 주는 작업을 한다. 모델은 가끔 틀린다. 보호층이 없으면 그 실수가 곧 사고다.

핵심 명제: 자율성과 안전은 트레이드오프다. 하네스의 일은 위험에 비례하는 마찰을

두는 것 — 안전한 작업은 자유롭게, 위험한 작업은 확인을 거쳐.

7.2 위험의 스펙트럼

모든 툴 호출이 같은 위험은 아니다. 위험도에 따라 다른 정책을 적용한다.

flowchart LR
    A["읽기 전용<br/>read_file, search"] -->|"자유 실행"| Safe["마찰 없음"]
    B["국소 부작용<br/>파일 수정, 로컬 커밋"] -->|"정책에 따라"| Med["허용 또는 확인"]
    C["되돌리기 어려움/외부<br/>삭제, push, 전송, 배포"] -->|"사전 확인"| High["명시적 승인 필요"]

판단 기준 두 축:

  • 되돌릴 수 있는가? — 삭제·덮어쓰기·전송은 취소가 어렵다.
  • 바깥에 나가는가? — 외부로 보내면 캐시·색인돼 회수 불가일 수 있다.

7.3 권한 모드 (Permission modes)

하네스는 보통 "툴 호출을 어디까지 자동 허용할지"를 모드로 묶어 제공한다. 전형적 스펙트럼:

모드동작쓰임
plan / read-only변경 없이 계획만신뢰 전, 탐색·설계 단계
기본(default)위험 작업은 확인 요청일상 작업
auto / accept-edits대부분 자동 실행신뢰된 반복 작업
허용목록(allowlist)특정 명령만 무확인자주 쓰는 안전 명령

📦 이 환경의 사례: 지금 이 세션은 한때 plan mode 였다 — "편집 금지, 읽기 전용,

이 지시가 다른 모든 것에 우선"이라는 강한 보호 모드. 또 설정으로 특정 Bash 명령을

허용목록 에 넣어 매번 묻지 않게 할 수 있다(스킬 fewer-permission-prompts가 이를 돕는다).

핵심은 모드가 정책을 묶어 사용자가 신뢰 수준을 한 번에 조절하게 한다는 점이다.

7.4 샌드박싱 (Sandboxing)

위험한 실행을 격리된 환경 에서 돌려 부작용을 차단한다. 파일시스템·네트워크 접근을 제한하거나, 일회용 컨테이너/워크트리에서 실행하는 식이다.

  • 읽기 샌드박스: 네트워크 차단된 채 명령 실행 → 정보 유출·외부 호출 방지.
  • 쓰기 격리: 별도 워크트리에서 변경 → 메인을 건드리지 않고 검증 후 병합(→ 9장).
  • 확인 우회와의 관계: 샌드박스 안이라면 더 자유롭게 실행하고, 샌드박스를 벗어나는

    • 작업(네트워크, 메인 브랜치)만 확인을 요구할 수 있다.

flowchart TB
    Cmd["모델이 명령 실행 요청"] --> Q{"부작용이<br/>샌드박스 안인가?"}
    Q -->|"예 (격리됨)"| Run["자유 실행"]
    Q -->|"아니오 (외부/영구)"| Ask["사전 확인 요구"]

7.5 사전 확인 — 무엇을, 어떻게

되돌리기 어려운 작업은 실행 전 사용자에게 확인을 받는다. 좋은 확인의 원칙:

  1. 무엇을·왜를 명확히 — "build/ 디렉토리를 삭제합니다(이전 빌드 정리)" 처럼 대상과

    2. 이유를 보여라.

  1. 한 맥락의 승인은 그 맥락에만 — 한 번 허락받았다고 다음번까지 확장하지 마라.
  1. 삭제·덮어쓰기 전엔 대상 확인 — 지우려는 것이 설명과 다르거나, 내가 만들지 않은

    2. 것이면 멈추고 사용자에게 알려라.

  1. 외부 전송은 게시로 간주 — 보내기 전 한 번 더.

📦 이 환경의 사례: 이 하네스의 지침은 "되돌리기 어렵거나 바깥에 영향을 주는

작업은, 지속적 승인이나 명시적 허가가 없으면 먼저 확인하라; 한 맥락의 승인이 다음으로

이어지지 않는다", "삭제·덮어쓰기 전에 대상을 살펴보고, 내가 만들지 않았거나 설명과

모순되면 그걸 먼저 보고하라"고 명시한다. 이것이 7.5 원칙의 실제 문장화다.

7.6 신뢰의 점진적 확대

처음부터 모든 권한을 줄 필요는 없다. 신뢰는 단계적으로 키운다.

flowchart LR
    A["plan mode<br/>계획만 보고 신뢰"] --> B["기본 모드<br/>확인하며 실행"] --> C["allowlist<br/>안전 명령 무확인"] --> D["auto<br/>검증된 작업 자동화"]

새 작업·새 코드베이스일수록 왼쪽에서, 익숙하고 검증된 반복일수록 오른쪽으로 이동한다.

7.7 흔한 실수 / 안티패턴

  • ❌ 균일한 정책 — 읽기와 삭제를 같은 마찰로 다루면, 너무 빡빡하거나 너무 위험하다.

    • 위험에 비례 시켜라.

  • ❌ 승인 범위 확장 — "아까 허락했으니 계속 해도 되겠지"는 사고의 지름길.
  • ❌ 확인 메시지가 모호 — "실행할까요?"만으론 사용자가 위험을 모른다. 대상·이유를 보여라.
  • ❌ 샌드박스 맹신 — 격리가 완벽하다고 가정하지 마라. 외부로 나가는 경로는 항상 점검.
  • ❌ 본 적 없는 것을 덮어쓰기 — 삭제·덮어쓰기 전 대상을 읽어 확인하라.

7.8 핵심 정리

  • 위험은 스펙트럼이다: 되돌릴 수 있나 · 바깥에 나가나 두 축으로 판단.
  • 권한 모드로 신뢰 수준을 묶어 조절(plan → 기본 → allowlist → auto).
  • 샌드박스로 부작용을 격리, 격리를 벗어나는 작업만 확인.
  • 되돌리기 어려운/외부 작업은 대상·이유를 보이며 사전 확인. 승인은 그 맥락에만.

8장 — 메모리 시스템 (Memory Systems)

한 줄 요약: 컨텍스트 윈도우는 세션이 끝나면 사라진다. 메모리는 세션을 넘어 지속되는

사실의 저장소다. 무엇을 저장하고, 어떻게 불러오고(recall), 무엇을 저장하지 않을지 가

메모리 설계의 핵심이다.

8.1 왜 중요한가

5장에서 본 컨텍스트는 휘발성이다 — 압축되면 잊히고, 세션이 끝나면 통째로 사라진다. 그런데 어떤 사실은 지속 되어야 한다: 사용자의 선호, 프로젝트의 제약, 반복되는 교훈. 이걸 매번 다시 알려주게 하는 것은 나쁜 경험이다. 메모리는 이 간극을 메운다.

핵심 명제: 메모리는 "윈도우 밖에 사는 컨텍스트"다. 항상 들고 다니기엔 비싸지만

(5장), 필요할 때 불러올 수 있어야 하는 사실 — 그것이 메모리에 들어간다.

8.2 메모리의 수명 계층

정보를 얼마나 오래 살아야 하는가 로 나누면 어디에 둘지 분명해진다.

flowchart TB
    A["한 턴<br/>툴 결과·중간 추론"] --> B["한 세션<br/>대화 이력 (압축 대상)"]
    B --> C["세션을 넘어<br/>★ 메모리 (파일 영속)"]
    C --> D["코드/문서에 이미 기록됨<br/>메모리에 넣지 말 것"]

메모리는 ③ 계층을 담당한다. ④(코드 구조, git 이력, README가 이미 말하는 것)는 메모리에 중복 저장하지 않는다 — 이미 영속적이고, 중복은 낡으면 거짓이 된다.

8.3 파일 기반 메모리 — 한 사실, 한 파일

견고한 패턴 하나: 각 메모리를 하나의 파일 로, 하나의 사실만 담아 저장한다. 구조화를 위해 frontmatter(메타데이터 머리말)를 붙인다.

---
name: <짧은-kebab-슬러그>
description: <한 줄 요약 — recall 때 관련성 판단에 쓰임>
metadata:
  type: user | feedback | project | reference
---

<사실 본문. feedback/project는 **Why:** 와 **How to apply:** 를 덧붙인다.
 관련 메모리는 [[다른-메모리-name]] 로 링크한다.>
  • name: 파일·링크의 식별자.
  • description: recall 시 "이게 지금 관련 있나?"를 빠르게 판단하게 하는 한 줄.
  • type: 분류 — 사용자(누구인가)·피드백(어떻게 일하라)·프로젝트(진행 맥락)·참조(외부 링크).
  • 본문: 사실 하나. 피드백/프로젝트는 왜 와 어떻게 적용 을 함께 — 6장의 "이유를

    • 붙여라"와 같은 이유로, 그래야 새 상황에 일반화된다.

📦 이 환경의 사례: 이 하네스는 정확히 이 구조의 파일 메모리를 쓴다. MEMORY.md

라는 인덱스에 한 줄짜리 포인터(- [제목](file.md) — 훅)를 두고, 각 사실은 frontmatter를

가진 개별 파일에 산다. 실제로 이 가이드북 작업 중 "다이어그램은 mermaid로"라는 사용자

피드백을 harness-guidebook-mermaid.md 파일로 저장하고 MEMORY.md에 포인터를 넣었다 —

8.3의 패턴 그대로다.

8.4 인덱스와 recall — 어떻게 "적시에" 떠오르나

메모리가 수백 개면 전부를 매 세션 컨텍스트에 넣을 수 없다(5장의 비용). 해법은 2단계다.

flowchart TB
    Idx["가벼운 인덱스 (MEMORY.md)<br/>각 메모리 한 줄 요약만 로드"] --> Q{"지금 작업에<br/>관련된 메모리가?"}
    Q -->|"있음"| Load["해당 파일 전체를 recall<br/>(컨텍스트에 주입)"]
    Q -->|"없음"| Skip["로드 안 함 — 윈도우 절약"]
  • 인덱스는 항상, 본문은 선택적 — description 한 줄만 늘 보이고, 본문은 관련될 때만 불러온다.

    • 4장의 지연 툴 과 똑같은 "이름은 싸고 내용은 비싸다" 원리다.

  • recall된 메모리는 배경 정보 — 사용자의 직접 지시가 아니라 과거에 참이었던 사실 이다.

    • 그래서 파일명·플래그를 언급하면, 추천 전에 아직 존재하는지 확인 해야 한다.

8.5 무엇을 저장할 것인가 (그리고 하지 말 것)

메모리의 질은 무엇을 넣지 않느냐 로 결정된다.

저장할 것

  • 사용자: 역할·전문성·선호.
  • 피드백: "이렇게 일해라"는 교훈(이유 포함).
  • 프로젝트: 코드에서 유추 안 되는 진행 맥락·제약(상대 날짜는 절대 날짜로 변환).
  • 참조: 외부 리소스 포인터(URL·대시보드·티켓).

저장하지 말 것

  • 코드 구조, 과거 수정 내역, git 이력, README가 이미 말하는 것.
  • 이번 대화에서만 의미 있는 것.
  • 틀린 것으로 판명된 메모리 → 삭제 하라.

누가 "이 파일 구조를 기억해"라고 하면, 그대로 저장하기 전에 "무엇이 비자명했나"를

물어 그 부분만 저장하라. 코드가 이미 말하는 것은 메모리의 낭비이자 부패의 씨앗이다.

8.6 메모리의 함정 — 부패(staleness)

메모리는 쓰인 시점의 진실 이다. 시간이 지나면 코드가 바뀌어 메모리가 거짓이 된다. 대비책:

  • 중복 금지 — 같은 사실을 코드와 메모리 양쪽에 두지 마라. 한쪽만 갱신되면 모순.
  • 사용 전 검증 — 메모리가 파일·함수·플래그를 지목하면, 추천 전 실존 확인.
  • 갱신·삭제 — 새 사실이 기존 파일을 덮으면 그 파일을 수정(중복 생성 금지), 틀린

    • 메모리는 삭제.

8.7 흔한 실수 / 안티패턴

  • ❌ 모든 걸 저장 — 코드가 이미 말하는 것까지 넣으면 부패와 잡음만 늘린다.
  • ❌ 한 파일에 여러 사실 — recall 정밀도가 떨어진다. 한 사실 = 한 파일.
  • ❌ 인덱스 없이 전부 로드 — 윈도우를 메모리가 잡아먹는다.
  • ❌ recall을 지시로 오인 — 메모리는 배경 사실이지 새 명령이 아니다.
  • ❌ 상대 날짜 저장 — "다음 주"는 나중에 무의미. 절대 날짜로 변환해 저장.

8.8 핵심 정리

  • 메모리 = 세션을 넘어 사는 컨텍스트. "한 사실, 한 파일 + 가벼운 인덱스".
  • recall은 지연 로딩: description으로 관련성 판단 → 관련될 때만 본문 주입.
  • 코드/이력이 이미 말하는 것은 저장하지 마라 — 중복은 부패한다.
  • 메모리는 쓰인 시점의 진실 — 사용 전 실존을 검증하라.

서브에이전트와 검증 루프의 조립법

9장 — 서브에이전트 (Subagents)

한 줄 요약: 한 에이전트로 모든 걸 하면 컨텍스트가 터지고 집중이 흩어진다.

작업을 서브에이전트에게 위임하면, 무거운 일을 격리하고 결론만 가져올 수 있다.

fan-out, fork vs fresh, 격리 — 이 장의 세 축이다.

9.1 왜 중요한가

5장에서 봤듯 컨텍스트는 유한하다. 그런데 어떤 작업은 본질적으로 방대하다 — "이 코드베이스 전체에서 이 패턴을 쓰는 곳을 다 찾아줘"는 수십 개 파일을 읽어야 한다. 그걸 메인 에이전트가 직접 하면 윈도우가 파일 덤프로 가득 차고, 정작 결론 을 담을 공간이 없다.

서브에이전트는 이걸 해결한다: 무거운 작업을 별도 컨텍스트 에서 처리하고, 메인에는 정제된 결과만 돌려준다. 동시에 여러 개를 띄우면 병렬화 도 된다.

핵심 명제: 서브에이전트는 두 가지를 산다 — 격리(메인 컨텍스트를 더럽히지 않음)와

병렬성(독립 작업 동시 진행). 위임의 비용은 조율 오버헤드다.

9.2 언제 위임하는가

flowchart TB
    Q{"이 작업이..."} --> A["여러 파일을 훑어야<br/>결론이 나온다"]
    Q --> B["서로 독립적인<br/>일이 여럿이다"]
    Q --> C["단일 사실 조회<br/>(파일·값을 이미 안다)"]
    A -->|"위임 → 결론만 회수"| Deleg["서브에이전트"]
    B -->|"병렬 위임"| Deleg
    C -->|"직접 처리"| Direct["메인이 바로"]
  • 위임이 이득: 광범위 탐색, 독립적 병렬 작업, 여러 파일을 가로지르는 조사.
  • 위임이 손해: 이미 파일·심볼을 아는 단일 조회 — 위임 오버헤드가 더 크다.
  • 중복 금지: 한 번 위임했으면 같은 검색을 메인이 또 하지 마라 — 결과를 기다려라.

📦 이 환경의 사례: 이 하네스의 Explore/Agent 툴 설명은 정확히 이 기준을 말한다 —

"답이 여러 파일·디렉토리를 쓸어야 나오고 결론만 필요하면 위임하라; 이미 파일·심볼·값을

아는 단일 조회면 직접 검색하라." 또 "한 번 위임했으면 메인이 같은 걸 또 돌리지 말고

결과를 기다리라"고 못 박는다.

9.3 fork vs fresh — 컨텍스트를 물려줄 것인가

서브에이전트를 띄우는 두 방식이 있다. 이 구분이 중요하다.

    subgraph Fork["fork — 컨텍스트 상속"]
        P1["부모의 대화·맥락"] --> F["서브가 전체 맥락을 안 채 시작"]
    end
    subgraph Fresh["fresh — 빈 컨텍스트"]
        P2["명시적으로 준 프롬프트만"] --> N["서브가 백지에서 시작"]

forkfresh
컨텍스트부모 전체 상속준 프롬프트만

  • fork: "내가 아는 걸 다 아는 부하"가 필요할 때. 일관된 톤·맥락이 중요한 작업(예:

    • 이 책처럼 정해진 스타일로 여러 장 쓰기)에 강하다.

  • fresh: 맥락 오염을 피하고 좁은 작업만 시킬 때. 대부분의 탐색·조사에 적합.

📦 이 환경의 사례: Agent 툴은 subagent_type: "fork"면 "현재 대화 맥락을 상속하고

항상 부모 모델로 실행"되며, 그 외 타입은 "백지에서 시작"한다. 또 이미 띄운 서브에이전트는

SendMessage로 맥락을 유지한 채 이어 말할 수 있고, 새 Agent 호출은 새로 시작한다.

9.4 격리 — 병렬 수정의 충돌 막기 (worktree)

서브에이전트들이 읽기만 하면 충돌이 없다. 하지만 여럿이 동시에 같은 파일을 수정 하면 서로를 덮어쓴다. 해법은 각자에게 분리된 작업 공간(worktree)을 주는 것이다.

flowchart TB
    Main["메인 저장소"] --> W1["worktree A<br/>서브1이 수정"]
    Main --> W2["worktree B<br/>서브2가 수정"]
    W1 -->|"검증 후 병합"| Main
    W2 -->|"검증 후 병합"| Main
  • worktree는 저장소의 격리된 사본 — 병렬 수정이 충돌하지 않는다.
  • 단 비싸다(설정 시간 + 디스크). 진짜로 병렬 수정이 충돌할 때만 써라. 읽기 전용

    • 탐색엔 불필요하다.

📦 이 환경의 사례: Agent/Workflow의 isolation: "worktree" 옵션이 이것이다.

설명은 "병렬로 파일을 변경해 충돌할 때만 쓰라 — 비싸다(설정 ~수백 ms + 디스크), 변경이

없으면 자동 정리"라고 정확히 경고한다.

9.5 위임의 비용과 한계

서브에이전트는 공짜가 아니다.

  • 조율 오버헤드: 띄우고, 프롬프트를 정하고, 결과를 합치는 비용.
  • 맥락 손실(fresh): 백지에서 시작하므로 프롬프트에 충분한 배경 을 줘야 한다.

    • 부실한 프롬프트 → 부실한 결과.

  • 결과는 사용자에게 안 보임: 서브의 최종 메시지는 나에게 오는 데이터다. 사용자에게

    • 필요한 건 내가 다시 전달해야 한다.

  • 깊이 제한: 보통 위임은 1단계만 — 서브가 또 위임하는 무한 중첩은 막혀 있다.

좋은 위임 프롬프트 = 배경 + 구체적 작업 + 원하는 출력 형식. fresh 서브는 내가 준

것만 안다는 걸 늘 기억하라.

9.6 흔한 실수 / 안티패턴

  • ❌ 단일 조회를 위임 — 이미 아는 파일 하나 읽기를 위임하면 오버헤드만 손해.
  • ❌ 위임하고 메인도 같은 일 — 중복. 위임했으면 기다려라.
  • ❌ 빈약한 프롬프트로 fresh 띄우기 — 맥락이 없어 헤맨다. 배경을 충분히.
  • ❌ 읽기 작업에 worktree — 불필요하게 비싸다. 병렬 수정 충돌 때만.
  • ❌ 서브 결과를 그대로 사용자에게 안 전달 — 서브의 출력은 나를 위한 것, 요약해 전하라.

9.7 핵심 정리

  • 서브에이전트 = 격리(컨텍스트 보호) + 병렬성. 비용은 조율 오버헤드.
  • 위임 기준: 여러 파일을 가로지르거나 독립 병렬 작업이면 위임, 단일 조회면 직접.
  • fork(맥락 상속) vs fresh(백지) — 작업 성격에 맞게.
  • 병렬 수정 충돌엔 worktree 격리, 단 비싸니 꼭 필요할 때만.

9.8 더 읽기

  • 다음: 10장 — 워크플로우와 파이프라인 — 여러

    • 서브에이전트를 결정적으로 조율하기.

  • 컨텍스트 격리의 동기: 5장 — 컨텍스트 엔지니어링.
  • 위임 결과의 검증: 11장 — 검증 루프.

10장 — 워크플로우와 파이프라인 (Workflows & Pipelines)

한 줄 요약: 여러 에이전트를 모델의 즉흥 판단 이 아니라 결정적 제어 흐름 으로

조율하는 것이 워크플로우다. 핵심 선택은 pipeline(흐름) vs parallel(배리어)이며,

대부분의 경우 pipeline이 정답이다.

10.1 왜 중요한가

9장의 서브에이전트를 "여러 개, 여러 단계로" 엮으면 강력해진다. 하지만 어떻게 엮느냐가 성능을 가른다. 단계 사이에 불필요한 동기화(배리어)를 두면, 빠른 작업이 느린 작업을 하염없이 기다리며 벽시계 시간을 낭비한다.

워크플로우는 이 조율을 코드(루프·조건·팬아웃)로 결정적으로 표현한다 — 모델이 매번 "다음에 뭐 하지"를 즉흥 결정하게 두는 대신.

핵심 명제: 제어 흐름이 결정적이어야 할 때(반복·분기·팬아웃)는 모델이 아니라

스크립트 가 조율하라. 그리고 단계 사이 배리어는 꼭 필요할 때만 둬라.

10.2 세 가지 기본 조합자

flowchart TB
    subgraph S["agent — 한 명"]
        a1["작업 → 결과"]
    end
    subgraph P["parallel — 배리어"]
        p1["작업1"] --> bar(("모두 대기"))
        p2["작업2"] --> bar
        p3["작업3"] --> bar
        bar --> pr["결과 배열"]
    end
    subgraph PL["pipeline — 배리어 없음"]
        i1["항목A: 1단계→2단계→3단계"]
        i2["항목B: 1단계→2단계→3단계"]
        note["A가 3단계일 때 B는 1단계일 수 있음"]
    end
  • agent(작업): 서브에이전트 하나. 결과(텍스트 또는 구조화 객체)를 돌려준다.
  • parallel(작업들): 여러 작업을 동시에, 모두 끝날 때까지 기다림(배리어). 결과를

    • 한꺼번에 모아야 할 때.

  • pipeline(항목들, 단계들): 각 항목을 모든 단계에 독립적으로 통과. **단계 사이

    • 배리어 없음.** 항목 A가 3단계를 도는 동안 항목 B는 1단계일 수 있다.

10.3 pipeline이 기본인 이유

다단계 작업의 기본값은 pipeline 이다. 벽시계 시간이 "가장 느린 단일 항목의 사슬"로 결정되지, "단계별 가장 느린 것의 합"이 아니기 때문이다.

flowchart LR
    subgraph Bad["parallel 연쇄 (배리어 2번) — 낭비"]
        b1["3 finder 모두 끝날 때까지 대기"] --> b2["그다음 verify 일괄"]
    end
    subgraph Good["pipeline — 배리어 없음"]
        g1["finder가 끝나는 즉시 그 항목 verify 시작"]
    end

다섯 개의 finder가 도는데 가장 느린 게 가장 빠른 것의 3배 걸린다면, 배리어는 빠른 finder들의 시간을 2/3나 버린다. pipeline은 끝나는 즉시 다음 단계로 흘려보낸다.

10.4 배리어(parallel)가 정당한 때 — 그리고 아닌 때

배리어는 단계 N이 단계 N-1의 전체 결과 를 가로질러 봐야 할 때만 옳다.

정당한 배리어

  • dedup/merge: 비싼 다음 단계 전에 전체 결과에서 중복 제거·병합.
  • early-exit: 총합이 0이면 다음 단계 전체를 건너뛰기("버그 0개 → 검증 생략").
  • 교차 참조: 다음 단계 프롬프트가 "다른 발견들과 비교"를 요구.

정당하지 않은 배리어 (흔한 착각)

  • "먼저 flatten/map/filter 해야 해" → 그건 pipeline 단계 안 에서 하면 된다.
  • "단계가 개념적으로 분리돼 있어" → 분리 ≠ 동기화 필요.
  • "코드가 더 깔끔해" → 배리어 지연은 실재하는 비용이다.

냄새 테스트: parallel → (교차 의존 없는 단순 변환) → parallel 패턴을 쓰고 있다면,

그 중간 변환은 배리어가 필요 없다. pipeline 단계 안으로 넣어라.

10.5 정전(canonical) 패턴 — 차원별 리뷰 후 즉시 검증

가장 자주 쓰는 다단계 모양: 각 차원(dimension)을 리뷰하고, 그 리뷰가 끝나는 즉시 검증으로 흘려보낸다.

pipeline(
  DIMENSIONS,
  d      => agent(리뷰 프롬프트(d), {schema: FINDINGS}),        // 1단계
  review => parallel(review.findings.map(f =>                  // 2단계
    () => agent(`적대적으로 검증: ${f}`, {schema: VERDICT})
  ))
)
// dimension 'bugs'의 발견이 검증되는 동안, 'perf' dimension은 아직 리뷰 중.
// 벽시계 낭비 없음.

핵심: 1단계(리뷰)와 2단계(검증)가 항목별로 독립적으로 흐른다. 한 차원의 결과를 다른 차원이 기다리지 않는다.

📦 이 환경의 사례: 이 하네스의 Workflow 툴이 정확히 이 모델이다 — agent(),

parallel(), pipeline() 조합자와 "DEFAULT TO pipeline(), 배리어는 교차 의존이 있을

때만"이라는 규칙을 그대로 제공한다. 동시 실행은 코어 수에 따라 캡(예: 최대 16)되고,

초과분은 큐잉돼 슬롯이 나면 실행된다.

10.6 루프 패턴 — 알 수 없는 크기의 작업

발견할 양을 미리 모를 때(버그, 엣지 케이스)는 카운터 루프나 예산 루프를 쓴다.

// 목표 개수까지 누적
bugs = []
while bugs.length < 10:
    r = agent("버그를 찾아라", {schema: BUGS})
    bugs.push(...r.bugs)

// 예산까지 (토큰 한도에 맞춰 깊이 조절)
while budget.remaining() > 50_000:
    ...

→ 더 정교한 "고갈까지 반복(loop-until-dry)"은 11장에서.

10.7 resume — 결정적 조율의 보너스

조율이 스크립트로 결정적이면, 중단된 워크플로우를 재개 할 수 있다. 바뀌지 않은 앞부분 단계는 캐시된 결과를 즉시 반환하고, 처음 바뀐 단계부터 다시 실행한다. 같은 스크립트 + 같은 입력 = 100% 캐시 적중.

이것이 "모델 즉흥 조율"이 줄 수 없는 결정적 워크플로우의 이점이다 — 재현성과 재개.

10.8 흔한 실수 / 안티패턴

  • ❌ 습관적 배리어 — parallel을 연쇄해 단계마다 모두 대기시키기. 벽시계 낭비.
  • ❌ 중간 변환을 위한 배리어 — flatten/map은 pipeline 단계 안에서.
  • ❌ 결정적이어야 할 흐름을 모델에 맡김 — 반복·분기는 스크립트로.
  • ❌ 무한 루프 무방비 — 카운터·예산·최대 횟수 안전장치를 둬라(2·21장).
  • ❌ 동시성 캡 무시 — 수백 개를 한 번에 띄워도 동시 실행은 캡까지. 큐잉을 이해하라.

10.9 핵심 정리

  • 워크플로우 = 에이전트를 결정적 제어 흐름 으로 조율(반복·분기·팬아웃).
  • 다단계의 기본값은 pipeline — 배리어 없이 항목별로 흐른다.
  • 배리어(parallel)는 단계 N이 N-1의 전체 를 봐야 할 때만(dedup·early-exit·교차참조).
  • 결정적 조율은 재개(resume)라는 보너스를 준다.

11장 — 검증 루프 (Verification Loops)

한 줄 요약: 모델은 그럴듯하지만 틀린 답을 자신 있게 내놓는다. 검증 루프 —

적대적 검증, 심판 패널, 고갈까지 반복 — 은 결과를 반증 하거나 교차 평가 해

거짓 양성을 걸러내고 누락을 줄인다.

11.1 왜 중요한가

1장의 다섯 책임 중 마지막이 확인(verifying) 이었다. 이유는 단순하다: LLM은 확신과 정확성이 비례하지 않는다. 버그를 "찾았다"고 단언하지만 실제론 아닌 경우, 코드를 "고쳤다"지만 테스트는 깨진 경우 — 검증 없이는 이 거짓 양성이 그대로 사용자에게 간다.

핵심 명제: 생성과 검증을 분리하라. 무언가를 만든 에이전트에게 "맞아?"를 물으면

자기 합리화를 한다. 다른 에이전트에게 "틀렸음을 증명해봐"라고 시켜라.

11.2 적대적 검증 (Adversarial verify)

핵심 기법. 결과를 확인할 때 "맞는지 봐줘"가 아니라 "이걸 반증해봐" 라고 시킨다. 회의론자를 N명 띄워 각자 반박 을 시도하게 하고, 과반이 반박하면 폐기한다.

flowchart TB
    Claim["주장: '여기 버그가 있다'"] --> V1["검증자1: 반증 시도"]
    Claim --> V2["검증자2: 반증 시도"]
    Claim --> V3["검증자3: 반증 시도"]
    V1 --> Vote{"과반이<br/>반박했나?"}
    V2 --> Vote
    V3 --> Vote
    Vote -->|"예"| Kill["폐기 (거짓 양성)"]
    Vote -->|"아니오"| Keep["확정 (살아남음)"]
votes = parallel(3개, () =>
    agent(`이것을 반증하라: ${claim}. 불확실하면 기본값은 '반박됨(refuted=true)'.`,
          {schema: VERDICT}))
survives = votes.filter(v => !v.refuted).length >= 2

포인트:

  • 불확실하면 반박 쪽으로 기울게 하라(기본값 refuted=true). 보수적 통과가 거짓 양성을 줄인다.
  • 독립적으로 띄워라 — 서로의 판단을 못 보게 해야 다양성이 산다.

📦 이 환경의 사례: 이 하네스의 Workflow/code-review 패턴은 "각 발견마다 N명의

독립 회의론자를 띄워 반박을 시도 하게 하고, 과반이 반박하면 폐기"하는 adversarial

verify를 표준으로 제시한다. 그럴듯하지만 틀린 발견이 살아남는 걸 막는다.

11.3 관점 다양화 검증 (Perspective-diverse verify)

한 발견이 여러 방식으로 틀릴 수 있으면, 똑같은 회의론자 N명보다 서로 다른 렌즈 를 가진 검증자가 낫다.

flowchart TB
    F["발견"] --> C["정확성 렌즈<br/>논리가 맞나"]
    F --> S["보안 렌즈<br/>취약점인가"]
    F --> R["재현성 렌즈<br/>실제로 재현되나"]
    C --> J{"종합 판정"}
    S --> J
    R --> J

중복(같은 질문 N번)이 못 잡는 실패 모드를 다양성이 잡는다. 정확성·보안·성능·재현성처럼 독립적 실패 축 마다 검증자를 배치하라.

11.4 심판 패널 (Judge panel)

생성에도 검증 원리를 쓴다. 한 답을 반복 개선하는 대신, 서로 다른 각도 의 답 N개를 독립 생성하고, 병렬 심판으로 점수 매겨, 최고를 골라 종합한다.

flowchart TB
    G1["시도1: MVP 우선"] --> Jdg["병렬 심판 채점"]
    G2["시도2: 리스크 우선"] --> Jdg
    G3["시도3: 사용자 우선"] --> Jdg
    Jdg --> Syn["승자 기반 종합<br/>+ 차점자의 좋은 아이디어 접목"]

해 공간이 넓을 때(설계·전략) "한 번 시도 후 반복"보다 강하다 — 서로 다른 출발점이 국소 최적에 갇히는 걸 막는다.

11.5 고갈까지 반복 (Loop-until-dry)

발견할 양을 모를 때, 단순 카운터(N개 찾을 때까지)는 꼬리 를 놓친다. 대신 K번 연속 새 발견이 없을 때까지 반복한다.

flowchart TB
    Start["라운드 시작"] --> Find["finder들 실행"]
    Find --> Fresh{"새 발견이<br/>있나? (seen과 대조)"}
    Fresh -->|"있음"| Reset["dry=0, seen에 추가, 검증"] --> Start
    Fresh -->|"없음"| Inc["dry++"]
    Inc --> Q{"dry >= K?"}
    Q -->|"아니오"| Start
    Q -->|"예"| Done["종료 — 고갈"]
seen = Set(); dry = 0
while dry < 2:
    found = parallel(FINDERS, ...).flatMap(r => r.bugs)
    fresh = found.filter(b => !seen.has(key(b)))   // 전체 seen과 대조
    if fresh.empty: dry++; continue
    dry = 0; fresh.forEach(b => seen.add(key(b)))
    confirmed.push(...verify(fresh))

⚠️ 함정: dedup은 confirmed가 아니라 seen과 대조하라. 안 그러면 심판이 기각한

발견이 매 라운드 다시 나타나 영원히 수렴하지 않는다.

11.6 검증을 워크플로우에 얹기

10장의 pipeline과 결합하면 자연스럽다 — 리뷰가 끝나는 즉시 검증으로 흘려보낸다(10.5). 배리어가 정당한 유일한 경우는 전체 발견을 dedup한 뒤 비싼 검증을 돌릴 때다.

all     = parallel(DIMENSIONS, ...).flatMap(r => r.findings)
deduped = dedupeByFileAndLine(all)        // ← 전체가 필요 → 배리어 정당
verified = parallel(deduped.map(f => () => agent(verifyPrompt(f))))
confirmed = verified.filter(v => v.isReal)

11.7 흔한 실수 / 안티패턴

  • ❌ 생성자에게 자기 검증 — 자기 합리화한다. 검증은 독립 에이전트에게.
  • ❌ "맞아?"라고 묻기 — 동의 편향. "틀렸음을 증명하라"로 물어라.
  • ❌ 불확실을 통과로 — 검증은 불확실하면 기각 으로 기울어야 거짓 양성이 준다.
  • ❌ 단순 카운터로 탐색 종료 — 꼬리를 놓친다. loop-until-dry를 써라.
  • ❌ confirmed로 dedup — 기각된 게 재등장해 무한 루프. seen으로 dedup하라.
  • ❌ 모든 것에 과도한 검증 — 비용↑. 위험·불확실성이 큰 결과에 집중하라.

11.8 핵심 정리

  • 생성과 검증을 분리하라 — 독립 에이전트가 반증 을 시도하게.
  • 적대적 검증: N명 회의론자, 불확실하면 기각, 과반 반박이면 폐기.
  • 다양한 실패 축엔 관점 다양화 검증, 넓은 해 공간엔 심판 패널.
  • 미지의 양엔 loop-until-dry (seen으로 dedup, K번 빈 라운드에 종료).

12장 — 멀티에이전트 패턴 (Multi-Agent Patterns)

한 줄 요약: 9~11장의 조각(위임·파이프라인·검증)을 더 큰 구조로 합성한다.

multi-modal sweep, completeness critic, 그리고 이들을 조립한 "포괄적 탐색" 하네스 —

한 컨텍스트가 담을 수 없는 일을 해내는 법.

12.1 왜 중요한가

어떤 작업은 한 에이전트의 능력·컨텍스트를 넘는다 — 거대한 코드베이스 감사, 모든 엣지 케이스 발굴, 한 사람이 다 읽을 수 없는 자료의 종합. 이때 필요한 건 더 똑똑한 모델이 아니라 더 나은 구조 다. 멀티에이전트 패턴은 그 구조의 어휘다.

핵심 명제: 멀티에이전트의 목적은 세 가지 — 포괄성(분해해 병렬 커버), 확신

(독립 관점 + 적대적 점검), 규모(한 컨텍스트가 못 담는 일). 패턴을 작업에 맞게 골라 합성하라.

12.2 multi-modal sweep — 서로 다른 방식으로 훑기

한 가지 검색 각도는 항상 무언가를 놓친다. 여러 에이전트가 각자 다른 방식 으로 동시에 훑게 하라 — 컨테이너별, 내용별, 엔티티별, 시간별. 각자는 남들이 무얼 찾는지 모른다.

flowchart TB
    Target["대상 (코드베이스/자료)"] --> A["방식1: 구조로 탐색<br/>(디렉토리·모듈)"]
    Target --> B["방식2: 내용으로 탐색<br/>(키워드·패턴)"]
    Target --> C["방식3: 엔티티로 탐색<br/>(함수·심볼)"]
    Target --> D["방식4: 시간으로 탐색<br/>(최근 변경·이력)"]
    A --> M["병합 + dedup"]
    B --> M
    C --> M
    D --> M

하나의 검색 전략으로는 전부 못 찾을 때 쓴다. 각 방식의 사각지대가 서로를 보완한다.

12.3 completeness critic — "무엇이 빠졌나"

마지막에 한 에이전트에게 생산물이 아니라 누락 을 묻는다: "안 돌린 방식은? 검증 안 된 주장은? 안 읽은 자료는?" 그 답이 다음 라운드의 작업이 된다.

flowchart TB
    Work["1차 결과물"] --> Critic["완전성 비평가<br/>'무엇이 빠졌나?'"]
    Critic --> Gaps{"빈틈이<br/>있나?"}
    Gaps -->|"있음"| More["빈틈을 다음 라운드 작업으로"] --> Work
    Gaps -->|"없음"| Done["완료"]

단순 "검토해줘"와 다르다 — 적극적으로 사각지대를 사냥 하게 만든다. loop-until-dry(11장)와 짝을 이룬다.

📦 이 환경의 사례: 이 하네스의 워크플로우 가이드는 "completeness critic — 안 돌린

modality, 검증 안 된 주장, 안 읽은 소스가 무엇인지 묻는 마지막 에이전트; 그것이 다음

작업이 된다"를 명시적 품질 패턴으로 든다. 또 "은밀한 캡 금지 — top-N·샘플링으로 커버리지를

제한했으면 무엇을 버렸는지 log 하라"고 경고한다(은밀한 잘림은 '다 했다'처럼 읽힌다).

12.4 합성 — 포괄적 탐색 하네스

조각들을 조립하면 강력한 탐색 하네스가 된다: find → seen과 dedup → 다양한 렌즈로 판정 → 고갈까지 반복.

flowchart TB
    Round["라운드 시작"] --> Sweep["multi-modal sweep<br/>(여러 finder 배리어)"]
    Sweep --> Dedup["전체 seen과 dedup"]
    Dedup --> Fresh{"새 발견?"}
    Fresh -->|"없음"| Dry["dry++ → K번이면 종료"]
    Fresh -->|"있음"| Judge["각 발견을 3개 렌즈로 판정<br/>(정확성·보안·재현)"]
    Judge --> Confirm["과반 통과만 확정"]
    Confirm --> Round
    Dry --> Critic["completeness critic<br/>마지막 누락 점검"]

이 한 그림이 9~12장의 종합이다:

  • multi-modal sweep(12.2) = 누락 줄이는 탐색
  • dedup vs seen(11.5) = 평범한 코드로, 에이전트 아님
  • 다양한 렌즈 판정(11.3) = 관점 다양화 검증
  • loop-until-dry(11.5) = 꼬리까지
  • completeness critic(12.3) = 마지막 그물

규모를 작업에 맞춰라. "버그 좀 찾아줘"엔 finder 몇 개 + 단일 투표 검증. "철저히

감사해줘"엔 큰 finder 풀 + 3~5표 적대적 검증 + 종합 단계. 과한 구조는 과한 비용이다.

12.5 다른 합성 패턴들

  • 토너먼트 브래킷: 후보가 많을 때 쌍대결로 점진적 좁히기.
  • 자가 수리 루프(self-repair): 생성 → 테스트 → 실패하면 실패 로그를 물려 재생성.
  • 단계적 에스컬레이션: 싼 모델/얕은 검증으로 먼저, 통과 못 하면 비싼 자원 투입.
  • 스카우트 후 팬아웃: 먼저 인라인으로 작업 목록을 발굴(파일 나열·범위 파악)한 뒤,

    • 그 목록 위로 파이프라인을 돌린다. 작업의 모양을 알기 전이 아니라, 조율 단계 전 에만 구조를 알면 된다.

📦 이 환경의 사례: 이 하네스의 Workflow 가이드는 "흔한 단일 단계 워크플로우"로

understand(병렬 reader → 구조 지도), design(독립 N개 설계 심판 패널), review(차원 →

발견 → 적대적 검증), research(다중모드 sweep → 정독 → 종합), migrate(사이트 발굴 →

worktree 격리 변환 → 검증)를 제시한다 — 모두 이 장 패턴들의 실제 조합이다.

12.6 흔한 실수 / 안티패턴

  • ❌ 작업에 안 맞는 과한 구조 — 간단한 질문에 5표 검증 패널은 낭비.
  • ❌ 은밀한 캡 — top-N·무재시도·샘플링으로 커버리지를 줄였으면 반드시 log. 안 하면

    • '전부 커버'로 오인된다.

  • ❌ dedup을 confirmed로 — 기각된 게 재등장(11장 함정 재확인).
  • ❌ completeness 점검 생략 — 마지막 그물 없이 "다 했다"고 단정.
  • ❌ finder들이 같은 각도 — 다양성이 없으면 multi-modal의 의미가 없다.

12.7 핵심 정리

  • 멀티에이전트의 목적: 포괄성·확신·규모.
  • multi-modal sweep로 누락을 줄이고, completeness critic로 마지막 빈틈을 메운다.
  • 포괄 탐색 = sweep → seen-dedup → 다양 렌즈 판정 → loop-until-dry → critic.
  • 패턴은 카탈로그가 아니라 어휘 — 작업에 맞게 합성하고, 규모를 비용에 맞춰라.

13장 — 훅 (Hooks)

한 줄 요약: 훅은 라이프사이클 이벤트(툴 호출 전후, 턴 종료 등)에 하네스가 자동

실행하는 사용자 정의 동작이다. 모델이 아니라 하네스가 보장하므로, "항상 X 하라"는

결정론적 자동화를 가능케 한다.

13.1 왜 중요한가

"커밋 전에 항상 린트를 돌려라", "위험한 명령은 항상 차단하라" 같은 요구를 모델에게 부탁 하면 가끔 빠뜨린다 — 모델은 확률적이다. 훅은 이걸 하네스의 결정론 으로 옮긴다. 이벤트가 발생하면 모델의 기분과 무관하게 반드시 실행된다.

핵심 명제: "항상/매번/~할 때마다"는 프롬프트가 아니라 훅으로 구현하라. 모델의

협조가 아니라 런타임의 보장이 필요한 일이다.

13.2 훅의 라이프사이클 위치

훅은 에이전트 루프(2장)의 이음매 에 끼어든다.

flowchart TB
    Start["턴 시작"] --> Pre["[훅] 툴 실행 전<br/>(검사·차단·수정)"]
    Pre --> Tool["툴 실행"]
    Tool --> Post["[훅] 툴 실행 후<br/>(포맷·로깅·후처리)"]
    Post --> Loop{"루프 계속?"}
    Loop -->|"예"| Pre
    Loop -->|"아니오"| Stop["[훅] 턴 종료<br/>(알림·정리·검증)"]

전형적 훅 시점:

  • 툴 실행 전(pre): 위험 명령 차단, 인자 검증·수정, 권한 게이트(7장의 자동화 버전).
  • 툴 실행 후(post): 코드 자동 포맷, 결과 로깅, 부수 작업 트리거.
  • 턴 종료(stop): 사용자 알림, 임시 파일 정리, 최종 검사.

📦 이 환경의 사례: 이 하네스의 설정에서 훅은 settings.json에 정의되며, "Stop 때

X를 보여줘", "특정 명령 전에 검사" 같은 자동 동작을 건다. 가이드는 명시한다 — "자동

동작('이제부터 X할 때마다')은 훅으로 구현해야 한다; 하네스가 실행하는 것이지 모델이

아니므로, 메모리·선호로는 이를 충족할 수 없다." 이것이 13.1 명제의 실제 근거다.

13.3 훅이 받는 것, 돌려주는 것

훅은 단순 트리거가 아니라 흐름에 개입 할 수 있다.

  • 입력: 어떤 이벤트인지, 어떤 툴이 무슨 인자로 호출되는지 등 맥락.
  • 출력/효과:
    • 통과 — 아무 것도 안 함, 정상 진행.
    • 차단(block) — 툴 실행을 막고 이유를 모델에 피드백(모델은 그걸 보고 경로 변경).
    • 수정 — 인자나 결과를 바꿔 전달.
    • 부수 효과 — 로그·알림 등 흐름과 무관한 동작.

훅의 출력은 사용자가 아니라 하네스/모델에게 가는 피드백 으로 다뤄진다. 차단 메시지는

4장의 "에러를 피드백으로"와 같은 정신 — 모델이 읽고 복구하게 쓴다.

13.4 훅 vs 다른 메커니즘 — 무엇을 언제

같은 목표를 프롬프트·메모리·훅으로 할 수 있을 때, 무엇이 맞나?

원하는 것메커니즘왜
"이런 스타일을 선호" (유연한 경향)시스템 프롬프트/메모리모델의 판단 여지를 남김
"항상/반드시 X" (결정론적 보장)훅모델 협조에 의존하면 안 됨
"이 능력을 필요할 때 로드"스킬 (14장)상황별, 모델이 호출
"외부 시스템 연결"MCP (15장)도구·데이터 통합

핵심 판별: 빠뜨리면 안 되는가? 그렇다면 훅이다. 경향 이면 프롬프트, 보장 이면 훅.

flowchart TB
    Q{"이 동작은..."} --> A["가끔 빠져도<br/>괜찮은 경향"]
    Q --> B["반드시 매번<br/>실행돼야 함"]
    A -->|"프롬프트/메모리"| Soft["모델이 판단"]
    B -->|"훅"| Hard["하네스가 보장"]

13.5 훅의 위험 — 자동화는 양날

7장의 교훈이 여기서도 적용된다: 보호 없는 자동화는 사고를 가속 한다.

  • 무한/연쇄 트리거: 훅이 또 다른 훅을 부르는 루프 주의.
  • 조용한 실패: 훅이 실패했는데 흐름은 진행되면 잘못된 안전감. 실패를 가시화하라.
  • 과도한 차단: 너무 빡빡한 pre-훅은 정상 작업까지 막아 에이전트를 무력화한다.
  • 느린 훅: 매 툴 호출마다 무거운 훅이 돌면 전체가 느려진다.

13.6 흔한 실수 / 안티패턴

  • ❌ "항상 X"를 프롬프트로 — 확률적으로 빠진다. 보장이 필요하면 훅.
  • ❌ 훅 실패를 무시 — 안전 훅이 조용히 죽으면 보호가 사라진 줄 모른다.
  • ❌ 차단 사유를 안 줌 — 모델이 왜 막혔는지 모르면 복구 못 한다. 이유를 피드백하라.
  • ❌ 모든 걸 훅으로 — 유연해야 할 경향까지 훅으로 박으면 에이전트가 경직된다.

13.7 핵심 정리

  • 훅 = 라이프사이클 이벤트에 하네스가 보장 하는 자동 동작.
  • "항상/매번"은 훅, "경향/선호"는 프롬프트·메모리.
  • pre(차단·검증)·post(포맷·로깅)·stop(알림·정리) 시점에 끼어든다.
  • 차단은 모델에게 이유를 피드백 으로 주고, 자동화의 위험(연쇄·조용한 실패)을 경계하라.

13.8 더 읽기

  • 자동화의 안전: 7장 — 권한과 안전.
  • 보장 vs 경향: 6장 — 시스템 프롬프트 · 8장 메모리.
  • 다음: 14장 — 스킬과 슬래시 커맨드.

14장 — 스킬과 슬래시 커맨드 (Skills & Commands)

한 줄 요약: 스킬은 재사용 가능한 능력·절차·지식을 필요할 때만 로드되는 단위로

패키징한 것이다. 시스템 프롬프트를 비대하게 만들지 않으면서 에이전트에 전문성을 더한다.

14.1 왜 중요한가

6장에서 "시스템 프롬프트는 짧게, 상황별 내용은 분리하라"고 했다. 그 분리처 가 스킬이다. 모든 절차를 시스템 프롬프트에 넣으면 매 턴 토큰·주의력을 쓰고(5장) 서로 충돌한다(6장). 스킬은 "PDF 다루는 법", "이 프로젝트 배포하는 법" 같은 전문 절차를 호출될 때만 펼친다.

핵심 명제: 스킬은 능력의 지연 로딩 이다 — 8장 메모리, 4장 지연 툴과 같은 원리.

이름·설명은 항상 보이고(싸다), 전체 절차는 발동할 때만 로드된다(비싸다).

14.2 스킬의 구조

스킬은 보통 셋으로 이뤄진다.

flowchart LR
    subgraph Skill["하나의 스킬"]
        Name["이름 + 트리거 설명<br/>'언제 이걸 쓰는가' (항상 노출)"]
        Body["본문 절차/지식<br/>(발동 시 로드)"]
        Assets["딸린 자원<br/>스크립트·템플릿·참고 (선택)"]
    end
    Name -.->|"평소엔 이것만"| Ctx["컨텍스트"]
    Body -.->|"발동 시 펼침"| Ctx
  • 트리거 설명 이 가장 중요하다(4장 툴 설명과 같다). 모델은 이걸 읽고 "지금 이 스킬을

    • 써야 하나"를 판단한다. "언제 쓰는지"를 분명히 — 좋은 트리거는 사용 시점을 정확히 짚는다.

  • 본문: 실제 단계별 절차·지식. 발동 시점에만 컨텍스트로 들어온다.
  • 자원: 스킬이 쓰는 스크립트·템플릿 등.

14.3 발동 방식 — 모델 호출 vs 사용자 호출

스킬은 두 경로로 발동된다.

flowchart TB
    A["사용자가 /명령 입력<br>(명시적)"] --> Run["스킬 발동"]
    B["모델이 트리거 설명을 보고<br>'지금 필요하다' 판단 (자동)"] --> Run
    Run --> Load["본문 절차 로드 → 실행"]
  • 사용자 호출(슬래시 커맨드): /code-review, /deep-research처럼 사용자가 직접 부른다.
  • 모델 호출: 작업이 스킬 설명과 맞으면 모델이 스스로 발동한다.

📦 이 환경의 사례: 이 하네스에는 deep-research(다중 소스 조사), code-review(diff

리뷰), verify(변경 동작 확인), update-config(설정 변경) 등 다수 스킬이 있다. 각

스킬은 "BEFORE invoking, check…", "When the user wants…" 같은 트리거 조건 을 설명에

담아 발동 시점을 명확히 한다. 사용자가 /이름을 치거나, 작업이 맞으면 모델이 호출한다.

14.4 스킬 vs 서브에이전트 vs 훅 — 경계

비슷해 보이는 확장 메커니즘들의 차이를 분명히:

메커니즘본질언제
스킬 (14장)지식·절차 의 지연 로딩"이 일을 어떻게 하는지"를 알려줘야 할 때
서브에이전트 (9장)격리된 실행 단위컨텍스트를 분리하거나 병렬화할 때
훅 (13장)결정론적 자동 동작"항상/매번" 보장이 필요할 때
툴 (4장)단일 외부 기능모델이 세계에 작용하는 원자적 통로

스킬은 종종 다른 것들을 조율 한다 — 예컨대 code-review 스킬의 본문이 서브에이전트 팬아웃(9장)과 검증 패턴(11장)을 지시한다. 스킬은 "절차서", 나머지는 "도구"인 셈이다.

14.5 좋은 스킬 설계

  • 트리거를 정밀하게 — 너무 넓으면 엉뚱한 때 발동, 너무 좁으면 안 쓰인다. "X일 때,

    • 단 Y는 제외" 식으로.

  • 본문은 자족적으로 — 발동 시 그것만 읽고도 수행 가능하게.
  • 스코프를 명확히 — 특정 디렉토리·프로젝트 한정이면 명시(가장 구체적인 것이 우선).
  • 다른 메커니즘을 가리켜라 — 스킬 안에서 "이 단계는 서브에이전트로", "이건 훅으로"를

    • 지시해 합성한다.

14.6 흔한 실수 / 안티패턴

  • ❌ 모든 절차를 시스템 프롬프트에 — 비대·충돌. 상황별 절차는 스킬로.
  • ❌ 모호한 트리거 — 모델이 언제 쓸지 모른다. 4장 툴 설명 원칙을 그대로 적용.
  • ❌ 스킬에 결정론적 보장을 기대 — 모델이 발동을 빠뜨릴 수 있다. 반드시면 훅(13장).
  • ❌ 비자족적 본문 — 발동했는데 외부 맥락이 필요하면 실패. 자족적으로 써라.

14.7 핵심 정리

  • 스킬 = 능력·절차의 지연 로딩 — 시스템 프롬프트를 가볍게 유지하며 전문성 추가.
  • 트리거 설명이 핵심: "언제 쓰는지"를 정밀하게.
  • 사용자(슬래시)·모델(자동) 양쪽으로 발동.
  • 스킬은 절차서로서 서브에이전트·훅·툴을 조율 한다.

14.8 더 읽기

  • 분리의 동기: 6장 시스템 프롬프트 · 5장 컨텍스트.
  • 스킬이 조율하는 것: 9장 서브에이전트 · 11장 검증.
  • 보장이 필요하면: 13장 — 훅.
  • 다음: 15장 — MCP와 외부 도구.

15장 — MCP와 외부 도구 (MCP & External Tools)

한 줄 요약: 하네스가 가진 내장 툴만으로는 부족하다. MCP(Model Context Protocol)는

외부 도구·데이터 소스를 표준화된 방식 으로 에이전트에 연결해, 생태계 전체를 툴로 만든다.

15.1 왜 중요한가

4장에서 툴을 설계했지만, 모든 툴을 직접 만들 수는 없다 — Slack, GitHub, 데이터베이스, 사내 시스템…. 각각을 일회성으로 붙이면 N×M 통합 지옥이 된다. MCP는 이걸 표준 프로토콜 하나로 푼다: 도구 제공자는 MCP 서버를 한 번 구현하고, 어떤 MCP 호환 하네스든 그걸 붙인다.

핵심 명제: MCP는 툴의 USB-C다. 하네스(호스트)와 도구(서버) 사이의 표준 커넥터로,

통합을 N×M에서 N+M으로 줄인다.

15.2 구조 — 호스트, 클라이언트, 서버

flowchart LR
    subgraph Host["하네스 (호스트)"]
        Agent["에이전트 루프"]
        Client["MCP 클라이언트"]
    end
    Client <-->|"MCP 프로토콜"| S1["MCP 서버: GitHub"]
    Client <-->|"MCP 프로토콜"| S2["MCP 서버: DB"]
    Client <-->|"MCP 프로토콜"| S3["MCP 서버: Slack"]
    S1 --> Ext1["GitHub API"]
    S2 --> Ext2["데이터베이스"]
    S3 --> Ext3["Slack API"]
  • 호스트: 에이전트를 돌리는 하네스. MCP 클라이언트를 품는다.
  • 서버: 특정 시스템(GitHub 등)을 MCP로 노출. 툴·리소스·프롬프트를 제공한다.
  • 프로토콜: 양쪽이 합의한 표준 — 서버가 무슨 툴을 갖고 어떻게 부르는지 기술.

MCP 서버가 노출하는 툴은, 에이전트 입장에선 내장 툴과 똑같이 보인다(4장의 name· description·schema). 그래서 4장의 모든 설계 원칙이 그대로 적용된다.

15.3 컨텍스트 비용 — 외부 툴도 토큰을 쓴다

MCP 서버를 붙이면 그 툴들의 스키마가 컨텍스트에 들어온다(5장). 서버가 많고 툴이 많으면 윈도우를 잡아먹는다. 해법은 4장에서 본 지연 로딩 이다.

flowchart TB
    Many["연결된 MCP 서버들<br>수십~수백 개 툴"] --> Defer["대부분 deferred<br>이름만 노출"]
    Defer --> Search["필요할 때 검색해<br>해당 툴 스키마 로드"]
    Search --> Call["호출"]

📦 이 환경의 사례: 이 하네스는 "워크플로우 에이전트가 세션에 연결된 모든 MCP 툴을

ToolSearch로 접근할 수 있고, 스키마는 에이전트별로 필요 시 로드된다"고 동작한다. 즉

MCP 툴도 4장의 지연 툴 메커니즘으로 다뤄 컨텍스트를 보호한다. 단 주의 — "대화형 인증을

쓰는 MCP 서버는 헤드리스/크론 실행에선 없을 수 있다"는 한계도 명시한다.

15.4 외부 연결의 안전

MCP는 에이전트에 실제 외부 시스템 에 닿는 힘을 준다. 7장의 안전 원칙이 더 중요해진다.

  • 외부 전송은 게시 — MCP를 통해 메시지를 보내거나 데이터를 외부에 쓰면 회수 불가일

    • 수 있다. 사전 확인 대상(7.5).

  • 권한 범위 — MCP 서버에 준 자격증명의 범위를 최소화. 읽기만 필요하면 쓰기 권한을 주지 마라.
  • 신뢰 경계 — 외부 서버가 돌려주는 데이터는 신뢰할 수 없는 입력일 수 있다(프롬프트

    • 인젝션). 그대로 명령으로 삼지 마라.

  • 인증 환경 차이 — 대화형 로그인이 필요한 서버는 자동/백그라운드 실행에서 빠질 수 있다.

15.5 언제 MCP인가, 언제 내장 툴인가

상황선택
핵심·고빈도 기능(파일·셸)내장 툴 — 항상 노출, 빠름
외부 시스템(SaaS·DB·API)MCP 서버
이미 MCP 서버가 있는 생태계MCP 재사용(바퀴 재발명 금지)
일회성·프로젝트 한정 스크립트단순 툴/스킬로 충분할 수도

15.6 흔한 실수 / 안티패턴

  • ❌ 모든 서버를 항상 풀 로드 — 컨텍스트 폭발. 지연 로딩하라.
  • ❌ 과도한 자격증명 — 최소 권한 원칙. 필요한 범위만.
  • ❌ 외부 데이터를 맹신 — 인젝션 위험. 외부 출력을 명령으로 직접 쓰지 마라.
  • ❌ 환경 차이 무시 — 대화형 인증 서버가 크론에서 없을 수 있음을 가정하라.
  • ❌ MCP 툴에 4장 원칙 미적용 — 외부 툴도 설명·스키마·에러 설계가 똑같이 중요.

15.7 핵심 정리

  • MCP = 툴의 표준 커넥터(USB-C). 통합을 N×M → N+M으로.
  • 호스트(하네스)·서버(도구 제공자)·프로토콜 구조. 외부 툴도 내장 툴처럼 보인다.
  • 컨텍스트 비용은 지연 로딩으로, 외부 연결의 위험은 7장 안전 원칙으로 관리.
  • 외부 출력은 신뢰 못 할 입력 — 인젝션을 경계하라.

15.8 더 읽기

  • 툴 설계 기본: 4장 — 툴 설계 (지연 툴 포함).
  • 연결의 안전: 7장 — 권한과 안전.
  • 컨텍스트 비용: 5장 — 컨텍스트 엔지니어링.
  • 다음: 16장 — 스케줄링과 백그라운드.

16장 — 스케줄링과 백그라운드 (Scheduling & Background)

한 줄 요약: 에이전트가 늘 대화에 묶여 있을 필요는 없다. 백그라운드 작업, 폴링,

크론 스케줄, 깨우기(wakeup)는 시간 축을 따라 에이전트를 동작시키는 도구다 — 핵심은

언제 깨어날지 를 비용에 맞춰 정하는 것.

16.1 왜 중요한가

지금까지의 에이전트는 동기적이었다 — 사용자가 말하면 답한다. 하지만 실제 작업엔 기다림 이 있다: CI가 끝나길, 배포가 완료되길, 매일 아침 리포트를 돌리길. 이걸 대화 앞에서 멍하니 기다리면 비싸고 느리다. 스케줄링·백그라운드는 에이전트를 시간 축 으로 확장한다.

핵심 명제: 비동기 작업의 설계 변수는 "무엇을 하느냐"가 아니라 "언제 깨어나느냐"다.

너무 자주 깨면 비용 낭비, 너무 드물면 반응이 느리다. 깨우기 간격을 *기다리는 대상의

변화 속도* 에 맞춰라.

16.2 세 가지 시간 패턴

flowchart TB
    A["백그라운드 실행<br>오래 도는 작업을 떼어 실행<br>(끝나면 재호출)"]
    B["폴링/깨우기<br>주기적으로 깨어 외부 상태 확인"]
    C["크론 스케줄<br>정해진 시각에 반복 실행<br>(매일 9시 등)"]
  • 백그라운드 실행: 빌드·테스트처럼 오래 걸리는 걸 떼어 돌리고, 끝나면 하네스가

    • 에이전트를 다시 깨운다. 폴링 불필요 — 하네스가 추적한다.

  • 폴링/깨우기: 하네스가 추적 못 하는 외부 상태(원격 CI, 외부 큐)를 주기적으로 확인.
  • 크론 스케줄: 시각 기반 반복. "매일 아침 PR 점검" 같은 루틴.

16.3 깨우기 간격 — 캐시가 정하는 분기점

5장에서 본 프롬프트 캐시 TTL 이 여기서 결정적이다. 깨우기 간격을 캐시 수명에 맞춰라.

flowchart TB
    Q{"무엇을 기다리나?"} --> Short["빠르게 변하는 외부 상태<br>(CI·배포·원격 큐)"]
    Q --> Long["천천히 변하거나<br>그냥 유휴"]
    Short -->|"캐시 유지 (~270초 이내)"| S["짧은 폴링<br>캐시 따뜻하게"]
    Long -->|"캐시 미스 감수 (1200초+)"| L["긴 대기<br>한 번의 미스로 오래 산다"]

원리:

  • 5분(TTL) 이내: 캐시가 살아 있다. 빠르게 변하는 외부 상태를 능동 폴링 할 때.
  • 5분~1시간: 캐시 미스를 감수. 더 일찍 볼 이유가 없는 대기, 또는 긴 하트비트.
  • 함정 — 정확히 300초: 최악. 캐시 미스는 나는데 그 비용을 오래 분산도 못 한다.

    • "5분만 기다리자"가 당기면, 차라리 270초(캐시 유지) 또는 1200초+(미스 한 번으로 길게)로.

📦 이 환경의 사례: 이 하네스의 ScheduleWakeup 가이드가 정확히 이 논리다 —

"Anthropic 캐시는 5분 TTL; 300초를 넘기면 다음 깨어남이 캐시 미스. 300초를 고르지 마라 —

270초로 캐시를 유지하거나 1200초+로 미스를 길게 분산하라. 유휴 틱의 기본값은 1200~1800초."

또 "하네스가 추적하는 백그라운드 작업을 폴링하려고 짧은 깨우기를 걸지 마라 — 끝나면

자동 재호출되니 낭비"라고 경고한다(16.2의 백그라운드 vs 폴링 구분).

16.4 자율 루프와 자기 페이싱

에이전트가 스스로 반복하며 페이스를 정할 때(예: "배포 끝날 때까지 점검 반복"), 매 사이클 끝에 다음 깨우기를 스케줄 한다. 안전장치:

  • 무엇을 기다리는지 명시 — "watching CI run"이 "waiting"보다 낫다. 페이스를 그에 맞춘다.
  • 긴 폴백 하트비트 — 주 신호(작업 완료 재호출)가 안 오면 멈추지 않도록, 길게(1200초+)

    • 폴백 깨우기를 둔다. 작업이 멈추거나 영영 안 끝나도 루프가 생존한다.

  • 종료 조건 — 무한 루프 방지(2·10장의 안전장치와 같은 정신).

16.5 비동기 작업의 보고

백그라운드/스케줄 작업의 결과는 사용자가 지금 화면을 안 볼 수도 있다. 그래서:

  • 명확한 완료 신호 — 끝났을 때 무엇을 했는지 요약·알림.
  • 실패도 보고 — 조용히 죽으면 사용자는 됐다고 오해한다(13장 훅의 조용한 실패와 같은 위험).
  • 결과의 영속화 — 다음 세션이 이어받을 수 있게 메모리·파일로(8장).

16.6 흔한 실수 / 안티패턴

  • ❌ 추적되는 작업을 폴링 — 백그라운드 작업은 끝나면 자동 재호출. 폴링 깨우기는 낭비.
  • ❌ 300초 깨우기 — 캐시 최악 지점. 270초 또는 1200초+로.
  • ❌ 라운드 숫자로 사고 — "5분"이 아니라 "캐시 윈도우"로 생각하라.
  • ❌ 폴백 하트비트 없음 — 주 신호가 안 오면 루프가 영영 멈춘다.
  • ❌ 조용한 백그라운드 실패 — 완료·실패를 가시화하라.

16.7 핵심 정리

  • 시간 축 확장: 백그라운드 실행 · 폴링/깨우기 · 크론 스케줄.
  • 설계 변수는 깨우기 간격 — 프롬프트 캐시 TTL(5분)이 분기점.
  • 빠른 외부 상태는 270초 이내, 느린/유휴는 1200초+. 300초는 피하라.
  • 추적되는 작업은 폴링 말 것. 비동기 결과는 완료·실패를 명확히 보고.

16.8 더 읽기

  • 캐시 원리: 5장 — 컨텍스트 엔지니어링 · 21장 비용.
  • 조용한 실패의 위험: 13장 — 훅.
  • 결과 영속화: 8장 — 메모리.
  • 다음 (Part 5): 17장 — 엔드투엔드 설계 예시.

엔드투엔드 설계, 평가, 비용, 프로덕션

17장 — 엔드투엔드 설계 예시 (End-to-End Design)

한 줄 요약: 앞의 16개 장을 하나의 하네스 로 조립한다. "코드베이스 PR 리뷰

에이전트"를 요구 정의부터 루프·툴·오케스트레이션·검증·안전·운영까지 처음부터 끝까지

설계하며, 매 결정마다 "왜 이 선택인가"를 짚는다.

17.1 왜 이 장인가

지금까지는 부품을 하나씩 봤다. 하지만 실전의 어려움은 부품을 고르는 것 이 아니라 조립하는 것 에 있다 — 어떤 걸 쓰고, 어떤 걸 생략하고, 어떻게 엮는가. 이 장은 가상의 실제 과제 하나를 끝까지 설계해, 1~16장이 어떻게 한 시스템에서 맞물리는지 보여준다.

이 장은 레시피가 아니라 사고 과정 이다. 결론보다 "왜 그렇게 결정했는가"에 주목하라.

17.2 과제 정의

목표: 개발팀을 위한 PR 리뷰 에이전트. PR이 열리면 변경된 코드를 리뷰해 (1)

정확성 버그와 (2) 단순화/재사용 기회를 찾아, 검증된 발견만 인라인 코멘트로 단다.

먼저 요구를 제약으로 번역한다(설계의 첫걸음).

요구제약/함의
변경된 코드를 리뷰diff를 읽는 툴 필요, 변경 파일만(전체 아님) → 컨텍스트 절약(5장)
버그 + 단순화 두 축차원별 리뷰(dimension) → 팬아웃(9·10장)
검증된 발견만적대적 검증 필수 → 거짓 양성 차단(11장)
인라인 코멘트(외부 게시)되돌리기 어려움/외부 → 안전 게이트(7장)
PR이 열리면 자동이벤트 트리거 → 훅/스케줄(13·16장)

이 표가 이미 설계의 뼈대다. 각 행이 어떤 장의 도구를 부른다.

17.3 단계별 설계

단계 1 — 루프와 시스템 프롬프트 (1·2·3·6장)

핵심은 단일 ReAct 루프(3장)지만, 이 에이전트의 역할 을 시스템 프롬프트로 좁힌다(6장).

SYSTEM_PROMPT = """
너는 PR 리뷰 에이전트다.
- 변경된 diff만 리뷰한다 (전체 코드베이스가 아니라).
- 두 가지만 찾는다: ① 정확성 버그 ② 단순화/재사용 기회.
- 추측하지 말고, 관련 파일을 읽어 근거를 확보한 뒤 발견을 보고하라.
- 발견은 반드시 검증을 거친다 — 확신만으로 코멘트하지 마라. (이유: 거짓 양성은
  개발자 신뢰를 깎는다)
- 우선순위: 정확성 > 단순화 > 스타일. 외부 게시 전 항상 확인.
"""

규칙에 이유 와 우선순위 를 박은 것(6.3)을 보라.

단계 2 — 툴 설계 (4장)

필요한 툴을 좁고 명확하게(4장):

get_pr_diff(pr)        → 변경된 파일·헝크 목록. (전체 파일 아님 → 컨텍스트 절약)
read_file(path, range) → 맥락이 필요할 때 특정 구간만. (거대 파일 통째 금지)
post_comment(file, line, body) → ⚠️ 외부 게시. 되돌리기 어려움 → 안전 게이트 대상.

post_comment의 설명에 "외부 게시, 되돌리기 어려움"을 명시(4장)해 모델이 위험을 알게 한다.

단계 3 — 오케스트레이션 (9·10·11·12장)

두 차원(버그·단순화)을 팬아웃 하고, 각 리뷰가 끝나는 즉시 검증으로 흘려보낸다 — 10.5의 정전 pipeline 패턴 그대로.

flowchart TB
    Diff["get_pr_diff"] --> P["pipeline (배리어 없음)"]
    P --> D1["차원1: 버그 리뷰"]
    P --> D2["차원2: 단순화 리뷰"]
    D1 --> V1["발견들 → 적대적 검증<br>(반증 시도, 과반이면 폐기)"]
    D2 --> V2["발견들 → 적대적 검증"]
    V1 --> C["확정 발견 수집"]
    V2 --> C
    C --> Gate["안전 게이트"]
results = pipeline(
  ["bugs", "simplify"],
  dim    => agent(리뷰_프롬프트(dim, diff), {schema: FINDINGS}),       // 1단계
  review => parallel(review.findings.map(f => () =>                   // 2단계: 즉시 검증
     agent(`이 발견을 반증하라: ${f}. 불확실하면 기각.`, {schema: VERDICT})
       .then(v => ({...f, verdict: v}))))
)
confirmed = results.flat().filter(f => f.verdict.isReal)

왜 pipeline인가? 버그 리뷰가 단순화 리뷰를 기다릴 이유가 없다(10.3). 버그 발견들은 단순화 차원이 아직 리뷰 중일 때 이미 검증된다. 배리어를 쓰면 벽시계를 낭비한다(10.4).

왜 적대적 검증인가? "검증된 발견만"이 요구였다(17.2). 생성한 에이전트에게 "맞아?"를 물으면 자기 합리화한다 — 독립 검증자가 반증 을 시도하게 한다(11.2). 불확실하면 기각으로 기울여 거짓 양성을 줄인다.

규모 조절(12.4): 작은 PR이면 검증 단일 투표로 충분. "철저히"가 요구면 3표 적대적 패널로.

단계 4 — 안전 게이트 (7장)

확정 발견을 게시 하기 전, 외부 작업이므로 확인을 건다(7.5).

flowchart TB
    C["확정 발견 N개"] --> Q{"외부 게시<br>(되돌리기 어려움)"}
    Q -->|"신뢰/자동화 모드"| Auto["바로 게시 (allowlist)"]
    Q -->|"기본 모드"| Ask["요약 보여주고 사전 확인"]
    Ask -->|"승인"| Post["post_comment 실행"]

신뢰 수준에 따라 모드를 달리한다(7.6) — 처음엔 확인하며(기본), 검증되면 자동(allowlist)으로.

단계 5 — 트리거와 운영 (13·16장)

"PR이 열리면 자동"은 경향이 아니라 보장 이다 → 훅/스케줄(13.4의 판별).

  • 훅: PR 오픈 이벤트에 리뷰 워크플로우를 건다(결정론).
  • 또는 크론(16장): "매시간 새 PR 점검". 결과는 게시 + 실패 시 알림(16.5).

단계 6 — 컨텍스트·메모리·비용 (5·8·21장)

  • 컨텍스트(5장): diff만, 파일은 구간만. 차원 리뷰는 서브에이전트로 격리해 메인을

    • 깨끗이(9장).

  • 메모리(8장): "이 레포는 X 컨벤션을 쓴다" 같은 프로젝트 사실을 저장해, 매번 다시

    • 설명 안 하게. 단 코드가 이미 말하는 건 저장 안 함(8.5).

  • 비용(21장): 차원당 한 번 리뷰 + 발견당 검증. 발견이 0이면 검증 단계를 건너뛴다

    • (early-exit 배리어가 정당한 경우, 11.6).

17.4 전체 그림

flowchart TB
    Trigger["PR 오픈 (훅/크론)"] --> Diff["get_pr_diff — 변경만"]
    Diff --> Pipe["pipeline: 차원별 리뷰"]
    Pipe --> Rev["버그 리뷰 / 단순화 리뷰<br>(서브에이전트 격리)"]
    Rev --> Ver["적대적 검증 — 거짓 양성 폐기"]
    Ver --> Conf["확정 발견"]
    Conf --> Gate["안전 게이트 — 외부 게시 확인"]
    Gate --> Post["인라인 코멘트 게시"]
    Post --> Report["완료/실패 보고 + 메모리 갱신"]

이 한 장의 그림 안에 1~16장이 다 들어 있다: 루프·툴(1~4) · 컨텍스트·안전(5·7) · 시스템 프롬프트(6) · 메모리(8) · 팬아웃·파이프라인· 검증(9~12) · 훅/스케줄(13·16) · 비용(21).

17.5 설계에서 배우는 원칙

  1. 요구를 제약으로 번역하라 — 17.2의 표가 곧 부품 선택표가 된다.
  1. 기본은 단순하게, 복잡성은 정당화하라 — pipeline·검증은 요구가 부른 것이지

    2. 멋있어서가 아니다. 작은 PR엔 더 줄여도 된다.

  1. 위험에 비례한 마찰 — 읽기는 자유, 외부 게시는 게이트(7장).
  1. 생성과 검증을 분리 — "검증된 발견만"은 독립 검증자로만 달성(11장).
  1. 보장과 경향을 구분 — "PR 열리면 자동"은 훅, "이런 톤으로"는 프롬프트(13장).

17.6 흔한 실수 / 안티패턴

  • ❌ 모든 파일을 읽기 — diff만으로 충분한데 전체를 읽어 컨텍스트 낭비.
  • ❌ 검증 생략하고 바로 게시 — 거짓 양성이 신뢰를 깎는다. 요구를 어긴다.
  • ❌ 차원 리뷰에 습관적 배리어 — pipeline이면 될 걸 parallel 연쇄로 묶어 벽시계 낭비.
  • ❌ 외부 게시를 무확인 자동으로 (처음부터) — 신뢰가 쌓이기 전엔 게이트를 둬라.
  • ❌ "자동 트리거"를 프롬프트로 — 빠뜨린다. 훅으로 보장하라.

17.7 핵심 정리

  • 설계 = 요구 → 제약 → 부품 선택 → 조립. 표 하나로 시작하라.
  • 기본은 단순, 복잡성은 요구가 정당화할 때만 추가.
  • 이 에이전트는 1~16장을 한 시스템으로 엮는다 — 부품이 아니라 맞물림 이 실력이다.

17.8 더 읽기

  • 설계 전 합의: 18장 — 계획과 플랜 모드.
  • 잘 되는지 측정: 19장 — 평가와 관측.
  • 무엇이 망가지나: 20장 — 실패 모드.
  • 빠른 참조: 부록 A 패턴 카탈로그 · 부록 B 체크리스트.

18장 — 계획과 플랜 모드 (Planning & Plan Mode)

한 줄 요약: 행동 전에 합의하는 것 — 그것이 계획이다. 플랜 모드는 에이전트가

변경 없이 이해·설계하고, 사람의 승인을 받은 뒤에야 실행하게 하는 안전·정렬 장치다.

18.1 왜 중요한가

7장은 위험한 개별 작업 을 막았다. 하지만 더 큰 위험은 방향 자체가 틀린 채 전속력으로 달리는 것이다 — 사용자가 원한 게 A인데 에이전트가 B를 완벽하게 구현하는 경우. 계획은 이 정렬 실패를 비싼 실행 전 에 잡는다.

핵심 명제: 되돌리기 어려운 것은 개별 명령만이 아니다 — 잘못된 방향에 쏟은 노력

도 그렇다. 계획은 가장 싼 단계(말)에서 정렬을 맞춰, 가장 비싼 단계(실행)의 낭비를 막는다.

18.2 계획 워크플로우의 구조

좋은 계획은 단계가 있다. 전형적 흐름:

flowchart TB
    P1["① 이해<br>읽기 전용 탐색으로 요구·코드 파악"] --> P2["② 설계<br>접근법 수립, 대안 검토"]
    P2 --> P3["③ 검토<br>핵심 파일 확인, 사용자에 질문"]
    P3 --> P4["④ 계획 확정<br>맥락·접근·검증법을 문서화"]
    P4 --> P5["⑤ 승인 요청<br>사용자가 보고 승인"]
    P5 -->|"승인"| Exec["실행 (이제 변경 허용)"]
    P5 -->|"수정 요청"| P2
  • ① 이해: 변경 없이 코드·요구를 파악. 탐색은 서브에이전트로 병렬화(9장).
  • ② 설계: 접근을 정하고 트레이드오프를 검토. 필요하면 설계 심판 패널(11.4).
  • ③ 검토: 핵심 파일을 직접 보고, 불명확한 요구는 사용자에게 질문. 가정 금지.
  • ④ 확정: 맥락(왜 하는가)·추천 접근·검증법을 담은 계획서 작성.
  • ⑤ 승인: 사용자가 계획을 보고 승인해야 실행으로 넘어간다.

📦 이 환경의 사례: 이 가이드북 자체가 이 워크플로우로 만들어졌다. 처음에 **plan

mode** 로 들어가 (①) 빈 디렉토리를 확인하고, (②③) AskUserQuestion으로 주제·형식·

독자·언어를 좁히고, (④) 계획 파일에 구조를 쓰고, (⑤) ExitPlanMode로 승인을 받은 뒤

비로소 챕터를 썼다. 승인 전엔 계획 파일 외 어떤 것도 변경하지 않았다 — 7장의 plan

mode 보호와 18장의 정렬이 겹치는 지점이다.

18.3 좋은 계획서의 요소

계획서는 사용자가 빠르게 훑되 실행은 충분히 할 수 있어야 한다.

  • 맥락(Context): 왜 이 변경을 하는가 — 문제·동기·목표. 가장 자주 빠지는 부분.
  • 추천 접근만: 모든 대안이 아니라 고른 하나 와 그 이유. (검토 과정에서 본 대안은

    • 요약만.)

  • 건드릴 핵심 파일: 반복 패턴이면 한 번 설명 + 대표 경로 몇 개(전부 나열 금지).
  • 재사용할 기존 자산: 새로 짜기 전에 이미 있는 함수·유틸을 가리켜라.
  • 검증법: 끝나고 어떻게 동작을 확인 할지(실행·테스트).

18.4 언제 계획하고, 언제 바로 하나

계획은 공짜가 아니다 — 왕복 시간이 든다. 작업에 비례시켜라.

flowchart TB
    Q{"작업이..."} --> A["사소·국소·되돌리기 쉬움<br>(오타, 한 줄 수정)"]
    Q --> B["범위 불확실·여러 영역·<br>방향이 갈릴 수 있음"]
    A -->|"바로 실행"| Now["계획 생략"]
    B -->|"먼저 계획·합의"| Plan["플랜 모드"]
  • 바로: 명백한 단일 작업, 되돌리기 쉬운 것.
  • 계획: 방향이 갈릴 수 있거나, 비싸거나, 여러 부분을 건드리거나, 요구가 모호할 때.

의심스러우면, 질문 하나 가 잘못된 구현 전체 보다 싸다(18.1).

18.5 질문하기 — 정렬의 핵심 도구

계획 단계의 가장 강력한 행동은 좋은 질문 이다. 단, 남발하면 피곤하다.

  • 질문할 것: 코드·요구·상식적 기본값으로 풀 수 없는, 사용자만 답할 수 있는 결정.
  • 질문하지 말 것: 관행적 기본값이 있거나, 코드에서 직접 확인 가능한 것 → 직접 정하고

    • 언급 하며 진행.

  • 추천을 곁들여라 — 선택지를 줄 때 추천안을 먼저 제시하면 사용자 부담이 준다.

📦 이 환경의 사례: 이 책을 만들 때 "주제 범위·형식·독자 수준·언어"는 *사용자만 답할

수 있는* 결정이라 물었다. 반면 "디렉토리 레이아웃·챕터 7요소 골격"은 합리적 기본값으로

직접 정하고 계획서에 적었다 — 18.5의 분별 그대로다.

18.6 흔한 실수 / 안티패턴

  • ❌ 사소한 작업에 과한 계획 — 오타 수정에 계획서는 마찰만.
  • ❌ 큰 작업을 계획 없이 돌진 — 방향이 틀리면 전부 낭비.
  • ❌ 맥락(왜) 누락 — "무엇을"만 있는 계획은 검토자가 판단 못 한다.
  • ❌ 가정으로 질문 회피 — 큰 가정은 큰 재작업을 부른다. 모호하면 물어라.
  • ❌ 질문 남발 — 기본값으로 될 걸 다 물으면 피곤하다. 분별하라.

18.7 핵심 정리

  • 계획 = 비싼 실행 전에 가장 싼 단계(말) 에서 방향을 정렬.
  • 워크플로우: 이해(읽기전용) → 설계 → 검토(질문) → 확정 → 승인 → 실행.
  • 계획서엔 맥락(왜) · 추천 접근 · 핵심 파일 · 재사용 자산 · 검증법.
  • 작업에 비례해 계획하라. 모호하면 질문 하나가 재작업 전체보다 싸다.

18.8 더 읽기

  • plan mode의 안전: 7장 — 권한과 안전.
  • 설계 실전: 17장 — 엔드투엔드 설계.
  • 설계 심판 패널: 11장 — 검증 루프.
  • 다음: 19장 — 평가와 관측.

19장 — 평가와 관측 (Eval & Observability)

한 줄 요약: 측정하지 않으면 개선할 수 없다. 평가(eval)는 하네스의 품질을 정량화하고,

관측(observability)은 실행 내부를 들여다보게 한다. 이 둘 없이는 "좋아진 것 같다"는

착각만 쌓인다.

19.1 왜 중요한가

하네스는 부품이 많고(1~16장) 동작이 확률적이다. 어떤 변경이 정말 나아지게 했는지, 어디서 망가지는지를 감 으로 판단하면 틀린다. 평가는 변경의 효과를 숫자로 보여주고, 관측은 실패의 원인을 추적 가능하게 한다.

핵심 명제: 에이전트 개발은 "프롬프트를 만지며 데모가 잘 되길 비는 것"이 아니다.

반복 가능한 측정 위에서 변경하라. 평가 없는 개선은 무작위 보행이다.

19.2 평가의 층위

flowchart TB
    Unit["① 컴포넌트 평가<br>툴 하나, 프롬프트 하나가 맞나"]
    Task["② 작업 평가<br>대표 작업 N개를 끝까지 수행하나"]
    E2E["③ 엔드투엔드 평가<br>실제 사용자 시나리오 성공률"]
    Unit --> Task --> E2E
  • 컴포넌트: 개별 툴·프롬프트·검증기의 정확도. 빠르고 싸다.
  • 작업: 대표 작업 세트(골든 셋)를 에이전트가 완수하는가. 회귀 탐지의 핵심.
  • 엔드투엔드: 실제 시나리오 전체의 성공률·품질. 가장 현실적이나 비싸고 잡음 많다.

19.3 무엇을 측정하나

작업 성격에 맞는 지표를 골라라.

차원지표 예
정확성작업 성공률, 거짓 양성/음성률(11장 검증 효과)
효율토큰/작업, 단계 수, 벽시계 시간(21장)
안전위험 작업 차단율, 미승인 외부 작업 0건(7장)
견고성에러 복구율(4장), 재현성
비용작업당 비용, 캐시 적중률(5·21장)

단일 지표를 좇지 마라. 정확성만 올리려다 비용이 폭증하면 의미 없다. 트레이드오프 를

함께 보라.

19.4 LLM을 심판으로 (LLM-as-judge)

정답이 딱 떨어지지 않는 출력(리뷰 품질, 요약의 충실성)은 다른 LLM 으로 채점한다. 11장의 심판 패널과 같은 원리 — 단 여기선 평가용 으로.

  • 기준을 명시 — "정확성·완전성·간결성을 1~5로" 처럼 루브릭을 줘라.
  • 편향 주의 — 길이·순서·자기 출력 선호 편향. 위치를 섞고 익명화하라.
  • 앵커 예시 — "5점은 이런 것, 1점은 이런 것" 예시로 척도를 고정.
  • 검증의 검증 — 심판이 맞는지 사람 표본으로 가끔 대조.

19.5 관측 — 실행을 들여다보기

평가가 "얼마나 좋은가"라면, 관측은 "왜 그런가"다. 실패를 디버깅하려면 내부가 보여야 한다.

flowchart TB
    Trace["트레이스<br>매 모델 호출·툴 호출·결과를 기록"] --> Inspect["사후 검사<br>어디서 틀렸나 되짚기"]
    Trace --> Metric["집계 지표<br>단계 수·토큰·실패 지점 분포"]
    Trace --> Replay["재현<br>같은 입력으로 다시 돌려보기"]

기록할 것:

  • 매 단계: 어떤 컨텍스트로 모델을 불렀고, 무슨 툴을 어떤 인자로 호출했고, 무슨 결과가

    • 돌아왔는지.

  • 결정 지점: 왜 그 툴을 골랐나(추론), 검증이 무엇을 기각했나(11장).
  • 자원: 단계별 토큰·시간·비용.
  • 은밀한 캡: 무엇을 잘랐는지(12.3) — 안 남기면 '다 했다'로 오인.

📦 이 환경의 사례: 이 하네스의 워크플로우는 /workflows로 라이브 진행을 보여주고,

각 에이전트 실행을 agent-<id>.jsonl 트레이스로 남긴다. 그래서 중단된 워크플로우를

재개(resume)하거나, 트레이스를 읽어 무엇이 어디서 일어났는지 사후 추적할 수 있다

(10.7의 결정적 조율이 관측·재현을 가능케 한 것).

19.6 평가를 개발 루프에 넣기

평가는 한 번 하고 마는 게 아니라 개발 루프의 일부 다.

flowchart LR
    Change["하네스 변경<br>(프롬프트·툴·구조)"] --> Eval["골든 셋 평가"]
    Eval --> Better{"지표가<br>나아졌나?"}
    Better -->|"예"| Keep["반영"]
    Better -->|"아니오/회귀"| Revert["되돌리고 재시도"]
    Keep --> Change

작은 골든 셋이라도 매 변경마다 돌리면, 회귀를 즉시 잡고 "좋아진 것 같다"는 착각을 막는다.

19.7 흔한 실수 / 안티패턴

  • ❌ 데모 한 번으로 판단 — 확률적이라 한 번의 성공은 신호가 약하다. 세트로 측정.
  • ❌ 단일 지표 최적화 — 정확성만 좇다 비용·안전을 잃는다. 트레이드오프를 보라.
  • ❌ 관측 없는 디버깅 — 트레이스가 없으면 추측만. 매 단계를 기록하라.
  • ❌ 심판 편향 무시 — 길이·순서·자기선호 편향이 평가를 오염시킨다.
  • ❌ 은밀한 캡 미기록 — 잘린 걸 안 남기면 커버리지를 과신한다.

19.8 핵심 정리

  • 측정 없는 개선은 무작위 보행 — 평가를 개발 루프에 넣어라.
  • 층위: 컴포넌트 → 작업(골든 셋) → 엔드투엔드. 트레이드오프를 함께 보라.
  • 정답 없는 출력은 LLM-as-judge로, 단 편향을 관리하라.
  • 관측 = 매 단계 트레이스. "얼마나 좋은가"(eval)와 "왜 그런가"(observability)를 둘 다.

19.9 더 읽기

  • 검증과의 관계: 11장 — 검증 루프.
  • 비용 지표: 21장 — 비용·토큰·캐시.
  • 무엇이 망가지나: 20장 — 실패 모드.
  • 재현의 토대: 10장 — 워크플로우.

20장 — 실패 모드와 안티패턴 (Failure Modes & Anti-Patterns)

한 줄 요약: 하네스는 정형화된 방식으로 망가진다. 이 장은 앞 장들에 흩어진 실패

모드를 한곳에 모아 증상 → 원인 → 처방 으로 정리한 진단 사전이다.

20.1 왜 중요한가

실패를 이름 으로 알아보면 디버깅이 빨라진다. "에이전트가 이상해"가 아니라 "이건 컨텍스트 오염이군, 5장 처방을 쓰자"가 된다. 이 장은 앞에서 본 안티패턴들을 증상 기반 으로 다시 엮어, 문제를 만났을 때 역으로 찾아보게 한다.

핵심 명제: 대부분의 에이전트 실패는 새롭지 않다. 알려진 실패 모드의 카탈로그를

가지면, 절반은 진단 즉시 해결된다.

20.2 실패 지도

flowchart TB
    F["에이전트가 잘못 동작"] --> C1["컨텍스트 계열<br>오염·망각·폭발"]

21장 — 비용·토큰·캐시 (Cost, Tokens & Caching)

한 줄 요약: 토큰은 돈이다. 비용을 모르고 짠 하네스는 빠르게 비싸진다. 캐시·예산·

모델 계층화 — 이 세 레버로 품질을 지키면서 비용을 통제한다.

21.1 왜 중요한가

데모는 싸지만 반복되는 production 은 비싸다. 멀티에이전트(9~12장)는 토큰을 N배로 쓰고, 긴 컨텍스트(5장)는 호출마다 누적되며, 잘못 설계한 캐시는 비용을 몇 배로 부풀린다. 비용을 설계 변수 로 다루지 않으면, 잘 동작하는데 쓸 수 없는 하네스가 된다.

핵심 명제: 비용은 사후에 줄이는 게 아니라 설계에 넣는 것이다. 토큰 경제학을

모르면 정확성·속도·비용의 트레이드오프를 의식적으로 못 한다.

22장 — 프로덕션과 스케일 (Production & Scale)

한 줄 요약: 데모와 production은 다른 동물이다. 신뢰성, 한 컨텍스트를 넘는 작업,

사람과의 협업, 운영 — 에이전트를 실제로 굴리는 단계의 과제를 다룬다.

22.1 왜 중요한가

앞 장들은 "잘 작동하는 에이전트"를 만들었다. 이 장은 그것을 수천 번, 사람들과 함께, 오래 굴릴 때 생기는 문제를 다룬다. 한 번의 성공과 지속 가능한 운영 사이에는 큰 간극이 있고, 그 간극이 대개 프로젝트의 성패를 가른다.

핵심 명제: production은 "더 잘 만드는 것"이 아니라 "나쁜 날에도 견디게 만드는 것"이다.

실패·재시작·사람 개입·규모를 전제 로 설계하라.

22.2 신뢰성 — 확률적 시스템을 믿을 만하게

모델은 확률적이라 가끔 틀린다. production은 이걸 예외가 아니라 정상 으로 다룬다.

  • 재시도와 백오프: 일시적 실패(API 오류·타임아웃)는 재시도. 단 부작용 있는 작업의

    • 재시도는 멱등성(idempotency)을 확보하라 — 같은 이메일을 두 번 보내면 안 된다.

  • 우아한 저하(graceful degradation): 서브에이전트 하나가 죽어도 전체가 멈추지 않게

    • (null로 떨어뜨리고 filter, 10장). 부분 결과라도 유용하게.

  • 타임아웃·서킷 브레이커: 멈춘 작업이 전체를 잡지 않게 상한을 둬라(2·10장).
  • 검증을 신뢰의 토대로: 출력을 믿을 수 있는 건 검증했기 때문이다(11장). production일수록

    • 검증이 더 중요.

    Pipe --> Resume["중단 시 resume<br>(완료분은 건너뜀)"]
    Resume --> State
  • 외부 상태: 진행 상황을 파일·메모리에 영속화(8장). 컨텍스트가 압축돼도(5장),

    • 세션이 끊겨도 이어간다.

  • 재개 가능성(resume): 결정적 워크플로우는 중단점부터 재개(10.7). 거대 작업의 생명줄.
  • 스카우트 후 팬아웃: 모양을 먼저 발굴하고 그 위로 파이프라인(12.5).
  • 은밀한 캡 금지: 규모를 제한했으면(top-N) 무엇을 안 했는지 반드시 보고(12·19장).

22.4 사람과의 협업 (Human-in-the-loop)

production 에이전트는 사람과 함께 일한다. 협업 지점을 설계 하라.

  • 계획 합의: 큰 작업은 실행 전 계획 승인(18장).
  • 위험 게이트: 되돌리기 어려운 작업은 사람 확인(7장).
  • 정직한 보고: 실패는 실패로, 건너뛴 건 건너뛴 걸로, 검증 안 된 건 그렇다고. 성공을

    • 과장하지 마라 — 신뢰는 정직에서 온다.

  • 개입 가능성: 오래 도는 작업은 사람이 중간에 멈추고 방향을 틀 수 있게(16장 자율 루프의

    • 중단 가능성).

정직한 보고는 단순한 예의가 아니라 운영 안전장치 다. 에이전트가 "다 됐다"고 했는데

실제로 아니면, 그 거짓이 곧 production 사고가 된다(19·20장).

22.5 진화하는 하네스 — 운영은 끝이 없다

하네스는 한 번 짜고 끝이 아니다. 코드가 바뀌고, 모델이 바뀌고, 요구가 바뀐다.

  • 평가를 회귀 방지선으로: 변경마다 골든 셋(19장). 모델 업그레이드도 평가로 검증 —

    • 새 모델이 항상 더 낫지는 않다.

  • 메모리·문서를 최신으로: 부패한 메모리는 거짓이 된다(8장). 주기적으로 검증·정리.
  • 트레이스로 회귀 추적: 실패가 늘면 관측 데이터에서 원인을(19장).
  • 점진적 신뢰 확대: 검증된 자동화만 권한을 넓힌다(7.6).

    Build["하네스 변경"] --> Eval["평가 회귀 점검"] --> Deploy["배포"] --> Observe["트레이스 관측"] --> Learn["실패→개선"] --> Build

22.6 흔한 실수 / 안티패턴

  • ❌ 실패를 예외로 취급 — 확률적 시스템에서 실패는 정상. 재시도·저하를 설계하라.
  • ❌ 부작용 작업의 무분별 재시도 — 멱등성 없이 재시도하면 중복 사고.
  • ❌ 상태를 컨텍스트에만 — 압축·중단에 다 날아간다. 외부 영속화하라.
  • ❌ 성공 과장 보고 — production 신뢰를 무너뜨린다. 정직하게.
  • ❌ 모델 업그레이드 맹신 — 평가 없이 갈아끼우면 조용한 회귀.
  • ❌ 한 번 만들고 방치 — 코드·모델·요구가 바뀐다. 진화시켜라.

22.7 핵심 정리

  • production = 나쁜 날에도 견디는 설계: 재시도·저하·타임아웃·검증.
  • 거대 작업은 외부 상태 + resume + 스카우트 후 팬아웃으로 한 컨텍스트를 넘는다.
  • 사람과의 협업 지점(계획·게이트·정직한 보고·개입)을 의도적으로 설계.
  • 하네스는 진화한다 — 평가를 회귀 방지선으로, 관측을 나침반으로.

B.1 새 하네스 설계 체크리스트

요구 → 제약

  • 요구를 제약/함의 표로 번역했는가 (17.2)
  • 각 제약이 어떤 부품을 부르는지 매핑했는가
  • 되돌리기 어려운/외부 작업을 식별했는가 (7장)

루프 & 툴 (1~4장)

  • 종료 조건 + 안전장치(MAX_STEPS/예산)가 있는가
  • 툴 설명이 "언제 쓰는지 · 경계 · 반환 · 제약"을 담는가
  • 에러를 결과로 주입 하는가 (throw 아님)
  • 거대 결과를 요약·잘림 처리하는가
  • 툴이 많으면 지연 로딩하는가

컨텍스트 & 메모리 (5·8장)

  • 무엇을 넣고 빼는지 예산 으로 관리하는가
  • 고정/가변을 분리해 캐시 적중률을 지키는가
  • 세션 넘는 사실을 메모리로, 코드가 말하는 건 저장 안 하는가

시스템 프롬프트 (6장)

  • 규칙에 이유 와 우선순위 가 있는가
  • 가변 정보(타임스탬프)를 안 넣었는가
  • 상황별 절차를 스킬·메모리로 분리했는가

안전 (7장)

  • 위험에 비례한 마찰인가 (읽기 자유, 외부 게이트)
  • 승인이 그 맥락에만 한정되는가
  • 삭제·덮어쓰기 전 대상을 확인하는가

오케스트레이션 (9~12장)

  • 위임이 정당한가 (광범위/병렬 — 단일 조회 아님)
  • 다단계가 pipeline 기본인가 (불필요한 배리어 없음)

부록 D — Superpowers로 하네스 만들기

superpowers는 “하네스 엔지니어링 전용 스킬 하나”라기보다, 하네스를 스킬 기반 능력 묶음 으로 설계하는 방식의 좋은 예시다. 모델에게 모든 규칙을 한 번에 밀어 넣는 대신, 상황에 맞는 스킬을 발견하고, 필요한 지침만 펼치고, 작업 전후의 게이트로 품질을 고정한다.

D.1 Superpowers란 무엇인가

이 책의 관점에서 superpowers 는 “LLM이 더 똑똑해지는 마법”이 아니다. 더 정확히는, 에이전트가 특정 종류의 일을 할 때 어떤 절차를 읽고, 어떤 순서로 행동하고, 무엇을 확인한 뒤 완료라고 말할지 를 스킬 단위로 포장한 하네스 패턴이다.

예를 들어 디버깅, 테스트 주도 개발, 코드 리뷰, 브랜치 마무리 같은 작업은 매번 즉흥적으로 해도 어느 정도는 된다. 하지만 좋은 하네스는 이런 작업을 “모델의 습관”에 맡기지 않는다. 디버깅이면 먼저 재현하고, 가설을 나누고, 증거를 모으고, 수정 뒤 같은 경로로 검증하게 만든다. 코드 리뷰면 변경 요약보다 위험과 회귀 가능성을 먼저 보게 한다. 이처럼 작업별 절차를 능력으로 묶은 것 이 superpowers식 스킬이다.

따라서 superpowers는 하네스 엔지니어링의 대상이자 예시다. 하네스 엔지니어링은 더 넓은 분야이고, superpowers는 그 안에서 “스킬 라우팅 + 점진적 문서 로딩 + 절차 게이트”를 강하게 밀어붙인 한 구현 방식이다.

오해더 정확한 이해
superpowers는 하네스 전용 만능 스킬이다여러 작업 스킬을 필요할 때 불러 쓰는 하네스 설계 방식이다
스킬은 프롬프트 조각이다스킬은 트리거, 읽을 문서, 실행 절차, 완료 조건을 담은 작업 계약이다
많은 규칙을 미리 넣을수록 좋다지금 필요한 규칙만 읽게 해야 컨텍스트와 충돌을 줄인다
모델이 알아서 잘 따르면 된다하네스가 절차와 검증 게이트를 제공해야 반복적으로 좋아진다

D.2 Superpowers가 해결하는 문제

긴 시스템 프롬프트 하나로 모든 행동을 통제하려 하면 세 가지 문제가 생긴다. 첫째, 매 턴 컨텍스트가 무거워진다. 둘째, 서로 다른 작업 규칙이 충돌한다. 셋째, 검증·계획·디버깅처럼 절차가 중요한 작업이 모델의 즉흥 판단에 맡겨진다.

superpowers식 하네스는 이 문제를 작은 능력 단위 로 푼다. 각 스킬은 언제 발동하는지, 무엇을 읽어야 하는지, 어떤 순서로 확인해야 하는지, 완료 전 어떤 검증을 해야 하는지를 가진다.

하네스 요소superpowers식 구현설계 효과
툴 발견작업 설명과 트리거로 필요한 스킬을 고른다항상 모든 규칙을 싣지 않는다
컨텍스트 로딩SKILL.md를 먼저 읽고, 필요한 참조만 추가로 연다점진적 공개로 토큰을 아낀다
절차 강제계획, TDD, 디버깅, 검증 같은 루프를 명시한다모델의 즉흥성을 실행 절차로 묶는다
완료 게이트검증 전 완료 보고를 금지한다그럴듯한 보고보다 증거를 우선한다

D.3 스킬을 하네스 부품으로 보기

스킬은 프롬프트 조각이 아니라 작업 계약 이다. 좋은 스킬은 다음 네 가지를 분명히 한다.

  1. 트리거: 어떤 사용자 요청이나 상황에서 발동하는가.
  2. 필수 입력: 어떤 파일, 문서, 상태를 먼저 읽어야 하는가.
  3. 절차: 탐색, 구현, 검증, 보고를 어떤 순서로 해야 하는가.
  4. 완료 조건: 무엇을 확인해야 끝났다고 말할 수 있는가.

이 구조는 14장의 스킬 설계와 18장의 계획 모드를 합친 것이다. 차이는 사람이 매번 기억해서 지시하는 것 이 아니라, 하네스가 작업 종류에 맞는 절차를 자동으로 끌어온다는 점이다.

D.4 Superpowers 스킬은 어떻게 만드는가

superpowers식 스킬은 “좋은 조언 모음”이 아니라 작업 절차를 담은 작은 매뉴얼 이다. 핵심은 모델이 매번 기억으로 행동하지 않게 만들고, 특정 상황에서 반드시 읽고 따라야 할 문서를 분리해두는 것이다.

skill/
  SKILL.md
  references/
    checklist.md
    examples.md
  scripts/
    verify.mjs

SKILL.md 구조:
1. name / description
   - 언제 이 스킬이 발동해야 하는지 짧고 분명하게 쓴다.

2. trigger rules
   - 사용자의 어떤 말, 파일 형태, 에러, 작업 종류가 이 스킬을 부르는지 적는다.

3. progressive disclosure
   - 처음에는 SKILL.md만 읽고, 필요할 때 references/*를 추가로 읽게 한다.

4. procedure
   - 탐색 → 설계 → 실행 → 검증 → 보고 순서를 명시한다.

5. completion gate
   - 무엇을 실제로 확인해야 완료라고 말할 수 있는지 적는다.

이 구조의 장점은 컨텍스트를 아끼는 데 있다. 모든 규칙을 시스템 프롬프트에 넣지 않고, “지금 작업에 필요한 규칙”만 파일에서 꺼내 읽는다. 즉 스킬은 지식 저장소이면서 동시에 라우터다.

D.5 나만의 MD 하네스 만들기

개인 작업방식으로서의 하네스는 거창한 서버나 프레임워크가 없어도 된다. 잘게 분리된 MD 파일을 만들고, 에이전트가 그 파일들을 효율적으로 발견하고 필요한 것만 읽게 하는 구조면 충분하다.

my-harness/
  README.md                 # 전체 지도: 무엇이 어디 있는가
  ROUTER.md                 # 어떤 요청이면 어떤 문서를 읽을지
  rules/
    coding.md               # 코딩할 때의 원칙
    writing.md              # 문서 작성 원칙
    review.md               # 리뷰/검증 원칙
  workflows/
    bugfix.md               # 버그 수정 루프
    feature.md              # 기능 추가 루프
    research.md             # 조사 루프
  context/
    project-map.md          # 폴더 구조와 핵심 파일
    glossary.md             # 자주 쓰는 용어
    decisions.md            # 확정된 설계 결정
  examples/
    good-task.md
    bad-task.md

이때 가장 중요한 파일은 ROUTER.md다. 모델에게 “모든 MD를 다 읽어라”라고 하지 말고, 어떤 상황에서 어떤 MD를 읽을지 알려준다.

# ROUTER.md 예시

요청이 버그 수정이면:
1. context/project-map.md를 읽는다.
2. workflows/bugfix.md를 읽는다.
3. 관련 파일을 rg로 찾는다.
4. 수정 전 실패 조건을 재현한다.
5. 수정 후 같은 조건을 다시 검증한다.

요청이 새 기능이면:
1. context/decisions.md를 읽어 기존 결정과 충돌하는지 본다.
2. workflows/feature.md를 읽는다.
3. 구현 범위를 작게 자른다.
4. 완료 전 사용자가 보는 표면을 실제로 확인한다.

요청이 글쓰기/정리면:
1. rules/writing.md를 읽는다.
2. context/glossary.md를 읽는다.
3. 원문을 요약하지 말고 구조를 재편집한다.
4. 누락된 제목/표/코드블록을 카운트한다.

이 방식은 8장의 파일 기반 메모리와 14장의 스킬 설계를 합친 것이다. MD 파일은 기억이고, ROUTER는 리콜 정책이며, workflow 문서는 실행 루프다.

D.6 MD 하네스가 잘 동작하게 하는 규칙

  • 한 파일은 한 역할만: 규칙, 절차, 예시, 결정 기록을 한 문서에 섞지 않는다.
  • README는 지도만: 자세한 설명보다 “어디를 읽어야 하는지”를 빠르게 알려준다.
  • ROUTER는 조건문처럼: “버그면 A, 기능이면 B, 문서면 C”처럼 선택 기준을 쓴다.
  • 예시는 작게: 좋은 예/나쁜 예를 1~2개씩 두면 모델이 스타일을 빠르게 맞춘다.
  • 검증을 문서화: “끝났다”의 기준을 각 workflow에 적어 둔다.
  • 카운트 가능한 것은 카운트: 파일 수, 제목 수, 코드블록 수, 링크 수처럼 누락을 잡는 숫자를 만든다.

요약하면 나만의 하네스는 “MD를 많이 쌓는 것”이 아니라, 에이전트가 필요한 MD를 고르고, 읽고, 적용하고, 검증하게 만드는 방식 이다.

예제 — Minimal ReAct Loop

3장 — 첫 하네스 만들기의 동반 예제.

"에이전트 = while 루프 + 툴 + 결과 주입"을 한 파일로 보여주는 언어 무관 의사코드다. 특정 SDK에 의존하지 않으며, 어떤 언어/모델 API로든 그대로 옮길 수 있도록 경계를 명확히 했다.

의사코드 전문

아래 코드블록이 기존 예제 파일의 전체 내용이다. 이 HTML 밖의 파일을 열 필요가 없다.

// =====================================================================
//  Minimal ReAct Loop — 언어 무관 의사코드
//  "에이전트 = while 루프 + 툴 + 결과 주입"
//  동반 챕터: part1-foundations/03-your-first-harness.md
// =====================================================================

// ---------------------------------------------------------------------
// (1) 설정 — 안전장치
// ---------------------------------------------------------------------
CONST MAX_STEPS = 12        // 무한 루프 / 비용 폭주 방지 (하네스가 두는 안전장치)

// ---------------------------------------------------------------------
// (2) 시스템 프롬프트 — 모델에게 역할·제약을 정의 (6장에서 심화)
// ---------------------------------------------------------------------
CONST SYSTEM_PROMPT = """
너는 파일 시스템을 탐색하는 보조 에이전트다.
- 질문에 답하려면 제공된 툴(read_file, list_dir)을 사용하라.
- 추측하지 말고, 필요한 파일을 직접 읽어 근거를 확보한 뒤 답하라.
- 충분한 정보를 얻으면 더 이상 툴을 호출하지 말고, 간결한 최종 답을 한국어로 제시하라.
"""

// ---------------------------------------------------------------------
// (3) 툴 정의 — 스키마(모델이 봄) + 구현(실제 실행) 의 쌍  (4장에서 심화)
// ---------------------------------------------------------------------
CONST TOOLS = [
  {
    name: "list_dir",
    description: "디렉토리의 파일/폴더 목록을 문자열 배열로 반환한다.",
    parameters: {
      path: { type: "string", description: "목록을 볼 디렉토리 경로" }
    },
    run: FUNCTION(args) {
      RETURN filesystem.list(args.path)         // ← 실제 구현으로 교체
    }
  },
  {
    name: "read_file",
    description: "주어진 경로의 텍스트 파일 내용을 반환한다.",
    parameters: {
      path: { type: "string", description: "읽을 파일 경로" }
    },
    run: FUNCTION(args) {
      RETURN filesystem.read(args.path)         // ← 실제 구현으로 교체
    }
  }
]

// ---------------------------------------------------------------------
// (4) 핵심 — ReAct 루프
// ---------------------------------------------------------------------
FUNCTION run_agent(user_input):

    // 컨텍스트 초기화: 시스템 프롬프트 + 사용자 입력
    messages = [
      { role: "system", content: SYSTEM_PROMPT },
      { role: "user",   content: user_input },
    ]

    FOR step IN 1..MAX_STEPS:

        // ① 추론(Reason): 모델 호출. 툴 스키마를 함께 전달.
        response = model.call(messages, tools = TOOLS)

        // 모델의 발화(텍스트/툴호출 의도)를 반드시 대화에 기록
        messages.append(response)

        // ③ 종료 판단: 툴 호출이 없으면 모델이 "끝"이라고 본 것
        IF response.tool_calls IS EMPTY:
            RETURN response.text

        // ② 행동(Act): 요청된 툴들을 실행하고 결과를 주입
        FOR call IN response.tool_calls:          // 여러 개면 병렬화 가능(연습 2)
            tool = find_by_name(TOOLS, call.name)

            TRY:
                result = tool.run(call.args)
            CATCH error:
                // ★ 에러는 throw 하지 않는다 — 결과로 주입해 모델이 복구하게 한다
                result = "ERROR: " + error.message

            // ★★ 결과 주입(result injection): 이 한 단계가 에이전트의 핵심.
            //     결과를 다시 messages에 넣어야 다음 호출에서 모델이 "본다".
            messages.append({
              role: "tool",
              tool_call_id: call.id,
              content: stringify(result),
            })

        // 루프 계속 → 다시 ①로 (모델이 새 결과를 보고 추론을 이어감)

    // 안전장치 발동: MAX_STEPS 안에 끝내지 못함
    RETURN "중단됨: 최대 단계 수(" + MAX_STEPS + ")를 초과했습니다."


// =====================================================================
//  실행 추적 예시
//  run_agent("src 폴더에 테스트 파일이 몇 개 있어?")
// ---------------------------------------------------------------------
//  step 1:
//    model → list_dir("src")
//    tool  → ["app.py","test_app.py","util.py","test_util.py"]   (주입됨)
//  step 2:
//    model → (툴 호출 없음) "test_ 로 시작하는 파일이 2개입니다."   → 종료
//  return: "src 폴더에 테스트 파일은 2개입니다 (test_app.py, test_util.py)."
// =====================================================================

실제 코드로 옮길 때 매핑할 3가지

의사코드의 다음 3곳만 실제 환경에 맞게 바꾸면 동작한다.

의사코드실제로 바꿀 것
model.call(messages, tools)쓰는 LLM API의 호출 + 툴 정의 전달 방식
tool.run(args)각 툴의 실제 구현 (파일 I/O, 셸 등)
response.tool_callsAPI가 반환하는 툴 호출 구조 파싱

직접 확장해보기 (연습)

  1. write_file 툴을 추가하고, 쓰기 전에 사용자 확인을 받게 하라. → 7장 안전
  1. 한 응답에서 여러 툴 호출을 병렬 로 실행하도록 루프를 고쳐라. → 2장 심화 박스
  1. 거대한 파일을 읽을 때 결과를 요약해 주입하라. → 5장 컨텍스트

예제 — Workflow Patterns

10장 — 워크플로우와 파이프라인과 11장 — 검증 루프의 동반 예제.

오케스트레이션 조합자(agent/parallel/pipeline)와 핵심 패턴을 의사코드로 모았다.

워크플로우 패턴 전문

아래 코드블록이 기존 예제 파일의 전체 내용이다.

// =====================================================================
//  Workflow Patterns — 오케스트레이션 의사코드 모음
//  동반 챕터: part3-orchestration/10·11·12
//  조합자: agent(작업) / parallel(작업들=배리어) / pipeline(항목들, 단계들)
// =====================================================================

// ---------------------------------------------------------------------
// 패턴 0 — 세 조합자
// ---------------------------------------------------------------------
result  = agent("작업", {schema: S})                 // 서브에이전트 하나
results = parallel([f1, f2, f3])                      // 동시 실행 + 모두 대기(배리어)
out     = pipeline(items, stage1, stage2)            // 항목별로 단계 통과, 배리어 없음


// ---------------------------------------------------------------------
// 패턴 1 — pipeline 기본 (10.3): 차원별 리뷰 → 즉시 검증
//   'bugs' 발견이 검증되는 동안 'perf'는 아직 리뷰 중. 벽시계 낭비 없음.
// ---------------------------------------------------------------------
DIMENSIONS = ["bugs", "perf", "simplify"]

results = pipeline(
  DIMENSIONS,
  dim    => agent(review_prompt(dim), {schema: FINDINGS}),            // 1단계: 리뷰
  review => parallel(review.findings.map(f => () =>                   // 2단계: 즉시 검증
     adversarial_verify(f).then(v => ({...f, verdict: v}))
  ))
)
confirmed = results.flat().filter(Boolean).filter(f => f.verdict.isReal)


// ---------------------------------------------------------------------
// 패턴 2 — 배리어가 정당한 경우 (10.4): 전체 dedup 후 비싼 검증
//   deduped는 '모든' 발견이 있어야 가능 → 배리어 정당.
// ---------------------------------------------------------------------
all     = parallel(DIMENSIONS.map(d => () => agent(review_prompt(d), {schema: FINDINGS})))
            .filter(Boolean).flatMap(r => r.findings)
deduped = dedupe_by_file_and_line(all)                  // ← 교차 항목 필요
verified = parallel(deduped.map(f => () => adversarial_verify(f)))
confirmed = verified.filter(v => v.isReal)


// ---------------------------------------------------------------------
// 패턴 3 — 적대적 검증 (11.2): 독립 N명이 반증 시도
// ---------------------------------------------------------------------
FUNCTION adversarial_verify(claim):
    votes = parallel(repeat(3, () =>
        agent(`이것을 반증하라: ${claim}.
               불확실하면 기본값은 refuted=true.`, {schema: VERDICT})))
    survives = votes.filter(Boolean).filter(v => !v.refuted).length >= 2
    RETURN { isReal: survives }
// 관점 다양화 버전 (11.3): 동일 회의론자 대신 서로 다른 렌즈
FUNCTION diverse_verify(claim):
    lenses = ["correctness", "security", "reproducibility"]
    votes = parallel(lenses.map(lens => () =>
        agent(`${lens} 렌즈로 판단: "${claim}" — 진짜인가?`, {schema: VERDICT})))
    RETURN { isReal: votes.filter(Boolean).filter(v => v.real).length >= 2 }


// ---------------------------------------------------------------------
// 패턴 4 — loop-until-dry (11.5): K번 연속 빈 라운드까지
//   ★ dedup은 seen으로 (confirmed로 하면 기각된 게 재등장해 무한 루프)
// ---------------------------------------------------------------------
seen = Set(); confirmed = []; dry = 0
WHILE dry < 2:
    found = parallel(FINDERS.map(f => () => agent(f.prompt, {schema: BUGS})))
              .filter(Boolean).flatMap(r => r.bugs)
    fresh = found.filter(b => NOT seen.has(key(b)))      // ← seen과 대조
    IF fresh.isEmpty: dry++; CONTINUE
    dry = 0; fresh.forEach(b => seen.add(key(b)))
    confirmed.push(...parallel(fresh.map(b => () => diverse_verify(b)))
                       .filter(v => v.isReal))


// ---------------------------------------------------------------------
// 패턴 5 — 예산 루프 (21.4): 예산이 남는 동안 깊이 확장
//   budget.total 가드 필수 — 없으면 remaining()이 무한대라 캡까지 돌진.
// ---------------------------------------------------------------------
bugs = []
WHILE budget.total AND budget.remaining() > 50_000:
    bugs.push(...agent("이 코드베이스의 버그를 찾아라", {schema: BUGS}).bugs)
    log(`${bugs.length} found, ${round(budget.remaining()/1000)}k remaining`)


// ---------------------------------------------------------------------
// 패턴 6 — 포괄적 탐색 합성 (12.4): sweep → dedup → 다양 렌즈 → dry → critic
// ---------------------------------------------------------------------
FUNCTION exhaustive_audit():
    seen = Set(); confirmed = []; dry = 0
    WHILE dry < 2:
        // multi-modal sweep: 서로 다른 방식의 finder들 (배리어)
        found = parallel(SWEEP_MODES.map(m => () => agent(m.prompt, {schema: BUGS})))
                  .filter(Boolean).flatMap(r => r.bugs)
        fresh = found.filter(b => NOT seen.has(key(b)))
        IF fresh.isEmpty: dry++; CONTINUE
        dry = 0; fresh.forEach(b => seen.add(key(b)))
        confirmed.push(...parallel(fresh.map(b => () => diverse_verify(b)))
                           .filter(v => v.isReal))
    // completeness critic: 마지막 누락 점검 (12.3)
    gaps = agent("무엇이 빠졌나? 안 돌린 방식·미검증 주장·안 읽은 소스는?",
                 {schema: GAPS})
    log(`확정 ${confirmed.length}건, 남은 빈틈: ${gaps.summary}`)
    RETURN confirmed

한 장 요약

  • 다단계의 기본은 pipeline — 항목별로 배리어 없이 흐른다 (10.3).
  • 배리어(parallel)는 교차 의존 때만 — dedup·early-exit·교차참조 (10.4).
  • 검증은 독립 적대적으로 — "반증하라", 불확실은 기각 (11.2).
  • 미지의 양은 loop-until-dry — seen으로 dedup, K번 빈 라운드에 종료 (11.5).
  • 규모를 작업·예산에 맞춰라 (12·21장).
