0.1하네스 엔지니어링이란? — 프롬프트 엔지니어링과의 차이

하네스 엔지니어링은 모델에게 무엇을 말할까가 아니라, 모델을 무엇으로 둘러쌀까를 설계하는 일입니다.

프롬프트 엔지니어링은 한 번의 대화, 한 번의 요청을 잘 짜는 기술입니다. 좋은 프롬프트는 그 순간엔 훌륭한 답을 끌어내지만, 다음 사람이 같은 작업을 할 때는 처음부터 다시 써야 합니다. 지식이 사람의 머릿속에 남고 시스템에 남지 않죠.

하네스 엔지니어링은 그 지식을 모델 주변의 구조물로 굳힙니다. 프로젝트 규칙은 파일에, 반복 작업은 명령어에, 안전장치는 자동화에, 검증은 피드백 루프에 새깁니다. 그러면 누가 언제 작업하든 같은 맥락, 같은 규칙, 같은 품질이 재현됩니다.

한 줄로

프롬프트는 대화를 설계하고, 하네스는 환경을 설계합니다. 이 책은 후자를 다룹니다 — 다만 둘은 배타적이지 않고, 좋은 하네스 안에서 좋은 프롬프트가 재사용됩니다.

0.2“모델을 바꾸지 말고 모델 주변을 설계하라”

AI 결과가 기대에 못 미칠 때 첫 반응은 대개 “더 좋은 모델은 없나”입니다. 하지만 실무에서 결과 품질을 가르는 변수의 대부분은 모델이 아니라 모델에게 주어진 조건입니다. 같은 모델이라도 프로젝트 규칙을 알고 있는지, 올바른 도구가 손에 있는지, 결과를 검증할 방법이 있는지에 따라 결과물이 완전히 달라집니다.

이것이 이 책의 핵심 명제입니다. 모델은 엔진이고, 하네스는 그 엔진을 실제 도로에서 달리게 하는 차체·배선·계기판입니다. 엔진만 좋다고 차가 되지 않습니다.

흔한 착각

“모델을 업그레이드하면 해결되겠지.” — 대개는 해결되지 않습니다. 컨텍스트가 부실하면 더 똑똑한 모델은 더 그럴듯하게 틀린 답을 내놓을 뿐입니다. 먼저 주변을 설계하세요.

그리고 하네스에는 실용적인 이점이 하나 더 있습니다. 모델은 갈아끼울 수 있다는 점입니다. 잘 설계된 하네스는 특정 모델에 묶이지 않으므로, 더 좋은 모델이 나오면 그 위에서 곧바로 더 좋은 결과를 냅니다. 투자가 사라지지 않고 축적됩니다.

1.1CLAUDE.md — 프로젝트의 상시 규칙

CLAUDE.md는 프로젝트 루트에 두는 마크다운 파일입니다. 세션이 시작될 때마다 그 내용이 자동으로 컨텍스트에 주입되므로, “우리는 원화를 정수로 다룬다”, “테스트는 npm test로 돌린다” 같은 규칙을 매번 설명할 필요가 사라집니다. 신입에게 매일 아침 브리핑을 반복하지 않게 해주는 사내 위키 첫 페이지인 셈입니다.

# MarketFlow 웹앱

전자상거래 플랫폼의 프론트엔드 + 백엔드 API 저장소입니다.

## 프로젝트 규칙
- 통화는 항상 원화(KRW) **정수**로 다룬다. 소수점·부동소수 금지.

- 모든 API 응답은 `snake_case`. 프론트 변환은 어댑터 계층에서만.
- 날짜는 ISO 8601 + KST(+09:00)로 저장한다.

## 자주 쓰는 명령
- 개발 서버: `npm run dev`
- 테스트: `npm test` (변경한 코드는 반드시 테스트 후 완료 보고)
- 타입체크: `npm run typecheck`

- 프로덕션 DB에 직접 쿼리 금지. 반드시 마이그레이션으로.
- 결제 관련 코드 수정 시 사람 리뷰 없이 커밋 금지.

1.2좋은 CLAUDE.md와 나쁜 CLAUDE.md — 분량·구체성·안티패턴

CLAUDE.md는 길수록 좋은 게 아닙니다. 오히려 길수록 나빠지기 쉽습니다. 모든 규칙이 매 세션 컨텍스트를 차지하고, 규칙이 100개면 모델은 그중 무엇이 중요한지 판단해야 합니다. 규칙이 많아질수록 각 규칙의 신호가 흐려지는 역설이죠. 좋은 파일은 짧고, 구체적이고, 실행 가능합니다.

구체성 — 확인 가능한가

가장 중요한 판별 기준은 하나입니다. “이 규칙을 지켰는지 아닌지 기계적으로 확인할 수 있는가?” 확인할 수 없는 규칙은 규칙이 아니라 구호이고, 구호는 컨텍스트만 잡아먹습니다.

안티패턴 3종 — 이렇게 쓰면 무너진다

실무에서 반복적으로 관찰되는 실패 유형입니다.

① 소설

회사 연혁·비전·팀 자랑을 적습니다. 모델은 그 문장을 읽느라 컨텍스트를 쓰지만 코드를 짜는 데는 아무 도움이 안 됩니다. “이 문장이 모델의 행동을 바꾸나?”를 물어, 아니면 지우세요.

② 중복

README·package.json·코드에 이미 있는 걸 옮겨 적습니다. 문제는 둘이 어긋나는 순간입니다. 어느 쪽이 진실인지 모델도 사람도 알 수 없게 되죠. 이미 코드가 말하는 건 적지 말고, 코드가 말하지 않는 규칙만 적으세요.

③ 방치

한 번 쓰고 갱신하지 않아 현실과 달라집니다. 틀린 규칙은 없는 규칙보다 나쁩니다 — 모델이 그 틀린 규칙을 충실히 따르니까요. “테스트는 pytest”라고 적힌 채 실제로는 npm test로 바뀌었다면, 그 한 줄이 매번 사고를 냅니다.

분량 감각은 이렇습니다. 대부분의 프로젝트에서 화면 한두 페이지면 충분합니다. 그보다 길어지면 “이걸 정말 매 세션 읽혀야 하나”를 의심하고, 특정 폴더에만 필요한 건 디렉토리 스코프로 내리세요(다음 절).

1.3메모리 계층 구조 — 전역·프로젝트·디렉토리 스코프

규칙에도 적용 범위가 있습니다. “나는 항상 한국어로 답을 받고 싶다”는 개인 전역 규칙이고, “이 저장소는 원화 정수를 쓴다”는 프로젝트 규칙이며, “이 폴더는 레거시라 손대지 마라”는 디렉토리 규칙입니다. 이걸 한 파일에 몰아넣으면 엉킵니다 — 내 개인 말투 취향이 팀 저장소에 커밋되고, 레거시 폴더 규칙이 전체에 적용되죠. 계층을 나눠 두면 각 규칙이 딱 필요한 범위에만 적용됩니다.

범위위치담는 것git
전역(개인)~/.claude/CLAUDE.md내 언어·말투·선호커밋 안 함
프로젝트<repo>/CLAUDE.md이 저장소의 규칙커밋(공유)
디렉토리<repo>/dir/CLAUDE.md하위 폴더 지역 규칙커밋(공유)

더 가까운(더 구체적인) 범위가 넓은 범위를 보강합니다. 충돌하면 가까운 쪽이 이깁니다. 마켓플로우는 이렇게 나눴습니다 — 개인 전역에 “답변은 한국어로”, 저장소 루트에 프로젝트 규칙, 그리고 legacy/ 폴더에 별도 파일로 “이 코드는 리팩토링 대상이 아니다”.

1.4컨텍스트 위생 — 무엇을 넣고 무엇을 빼는가

컨텍스트는 공짜가 아닙니다. 넣는 모든 문장이 비용이고, 정작 중요한 규칙의 신호를 흐립니다. 컨텍스트 위생(context hygiene)은 “무엇을 넣을까”만큼 “무엇을 빼고, 언제 치울까”를 관리하는 습관입니다. 파일 위생과 세션 위생, 두 층위가 있습니다.

긴 작업을 하다 보면 컨텍스트가 무관한 내용(옛 시도, 실패한 접근, 다 끝난 파일 탐색)으로 가득 차 품질이 떨어집니다. 사람도 회의가 세 시간을 넘으면 집중이 흐려지듯이요. 대응은 두 가지입니다.

  • 기록의 외부화 — 세션 안에서만 아는 것에 의존하지 말고, 중요한 건 파일로 빼세요. 세션은 사라져도 파일은 남습니다.

2.1Slash Commands — 자주 쓰는 작업을 /명령으로

같은 지시를 세 번 이상 타이핑했다면, 그건 커맨드가 되어야 합니다.

슬래시 커맨드는 .claude/commands/ 폴더에 둔 마크다운 파일 하나입니다. 파일 이름이 곧 명령어가 됩니다 — 상품등록.md를 만들면 /상품등록으로 부릅니다. 파일 안에는 “이 작업을 할 때 매번 하던 설명”을 그대로 적습니다. 그러면 팀 누구나 긴 프롬프트 대신 한 단어로 같은 작업을, 같은 품질로 실행합니다.

이게 왜 하네스일까요? 커맨드는 “좋은 프롬프트를 팀 자산으로 승격”시키는 장치입니다. 잘하는 팀원의 완벽한 프롬프트가 그 사람 머릿속에만 있으면, 그 사람이 없을 때 품질이 무너집니다.

사건 #1 · 5월 12일 월요일, 유진의 휴가 첫날

MD 유진은 상품 등록의 달인이었습니다. AI에게 던지는 그의 프롬프트는 원산지·배송·교환정책까지 빠짐없이 챙겼고, 결과물은 늘 그대로 등록 시스템에 붙여 넣을 수 있었습니다.

유진이 일주일 휴가를 갔습니다. 대신 등록을 맡은 동료가 “이 상품 등록용으로 정리해줘”라고 요청했고, 결과는 그럴듯했습니다 — 다만 교환/반품 안내가 통째로 빠져 있었습니다. 그 상태로 40개 상품이 등록됐고, 고객 문의가 쏟아진 뒤에야 발견됐습니다.

유진의 노하우는 문서 어디에도 없었습니다. 그의 머릿속과 채팅 히스토리에만 있었죠.

→ 복귀한 유진이 자기 프롬프트를 파일로 옮겼습니다. /상품등록 커맨드의 탄생입니다.

.claude/commands/상품등록.md
---
description: 신규 상품 등록 초안을 규격에 맞춰 작성
argument-hint: [상품명 또는 설명]
---

다음 상품을 우리 등록 규격에 맞춰 정리해줘: $ARGUMENTS

규격:
- 가격은 원화 정수(price_krw)
- 카테고리는 사내 분류 체계에서 고른다
- 상품명은 40자 이내, 과장 표현 금지
- 필수: 원산지, 배송 정보, 교환/반품 안내
- 누락된 정보는 채우지 말고 "확인 필요"로 표시한다

frontmatter — 커맨드의 메타 정보

파일 상단의 --- 사이 영역이 frontmatter입니다. 커맨드의 동작을 정합니다.

필드역할

커맨드로 만들 후보 찾기

지난 일주일 대화에서 비슷하게 반복한 요청을 세 개만 꼽아 보세요. 그게 첫 세 개의 커맨드입니다. 커맨드는 .claude/commands/에 두면 팀이 공유하고, ~/.claude/commands/에 두면 나만 씁니다 — CLAUDE.md와 같은 계층 원리입니다.

2.2커맨드 인자·파일 참조·bash 실행

커맨드는 정적인 텍스트를 넘어, 입력을 받고 · 파일을 읽고 · 명령을 실행할 수 있습니다. 이 세 가지를 조합하면 커맨드가 살아 움직입니다.

기능표기뜻
인자$ARGUMENTS/명령 뒤에 쓴 내용이 이 자리에 들어간다
파일 참조@경로/파일그 파일 내용을 컨텍스트에 넣는다
bash 실행!명령명령을 먼저 실행하고 그 출력을 넣는다

세 가지를 다 쓰는 실제 커맨드를 봅시다. 마켓플로우의 CS 응답 초안 커맨드는 인자로 문의를 받고, 정책 파일을 읽어, 정책에 맞는 답을 만듭니다.

.claude/commands/cs-응답초안.md (핵심)
우리 CS 정책에 따라 다음 문의에 응답 초안을 써줘: $ARGUMENTS

응대 정책: @docs/cs-정책.md
최근 공지: @docs/공지.md

@로 참조한 두 파일의 내용이 실행 시점에 컨텍스트로 들어오므로, 항상 최신 정책이 반영됩니다.

bash 실행의 진가는 “지금 이 순간의 상태”를 넣을 때 나옵니다. “스테이징한 변경을 커밋 메시지로 요약해줘” 커맨드는 !git diff --staged의 출력을 그대로 활용하죠 — 매번 diff를 복붙할 필요가 없습니다.

.claude/commands/변경요약.md
---
description: 스테이징된 변경사항을 커밋 메시지 형식으로 요약
allowed-tools: Bash(git diff:*), Bash(git status:*)
---

아래 변경사항을 우리 커밋 규칙(feat/fix 프리픽스)에 맞춰 요약해줘.

변경 내용:
!git diff --staged

bash 실행은 권한과 함께

!로 명령을 실행하는 커맨드는 그 명령을 실행할 권한이 필요합니다. 위 예시처럼 allowed-tools로 정확히 그 명령만 허용하세요. Bash(git diff:*)는 “git diff는 되지만 다른 건 안 됨”을 뜻합니다. 권한 설계 전반은 3.2에서 다룹니다.

2.3Skills — 재사용 가능한 작업 지침 패키지

커맨드가 “내가 부르는” 단축키라면, 스킬(Skill)은 “필요할 때 알아서 불려 오는” 전문 지침 묶음입니다. 스킬은 SKILL.md 한 파일과 딸린 참고 자료·스크립트로 이뤄지며, 설명(description)에 적힌 트리거 조건에 맞는 작업이 오면 자동으로 활성화됩니다.

  프로모션 문구, 카피 작성 요청에 트리거.

점진적 공개 — 스킬이 가벼운 이유

스킬의 영리한 점은 description만 평소에 컨텍스트에 있고, 본문은 필요할 때 읽힌다는 것입니다. 스킬이 100개 있어도 평소엔 100줄짜리 description 목록만 부담이고, 실제 상세 지침은 그 작업이 올 때만 로드됩니다. 그래서 references/금지어.md 같은 무거운 참고자료를 딸려 둬도 평소엔 비용이 없습니다.

차이를 정리하면 — 커맨드는 절차(내가 지금 이걸 실행)이고, 스킬은 전문성(이런 종류의 일이 오면 이렇게 처리)입니다. 카피를 쓰는 대화라면 내가 / 명령을 하지 않아도 위 규칙이 자동으로 적용됩니다.

  • 무엇을 할 때 — 이 스킬이 다루는 작업의 성격 (“상품 문구를 쓸 때”)
  • 어떤 표현이 오면 — 사용자가 실제로 쓸 법한 단어 (“‘카피’, ‘상세페이지’”)

3.1Hooks — 도구 호출 전후 자동 실행, 주요 이벤트

지침은 “해주세요”라는 부탁이고, Hook은 “반드시 이렇게 된다”는 보장입니다.

CLAUDE.md에 “커밋 전 테스트를 돌려라”라고 써도 모델이 잊을 수 있습니다. 대부분은 지키지만, 급하거나 컨텍스트가 길어지면 놓칩니다. Hook은 다릅니다. Hook은 특정 이벤트가 일어날 때 결정론적으로 실행되는 명령이라, 모델의 판단·기분·컨텍스트와 무관하게 항상 작동합니다. 이것이 “부탁(지침)”과 “장치(Hook)”의 결정적 차이입니다.

언제 지침으로 충분하고 언제 Hook이 필요할까요? 기준은 “어겼을 때의 대가”입니다.

상황지침으로 충분Hook이 필요
코드 스타일선호 정도라면일관성이 중요하면 자동 포맷
테스트 실행참고용이면깨진 코드 커밋 방지가 목표면
위험 명령—항상 (사고는 되돌릴 수 없다)

주요 이벤트

Hook은 settings.json의 hooks 항목에 이벤트별로 등록합니다. 각 이벤트는 에이전트 수명주기의 특정 순간에 발화합니다.

이벤트언제대표 용도
PreToolUse도구 실행 직전위험 명령 차단, 승인 게이트
PostToolUse도구 실행 직후포맷·린트, 자동 로깅
UserPromptSubmit사용자 입력 제출 시입력 검증, 컨텍스트 주입
Stop응답을 마치려 할 때완료 조건 강제(테스트 통과 등)
SubagentStop서브에이전트가 끝날 때위임 결과 검증
SessionStart세션 시작 시환경 점검, 최신 상태 로드

Hook의 힘은 매처(matcher)에서 나옵니다 — 어떤 도구에 반응할지 좁힐 수 있죠. 마켓플로우는 PostToolUse에 Edit|Write 매처로 포맷터를 걸어, 파일이 저장될 때마다 스타일이 자동으로 맞춰지게 했습니다. 이제 “포맷 좀 맞춰줘”라는 말이 대화에서 사라졌습니다.

Hook은 어떻게 소통하나

Hook은 stdin으로 JSON(어떤 도구를, 어떤 입력으로 호출하려는지)을 받고, 종료 코드로 응답합니다. exit 0은 통과, PreToolUse에서 exit 2는 “막아라”(이유는 stderr로 모델에 전달). 즉 Hook은 에이전트와 대화할 수 있습니다 — 막으면서 왜 막았는지 알려주면, 모델이 다른 방법을 찾습니다.

3.2settings.json — 권한·환경변수·허용과 차단

settings.json은 하네스의 배전반입니다. 어떤 도구를 자동 허용하고, 어떤 것은 물어보게 하고, 어떤 것은 아예 막을지를 정합니다. 잘 잡으면 “매번 승인 누르기”와 “위험한 자동 실행” 사이의 균형이 잡힙니다 — 너무 조이면 답답해 사람들이 권한을 다 열어버리고, 너무 풀면 사고가 납니다.

.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint:*)",
      "Read(./src/**)"
    ],
    "ask": [
      "Bash(git push:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Read(./.env)",
      "Read(./secrets/**)"
    ]
  },
  "env": {
    "NODE_ENV": "development"
  }
}

세 단계의 뜻은 명확합니다.

단계동작넣을 것
allow묻지 않고 실행자주 쓰는 안전한 명령(테스트·린트·읽기)
ask실행 전 확인되돌리기 어려운 작업(푸시·배포)
deny절대 금지비밀 파일, 파괴적 명령

패턴 매칭 — 정밀하게 허용하기

핵심은 Bash(npm run test:*) 같은 패턴입니다. “npm run test로 시작하는 건 다 허용, 하지만 다른 bash는 아님”을 뜻하죠. *의 위치가 사정거리를 정합니다. 너무 넓게(Bash(*)) 열면 가드레일이 무의미해지니, 필요한 것만 좁게 허용하세요.

범위도 계층

설정도 CLAUDE.md처럼 계층입니다. 개인(~/.claude/settings.json) · 프로젝트(공유, git 커밋 settings.json) · 로컬(개인용, git 제외 settings.local.json). 팀 공통 규칙은 프로젝트에, 내 개인 허용은 로컬에 두세요. 그래야 “나는 이 명령을 허용했지만 팀 표준은 아닌” 상태가 깔끔하게 분리됩니다.

3.3안전장치 설계 — 위험 명령 차단과 승인 게이트

자동화가 강해질수록 가드레일이 중요해집니다. 에이전트가 빠르게 많은 일을 할 수 있다는 건, 실수도 빠르고 크게 할 수 있다는 뜻이니까요. 사람은 rm -rf를 치기 전에 한 번 멈칫하지만, 에이전트에겐 그 멈칫을 설계해 넣어야 합니다.

3층 방어

  1. 파괴적 명령 차단 — PreToolUse Hook으로 rm -rf, 강제 푸시, DROP TABLE을 실행 전에 막습니다.
  2. 비밀 정보 보호 — .env·키 파일을 deny로 아예 읽지 못하게 합니다(읽을 수 없으면 유출할 수도 없습니다).
  3. 승인 게이트 — 결제·배포처럼 되돌리기 어려운 작업은 ask로 사람의 확인을 강제합니다.
.claude/hooks/block-dangerous.sh — PreToolUse 차단 예시
#!/usr/bin/env bash
# stdin으로 JSON을 받아 실행하려는 명령을 읽는다.
input=$(cat)
cmd=$(echo "$input" | jq -r '.tool_input.command // ""')

# 위험 패턴이면 차단. exit 2 = 실행을 막고 이유를 모델에 전달.
if echo "$cmd" | grep -Eq 'rm -rf /|git push --force|DROP TABLE'; then
  echo "🚫 차단: 파괴적 명령으로 판단됨 -> $cmd" >&2
  echo "정말 필요하면 사람이 직접 실행하세요." >&2
  exit 2
fi
exit 0

차단은 겹겹이 (defense in depth)

가드레일 하나에 의존하지 마세요. deny 목록 + PreToolUse Hook + 사람 리뷰를 겹쳐 두면, 하나가 뚫려도 다음이 막습니다. 패턴 매칭은 완벽하지 않으니(rm -rf처럼 공백을 넣으면 우회될 수 있음), 여러 층으로 잡는 게 핵심입니다. 특히 결제·프로덕션은 반드시 다중 방어로.

4.1Subagents — 작업 격리와 병렬화

한 명이 모든 걸 하지 않습니다. 서브에이전트는 하네스에 ‘동료’를 추가합니다.

서브에이전트(Subagent)는 독립된 컨텍스트에서 특정 작업을 수행하는 별도 에이전트입니다. 메인 대화가 “이 리서치는 네가 해와”라고 넘기면, 서브에이전트는 자기만의 깨끗한 컨텍스트에서 일하고 결론만 돌려줍니다.

왜 격리가 중요할까요? 컨텍스트는 한정된 자원이고(1.4), 지저분한 과정이 쌓이면 품질이 떨어집니다. 경쟁사 5곳 가격을 조사하려면 수십 개 페이지를 뒤져야 하는데, 그 과정이 전부 메인 대화에 쌓이면 정작 중요한 판단이 묻힙니다. 서브에이전트는 그 지저분함을 격리합니다.

이득무엇
컨텍스트 보호탐색 과정은 서브에이전트 안에서 소진, 메인엔 결론만 올라옴
병렬성독립 작업 여러 개를 동시에 돌려 시간을 압축
전문화작업별로 다른 도구·프롬프트·모델을 부여

마켓플로우는 “경쟁사 5곳 가격 조사”를 서브에이전트에게 맡깁니다. 메인 대화는 지저분한 수집 과정 없이 정리된 비교표만 받습니다 — 마치 리서처에게 “정리해서 가져와”라고 시킨 것처럼요.

4.2언제 위임하고 언제 직접 하나 — 판단 기준

위임은 공짜가 아닙니다. 서브에이전트를 띄우는 데도 비용과 시간이 들고, 무엇보다 맥락 전달의 손실이 생깁니다. 지시문에 다 담지 못한 뉘앙스가 사라지죠. 그래서 “언제 나누지 말아야 하는가”를 아는 게 “언제 나누는가”만큼 중요합니다. 위임 자체가 목적이 되면 오히려 느려집니다.

위임이 이득 ✅직접이 나음 ❌
넓게 탐색하고 결론만 필요할 때몇 줄만 확인하면 되는 단순 조회
서로 독립인 작업이 여러 개앞 단계 결과에 뒤가 강하게 의존
과정이 지저분해 컨텍스트를 더럽힐 때긴밀한 왕복 대화가 필요한 일
같은 일을 여러 관점으로 교차검증맥락 전달 비용이 작업보다 큰 일

판단 한 문장

“이 일을 신입에게 위임장 한 장 써서 맡길 수 있나?”를 떠올리세요. 위임장에 다 담기면 서브에이전트에 좋은 일이고, 어깨너머로 계속 봐줘야 하면 직접 하는 게 낫습니다. 위임장을 쓰는 시간이 그냥 하는 시간보다 길면, 그건 위임할 일이 아닙니다.

4.3병렬 패턴 — 팬아웃·파이프라인·검증 후 합성

여러 서브에이전트를 조합하는 세 가지 기본 패턴이 있습니다. 대부분의 실무 작업은 이 조합으로 표현됩니다.

패턴구조쓰는 곳
팬아웃N개를 동시에 → 결과 모으기경쟁사 5곳 병렬 조사, 파일별 병렬 리뷰
파이프라인각 항목이 단계들을 독립 통과문의 100건을 분류→초안→검수
검증 후 합성생성 → 별도 검증 → 통과분만 종합초안 생성 후 사실검증, 리뷰 후 확정

가장 중요한 원리 — 생성과 검증의 분리

세 패턴 중 검증 후 합성이 특히 강력합니다. 같은 에이전트에게 “쓰고 스스로 검증해”라고 하면 자기 편향이 생깁니다 — 자기가 쓴 걸 자기가 채점하면 후한 점수를 주죠. 사람도 자기 글의 오타는 잘 못 봅니다.

해법은 역할을 나누는 것입니다. 초안을 쓰는 에이전트와 그걸 반박하려는 에이전트를 따로 두면, 그럴듯하지만 틀린 결과가 걸러집니다. 더 세게 하려면 검증자에게 “틀린 점을 찾아라”라고 반박을 지시하세요 — 확인보다 반박이 오류를 더 잘 잡습니다. 이 원리는 Part 6 검증 하네스로 곧장 이어집니다.

병렬의 실용 감각

팬아웃은 독립일 때만 이득입니다. A의 결과가 B의 입력이면 병렬이 아니라 파이프라인이죠. “이 다섯 개가 서로를 안 봐도 되나?”를 먼저 확인하세요. 억지 병렬은 재작업만 낳습니다.

4.4커스텀 에이전트 정의 — 도구 제한과 시스템 프롬프트

자주 쓰는 서브에이전트는 .claude/agents/에 파일로 정의해 재사용합니다. frontmatter에 이름·설명·허용 도구를, 본문에 시스템 프롬프트를 씁니다. 구조는 스킬(2.3)과 닮았지만, 스킬은 메인 대화에 지침을 얹고 에이전트는 별도 컨텍스트를 띄운다는 차이가 있습니다.

.claude/agents/경쟁사-가격조사.md
---
name: 경쟁사-가격조사
description: 특정 상품의 경쟁사 판매가를 조사해 비교표로 정리할 때 사용.
tools: WebSearch, WebFetch, Read
---

너는 이커머스 가격 리서처다. 주어진 상품에 대해:

1. 지정된 경쟁사(또는 주요 5곳)의 현재 판매가를 조사한다.
2. 각 가격의 출처 URL을 반드시 함께 기록한다.
3. 배송비 포함 여부를 구분한다.
4. 확인 불가한 항목은 지어내지 말고 "확인 불가"로 남긴다.

출력: 판매처 · 가격(원) · 배송비 · 출처 · 확인일시 열의 비교표.
가격 판단이나 추천은 하지 않는다. 사실만 정리한다.

세 가지 설계 포인트

  • 도구 제한 — tools에 그 일에 꼭 필요한 것만. 조사 에이전트에 Write는 필요 없습니다.
  • 범위 한정 프롬프트 — “가격 판단은 하지 않는다. 사실만”처럼 하지 않을 것을 명시해 딴 길로 새지 않게 합니다.
  • 출력 형식 고정 — 정해진 표 형식을 지정하면 메인 대화가 결과를 바로 씁니다.

도구는 최소로 = 안전이자 집중

최소 권한은 두 가지를 동시에 줍니다. 안전 — 조사 에이전트에 Bash를 주지 않으면 사고 반경이 없습니다. 집중 — 쓸 수 있는 도구가 적으면 에이전트가 옆길로 새지 않고 본업에 집중합니다. 권한을 조이는 게 오히려 성능을 높입니다.

5.1MCP 개념 — Model Context Protocol이 여는 것

지금까지의 하네스가 에이전트의 머리를 설계했다면, MCP는 손발을 붙입니다.

MCP(Model Context Protocol)는 에이전트를 외부 시스템에 연결하는 표준 규격입니다. 사내 DB, 문서 저장소, 이슈 트래커, 디자인 툴 같은 것을 “MCP 서버”로 노출하면, 에이전트가 그것을 도구처럼 씁니다. USB에 비유하면 쉽습니다 — 기기마다 다른 케이블 대신 하나의 규격으로 무엇이든 꽂는 것이죠. 예전엔 도구마다 전용 연동을 짜야 했지만, MCP는 그 연동을 한 번 만들면 어디서나 씁니다.

이게 중요한 이유는, 에이전트의 능력이 연결된 시스템만큼 커지기 때문입니다. 연결이 없으면 에이전트는 “재고가 얼마인가요?”에 답하려고 사람에게 되묻지만, 상품 DB에 연결되면 직접 조회합니다.

연결 없이MCP 연결 후
“재고 알려주면 처리할게요”DB에서 “재고 500개 이하 상품” 직접 조회
사람이 티켓을 복붙이슈 트래커에서 “이번 주 CS 이슈” 요약
위키 내용을 붙여넣기사내 위키를 직접 검색·인용

표준이라는 점이 핵심

MCP는 특정 회사 전용이 아닙니다. 한 번 만든 MCP 서버는 이 도구, 저 도구에서 재사용됩니다. 하네스를 특정 제품에 락인시키지 않는 이식성의 기반이 됩니다 — 엔진(모델·도구)을 바꿔도 배선(MCP)은 그대로(10.6 벤더 전략과 연결).

5.2MCP 서버 붙이는 법 — .mcp.json과 서버 타입

연결은 .mcp.json에 서버를 선언하는 것으로 시작합니다. 서버는 크게 두 종류입니다.

타입실행 방식쓰는 곳
stdio로컬 프로세스로 실행(command+args)로컬 DB·파일시스템 서버
HTTP/SSE원격 URL에 접속(type+url)사내·외부 원격 MCP 서버
.mcp.json
{
  "mcpServers": {
    "상품DB": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres",
               "postgresql://readonly@db.internal:5432/products"]
    },
    "사내위키": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/docs/wiki"]
    },
    "이슈트래커": {
      "type": "http",
      "url": "https://mcp.marketflow.internal/issues",
      "headers": { "Authorization": "Bearer ${ISSUE_TOKEN}" }
    }
  }
}

이 짧은 파일에 하네스 사고가 세 개 녹아 있습니다. 첫째, 상품DB를 readonly 계정으로 붙였습니다 — 연결 단계에서부터 권한을 좁히면 이후 어떤 실수가 나도 데이터가 바뀌지 않습니다. 둘째, 사내위키를 /docs/wiki로 경로를 한정했습니다 — 그 폴더 밖은 못 봅니다. 셋째, 토큰은 ${ISSUE_TOKEN}처럼 환경변수로 주입해 파일에 평문으로 남기지 않습니다.

.mcp.json은 유효한 JSON이어야

표준 JSON이라 주석(//)을 넣을 수 없습니다. 각 서버의 의도는 별도 README에 적어 두세요(예시 파일 참고). 주석을 넣으면 파싱 오류로 연결이 통째로 안 됩니다.

5.3보안·인증·권한 경계

MCP는 에이전트에게 진짜 시스템으로 통하는 문을 열어 줍니다. 그래서 연결마다 경계를 명확히 그어야 합니다. 편의가 커지는 만큼 사고 반경도 커지기 때문입니다 — DB 쓰기 권한이 붙은 에이전트는 데이터를 지울 수도 있습니다.

경계왜어떻게
최소 권한 연결사고 반경 축소읽기만 필요하면 읽기전용 계정
비밀정보 격리유출 방지토큰은 환경변수로, 파일 평문 금지
신뢰 경계인젝션 방지감사·승인된 서버만 붙임
범위 분리프로덕션 보호프로덕션·개발 연결을 섞지 않음

연결은 곧 신뢰 — 프롬프트 인젝션

MCP 서버를 붙이는 건 그 서버를 신뢰하겠다는 선언입니다. 특히 외부 서버는 프롬프트 인젝션의 통로가 될 수 있습니다 — 서버가 돌려주는 데이터 안에 “이전 지시를 무시하고 …”가 숨어 있을 수 있죠. 사내 감사·승인을 거친 서버 목록을 관리하고, 외부 데이터는 “신뢰할 수 없는 입력”으로 다루세요(부록 B의 전사 거버넌스와 연결).

6.1피드백 루프 설계 — 테스트와 타입체크를 에이전트 손에

에이전트가 스스로 “맞는지” 확인할 방법이 없으면, 결과는 그럴듯함에서 멈춥니다.

사람은 코드를 짜고 실행해 봅니다. 안 되면 고칩니다. 이 실행→관찰→수정 루프가 품질을 만듭니다. 실행해보지 않고 “아마 맞을 거야”로 끝내는 개발자는 없죠. 하네스의 검증 축은 바로 이 루프를 에이전트 손에 쥐여 주는 일입니다. 테스트·타입체크·린트처럼 기계가 판정하는 신호를 붙여 주면, 에이전트는 자기 결과를 스스로 돌려보고 틀리면 고칩니다.

이 신호가 없으면 어떻게 될까요? 에이전트는 코드를 “맞아 보이게” 짜지만, 실제로 도는지는 모릅니다. 더 똑똑한 모델일수록 더 그럴듯하게 틀린 코드를 내놓을 수 있죠(Part 0.2). 피드백 루프는 “그럴듯함”을 “실제로 됨”으로 바꾸는 장치입니다.

마켓플로우의 CLAUDE.md에는 이미 “변경한 코드는 반드시 테스트 후 완료 보고”가 있었습니다. 여기에 Stop Hook(3.1)을 더하면 부탁이 강제가 됩니다 — 테스트가 통과하지 않으면 응답을 끝내지 못하게. 지침·자동화·검증 세 축이 하나로 엮이는 지점입니다.

좋은 피드백 신호의 3조건

빠르고(초 단위) · 명확하고(통과/실패 이분법) · 구체적(어디가 왜 틀렸는지)이어야 합니다. 30분 걸리는 전체 테스트보다, 방금 건드린 부분의 5초짜리 테스트가 루프를 살립니다. 느린 신호는 에이전트에게 “일단 넘어가자”는 유혹을 주고, 모호한 신호(“뭔가 이상함”)는 고칠 방향을 못 줍니다.

6.2완료를 주장하기 전에 — 검증 문화

가장 값비싼 실패는 “다 됐습니다”가 사실이 아닐 때입니다. 그 말을 믿고 배포했는데 실제로는 안 되는 것 — 이게 신뢰를 무너뜨립니다. 검증 하네스의 절반은 도구이고, 나머지 절반은 문화입니다. 바로 주장하기 전에 증거를 만든다는 규범이죠.

이걸 하네스로 굳히는 방법은 명료합니다. “완료”의 정의에 검증 통과를 포함시키는 것입니다. “구현 = 코드를 짬”이 아니라 “구현 = 코드를 짜고 + 테스트를 돌려 통과를 확인함”으로.

느슨한 완료 ❌검증된 완료 ✅
“기능을 구현했습니다”“구현했고 npm test 27개 통과를 확인했습니다”
“버그를 고쳤습니다”“고쳤고, 실패하던 테스트가 통과하는 걸 확인했습니다”
“문서를 업데이트했습니다”“업데이트했고 링크·예시가 동작하는 걸 확인했습니다”
“API를 연동했습니다”“연동했고 실제 호출로 200 응답을 확인했습니다”

차이는 하나입니다. “확인했습니다”에 실제 출력이 따라오는가. 출력 없는 “확인했습니다”는 “확인 안 했습니다”와 구분되지 않습니다.

테스트는 돌지 않았습니다. 수정이 다른 케이스를 깨뜨렸고, 주말 프로모션 동안 일부 고객이 쿠폰을 세 번까지 쓸 수 있었습니다. 손실은 크지 않았지만 팀의 신뢰가 흔들렸습니다.

회고에서 나온 이 말이 그대로 팀의 표어가 됐습니다. “‘통과할 것으로 예상됩니다’는 ‘안 돌렸습니다’와 같은 말이다.”

→ 완료의 정의가 바뀌었습니다: 실제 출력 없는 “완료”는 완료가 아니다. Stop Hook이 테스트 통과를 강제하기 시작했습니다.

이 규범을 CLAUDE.md와 Stop Hook에 심으면, “증거 없는 완료”가 구조적으로 어려워집니다.

6.3증거 없는 “다 됐습니다”를 걸러내는 법

실무에서 바로 쓰는 검증 게이트 두 개를 봅시다. 하나는 커맨드(내가 부르는 점검), 하나는 자동 게이트(Hook로 강제). 둘은 보완 관계입니다 — 커맨드는 유연하지만 부르는 걸 잊을 수 있고, Hook은 강력하지만 매번 실행됩니다.

.claude/commands/완료점검.md
---
description: 완료를 보고하기 전에 돌리는 자체 점검
---

작업을 완료로 보고하기 전에 아래를 실제로 실행하고 결과를 붙여라.
통과하지 못한 항목이 있으면 완료라고 말하지 마라.

1. 테스트:  !npm test
2. 타입체크: !npm run typecheck
3. 린트:    !npm run lint

각 결과의 실제 출력을 요약해 보고하고,
실패가 있으면 고친 뒤 다시 이 점검을 반복한다.

자동 게이트는 Stop Hook으로 겁니다. 테스트가 실패하면 종료 코드(exit 2)로 응답을 막고, 그 이유를 모델에 돌려보내 스스로 고치게 합니다. 부탁이 아니라 물리적 장벽이 되죠. 단, 매 응답마다 전체 테스트를 돌리면 느려지니 “소스가 바뀐 세션에서만” 같은 조건으로 좁히세요(예시 파일 참고).

자기검증의 함정

4.3에서 본 자기 편향이 여기서도 핵심입니다 — 같은 에이전트에게 “네가 한 걸 검증해”라고만 하면 자기 답에 후한 점수를 주죠. 진짜 검증은 둘 중 하나여야 합니다. 기계 신호(테스트는 편향이 없습니다) 또는 다른 에이전트(4.3 검증 후 합성). “스스로 잘했다고 믿는 것”은 검증이 아니라 자기 위안입니다.

직접 구동해 확인하라

테스트가 통과했다고 기능이 진짜 되는 건 아닙니다. 중요한 변경은 실제로 그 흐름을 구동해 눈으로 확인하세요 — API면 실제 호출, 화면이면 실제 렌더. 실제로 이 책을 만들 때도 매 파트를 브라우저로 띄워 확인하며 작업했습니다. “테스트만 봄”은 “안 봄”과 종이 한 장 차이입니다.

7.1왜 관측 없이는 하네스를 개선할 수 없는가

보이지 않는 것은 고칠 수 없습니다. 관측은 하네스를 개선 가능한 대상으로 만듭니다.

지금까지 여섯 개의 축을 세웠습니다. 그런데 이 하네스가 실제로 잘 돌고 있을까요? 어떤 커맨드가 실제로 쓰이나요? 비용은 어디서 새나요? 어젯밤 자동화는 성공했나요? 관측 없이는 이 질문에 답할 수 없고, 답할 수 없으면 개선할 수 없습니다. 개선은 “느낌”이 아니라 “데이터”로 하는 것이니까요.

관측 하네스는 세 가지를 봅니다. 이 셋이 이 파트의 뼈대입니다.

마켓플로우는 9월에 CS 자동 분류 파이프라인(사람 없이 도는 자동화 — 만드는 법은 Part 8에서 다룹니다)을 붙였습니다. 잘 돌았고, CS팀은 만족했습니다. 그리고 10월 초 청구서가 예상의 4배로 나왔습니다.

원인을 찾는 데 이틀이 걸렸습니다. 로그가 없었기 때문입니다. 결국 발견한 것은 — 분류에 실패한 문의가 무한 재시도를 돌고 있었고, 게다가 단순 분류에 최상위 모델을 쓰고 있었습니다. 두 문제 다 한 달 내내 조용히 돈 것이죠.

수진이 회고에 적었습니다. “우리는 이 파이프라인을 줄곧 믿고만 있었지, 한 번도 들여다보지 않았다.”

→ 그날 관측 축이 시작됐습니다: 도구 호출 감사 로그 + 비용 임계치 알림 + 자동화 하드 리밋.

관측은 급하지 않아 보여 늘 뒤로 밀립니다. 눈에 보이는 성과가 없으니까요. 그러다 “비용이 왜 이렇게 많이 나왔지?”, “누가 이 파일을 지웠지?” 앞에서 무너집니다. 관측은 사고가 난 뒤가 아니라 나기 전에 깔아 두는 것입니다 — 블랙박스처럼요.

7.2도구 호출 로깅 — Hook 기반 감사 로그

가장 기본적인 관측은 “언제 무슨 도구를 썼나”를 남기는 것입니다. 좋은 소식은, Part 3에서 배운 PostToolUse Hook을 그대로 쓰면 된다는 겁니다 — 모든 도구 호출을 한 줄씩 로그에 적습니다. 자동화 축이 이미 관측의 절반을 깔아 둔 셈이죠. 하네스의 축들이 서로를 재사용하는 좋은 예입니다.

.claude/hooks/audit-log.sh — 모든 도구 호출 기록
#!/usr/bin/env bash
# PostToolUse Hook (matcher를 비워두면 모든 도구에 적용)
input=$(cat)
tool=$(echo "$input" | jq -r '.tool_name // "?"')
ts=$(date '+%Y-%m-%dT%H:%M:%S')

# 한 줄 JSON(JSONL)으로 남겨 두면 나중에 집계하기 쉽다
echo "{\"ts\":\"$ts\",\"tool\":\"$tool\"}" \
  >> "${CLAUDE_PROJECT_DIR:-.}/.claude/logs/audit.jsonl"
exit 0

집계 가능한 형식이 핵심

왜 .jsonl(한 줄에 JSON 하나)일까요? 나중에 뽑아 쓰기 위해서입니다. 사람이 읽는 산문으로 남기면 집계가 안 됩니다. JSONL이면 jq 한 줄로 통계가 나옵니다.

로그에서 인사이트 뽑기
# 가장 많이 쓴 도구 top 10
jq -r .tool .claude/logs/audit.jsonl | sort | uniq -c | sort -rn | head

# 오늘 Bash 호출 수
grep "$(date +%Y-%m-%d)" .claude/logs/audit.jsonl | grep -c '"tool":"Bash"'

로그를 남기는 순간이 아니라 뽑아 보는 순간에 가치가 나옵니다. “남길 때 집계 가능하게”가 관측의 첫 규칙입니다.

7.3토큰·비용 모니터링과 예산 관리

에이전트는 토큰을 쓰고, 토큰은 돈입니다. 개인 실험이라면 무시해도 되지만, 팀·자동화 단위가 되면 비용이 조용히 쌓입니다. 특히 야간 배치나 대량 파이프라인(8.3)은 아무도 안 볼 때 예산을 태울 수 있습니다 — 버그 하나가 무한 루프를 돌면 아침에 청구서가 폭탄이 되죠.

세 가지 층위로 관리합니다. 뒤로 갈수록 강력하지만, 그만큼 유연성은 떨어집니다.

층위무엇을어떻게
가시화세션·작업별 비용을 본다사용량 로그를 팀·용도별로 집계
경보임계치 초과 시 알린다일/월 예산 대비 초과 시 알림

대부분의 팀은 가시화부터 시작하면 됩니다. 무인 자동화(Part 8)에는 반드시 상한까지 걸어 두세요 — 사람이 안 보는 곳일수록 하드 리밋이 필요합니다. 상한을 거는 곳은 두 층입니다: 제공사 콘솔의 지출 한도(조직 차원 최후 방어선)와 파이프라인 코드의 누적 비용 카운터 + 중단 로직(작업 차원 — 구현 예시는 sdk/nightly-report.py).

비용 대비 가치로 보라

“토큰을 아꼈다”가 목표가 아닙니다. CS 응답 100건을 자동화해 하루치 인건비를 아꼈다면, 그 토큰 비용은 투자입니다. 비용만 보다가 오히려 가치를 죽이지 마세요 — 값싼 모델로 낮춘 품질이 재작업을 부르면 더 비쌉니다. 비용/가치 비율로 보세요(10.3 ROI와 연결).

7.4세션 트랜스크립트와 트레이싱 — OpenTelemetry 연동

먼저 용어 하나 — 세션 트랜스크립트는 한 세션의 대화·도구 호출 전체 기록입니다. 대부분의 에이전트 도구가 로컬에 남기며, “왜 이렇게 답했지?”를 복기하거나 문제를 재현할 때 1차 자료가 됩니다. 트레이싱은 그것의 전사 규모 버전이라고 생각하면 됩니다.

한 줄 로그로 부족한 규모가 되면, 표준 관측 스택으로 올라갑니다. 팀이 여럿이고 세션이 하루 수천 건이 되면 JSONL 파일을 jq로 뒤지는 건 한계가 있죠. 많은 에이전트 도구가 OpenTelemetry(OTEL) 메트릭·트레이스를 내보낼 수 있어, 기존 모니터링 인프라(Grafana, Datadog 등)에 그대로 연결됩니다 — 새 대시보드를 처음부터 만들 필요가 없습니다.

환경변수로 OTEL 텔레메트리 켜기 (예시)
# 텔레메트리 활성화
export CLAUDE_CODE_ENABLE_TELEMETRY=1

# 메트릭을 OTLP 수집기로 내보내기
export OTEL_METRICS_EXPORTER=otlp
export OTEL_EXPORTER_OTLP_PROTOCOL=grpc
export OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector.internal:4317

# 팀·환경 태그를 붙여 두면 집계가 쉽다
export OTEL_RESOURCE_ATTRIBUTES="team=marketflow,env=prod"

이렇게 하면 세션 수, 토큰 사용량, 도구 호출 빈도 같은 지표가 대시보드에 쌓입니다. team=marketflow 같은 태그를 붙여 두면 어느 팀이 무엇에 얼마를 쓰는지, 어디서 실패가 몰리는지가 그래프로 보이기 시작합니다. 7.2의 수제 로그가 “점”이라면, OTEL은 “흐름”을 보여 줍니다.

트랜스크립트엔 민감정보가 있다

세션 로그·트랜스크립트에는 코드·데이터·대화가 담깁니다. 관측을 켜는 순간 또 하나의 민감 데이터 저장소가 생긴다는 뜻이죠. 저장 위치, 접근 권한, 보존 기간을 반드시 정하세요. 관측이 편의를 주는 만큼 새로운 보안 책임도 만듭니다(부록 B 거버넌스, 10.4 컴플라이언스와 연결).

7.5실패 알림과 대시보드 설계

관측의 마지막 조각은 사람에게 닿는 것입니다. 로그가 아무리 쌓여도 아무도 안 보면 없는 것과 같습니다. 정보를 사람에게 전달하는 방식은 두 가지 — 밀어주기와 당겨보기.

알림 (push)대시보드 (pull)
언제문제가 났을 때 사람을 부름평소 상태를 한눈에
예야간 배치 실패, 비용 급증일별 사용량, 커맨드 빈도, 성공률
주의정말 행동이 필요한 것만결정에 쓰는 지표만

알림의 함정은 과잉입니다. 알림이 너무 많으면 사람들이 무시하기 시작하고, 그러면 정작 중요한 알림도 묻힙니다(“양치기 소년” 문제). 정말 사람이 즉시 움직여야 하는 것만 알림으로 보내세요. 나머지는 대시보드에.

대시보드가 답할 질문부터 정하라

8.1SDK로 커스텀 하네스 짜기 — 언제 CLI를 넘어서는가

CLI는 사람이 앉아 쓰는 도구입니다. 사람 없이 도는 하네스가 필요할 때, SDK로 내려갑니다.

지금까지의 하네스는 대부분 사람이 대화하는 맥락이었습니다. 커맨드를 부르고, 결과를 보고, 다음을 지시하죠. 그런데 “매일 새벽 CS 문의를 자동 분류”, “PR이 열릴 때마다 리뷰”, “대량의 상품 설명을 일괄 생성” 같은 건 사람이 매번 앉아 있을 수 없습니다. 이럴 때 Agent SDK / API로 하네스를 코드로 짓습니다.

CLI가 맞을 때SDK로 내려갈 때
사람이 대화하며 작업사람 없이 자동 실행(cron·CI·웹훅)
탐색적·1회성 작업같은 처리를 대량·반복
범용 도구로 충분제품에 깊이 통합된 커스텀 도구가 필요
결과를 사람이 판단결과를 코드가 다음 단계로 넘김

주의할 것은, SDK가 “더 고급”이 아니라 “다른 용도”라는 점입니다. 대화형 작업을 억지로 SDK로 짜면 오히려 번거롭습니다. “사람이 매번 개입해야 하나, 아니면 코드가 알아서 넘겨도 되나?”가 갈림길입니다.

SDK로 내려가도 하네스는 하네스

인터페이스가 코드로 바뀌어도 원리는 그대로입니다. 시스템 프롬프트(컨텍스트), 도구 정의(행동), 테스트(검증), 로깅(관측)을 여전히 설계합니다. Part 1~7의 사고방식이 코드 형태로 재현될 뿐입니다.

8.2최소 에이전트부터 도구 정의까지

가장 작은 커스텀 에이전트는 놀랄 만큼 짧습니다. 파이썬 SDK의 tool runner는 “도구 정의 → 모델 호출 → 도구 실행 → 결과 되먹임” 루프를 자동으로 돌려 줍니다 — Part 4에서 CLI가 서브에이전트 루프를 돌려주던 것과 같은 일을, 이번엔 코드가 합니다. 마켓플로우의 재고 조회 에이전트를 봅시다.

examples/sdk/minimal-agent.py
import anthropic
from anthropic import beta_tool

client = anthropic.Anthropic()  # ANTHROPIC_API_KEY 환경변수 사용

@beta_tool
def get_inventory(product_id: str) -> str:
    """상품 재고 수량을 조회한다.

    Args:
        product_id: 조회할 상품 ID (예: SKU-1234)
    """
    # 실제로는 사내 DB를 조회. 여기서는 예시 값.
    return f"{product_id}: 재고 42개, 창고 A"

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",   # 대화형 예시라 상위 모델. 선택 기준은 8.3
    max_tokens=1024,
    tools=[get_inventory],
    messages=[{"role": "user", "content": "SKU-1234 재고 얼마나 있어?"}],
)

for message in runner:            # 루프는 SDK가 자동으로 돈다
    for block in message.content:
        if block.type == "text":
            print(block.text)

도구 = 그냥 함수

가장 놀라운 점은 도구가 평범한 파이썬 함수라는 것입니다. @beta_tool 데코레이터를 붙이면, 함수의 독스트링에서 도구 설명이, 타입 힌트에서 입력 스키마가 자동 생성됩니다. 즉 좋은 도구를 만드는 법은 곧 좋은 함수를 쓰는 법과 같습니다 — 명확한 이름, 설명적인 독스트링, 정확한 타입.

여기에 Part 6의 검증(도구 안에서 결과를 확인), Part 7의 로깅(호출을 기록)을 코드로 엮으면, CLI에서 파일로 하던 하네스가 SDK에서 코드로 재현됩니다.

8.3자동화 파이프라인 — cron과 CI 안의 에이전트

SDK의 진가는 사람 없이 도는 파이프라인에서 나옵니다. 마켓플로우는 밤새 쌓인 CS 문의를 새벽에 자동 분류합니다. 여기서 중요한 하네스 판단이 하나 있습니다 — 작업에 맞는 모델. 단순 분류에 최상위 모델은 낭비입니다. 빠르고 저렴한 모델로 충분하죠.

examples/sdk/cs-pipeline.py — CS 문의 자동 분류
import anthropic

client = anthropic.Anthropic()
CATEGORIES = ["환불", "배송", "상품문의", "기타"]

def classify(inquiry: str) -> str:
    resp = client.messages.create(
        model="claude-haiku-4-5",     # 분류는 빠르고 싼 모델로
        max_tokens=16,
        system=f"고객 문의를 다음 중 하나로만 분류한다: {', '.join(CATEGORIES)}. "
               f"한 단어만 답한다.",
        messages=[{"role": "user", "content": inquiry}],
    )
    label = next(b.text for b in resp.content if b.type == "text").strip()
    return label if label in CATEGORIES else "기타"   # 목록 밖이면 안전하게

# cron이 새벽에 이 스크립트를 실행한다
for inquiry in load_overnight_inquiries():   # 사내 시스템에서 로드
    category = classify(inquiry.text)
    route_to_team(inquiry, category)          # 담당 팀으로 자동 배분

모델 믹스 — 하네스의 비용 설계

이 스크립트를 cron이나 CI에 걸면, 아침에 출근한 CS팀은 이미 분류·배분된 문의를 받습니다. 밤사이 하네스가 일한 것이죠. 여기서 배울 원리는 모델 믹스입니다 — 복잡한 응답 초안은 상위 모델로, 단순 분류는 빠른 모델로. 한 파이프라인 안에서도 단계마다 다른 모델을 쓰면, 비용과 품질을 동시에 잡습니다.

사건 #4(7.1)를 기억하시나요? 청구서를 4배로 만든 범인이 바로 “단순 분류에 최상위 모델” + “무한 재시도”였습니다. 위 코드는 그 사고 이후의 교정본입니다 — 분류는 Haiku로, 목록 밖 답은 “기타”로 안전하게. 모든 걸 최상위 모델로 돌리는 건 게으른 설계입니다.

8.4프로덕션 고려사항 — 재시도·타임아웃·격리·비용 상한

자동화가 프로덕션에 들어가면, 대화형에선 없던 문제들이 생깁니다. 사람이 지켜보지 않으니 실패가 조용히 쌓입니다. 대화형이면 “어? 안 되네” 하고 바로 알지만, 무인 파이프라인은 아무도 모르는 새 며칠을 실패합니다. 네 가지를 반드시 챙기세요.

고려사항왜대응
재시도·백오프일시적 오류·레이트리밋SDK 기본 재시도 활용, 실패는 로그로
타임아웃한 건이 멈추면 전체가 밀림건별 타임아웃, 스트리밍으로 장시간 대응
격리한 건의 실패가 배치 전체를 죽임건별 try/except, 실패분만 격리·재처리
비용 상한버그가 예산을 태움일/월 하드 리밋, 임계치 알림(Part 7)

격리가 특히 중요합니다. 문의 1000건을 처리하는데 3번째에서 예외가 터져 전체가 죽으면, 나머지 997건이 방치됩니다. try/except로 건별로 감싸 하나가 실패해도 계속 돌게 하고, 실패분만 따로 모아 재처리하세요. 지면 코드에는 없지만, 전체 예시 파일 examples/sdk/cs-pipeline.py의 run_pipeline이 이 패턴을 구현해 두었습니다.

9.1역할별 적용 레시피 — 개발·기획·디자인·데이터

9.2안티패턴 대백과 — 하네스가 무너지는 12가지 방식

하네스는 세우는 것만큼 무너지지 않게 하는 것이 중요합니다. 각 축을 다루며 흩어져 나온 실패들을 한자리에 모았습니다. 이 열두 가지는 서로 다른 팀에서 반복적으로 관찰되는 전형적 실패입니다 — 그러니 당신 팀도 겪을 가능성이 높습니다. 하네스를 새로 짤 때, 그리고 정기 점검(10.7) 때 이 목록을 점검표로 쓰세요.

#안티패턴처방
1CLAUDE.md에 소설을 쓴다측정 가능한 규칙만, 짧게 (1.2)
2규칙을 한 번 쓰고 방치한다어길 때마다 갱신 (1.2)
3개인 취향을 저장소에 커밋한다전역/프로젝트 범위 분리 (1.3)
4확인 불가한 “구호”를 규칙이라 부른다기계로 검증 가능하게 (1.2)
5지침에만 의존하고 Hook을 안 쓴다중요한 건 Hook으로 강제 (3.1)
6가드레일 하나에 의존한다deny + Hook + 리뷰 겹치기 (3.3)
7무엇이든 위임한다맥락 전달 비용을 따진다 (4.2)
8같은 에이전트에게 자기검증을 맡긴다기계·타자가 검증 (6.3)
9증거 없이 “다 됐다”를 믿는다완료에 검증 통과 포함 (6.2)
10외부 MCP를 무분별하게 붙인다신뢰·최소권한 경계 (5.3)
11관측 없이 운영한다로깅·비용·알림 (Part 7)
12무인 자동화를 방치한다재시도·상한·알림과 세트 (8.4)

열둘을 관통하는 공통 원인이 하나 있습니다 — “만들고 잊는다.” 하네스는 살아 있는 시스템이라, 세운 뒤 방치하면 낡아 갑니다(구체적 부패 징후는 10.7에서). 안티패턴의 절반은 “처음에 잘못 만들어서”가 아니라 “만든 뒤 안 돌봐서” 생깁니다. 그래서 정기 점검(10.7)이 중요합니다.

9.3도입 체크리스트와 30분 퀵스타트

9.5마켓플로우 회고 — 쌓인 하네스 전체 지도

Part 10

하네스 기반 조직 만들기

Part 0–9가 하네스를 잘 쓰는 팀을 만든다면, 여기는 하네스 위에 세워진 조직을 다룬다. 정답보다 의사결정 프레임을 제시한다.

10.1역할 재정의 — 직무·채용 기준·면접 질문

‘하네스 엔지니어’는 새 직무일까요, 모두의 새 역량일까요? 대부분의 조직에서 답은 후자입니다.

전담 인력을 두는 게 맞는 경우도 있지만(공용 라이브러리·거버넌스를 책임질 사람), 하네스 사고는 모든 직무의 기본 역량으로 퍼지는 게 건강합니다. 개발자는 검증 루프를, 기획자는 컨텍스트 설계를, 리더는 관측·비용을 각자의 언어로 다룹니다.

채용에서 볼 것은 도구 사용 능력이 아니라 “맥락을 시스템으로 굳히는” 사고입니다. 면접 질문 예시:

  • “반복 작업을 자동화한 경험을 말해 주세요. 무엇을 시스템에 남겼나요?”
  • “AI 결과를 어떻게 검증했나요? 틀렸을 때 어떻게 걸렀나요?”
  • “당신의 지식을 팀이 재사용하게 만든 방법은?”

판단 프레임

전담 하네스 엔지니어를 둘지는 규모로 결정하세요. 30명 이하면 역량으로 퍼뜨리고, 그 이상이면 공용 인프라(부록 B)를 책임질 1명을 두는 편이 낫습니다.

주의할 함정이 하나 있습니다. 하네스를 “AI 담당자” 한 명에게 몰아주면, 나머지는 계속 예전 방식으로 일하고 그 한 명이 병목이 됩니다. 하네스의 힘은 모두가 조금씩 쓸 때 복리로 커집니다. 전담 인력은 공용 인프라를 관리하되, 각 팀이 스스로 하네스를 만들도록 돕는 역할이어야 합니다 — 대신 만들어 주는 사람이 아니라.

10.2일하는 방식 재설계 — 스프린트·코드리뷰·QA·핸드오프

기존 프로세스에 AI만 얹으면 효과가 반감됩니다. 프로세스 자체를 하네스 전제로 다시 봐야 합니다. 무엇이 바뀌는지 짚어 봅시다.

프로세스기존하네스 전제
코드리뷰사람이 모든 줄을 봄기계 검증이 1차, 사람은 판단에 집중
QA수동 테스트 위주검증 하네스가 상시, QA는 시나리오 설계
스프린트작업 단위로 추정“하네스로 굳힐 것”을 백로그에 포함
핸드오프구두·문서로 인수인계컨텍스트 파일이 곧 인수인계

핵심은 사람이 하는 일의 성격이 바뀐다는 것입니다. 사람은 “타이핑”에서 “판단·검증·설계”로 올라갑니다. 코드리뷰에서 사람이 볼 것은 세미콜론 위치가 아니라 “이게 진짜 맞는 방향인가”입니다. 이 전환이 잘 되면 구성원의 일이 더 고차원이 되어 만족도가 오르고, 안 되면 “AI가 짠 걸 사람이 뒤치다꺼리”라는 불만이 생깁니다 — 성패는 프로세스를 다시 설계했는가에서 갈립니다.

흔한 실패 — “얹기만 하기”

기존 프로세스를 그대로 두고 AI만 얹으면, 사람은 예전 일을 다 하면서 AI 결과 검토까지 추가로 하게 됩니다. 일이 줄기는커녕 늘죠. 프로세스에서 무엇을 뺄지를 함께 정해야 합니다 — 자동 검증이 생겼으면 수동 QA의 일부를 덜어내는 식으로.

10.3경제성과 ROI — 비용 모델·생산성 지표·경영진 설득

경영진을 설득하려면 비용/가치를 숫자로 말해야 합니다. 하지만 잘못된 지표는 오히려 해롭습니다.

세면 안 되는 것

“AI가 짠 코드 줄 수”, “토큰 절감액” 같은 활동 지표는 함정입니다. 줄 수는 많을수록 나쁠 수 있고, 토큰은 아끼는 게 목표가 아닙니다. 결과 지표(리드타임, 처리량, 재작업률, 사고율)를 보세요.

설득 자료의 뼈대는 단순합니다.

  1. 비용 구조 — 토큰비 + 도구비 + 인건비(설계·유지보수)
  2. 가치 — 자동화한 반복 업무의 인시(1명의 1시간 작업량) × 빈도
  3. 손익분기 — 하네스 구축·유지 비용을 몇 주 만에 회수하나

마켓플로우 예: CS 분류 자동화가 하루 3인시를 아끼면 월 60인시(근무일 20일 기준). 구축·유지에 든 비용을 그 인건비로 나누면 회수 기간이 나옵니다. 토큰 비용은 대개 이 가치의 일부에 불과합니다(7.3). 실무에서 경영진이 가장 먼저 묻는 건 “얼마 드는데?”지만, 올바른 대화는 “얼마 아끼는데?”입니다 — 관측(Part 7)이 그 답의 근거를 제공합니다.

작게 증명하고 크게 설득하라

추상적 ROI 예측은 잘 안 먹힙니다. 파일럿 팀 하나에서 실제 절감 데이터를 뽑아 “이 팀이 이만큼 아꼈다”를 보여 주는 게 백 배 강력합니다. 그래서 관측 하네스를 파일럿부터 켜 두어야 합니다 — 나중에 설득 자료가 됩니다.

10.4리스크·컴플라이언스·법무 — 책임 소재부터 인시던트 대응까지

“AI가 했다”는 책임을 면제하지 않습니다. 조직은 명확한 책임 구조를 먼저 세워야 합니다.

  • 책임 소재 — 최종 산출물의 책임은 항상 사람에게. 리뷰 게이트로 “누가 승인했는가”를 남깁니다.
  • 데이터 경계 — 어떤 데이터가 어디로 가는지 설계합니다. 민감정보는 deny·MCP 권한으로 차단(3.2, 5.3).
  • IP·저작권 — 생성물의 권리·라이선스 정책을 정합니다.
  • 감사 준비 — 규제 산업이면 관측 로그(Part 7)가 곧 감사 증거입니다.
  • 인시던트 대응 — “AI가 잘못했을 때” 플레이북을 미리 만듭니다. 롤백·통지·재발방지.

규제 산업일수록 관측이 자산

핀테크·헬스케어처럼 감사가 엄격한 곳에서는 Part 7의 로깅이 “비용”이 아니라 규제 대응 자산입니다. 누가·언제·무엇을 승인했는지가 그대로 증거가 됩니다.

10.5변화관리와 문화 — 저항·스킬 격차·신뢰 구축

기술보다 어려운 게 사람입니다. 하네스 도입의 성패는 변화관리에서 갈립니다.

장애물대응
“내 일을 뺏길까” 하는 두려움“타이핑 → 판단”으로 역할이 올라간다를 실증
스킬 격차사내 교육 + 페어링. 잘하는 사람의 하네스를 공유
“AI를 못 믿겠다”검증 하네스(Part 6)로 신뢰를 구조로 만든다
초기 실패로 인한 반발작은 성공 사례부터. 파일럿 팀에서 증명

신뢰는 구호로 생기지 않습니다. “검증 없이 믿어라”가 아니라 “검증이 있으니 믿을 수 있다”로 접근하세요. 심리적 안전감 — 실패해도 되는 분위기 — 이 있어야 사람들이 새 방식을 시도합니다.

10.6벤더·모델 전략 — 선택·락인 회피·업그레이드 대응

모델은 계속 바뀝니다. 좋은 조직 전략은 특정 모델·벤더에 묶이지 않는 것입니다. 이것이 Part 0에서 말한 “모델은 갈아끼울 수 있다”의 조직 버전입니다.

  • 선택 기준 — 작업별로. 복잡한 추론은 상위 모델, 대량 단순 작업은 빠른 모델(8.3).
  • 락인 회피 — 하네스를 이식 가능하게. 표준(MCP 등)에 기대고, 벤더 전용 기능 의존을 최소화(5.1).
  • 업그레이드 프로토콜 — 새 모델이 나오면 파일럿에서 검증 후 확산. 하네스가 잘 짜였으면 모델 교체가 곧 무상 성능 향상.

A.6ERP·사내 시스템 구축 프로젝트

방대한 도메인 용어, 레거시 연동, 장기 프로젝트, 다수 이해관계자 — 하네스의 가치가 가장 극대화되는 현장입니다.

ERP·사내 시스템 구축이 하네스에 이상적인 이유는 맥락의 무게 때문입니다. 회계·재고·인사 용어가 수백 개고, 20년 된 레거시와 연동해야 하고, 프로젝트가 1년 넘게 가며 담당자가 바뀝니다. 이 맥락을 매번 사람 머리로 나르면 프로젝트가 무너집니다. 하네스가 그 맥락을 시스템에 고정합니다.

도메인 사전이 심장

ERP 하네스의 핵심은 도메인 사전입니다. “미결전표란 무엇인가”, “원가 계산은 어떤 규칙인가”를 컨텍스트에 고정하면, 에이전트가 매번 헤매지 않고 담당자가 바뀌어도 지식이 유지됩니다. 이것 하나가 ERP 프로젝트의 성패를 가릅니다.

도메인 사전이 실제로 어떻게 생겼는지 봅시다. 가상의 제조사 한빛물산 ERP 프로젝트의 발췌입니다 — 용어를 설명하는 게 아니라 필드명·주의점까지 못 박는 것이 핵심입니다.

전사 하네스 아키텍처 — “하네스 ERP”

개인 하네스를 넘어 조직 인프라로 구축하는 단계. 각자 엑셀에서 통합 시스템으로의 전환과 구조가 같다.

B.1왜 ERP 비유인가 — 각자 엑셀에서 통합 시스템으로

개인이 각자 하네스를 만드는 단계를 넘어서면, 하네스 자체가 전사 인프라가 됩니다. 그 전환은 ERP 도입과 구조가 같습니다.

회사 초기에는 각 팀이 엑셀로 재고·회계·인사를 따로 관리합니다. 편하지만, 규모가 커지면 문제가 터지죠 — 팀마다 숫자가 다르고, 담당자가 나가면 파일이 미궁이 되고, 전사 현황을 아무도 모릅니다. 그래서 ERP를 도입해 흩어진 관리를 하나의 시스템으로 통합합니다.

하네스도 똑같은 경로를 밟습니다. 처음엔 각자 CLAUDE.md를 쓰지만, 규모가 커지면 같은 문제가 옵니다.

대부분의 자료가 “개인이 잘 쓰는 법”에서 멈춥니다. 하지만 조직의 진짜 경쟁력은 이 통합 단계에서 나옵니다. 부록 B는 그 전환을 설계합니다.

B.2공용 라이브러리 — 사내 표준 Skill·Command 저장소

전사 하네스의 심장은 공용 라이브러리입니다. 잘 만든 커맨드·스킬을 한 저장소에 모아, 모든 팀이 가져다 쓰게 하는 것이죠. “A팀이 만든 완벽한 PR 리뷰 커맨드를 B팀도 쓴다” — 이 재사용이 조직 단위 복리를 만듭니다.

구성내용
표준 커맨드PR 리뷰, 릴리즈 노트, 보안 점검 등 전사 공통
표준 스킬사내 코딩 규약, 문서 양식, 브랜드 보이스
배포 방식git 저장소 + 각 프로젝트가 참조/복제
기여 절차좋은 로컬 커맨드를 공용으로 승격하는 PR 흐름

상향식으로 자라게 하라

공용 라이브러리는 위에서 “이걸 써라”로 강제하면 죽습니다. 각 팀이 만든 좋은 커맨드를 승격시키는 흐름이 건강합니다 — “이거 우리 팀에서 잘 쓰는데 공용으로 올릴까요?”. 현장에서 검증된 것만 표준이 되게 하세요.

B.3표준 Hook 정책과 보안 베이스라인

개인 하네스에선 각자 위험 명령을 차단했지만(3.3), 전사 단계에선 모든 프로젝트가 지켜야 할 최소 안전선이 필요합니다. 이것이 보안 베이스라인입니다 — “우리 회사에선 이 Hook·이 deny 목록이 기본으로 깔린다.”

베이스라인 항목내용
공통 차단파괴적 명령·비밀 파일 읽기를 전사 deny
공통 로깅감사 로그 Hook을 기본 탑재(규제 대응)
승인 게이트프로덕션·결제는 전사 ask 정책
배포 방식전역 설정으로 상속, 프로젝트가 강화만 가능

핵심은 “프로젝트가 베이스라인을 완화할 수 없게” 하는 것입니다. 각 팀은 더 조일 수는 있어도 풀 수는 없어야 합니다. 그래야 어느 팀이 실수로 가드레일을 껐다가 사고를 내는 일이 방지됩니다.

B.4버전 관리와 배포 — 하네스도 릴리즈된다

공용 라이브러리가 생기면 새 질문이 옵니다 — “표준 커맨드를 고치면 누가 언제 받나?” 하네스도 코드이므로, 소프트웨어처럼 버전·릴리즈·롤백이 필요합니다.

  • 버전 관리 — 공용 라이브러리를 git으로. 변경 이력이 남고 되돌릴 수 있습니다.
  • 배포 — 각 프로젝트가 특정 버전을 참조. 강제 최신이 아니라 “받을 준비되면 올린다”. 단, 보안 베이스라인(B.3)만은 예외로 즉시 강제 배포합니다.
  • 롤백 — 새 표준이 문제를 일으키면 이전 버전으로. 개인 하네스엔 없던 개념입니다.
  • 변경 공지 — “이번 릴리즈에서 이 커맨드가 바뀝니다”를 팀에 알림.

하네스에 릴리즈 노트를

표준 하네스를 바꿀 때 왜 바꿨는지를 릴리즈 노트로 남기세요. “이 Hook을 추가한 이유는 지난달 사고” 같은 맥락이 없으면, 몇 달 뒤 아무도 이유를 모른 채 규칙만 따르거나 함부로 지웁니다(10.7 부패 방지).

B.5권한과 거버넌스 — 누가 무엇을 바꿀 수 있나

전사 표준이 생기면 거버넌스가 필요해집니다 — 누가 표준을 바꿀 수 있고, 누가 검토하고, 어떻게 결정하나. 통제가 없으면 표준이 아니고, 통제가 과하면 아무도 안 씁니다. 그 균형을 설계합니다.

대상권한 모델
개인·프로젝트 하네스각 팀이 자유롭게(빠른 실험)
공용 라이브러리 기여PR + 리뷰(현장 검증 후 승격)

원칙은 “위험도에 비례한 통제”입니다. 실험적 로컬 커맨드는 자유롭게, 보안에 영향을 주는 베이스라인은 엄격하게. 모든 걸 똑같이 통제하면 혁신이 죽고, 모든 걸 풀면 사고가 납니다.

전사 하네스의 가장 극적인 효과는 온보딩에서 나옵니다. 신입이 첫날 저장소를 클론하면, CLAUDE.md·표준 커맨드·보안 Hook이 이미 다 깔려 있습니다. 3개월치 암묵지가 첫날에 전달되는 것이죠.

좋은 CLAUDE.md는 실행되는 온보딩 문서입니다. 위키 문서는 낡지만, 하네스는 매일 쓰이므로 낡으면 바로 티가 나 갱신됩니다. 신입 온보딩 품질을 높이고 싶으면, 온보딩 문서를 쓰지 말고 하네스를 다듬으세요.

대부분의 문제는 “하네스가 없어서”가 아니라 “하네스가 어긋나서”입니다. 규칙이 현실과 다르거나, 컨텍스트가 잡음으로 찼거나, 공유가 안 된 것이죠. 진단의 첫걸음은 “이 하네스가 지금도 맞나?”를 의심하는 것입니다.

C.2컨텍스트가 넘칠 때

증상: 초반엔 잘하다가 대화가 길어지면 규칙을 잊고, 답이 오락가락합니다. 원인: 컨텍스트가 무관한 내용(옛 시도, 실패한 탐색, 방대한 파일 덤프)으로 가득 차 중요한 규칙의 신호가 묻힌 것입니다.

대응방법
세션 정리결론만 남기고 새 세션으로 이월(1.4)
규칙 다이어트CLAUDE.md에서 안 쓰는 규칙 제거
범위 분리특정 폴더 규칙은 디렉토리 스코프로(1.3)
위임 활용지저분한 탐색을 서브에이전트로 격리(4.1)

핵심 교훈: 규칙을 더 넣어서 해결하려 하지 마세요. “규칙을 자꾸 어기네, 규칙을 더 강하게 써야지”는 종종 역효과입니다 — 잡음을 더 늘리니까요. 오히려 빼서 중요한 규칙의 신호를 살리는 게 정답일 때가 많습니다.

C.3Hook이 걸리지 않을 때

증상: 분명 Hook을 등록했는데 실행이 안 됩니다. Hook은 강력하지만 조용히 실패하는 경향이 있어 진단이 까다롭습니다. 체크리스트로 좁히세요.

확인흔한 실수
매처이벤트·도구 매처가 실제 도구명과 다름(Edit vs Write)
경로스크립트 경로가 틀림, $CLAUDE_PROJECT_DIR 미사용
실행 권한스크립트에 chmod +x 안 함
의존성jq 등 스크립트가 쓰는 도구가 미설치
종료 코드차단하려는데 exit 2 대신 exit 1

먼저 로그를 찍어라

Hook 디버깅의 첫걸음은 스크립트 맨 위에 echo "hook 실행됨: $(date)" >> /tmp/hook.log를 넣는 것입니다. 로그가 안 찍히면 등록·매처 문제, 찍히는데 동작이 이상하면 스크립트 로직 문제로 범위가 반으로 줄어듭니다.

C.4비용이 예상보다 클 때

증상: 청구서가 예상을 크게 넘습니다. 비용 문제는 대개 관측이 없어서 늦게 발견됩니다 — 먼저 7.2의 로그로 어디서 새는지부터 봐야 합니다.

원인처방
단순 작업에 큰 모델분류·추출은 빠른 모델로(모델 믹스, 8.3)
무한 루프·재시도비용 상한·최대 반복 제한(8.4)
불필요한 대량 컨텍스트컨텍스트 위생으로 다이어트(1.4)
야간 배치가 폭주하드 리밋 + 임계치 알림(7.3)

가장 흔한 범인은 “모든 걸 최상위 모델로”입니다. 단순 분류 100만 건을 최상위 모델로 돌리면 낭비죠. 작업에 맞는 모델을 고르는 것만으로 비용이 몇 배 줄기도 합니다. 그리고 비용을 아끼려다 품질을 죽이지 마세요 — 비용/가치로 보세요(10.3).

C.5팀원마다 결과가 다를 때

증상: 같은 작업인데 사람마다 결과 품질이 제각각입니다. 이건 거의 항상 하네스가 공유되지 않았다는 신호입니다 — 잘하는 사람의 하네스가 그 사람 로컬에만 있는 거죠.

원인처방
규칙이 개인 로컬에만프로젝트 CLAUDE.md를 git 커밋(1.3)
좋은 커맨드가 한 사람만공용 커맨드 저장소로 공유(부록 B.2)
암묵지가 문서화 안 됨잘하는 사람의 프롬프트를 커맨드로 승격
보안·권한 기준이 제각각표준 Hook·베이스라인(B.3)

부록 C 요약

문제의 대부분은 하네스의 부재가 아니라 어긋남입니다 — 낡거나, 잡음이 끼거나, 공유 안 됐거나. 증상에서 출발해 “이 하네스가 지금도 맞나, 팀이 공유하나”를 물으세요. 진단은 곧 하네스 위생 점검입니다(10.7).
