# 핀테크 하네스 세트

규제·감사가 지배하는 도메인의 설정 예시입니다. 다른 도메인과 다른 점은
**"편의보다 통제가 우선"** 이고, **관측이 첫날부터 필수**라는 것입니다.

## settings.json 의 설계 의도

| 항목 | 왜 이렇게 |
|------|-----------|
| `ask` 에 원장·이체·정산 코드 편집 | 돈이 움직이는 코드는 사람이 반드시 확인 |
| `deny` 에 `psql`/`mysql` | DB 직접 접근 자체를 차단. 마이그레이션만 허용 |
| `deny` 에 KYC 데이터 경로 | 고객 신원정보는 읽을 수조차 없게 (읽을 수 없으면 유출도 불가) |
| `PostToolUse` 감사 로그를 **매처 없이** | 규제 대응상 **모든** 도구 호출을 기록해야 함 |
| `Stop` 검증 게이트 | 테스트 통과 없이 완료 보고 불가 |

## 첫날부터 켜야 하는 것

다른 도메인에서는 관측을 나중에 붙여도 되지만, 핀테크는 **처음부터** 켭니다.
감사 시점에 "기록이 없다"는 치명적입니다.

- [ ] 전체 도구 호출 감사 로그 (`audit-log.sh`)
- [ ] 로그 보존 기간 정책 수립 (규제 요구 기간 확인)
- [ ] 로그 접근 권한 제한 (로그 자체가 민감정보)
- [ ] 승인 이력이 "누가·언제·무엇을" 남기는지 확인

## CLAUDE.md 에 넣을 규칙 (발췌)

```markdown
## 금액·정산 규칙
- 금액은 최소 화폐단위 정수(원). 부동소수 절대 금지.
- 이자·수수료 계산은 반올림 규칙을 명시적으로 지정한다(은행식 반올림).
- 모든 금액 이동은 복식부기로 기록한다. 단식 기록 금지.

## 감사 규칙
- 모든 상태 변경에 who/when/why 를 남긴다.
- 로그를 지우거나 수정하는 코드를 작성하지 않는다.

## 금지
- 프로덕션 데이터를 로컬로 내려받지 않는다.
- 고객 식별정보를 로그·에러 메시지에 포함하지 않는다.
```
