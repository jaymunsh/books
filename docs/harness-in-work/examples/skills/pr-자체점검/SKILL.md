# PR 올리기 전 자체 점검

사람 리뷰어의 시간은 비쌉니다. **기계가 잡을 수 있는 것은 기계가 잡고**,
사람은 설계 판단에 집중하게 하는 것이 이 스킬의 목적입니다.

## 1. 기계 검증 (반드시 실제 실행)

- 테스트: `npm test`
- 타입체크: `npm run typecheck`
- 린트: `npm run lint`

세 가지를 **실제로 실행하고 출력을 확인**한다. "통과할 것으로 예상"은 금지.
하나라도 실패하면 PR을 올리지 않는다.

## 2. 변경 범위 점검

- [ ] 이 PR이 **한 가지 일**만 하는가? 성격이 다른 변경이 섞였으면 나누라고 제안한다.
- [ ] 요청되지 않은 리팩토링·포맷 변경이 섞이지 않았는가? (리뷰 노이즈)
- [ ] 디버그 코드, `console.log`, 주석 처리된 코드가 남아 있지 않은가?
- [ ] 커밋되면 안 되는 파일(`.env`, 로컬 설정, 대용량 산출물)이 없는가?

## 3. 프로젝트 규칙 대조

`CLAUDE.md`의 규칙과 대조한다. 특히 자주 어기는 항목:

- [ ] 명명 규칙(필드명·파일명)을 따랐는가
- [ ] 금지된 패턴을 쓰지 않았는가
- [ ] 결제·인증 등 민감 영역을 건드렸다면 사람 리뷰가 필요하다고 표시했는가

## 4. PR 설명 작성

리뷰어가 **무엇을 봐야 하는지** 알 수 있게 쓴다.

```
## 무엇을
(한 문장)

## 왜
(배경·문제)

## 어떻게 확인했나
(실행한 검증과 실제 결과 — 예: 테스트 27개 통과)

## 리뷰 포인트
(리뷰어가 특히 봐줬으면 하는 판단 지점)
```

## 출력

점검 결과를 항목별 ✅/❌로 보고하고, ❌가 있으면 **PR을 올리지 말라고 명시**한다.
모두 통과하면 위 형식의 PR 설명 초안을 함께 낸다.
