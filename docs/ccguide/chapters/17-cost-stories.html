Chapter 17 · 사례집

비용 효율화 사례집 — 절감 스토리와 AI 활용 원칙

15장이 기법이라면, 17장은 이야기와 원칙입니다. 실제 절감 시나리오 4개, 프로젝트 단계별 효율화 가이드, 그리고 Claude에 국한되지 않는 AI 활용 일반 원칙 8가지까지 다룹니다.

📌 이 장의 사례에 대해

아래 절감 사례들은 커뮤니티에 자주 보고되는 패턴을 합성한 가상의 케이스입니다. 숫자는 실제 ccusage·청구서 데이터에서 흔히 관찰되는 범위이며, 본인 환경에 그대로 옮기면 비슷한 효과를 기대할 수 있습니다.

절감 사례 4개 — 한눈 비교

사례핵심 액션전후절감
① 스타트업전사 디폴트 Opus → Sonnet 다운그레이드월 $2,400월 $720-70%
② 오픈소스CI에 변경 크기 차등 적용월 $820월 $210-74%
③ 솔로 개발자CLAUDE.md 250줄 → 90줄 다이어트세션당 18k세션당 6k-66%
④ 데이터팀대량 분류 작업을 Haiku 서브에이전트로일 $34일 $6-82%

📖 사례 ① · 스타트업 — 전사 디폴트 모델 한 줄로 70% 절감

배경시리즈 A 스타트업, 백엔드·프론트 합쳐 12명. Claude Code 도입 3개월차.
증상월 청구액이 도입 첫 달 $800 → 둘째 달 $1,600 → 셋째 달 $2,400로 가파르게 상승.
가설"모두가 알아서 잘 쓰고 있을 거다"는 막연한 믿음.

🔍 진단 (1시간 투자)

플랫폼 엔지니어 H가 ccusage로 사용자별 모델 분포를 뽑았다.

npx ccusage@latest monthly --json | jq '.entries | group_by(.model)'

# 결과:
# Opus 4.7  : 73% of tokens, 91% of cost
# Sonnet 4.6: 24%, 8%
# Haiku 4.5 : 3%, 1%

충격적이었다. "기본값이 Opus였다"는 것을 아무도 몰랐다 — 도입 시 누군가가 settings.json에 "model": "claude-opus-4-7"을 박아 두고 잊었다.

🛠 적용 (30분 작업)

// .claude/settings.json (팀 공유)
{
  "model": "claude-sonnet-4-6",   // ← Opus → Sonnet
  ...
}

동시에 팀 슬랙에 한 줄 가이드: "설계·복잡 디버깅만 /model로 Opus 전환. 끝나면 다시 Sonnet."

📊 결과 (한 달 후)

  • 월 청구액: $2,400 → $720 (-70%)
  • 주관적 품질 변화: 거의 없음 (출시 속도·버그 비율 동일)
  • Opus 사용 빈도: 73% → 12% (실제 필요한 곳에만)
🎓 강사 노트 — "기본값의 폭력"

가장 큰 비용 누수는 아무도 의식하지 않는 기본값입니다. 도입 첫 주에 settings.json을 한 번 같이 읽으세요. "왜 이 값이지?" 질문이 90% 비용 사고를 막습니다.

📖 사례 ② · 오픈소스 — CI 차등화로 74% 절감

배경스타 2.4k짜리 오픈소스 라이브러리. 메인테이너 2명. claude-code-action으로 PR 자동 1차 리뷰 운영 중.
증상주말 동안 외부 PR이 23개 들어오니 일요일 청구액만 $74. 월 청구액이 $820까지 치솟음. 후원금으로 운영되는 프로젝트라 부담.

🔍 진단

메인테이너가 gh pr list --json additions,deletions,changedFiles로 PR 크기 분포를 살폈다.

  • 전체 PR 중 62%가 5파일 이하의 소형 (오타·문서·작은 버그픽스)
  • 나머지 38%만 실제 코드 변경 큰 PR
  • 그런데 워크플로우는 모든 PR에 동일하게 --effort high + Opus 사용

🛠 적용 (워크플로우 1개 추가)

# .github/workflows/claude.yml
on: pull_request
jobs:
  size-check:
    runs-on: ubuntu-latest
    outputs:
      tier: ${{ steps.classify.outputs.tier }}
    steps:
      - uses: actions/checkout@v4
      - id: classify
        run: |
          FILES=$(gh pr view ${{ github.event.number }} --json changedFiles -q .changedFiles)
          if [ "$FILES" -le 5 ]; then echo "tier=light" >> $GITHUB_OUTPUT
          elif [ "$FILES" -le 30 ]; then echo "tier=medium" >> $GITHUB_OUTPUT
          else echo "tier=heavy" >> $GITHUB_OUTPUT
          fi

  review:
    needs: size-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          prompt: |
            ${{ needs.size-check.outputs.tier == 'light'  && '/code-review --effort low'    || '' }}
            ${{ needs.size-check.outputs.tier == 'medium' && '/code-review --effort medium' || '' }}
            ${{ needs.size-check.outputs.tier == 'heavy'  && '/code-review --effort high'   || '' }}

여기에 보안 민감 경로(auth/·crypto/)는 무조건 high 강제 룰 추가.

📊 결과

  • 월 청구액: $820 → $210 (-74%)
  • 외부 컨트리뷰터 피드백: "리뷰가 더 빨라졌다" (light tier가 30초 안에 응답)
  • 발견된 버그 수: 거의 동일 (큰 PR엔 여전히 high)
🎓 강사 노트 — "모든 PR이 같은 무게가 아니다"

CI 비용 폭주의 99%는 "오타 1줄짜리 PR에도 Opus + effort high"입니다. 변경 크기·민감도를 코드로 분기하면, 평균 리뷰 품질은 유지하면서 비용만 떨어집니다.

📖 사례 ③ · 솔로 개발자 — CLAUDE.md 다이어트로 매 세션 66% 절약

배경인디 SaaS 개발자 M. 1년째 운영, 매일 3~5세션 사용.
증상최근 세션마다 첫 응답이 느려짐. /context 찍어보니 대화 시작도 안 했는데 이미 22%(44k)가 차 있음.

🔍 진단

범인은 M이 1년간 "이것도 적어둬야지" 하다 누적된 CLAUDE.md였다.

wc -l CLAUDE.md
# 253 CLAUDE.md   ← 250줄 초과

# 내용 검토 결과:
# - "옛날에 한 번 막혔던 버그 회피법" 12개 (지금은 안 막힘)
# - 더 이상 안 쓰는 라이브러리 컨벤션 (마이그레이션 끝남)
# - 같은 의미의 규칙이 다른 표현으로 3번 반복
📌 짚고 가기 — CLAUDE.md vs auto memory, 누가 늘리나

흔한 오해 정정. CLAUDE.md는 Claude가 자동으로 늘리지 않습니다. 사람이 직접 편집해야 합니다. M의 경우처럼 "이슈 만날 때마다 한 줄씩 추가" 패턴이 1년이면 250줄로 누적되는 거지, 가만히 둔다고 커지지 않아요.

반면 auto memory(~/.claude/projects/<proj>/memory/MEMORY.md)는 Claude가 세션마다 스스로 학습한 내용을 자동으로 추가합니다. 이쪽이 진짜 "쓸수록 자동으로 늘어나는" 파일입니다. 둘 다 컨텍스트를 차지하므로, 분기 다이어트할 땐 /memory로 둘 다 점검하세요.

🛠 적용 (30분 가지치기)

M은 Claude한테 직접 다이어트를 시켰다.

> "이 CLAUDE.md를 100줄 이내로 다이어트해줘. 기준:
   1) 지금도 실제로 적용되는 규칙만 남기기
   2) 중복·유사 규칙은 하나로 통합
   3) "과거에 막혔던" 회상형 메모는 모두 삭제 (필요하면 .claude/rules/incidents.md로 분리)
   4) '왜 이 규칙인가'가 한 줄로 안 떠오르면 후보로 표시 → 내가 확인"

Claude가 90줄로 줄인 초안 + 17개 "삭제 후보" 표 제시. M은 그 중 4개만 살리고 나머지 삭제.

📊 결과

  • 세션 시작 컨텍스트: 18k → 6k (-66%)
  • 월 토큰 사용량: 약 30% 감소 (매 세션 절약 × 100+ 세션)
  • 체감 응답 속도: 첫 토큰까지 1초 빨라짐
🎓 강사 노트 — "분기 다이어트는 의무"

CLAUDE.md는 살아 있는 문서라 자연스럽게 비대해집니다. 3개월마다 한 번씩 강제 가지치기가 답. "삭제하면 큰일 나나?"를 자문해 보고, 답이 "아니"면 그건 삭제 대상입니다.

📖 사례 ④ · 데이터팀 — 대량 분류 작업을 Haiku로 격리해서 82% 절감

배경데이터 팀, 매일 새 고객 피드백 ~3,000건을 카테고리 6종으로 분류해야 함.
증상일일 분류 작업에 Claude Code(Sonnet) 사용 — 청구액 일 $34. 월 $1,000 넘게 누적.

🔍 진단

분류 작업은 단순한 "텍스트 → 라벨" 매핑이라 Sonnet은 오버킬. Haiku로 충분.

🛠 적용 — 서브에이전트로 분리

# .claude/agents/classifier/agent.md
---
name: classifier
description: "고객 피드백 → 6개 카테고리 분류 전용"
model: claude-haiku-4-5-20251001  # ← Haiku 명시
tools: [Read, Bash]
---

고객 피드백 한 건을 받아 다음 6개 카테고리 중 하나로 분류:
[Bug, Feature, Pricing, UX, Performance, Other]

출력 형식: JSON `{ "id": ..., "category": "Bug", "confidence": 0.85 }`
설명·이유 X. 분류만.
# 매일 실행 (스케줄)
> /schedule "0 2 * * *" sh -c '
  cat new_feedback.jsonl | while read line; do
    claude -p "/classifier $line" --output json
  done > classified.jsonl
'

📊 결과

  • 일일 비용: $34 → $6 (-82%)
  • 분류 정확도: Sonnet 96% → Haiku 94% (현실적으로 차이 없음)
  • 처리 속도: 오히려 30% 빨라짐 (Haiku가 더 빠름)
🎓 강사 노트 — "Haiku를 두려워하지 마라"

"Haiku는 품질 떨어지지 않을까?"는 가장 비싼 오해입니다. 분류·추출·변환·요약 같은 정형 작업에선 Haiku와 Sonnet의 차이가 사람 눈에 안 보이는 수준입니다. 대신 추론·창작·복잡 디버깅엔 절대 쓰지 마세요. 도구를 도구답게.

🔄 프로젝트 단계별 효율 시나리오

위 사례들이 "한 시점의 절감"이라면, 이번 섹션은 프로젝트 생애주기 전체에 적용할 가이드입니다.

1단계 · 설계 (Discovery)

  • Opus를 짧고 굵게 — 며칠 동안 매일 1~2회, 1시간씩 집중 사용
  • 출력: 아키텍처 문서·ADR·기술 스택 결정. 코드 X
  • 마치면 곧장 Sonnet으로 내려오기 (Opus를 디폴트로 두지 말 것)
  • 예상 비용: 보통 $20~50로 끝남. 가장 비싸 보이지만 가장 ROI 큼

2단계 · 구축 (Build)

  • Sonnet이 70~80% 작업 — 일반 기능·리팩터링·테스트
  • 막힐 때만 1~2회 Opus 호출 (디버깅·아키텍처 검증)
  • Plan mode 디폴트 ON. 위험 작업은 절대 끄지 말 것
  • 매 PR /code-review → /verify 사이클 강제
  • 예상 비용: 1인 풀타임 기준 월 $100~200

3단계 · 출시 (Launch)

  • /security-review를 Opus로 1회 — "값진 $5"
  • CI에 claude-code-action 부착 (effort 차등화 필수)
  • ccusage 베이스라인 측정 시작

4단계 · 운영 (Operate)

  • 모니터링 자동화는 Haiku — 로그 분류·이상 탐지 한 줄 요약
  • 인시던트 대응은 Opus + xhigh — "이때 아껴서 손해 보지 마라"
  • 주간 ccusage 리포트 자동화 (Slack)
  • 분기마다 CLAUDE.md·스킬·MCP 정리 (사례 ③ 다이어트)

5단계 · 회고 (Retro)

  • 분기에 한 번, 30분 회고: "지난 분기 비싸게 쓴 패턴 3가지"
  • 안티패턴은 settings.json·CLAUDE.md에 룰로 박기 (사람 의지 의존 X)
  • 다음 분기 예산 가이드라인 합의

🧭 AI 활용 일반 원칙 8가지 (Claude 무관)

여기서부터는 GPT·Gemini·Cursor·Copilot — 어떤 AI 도구를 쓰든 적용되는 원칙입니다.

원칙 1 · 도구의 무게는 작업의 무게에 맞춰라

오타 수정에 GPT-5 max thinking을 호출하지 마세요. 잘 모르는 분야의 설계는 작은 모델로 하지 마세요. "이 작업의 ROI가 이 모델 비용을 정당화하는가?"를 매 호출마다 물으세요.

원칙 2 · "비싼 모델 = 좋은 결과"는 신화

같은 프롬프트로 Opus와 Haiku를 비교해 본 적이 있나요? 분류·추출 같은 정형 작업에선 차이가 없습니다. 비싼 모델이 진가를 발휘하는 영역은 모호한 문제·다단계 추론·창의적 합성 정도입니다.

원칙 3 · 사람 시간 vs API 비용 트레이드오프

월 $200 더 쓰고 개발자 시간을 주 5시간 절약하면 명백히 이득입니다. 비용 자체를 줄이는 게 아니라, "사람 시간 환산 비용"을 줄이는 것이 진짜 최적화입니다.

# 한 줄 환산 공식
실효 비용 = API 비용 - (절약된 사람 시간 × 시간당 인건비)
# 음수면? 더 써도 됨. 무조건 흑자다.

원칙 4 · 측정 없는 최적화는 추측

"왠지 비용이 많이 나가는 거 같아"로 시작한 최적화는 보통 헛수고입니다. ccusage든 OpenAI usage dashboard든 숫자 먼저, 액션 나중. 비싼 세션 톱 5만 봐도 90% 패턴이 드러납니다.

원칙 5 · 컨텍스트가 곧 비용이다

모든 LLM의 비용은 "입력 토큰 × 단가"에서 시작합니다. 시스템 프롬프트·메모리·첨부 파일·이전 대화 — 다 입력 토큰입니다. "이 정보를 매번 보낼 필요가 있나?"가 가장 효과적인 비용 질문입니다.

원칙 6 · 캐싱은 무조건 켜라

같은 시스템 프롬프트·문서를 반복 사용한다면 캐싱은 무조건 활성화. 대부분 API에서 캐시 토큰은 50~90% 저렴합니다. "한 번에 끝낼 작업"이 아니라 "여러 번 호출되는 작업"에서 본전 뽑힙니다.

원칙 7 · 배치·비동기로 분리할 수 있는 건 분리

"지금 당장" 필요 없는 작업은 배치 API로 보내면 50% 절감되는 게 흔합니다(Claude·OpenAI 모두). 야간 분류·일일 요약·주간 리포트 같은 작업이 후보. 실시간성을 자기에게 강요하지 마세요.

원칙 8 · 모델·벤더에 락인되지 마라

오늘 Claude가 좋아도 6개월 뒤엔 다른 모델이 더 적합할 수 있습니다. 비용 구조도 바뀝니다. 프롬프트·스킬·CLAUDE.md를 평문 마크다운으로 유지하고, 도구 종속 기능(예: superpowers)도 한 곳에 격리. 그래야 갈아탈 때 비용이 며칠로 끝납니다.

🎓 강사 노트 — 8원칙의 한 줄 요약

"측정하고, 작업에 맞는 도구를 골라, 컨텍스트를 절제하고, 사람 시간으로 환산해서 판단한다." 이 한 문장이 모든 AI 도구의 비용·효율 사고의 시작점입니다.

💰 ROI 빠른 계산 — 사람 시간 환산

매번 "이 정도 비용이 적정한가?"를 물을 때 쓰는 공식.

항목예시 값
(A) 월 Claude Code 비용$200
(B) Claude로 절약된 개발자 시간 (월)20시간
(C) 개발자 시간당 비용 (인건비/8h)$50
(D) 절약 가치 = B × C$1,000
실효 ROI = (D − A) / A+400%

가이드라인:

  • ROI < 0% → 무언가 잘못됨. ccusage로 패턴 진단 필수
  • 0% ~ 100% → 정상이지만 개선 여지. 안티패턴 점검
  • 100% ~ 500% → 건강한 사용. 현 상태 유지
  • 500%+ → 아주 잘 쓰고 있음. 다른 팀에 베스트 프랙티스 공유

📊 팀 규모·예산 가이드라인

"우리 팀은 얼마가 적정한가?"의 대략적 벤치마크. 환경마다 다르니 참고용.

팀 규모적정 월 예산대표 사용 패턴
1인 (사이드)$10~50주 1~2회 사용, Sonnet 위주, 가끔 Opus
1인 (풀타임)$50~200매일 4~6시간, Sonnet 디폴트, CLAUDE.md·스킬 활용
2~5인 스타트업$200~600설계는 Opus, 구축은 Sonnet, CI 자동 리뷰 적용
5~20인 팀$600~2,000위 + 비용 정책 + ccusage 주간 모니터링
20인+ 조직$2,000~10,000+위 + 엔터프라이즈 settings + 감사 로그 hooks + 분기 회고
💡 가이드라인의 70%를 넘는다면

위 표의 상한선을 30% 이상 초과한다면 사례 ①~④ 중 하나가 발생 중일 가능성이 높습니다. 오늘 30분 시간 내서 ccusage 진단부터 돌려보세요.

🎯 닫는 글 — 비용을 "사람 시간"으로 환산하는 습관

이 장의 모든 사례·원칙은 결국 한 가지 사고방식으로 수렴합니다: "AI 비용을 절약된 사람 시간으로 환산해서 판단한다."

월 $200이 비싸 보이지만 개발자 5시간을 살린다면 $1,000짜리 의사결정입니다. 반대로 월 $50이 싸 보이지만 잘못된 모델 선택으로 사람이 10시간을 검토해야 한다면 $500짜리 실수입니다.

숫자 자체보다 그 숫자가 어떤 시간을 사고 있는가를 보세요. 그게 비용 효율의 핵심이고, AI 도구가 바뀌어도 변하지 않을 원칙입니다.
