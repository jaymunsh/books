Chapter 07

Claude Agent SDK — 본인 서비스에 에이전트 임베드

Claude Code(CLI)가 "터미널에서 사용"이라면, Agent SDK는 "본인 코드에 라이브러리로 임베드"입니다. 슬랙봇·CI 자동화·내부 도구 — 어디든 Claude의 추론을 끼워 넣을 수 있습니다.

📌 패키지·API명은 공식 문서 우선

SDK는 비교적 새롭고 빠르게 변합니다. 본 장의 패키지명·메서드명은 작성 시점 기준이며, 실제 사용 전 반드시 공식 문서의 최신 버전을 확인하세요. 구조와 사고방식은 동일하니, API 시그니처만 매핑하면 됩니다.

CLI 헤드리스 vs Agent SDK — 언제 무엇

Claude Code를 본인 시스템에 통합하는 두 가지 길이 있습니다. 잘 골라야 합니다.

방식적합한 상황예시
CLI 헤드리스
(claude -p ...)		 쉘에서 호출 가능한 단발성 작업. 셋업 5분. CI에서 PR 리뷰, 크론으로 매일 보안 스캔
Agent SDK
(코드 임베드)		 본인 서비스 안에서 호출, 도구·권한·상태를 프로그램으로 통제 슬랙봇, 내부 데이터 분석 봇, 자체 IDE 통합
🎓 강사 노트 — 90%는 헤드리스로 충분

"내 시스템에 임베드해야지" 싶은 작업의 상당수는 사실 claude -p + 쉘 스크립트로 끝납니다. SDK는 쉘로 표현 어려운 통제(도구 동적 선택·세션 상태·실시간 스트리밍)가 필요할 때만 꺼내세요. 과한 도구 선택은 유지보수 부담만 늘립니다.

설치

# Python 3.10+
pip install claude-agent-sdk
# 패키지명은 공식 문서 확인 권장 (변경될 수 있음)

# API 키는 환경변수로
export ANTHROPIC_API_KEY=sk-ant-...
# Node 20+
npm i @anthropic-ai/claude-agent-sdk
# 패키지명은 공식 문서 확인 권장

export ANTHROPIC_API_KEY=sk-ant-...

도구(Tool) 정의 — 외부 API·DB 연결

SDK의 핵심 강점은 본인 함수를 Claude가 도구로 호출하게 만들기입니다. Claude가 "이 정보가 필요해" 판단하면 자동으로 함수를 호출하고 결과를 받아 다음 단계에 활용합니다.

from claude_agent_sdk import ClaudeAgent, Tool

def get_user(user_id: str) -> dict:
    """DB에서 사용자 정보 조회"""
    return db.query("SELECT * FROM users WHERE id = ?", [user_id])

def refund(order_id: str, reason: str) -> dict:
    """주문 환불 (위험 작업)"""
    return stripe.refunds.create(charge=order_id, reason=reason)

agent = ClaudeAgent(
    model="claude-sonnet-4-6",
    system_prompt="고객 지원 봇. 환불 시 반드시 사용자 확인.",
    tools=[
        Tool.from_function(get_user),
        Tool.from_function(refund, requires_approval=True),  # 위험 작업은 승인 필수
    ],
)

result = agent.run("user_123 의 가장 최근 주문을 환불해줘. 이유는 '배송 누락'")
import { ClaudeAgent, Tool } from "@anthropic-ai/claude-agent-sdk";

const getUser = Tool.define({
  name: "get_user",
  description: "DB에서 사용자 정보 조회",
  parameters: { type: "object", properties: { user_id: { type: "string" } }, required: ["user_id"] },
  execute: async ({ user_id }) => db.query("SELECT * FROM users WHERE id = $1", [user_id]),
});

const refund = Tool.define({
  name: "refund",
  description: "주문 환불",
  parameters: { type: "object", properties: { order_id: { type: "string" }, reason: { type: "string" } }, required: ["order_id", "reason"] },
  requiresApproval: true,  // 위험 작업
  execute: async ({ order_id, reason }) => stripe.refunds.create({ charge: order_id, reason }),
});

const agent = new ClaudeAgent({
  model: "claude-sonnet-4-6",
  systemPrompt: "고객 지원 봇. 환불은 반드시 사용자 확인.",
  tools: [getUser, refund],
});

const result = await agent.run("user_123 의 최근 주문 환불, 이유 '배송 누락'");

권한·안전 장치

임베드된 에이전트는 본인 코드 안에서 도구를 직접 호출합니다. 권한 설정은 곧 안전입니다.

안전 장치설명
requires_approval도구별로 호출 직전 콜백 발생 → 사용자(또는 시스템) 승인 후 실행
도구 동적 노출사용자·요청 컨텍스트에 따라 노출 도구 다르게 (관리자만 refund 가능 등)
입력 검증SDK가 schema 검증 + 본인 함수 안에서 추가 검증
실행 제한최대 도구 호출 수·최대 토큰 cap 설정
로깅모든 도구 호출을 감사 로그로 (특히 prod 환경)

실전 시나리오 3가지

시나리오 1 · Slack 봇

채널에서 @ClaudeOps 멘션 시 작동하는 운영 봇. DB·로그·메트릭 조회 도구를 가짐.

# app.py — FastAPI + slack-bolt 가정
from claude_agent_sdk import ClaudeAgent, Tool

def query_metric(metric: str, since_minutes: int = 60) -> dict:
    """Prometheus에서 메트릭 조회"""
    return prom.query(metric, since_minutes)

def fetch_logs(service: str, query: str, limit: int = 50) -> list:
    """로그 검색"""
    return loki.query_range(service, query, limit=limit)

agent = ClaudeAgent(
    model="claude-sonnet-4-6",
    system_prompt="""당신은 SRE 어시스턴트.
    사용자가 장애 의심 신고하면:
    1) 관련 메트릭 조회
    2) 의심 시간대 로그 추출
    3) 가설 3개 제시 (상→하 가능성)
    실제 수정·재시작 도구는 없음 — 분석만.""",
    tools=[Tool.from_function(query_metric), Tool.from_function(fetch_logs)],
    max_tool_calls=10,  # 안전 cap
)

@app.event("app_mention")
def handle(event, say):
    result = agent.run(event["text"])
    say(result.text)

시나리오 2 · 실패 테스트 자동 수정 봇 (CI)

GitHub Actions에서 테스트 실패 시 SDK 기반 봇이 자동으로 분석 → 수정 PR.

# fix-failing-tests.py — Actions에서 호출
import subprocess, os
from claude_agent_sdk import ClaudeAgent, Tool

def run_cmd(cmd: str) -> str:
    return subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=120).stdout

def read_file(path: str) -> str:
    return open(path).read()

def write_file(path: str, content: str) -> str:
    open(path, "w").write(content)
    return f"wrote {path}"

agent = ClaudeAgent(
    model="claude-sonnet-4-6",
    system_prompt="""실패한 테스트 분석가.
    1) `npm test` 실행해 실패 메시지 수집
    2) 관련 파일 read 해서 원인 추정
    3) 가장 명백한 수정만 적용 (불확실하면 멈추고 보고)
    4) 다시 테스트 실행해 그린 확인""",
    tools=[Tool.from_function(run_cmd), Tool.from_function(read_file), Tool.from_function(write_file)],
)

result = agent.run("실패한 테스트를 분석하고 수정하라.")
print(result.text)
# GitHub Action 후속 step에서 gh pr create

시나리오 3 · 데이터 분석 파이프라인

매일 새 데이터를 받아 분류·요약·이상치 탐지 → 슬랙 리포트.

from claude_agent_sdk import ClaudeAgent, Tool
import pandas as pd

def load_csv(path: str) -> dict:
    df = pd.read_csv(path)
    return { "head": df.head(20).to_dict(), "shape": df.shape, "dtypes": str(df.dtypes) }

def aggregate(path: str, group_by: str, metric: str) -> dict:
    df = pd.read_csv(path)
    return df.groupby(group_by)[metric].agg(["sum", "mean", "count"]).to_dict()

agent = ClaudeAgent(
    model="claude-haiku-4-5-20251001",  # 정형 분석은 Haiku로 충분
    system_prompt="데이터 분석가. 결과는 마크다운 표·짧은 결론.",
    tools=[Tool.from_function(load_csv), Tool.from_function(aggregate)],
)

report = agent.run("data/daily_sales.csv 의 어제 매출을 region·product 별로 분석하고 이상치 강조")
slack.post("#daily-reports", report.text)

스트리밍 응답

긴 응답을 토큰 단위로 받아 사용자에게 점진적 표시. UX에 중요.

# Python
for chunk in agent.stream("긴 문서 요약"):
    print(chunk.text, end="", flush=True)
// TypeScript
for await (const chunk of agent.stream("긴 문서 요약")) {
  process.stdout.write(chunk.text);
}

SDK 사용 시 자주 만나는 함정

함정해결
도구가 너무 많아 Claude가 헷갈림한 에이전트당 도구 5~10개 이내 권장. 많으면 역할 분리
도구가 실패해도 무한 재시도max_tool_calls로 cap. 실패 시 에러를 텍스트로 반환
비밀이 도구 결과로 새어 나감함수 내부에서 마스킹 후 반환. 비밀은 절대 raw 노출 X
한 세션의 응답이 너무 길어 컨텍스트 폭발긴 작업은 여러 agent.run()으로 분리 + 중간 요약
도구 정의가 길어 토큰 낭비description은 1~2줄. 자세한 사용법은 system_prompt로
🎬 시나리오 · "SDK 임베드 → 헤드리스 회귀"
상황: 인덱싱 봇을 SDK로 만들었는데, 한 달 운영해 보니 그냥 claude -p + 쉘 스크립트면 됐을 작업. 회귀 결정: SDK 코드 200줄 → 쉘 30줄로 단순화. 유지보수 부담 사라짐. 교훈: SDK는 "이게 정말 필요한가?"를 매번 자문. 헤드리스로 안 되는 게 있을 때만 꺼내세요.

7장 체크리스트

  • ☐ CLI 헤드리스 vs SDK — 본인 작업에 맞는 길 선택
  • ☐ 도구는 5~10개 이내, description은 1~2줄
  • ☐ 위험 도구(결제·삭제 등)에 requires_approval 또는 동적 노출 제한
  • ☐ max_tool_calls로 안전 cap 설정
  • ☐ 비밀 정보 마스킹 — 도구 결과로 raw 노출 금지
  • ☐ 패키지명·메서드명은 공식 문서로 최종 확인
