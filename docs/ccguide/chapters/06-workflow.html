Chapter 06

Workflow — 멀티 에이전트 오케스트레이션

서브에이전트가 "한 명에게 위임"이라면, Workflow는 "수십~수백 명의 팀을 코드로 지휘"합니다. JavaScript 스크립트로 결정적 흐름을 짜서 코드베이스 전체 감사·대규모 리팩터링·다중 검증 같은 무거운 작업을 자동화합니다.

Workflow가 무엇인가

Workflow는 Claude Code의 도구 중 하나로, "이 단계 다음엔 무엇" 같은 흐름을 코드로 작성해 런타임이 결정적으로 실행합니다. 메인 Claude의 판단에 맡기는 것과 달리, 스크립트가 정한 순서·병렬성·반복 조건이 그대로 실행됩니다.

Skill · Subagent · Workflow · Agent Teams — 4가지 비교

도구무엇다음 단계 결정스케일
Skill재사용 가능한 지시문Claude가 본문 따라 판단1단계 작업
Subagent독립 컨텍스트의 위임자Claude가 턴마다 판단몇 개 / 턴
WorkflowJS 스크립트로 짠 오케스트레이션스크립트가 정함수십~수백 에이전트
Agent Teams여러 독립 세션 협업Lead 에이전트가 조율장시간·대규모

언제 Workflow를 쓰는가 (사용자 opt-in 필수)

⚠️ 중요 — 사용자가 명시적으로 요청해야

Workflow는 수십~수백 에이전트를 만들 수 있어 토큰을 많이 씁니다. 때문에 Claude는 사용자가 "workflow"·"orchestrate"를 직접 언급하거나 관련 슬래시 명령(/deep-research·/code-review ultra 등)을 실행할 때만 자동으로 Workflow 도구를 호출합니다. 반대로 말하면 — 일반 작업은 Subagent로도 충분합니다.

Workflow가 빛나는 상황 4가지

  • 대규모 병렬 작업 — "코드베이스 전체 보안 감사" (100+ 파일 동시 스캔)
  • 복잡한 단계 의존 — 검색 → 검증 → 투표 → 보고서, 각 단계가 이전 결과에 의존
  • 다중 검증·합의 — Judge panel로 결론 도달 전 N명 독립 검토
  • 반복 자동화 — 같은 오케스트레이션을 매주 실행 (스크립트만 다시 돌림)

쓰지 말아야 할 때

  • 간단한 1~2단계 작업 (Subagent로 충분)
  • 사용자와 대화형 상호작용이 많이 필요한 작업
  • 중간에 사람 판단이 자주 들어가야 하는 작업

스크립트 안에서 쓰는 hooks

Workflow 스크립트는 비동기 JS 환경에서 실행되며, 다음 함수들을 제공합니다.

훅역할
agent(prompt, opts?)서브에이전트 1개 생성. schema 옵션으로 구조화 출력 강제 가능
parallel(thunks[])여러 작업 동시 실행 (배리어 — 전부 끝나야 다음)
pipeline(items, stage1, stage2, ...)각 item이 모든 stage를 독립적으로 통과. 배리어 없음
phase(title)이후 agent() 호출들을 진척도 박스에 그룹핑
log(msg)사용자 화면에 진행 메시지 출력
argsWorkflow 실행 시 전달된 인자
budget이번 턴 토큰 예산 (budget.remaining())

패턴 1 · Pipeline (기본 — 배리어 없음)

가장 자주 쓰는 패턴. 각 item이 stage1 → stage2 → ... 를 독립적으로 통과합니다. item A가 stage 3에 있는 동안 item B는 아직 stage 1일 수 있어요. 전체 wall-clock = 가장 느린 단일 체인.

export const meta = {
  name: 'review-changes',
  description: '변경 파일을 차원별로 리뷰 + 발견 즉시 검증',
  phases: [{ title: 'Review' }, { title: 'Verify' }],
}

const DIMENSIONS = [
  { key: 'bugs', prompt: 'src/ 의 잠재 버그 5가지' },
  { key: 'perf', prompt: 'src/ 의 성능 핫스팟 3가지' },
  { key: 'sec',  prompt: 'src/ 의 보안 이슈 3가지' },
]

const results = await pipeline(
  DIMENSIONS,
  // Stage 1: 각 차원별 리뷰
  d => agent(d.prompt, { label: `review:${d.key}`, phase: 'Review', schema: FINDINGS_SCHEMA }),
  // Stage 2: 그 차원의 findings를 즉시 검증
  review => parallel(review.findings.map(f => () =>
    agent(`Verify: ${f.title}`, { phase: 'Verify', schema: VERDICT_SCHEMA })
      .then(v => ({ ...f, verdict: v }))
  ))
)

return results.flat().filter(f => f.verdict?.isReal)

패턴 2 · Parallel (배리어 — 다 끝나야 다음)

모든 결과를 모아서 다음 단계로 가야 할 때 사용. Pipeline보다 wall-clock이 길어질 수 있어요.

// 7명의 finder가 다 끝나야 dedup이 의미 있음
const all = await parallel(
  FINDERS.map(f => () => agent(f.prompt, { schema: FINDINGS_SCHEMA }))
)
const deduped = dedupeByFileAndLine(all.filter(Boolean).flatMap(r => r.findings))
const verified = await parallel(
  deduped.map(f => () => agent(verifyPrompt(f), { schema: VERDICT_SCHEMA }))
)
🎓 강사 노트 — "Pipeline이 디폴트"

중간에 cross-item 정보(예: 전체 dedup, 전체 count)가 필요한 게 아니라면 거의 모든 케이스에서 pipeline이 유리합니다. Parallel을 쓰고 싶다 싶을 때 "정말 배리어가 필요한가?" 한 번 더 자문하세요.

패턴 3 · Judge Panel (다중 검증)

한 결정을 여러 에이전트가 독립적으로 검토 후 투표. 중요한 결론에 신뢰도 확보용.

// 각 발견에 대해 3명의 다른 관점으로 검증
const verified = await parallel(findings.map(f => () =>
  parallel(['correctness', 'security', 'reproducibility'].map(lens => () =>
    agent(`${lens} 관점에서 이 발견이 진짜인지 판단: ${f.desc}`, { schema: VERDICT })
  )).then(verdicts => ({
    finding: f,
    real: verdicts.filter(v => v?.isReal).length >= 2  // 다수결
  }))
))

핵심: 같은 프롬프트로 3번 묻는 게 아니라 각자 다른 렌즈를 주세요. "다양성 있는 검증자"가 단순 반복보다 훨씬 효과적입니다.

패턴 4 · Loop-Until-Dry (탐색·발견형)

"버그를 다 찾을 때까지" 같이 끝점을 모르는 작업에 사용. 새 결과가 없는 라운드가 N번 연속이면 종료.

const seen = new Set()
const confirmed = []
let dryRounds = 0

while (dryRounds < 2) {  // 2라운드 연속 빈 손이면 종료
  const found = await parallel(FINDERS.map(f => () => agent(f.prompt, { schema: BUGS })))
  const fresh = found.flatMap(r => r?.bugs ?? []).filter(b => !seen.has(key(b)))

  if (fresh.length === 0) { dryRounds++; continue }
  dryRounds = 0
  fresh.forEach(b => seen.add(key(b)))
  confirmed.push(...fresh)
  log(`Round done — ${fresh.length} new bugs, ${confirmed.length} total`)
}
return confirmed

패턴 5 · Budget-aware Loop (예산 한도까지 깊이 늘리기)

"+500k 토큰 쓸 수 있으면 더 깊게" 같은 사용자 지시를 동적으로 처리.

const bugs = []
while (budget.total && budget.remaining() > 50_000) {  // 50k 여유 있으면 계속
  const result = await agent('코드베이스의 버그를 찾아줘', { schema: BUGS_SCHEMA })
  bugs.push(...result.bugs)
  log(`${bugs.length}개 발견, ${Math.round(budget.remaining() / 1000)}k 남음`)
}

실전 예시 — 코드베이스 전체 보안 감사

5가지 패턴을 조합한 실제 워크플로우. 한 번 등록해 두면 매주 실행으로 보안 baseline 유지.

export const meta = {
  name: 'security-audit',
  description: '코드베이스 전체 보안 감사 — 다중 finder + 다관점 검증 + dedup',
  phases: [{ title: 'Find' }, { title: 'Verify' }, { title: 'Report' }],
}

// 1) 5가지 보안 영역을 병렬로 finder 띄움
phase('Find')
const findings = (await parallel([
  () => agent('src/auth/ 의 인증·세션 취약점', { schema: BUGS }),
  () => agent('src/api/ 의 입력 검증 누락·SQLi 가능성', { schema: BUGS }),
  () => agent('src/ 전체의 비밀 정보 하드코딩', { schema: BUGS }),
  () => agent('src/ 전체의 의존성 취약점 (package.json 기반)', { schema: BUGS }),
  () => agent('docker·CI 설정 파일의 보안 미스컨피그', { schema: BUGS }),
])).flatMap(r => r?.bugs ?? [])

// 2) dedup (cross-item이라 배리어 필요)
const unique = dedupeByFileAndLine(findings)
log(`${unique.length} unique findings (raw: ${findings.length})`)

// 3) 각 발견을 3-judge로 검증 (다관점)
phase('Verify')
const verdicts = await parallel(unique.map(f => () =>
  parallel(['exploitability', 'impact', 'falsePositive'].map(lens => () =>
    agent(`${lens} 관점: ${JSON.stringify(f)}`, { schema: VERDICT })
  )).then(vs => ({ finding: f, real: vs.filter(v => v?.isReal).length >= 2 }))
))

const confirmed = verdicts.filter(v => v.real).map(v => v.finding)

// 4) 보고서 생성
phase('Report')
return await agent(
  `다음 ${confirmed.length}개 확정 발견을 심각도별 그룹핑 마크다운 보고서로:\n${JSON.stringify(confirmed)}`
)

이 한 스크립트를 /schedule "0 6 * * 1" workflow security-audit 로 등록하면 매주 월요일 새벽에 자동 실행되고, 보고서가 슬랙으로 옵니다.

모범 사례 & 안티패턴

✅ 모범❌ 안티패턴
Pipeline을 디폴트로모든 단계에 Parallel — wall-clock 손해
Judge에 다양한 렌즈 부여같은 프롬프트로 3번 묻기 (편향 동일)
Loop-until-dry로 끝점 자동 결정임의로 "10라운드까지만" 같은 cap (미완 위험)
각 agent에 schema 부여자유 텍스트 반환 → 다음 단계가 파싱 실패
중간에 log()로 진행 표시10분 침묵 — 사용자가 끊고 싶음
예산 가드 (budget.remaining())무한 루프로 토큰 폭주

6장 체크리스트

  • ☐ Skill / Subagent / Workflow의 차이를 한 줄로 설명할 수 있다
  • ☐ 5가지 패턴(Pipeline·Parallel·Judge·Loop-until-dry·Budget-aware) 중 본인 작업에 맞는 것 고르기
  • ☐ "Pipeline이 디폴트" 원칙 체득
  • ☐ Judge panel에 항상 서로 다른 렌즈를 부여
