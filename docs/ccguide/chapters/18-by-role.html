직군별 가이드 — 본인 환경에 맞게

# [Project] Frontend Guide

## Stack
- React 19 + TypeScript + Vite
- TailwindCSS + shadcn/ui
- TanStack Query (서버 상태)
- Zustand (클라이언트 상태)

## Patterns
- 컴포넌트는 함수형 + hooks만 (class 금지)
- 한 파일 200줄 넘기면 분리 검토
- 모든 API 호출은 src/api/ 의 React Query hook 경유
- 스타일은 Tailwind class 우선, 인라인 스타일 금지

## Accessibility
- 모든 button·a·input에 적절한 aria
- 키보드 only 사용 가능해야 (focus ring 보존)

## Forbidden
- innerHTML, dangerouslySetInnerHTML (XSS 위험)
- 직접 fetch — 항상 react-query 경유

자주 쓰는 워크플로우 3가지

  1. "이 컴포넌트 만들어줘 → Playwright로 스크린샷 → 다듬기" 
    > "src/components/UserCard.tsx 를 만들어줘. props: name, avatar, bio. 모바일 우선."
> "localhost:5173/storybook?path=UserCard 에서 스크린샷 찍어줘"
# → Playwright MCP가 헤드리스 브라우저로 캡처 → 첨부
> "위 스크린샷 보고 모바일 폭(360px)에서 깨진 부분 수정"
  2. "디자인 → 코드" (Figma MCP) 
    > "Figma 파일 URL: https://... 의 'Login Screen' 프레임을 React + Tailwind 컴포넌트로"
  3. 접근성 자동 점검 
    > "src/components/Header.tsx 를 키보드 only로 탐색 가능한지 점검. axe-core 룰 기준."
🎬 미니 시나리오 · "디자인 시안 → 작동 PR 1시간"
상황: 디자이너에게 새 회원가입 폼 Figma 시안 받음. 6개 필드·검증·에러 메시지. 적용: 
> "Figma 시안 [URL]을 React + Tailwind + react-hook-form 으로.
   검증은 zod, 에러 메시지는 한국어, 모바일 우선.
   완료 후 Playwright로 정상·에러 케이스 스크린샷."
> /code-review --effort medium
> gh pr create
결과: 1시간 안에 PR. 디자이너와 캡처 보면서 1라운드 수정 → 머지.

# [Project] Backend Guide

## Stack
- Node 22 + TypeScript + Hono
- PostgreSQL 16 + Drizzle ORM
- Redis 7 (세션·캐시)
- Zod (입력 검증)

## Conventions
- 모든 엔드포인트: Zod 검증 → 핸들러 → 응답 헬퍼
- 응답 형식: { ok: boolean, data?, error?: { code, message } }
- 트랜잭션은 db.transaction(), 절대 raw query 금지
- 환경변수는 src/env.ts 에서 검증 후 export

## Security
- raw SQL 금지 (Drizzle ORM only)
- 비밀번호: argon2id
- JWT secret: env 만, 코드에 금지
- 사용자 입력은 항상 schema 검증 후 사용

## Forbidden
- prod DB에 직접 SQL 실행
- console.log 로그 (winston/pino 사용)
- 동기 파일 IO (fs.readFileSync 금지)

자주 쓰는 워크플로우 3가지

  1. 마이그레이션 자동 생성 
    > "users 테이블에 phone_number(string, nullable, indexed) 컬럼 추가. Drizzle 마이그레이션 파일 생성 + 검증 스키마(zod) 갱신."
  2. N+1 발견 (DB MCP 활용) 
    > "GET /orders 엔드포인트의 SQL 실행 로그를 보고 N+1 의심 가는 부분 찾아줘.
   필요하면 postgres MCP로 EXPLAIN 실행."
  3. API 변경 시 자동 점검 체인 
    > "src/api/users.ts 에 새 PATCH 엔드포인트 추가. 작업 후:
   1) /code-review
   2) /security-review (인증·권한·검증 누락)
   3) /verify (실제 호출해 200/400 케이스)
   4) gh pr create"
🎬 미니 시나리오 · "N+1을 30분에 박멸"
상황: /orders 가 평소 200ms → 사용자 많아지면 8초. 의심은 N+1. 적용: 
> "src/api/orders.ts 의 list 핸들러에서 발생하는 모든 SQL을 분석.
   pg MCP로 EXPLAIN 실행. N+1 패턴 발견 시 with(relations) 또는 JOIN으로 수정."
> /verify # 동일 응답 보장 확인
결과: 8초 → 180ms. 같은 패턴이 다른 endpoint에도 있나 추가 점검까지 30분.

# [Project] Data/ML Guide

## Stack
- Python 3.12 + uv (패키지 관리)
- pandas / polars (Pol 우선, 대량은 polars)
- PyTorch (모델), Hugging Face transformers
- DuckDB (로컬 분석), BigQuery (운영)

## Notebook Rules
- 셀 하나 = 한 가지 작업
- 출력 결과는 변수에 저장 후 다음 셀에서 사용 (의존성 명시)
- 무거운 셀은 캐시 (joblib.Memory 또는 pickle)

## Reproducibility
- 시드 고정 (np.random.seed, torch.manual_seed)
- 환경: uv.lock 항상 커밋
- 데이터 버전: DVC 또는 명시적 스냅샷

## Cost
- 대량 분류·라벨링은 .claude/agents/classifier (Haiku)
- 모델 학습 코드 생성은 Opus, 실행은 직접

자주 쓰는 워크플로우 3가지

  1. EDA 자동화 
    > "data/sales_2026.csv 의 EDA를 마크다운으로:
   1) shape · dtypes
   2) 각 컬럼 분포 한 줄 요약
   3) 이상치 의심 톱 5
   4) 추가 분석할 가치 있는 질문 3가지"
  2. 대량 분류 (Haiku 서브에이전트) 
    > "data/feedback.jsonl 의 3,000건을
   .claude/agents/classifier 로 [Bug/Feature/UX/Pricing/Other] 분류.
   결과는 classified.jsonl 로 저장. 비용은 ccusage로 확인 가능."
  3. 모델 평가 자동 비교 
    > "experiments/ 의 최근 5개 run의 metrics.json을 읽어 비교 표 생성:
   [run_id | val_loss | f1 | latency_ms | model_size_mb]
   최고 성능 + 추천 이유 1줄."
🎬 미니 시나리오 · "Jupyter 안에서 Claude로 EDA"
상황: 새 데이터셋을 받았는데 EDA에 1시간씩 흘려 보내고 있음. 적용: 노트북 옆 터미널에서 claude 켜고, 다음을 시킴. 
> "data/new_dataset.parquet 의 EDA 노트북 eda.ipynb 를 만들어줘.
   셀 단위로 분리하고, 각 셀의 출력은 다음 셀에서 참조 가능하게.
   matplotlib + seaborn, 모바일 화면에서도 보기 좋은 figsize."
> "실행해서 결과 보고, 흥미로운 패턴 3가지를 노트북 마지막에 markdown 셀로"
결과: 1시간 → 15분. 사람은 "흥미로운 패턴 3가지"에서 시작해 깊은 분석으로 진입.

# [Org] Platform Guide

## Infra
- AWS (ap-northeast-2 주, us-west-2 DR)
- EKS 1.30, Argo CD, Helm
- Terraform (terraform/modules/ 공통, environments/{dev,stg,prod})
- Prometheus + Grafana + Loki

## Conventions
- 모든 변경은 terraform plan 결과 PR 본문에 첨부
- helm values는 environments/ 별로만 다름, charts는 공통
- 시크릿은 SOPS + age (KMS는 prod만)

## Forbidden
- prod에 kubectl apply 직접 (Argo CD 경유)
- terraform apply 자동화 X (사람 승인 필수)
- 콘솔에서 manual 변경 (drift 발생)

## Incident Response
- 처음 5분: 영향 범위 파악 + Slack #incident 채널 알림
- /incident-triage 스킬 사용 권장

자주 쓰는 워크플로우 3가지

  1. 인시던트 트리아지 
    > /incident-triage
# 스킬이 다음을 수행:
# 1) Sentry MCP 로 최근 1시간 에러 톱 5
# 2) Grafana 메트릭 확인 (API p99, error rate)
# 3) 최근 배포 (gh release list)
# 4) 가설 3개 제시 + 다음 확인 단계
  2. Terraform 리뷰 
    > "이 PR의 terraform plan 결과를 보고:
   1) 의도된 변경 vs 의외 변경 구분
   2) 비용 영향 추정
   3) downtime 가능성
   4) 안전한 적용 순서 제안"
  3. K8s 매니페스트 디버깅 
    > "my-service Pod가 CrashLoopBackOff. kubectl describe 와 logs 마지막 50줄 첨부:
   [붙여넣기]
   가능한 원인 3가지 + 각각 검증 명령"
🎬 미니 시나리오 · "5분 만에 장애 원인 좁히기"
상황: 새벽 3시 페이지. /api/* 에러율 급증. 적용: 졸린 머리로 한 줄. 
> "sentry MCP 로 최근 30분 에러 그룹화. 가장 자주 발생한 톱 3.
   각각: 첫 발생 시각, 영향 endpoint, stack trace 핵심 한 줄.
   그리고 gh로 같은 시간대 머지된 PR 확인."
결과: 5분 안에 "새벽 2:55에 머지된 PR #234가 DB connection pool 설정 변경" 식별. 롤백 1분.

# [Project] Mobile Guide (RN)

## Stack
- React Native 0.75 + TypeScript
- Expo SDK 51 (managed)
- Zustand + MMKV (상태)
- Reanimated 3 (애니메이션)

## Build
- iOS dev: `pnpm ios` (M-series Mac 필수)
- Android dev: `pnpm android` (Pixel 에뮬레이터)
- 배포: EAS Build (`eas build --profile production`)

## Platform Rules
- 플랫폼 분기는 Platform.select() 또는 .ios.tsx / .android.tsx
- iOS 의 safe-area는 react-native-safe-area-context
- Android 의 back button은 BackHandler 필수 핸들링

## Submission
- iOS: 매번 What's New 작성, 거부 사유 자주 보는 카테고리 체크
- Android: 정책 변경 분기마다 점검 (특히 광고·IAP)

자주 쓰는 워크플로우 3가지

  1. 플랫폼별 분기 코드 자동 생성 
    > "이 hook 의 햅틱 피드백을 iOS·Android 각각 적절한 방식으로 분기.
   iOS: react-native-haptic-feedback, Android: Vibration API.
   타입 안전성 유지."
  2. 심사 거부 분석 
    > "App Store 거부 메일 첨부: [붙여넣기]
   1) 핵심 사유 한 줄
   2) 우리 코드에서 영향받는 부분 추정
   3) 수정 + 재제출 시 강조할 What's New 초안"
  3. 오프라인 우선 패턴 적용 
    > "src/screens/FeedScreen.tsx 를 오프라인 우선으로 리팩토링:
   - MMKV 캐시
   - 네트워크 복귀 시 자동 sync
   - 옵티미스틱 업데이트 + 롤백"
🎬 미니 시나리오 · "심사 거부 → 재제출 2시간"
상황: App Store 심사 Guideline 4.0 위반으로 거부. 코드 수정이 필요한지 What's New만 고치면 되는지 모호. 적용: 거부 메일·관련 화면 스크린샷 첨부 후 분석 → 수정 → Fastlane으로 재제출. 결과: 사람만 했으면 하루 걸릴 일이 2시간. Claude의 "Guideline 4.0 흔한 패턴" 지식이 큰 도움.
🎓 강사 노트 — 모바일의 함정

빌드 명령은 5~30분 걸립니다. Claude한테 "빌드해서 확인"을 시키면 컨텍스트가 빠르게 소진됩니다. 빌드는 별도 터미널에서 본인이 돌리고, Claude는 코드만 짜게 분리하세요. 또한 시뮬레이터·실기기 차이는 Claude가 잘 모릅니다 — 실기기 테스트는 사람이.

🎓 신입·주니어 학습자

경험 적은 단계에서 가장 큰 위험은 "Claude가 짠 코드를 이해 못 하고 머지"입니다. 반대로 잘 활용하면 최고의 학습 도우미가 됩니다. 균형이 핵심.

CLAUDE.md 패턴 (학습용)

# My Learning Project

## My Level
- 백엔드 1년차, Python 익숙, TypeScript 학습 중

## How to help me
- 코드를 줄 때는 항상 짧은 설명도 함께
- 새 개념이 등장하면 한 문단 설명 + 공식 문서 링크
- 내가 작성한 코드의 문제점은 "왜" 부터 설명
- 모르는 게 있으면 "모른다"고 답해줘 (추측 X)

## Project
- 이 프로젝트는 학습 목적, 출시 X
- 안전·완성도보다 "이해" 우선

학습에 좋은 프롬프트 패턴 5가지

  1. 이해 우선 
    > "이 코드 작성하기 전에:
   - 이 기능에 쓸 만한 접근 2가지를 비교 (각각 장단점 2개씩)
   - 추천 + 이유
   - 그 다음 코드"
  2. "왜 이렇게 했나" 묻기 
    > "위 코드에서 X 대신 Y를 쓴 이유를 초보자도 알 수 있게 설명. 다른 선택지가 왜 별로인지도."
  3. 실수 추적 
    > "내가 짠 이 함수가 작동 안 함. 답을 주지 말고:
   1) 어디부터 디버깅 시작할지 힌트
   2) 어떤 출력을 봐야 하는지
   순서대로 알려줘. 내가 직접 풀어볼게."
  4. 코드 리뷰 받기 (학습용) 
    > "내가 짠 src/utils/date.ts 를 시니어 관점에서 리뷰.
   각 지적마다 '왜 더 좋은가'까지. 스타일이 아니라 사고방식 위주."
  5. 새 기술 학습 
    > "React Query 를 처음 쓴다. 5분 안에 핵심 이해할 수 있게:
   1) 해결하는 문제
   2) useState vs useQuery 차이
   3) 핵심 hook 3가지 + 한 줄 설명
   4) 첫 실수하기 쉬운 함정 2가지"
🎬 미니 시나리오 · "TypeScript 처음, 안전하게 첫 PR"
상황: Python만 하다 처음 TS로 작업. 시니어가 "한번 만들어봐"라고 작은 작업 줌. 잘못된 적용: "Claude한테 다 시켜서 PR" → 리뷰에서 "이게 왜 이렇게 됐어?" → 답 못 함. 좋은 적용: 
> "이 기능을 TypeScript로 만들고 싶다. Python 1년차 관점에서:
   1) 이 작업의 TS 접근 방법 설명 (Python과 다른 점)
   2) 의사 코드(영어 주석) 먼저
   3) 내가 의사 코드 보고 직접 TS로 변환
   4) 변환 후 너에게 리뷰 요청"
결과: 시간은 1.5배 들지만 PR 리뷰에서 모든 라인을 설명 가능. 실력이 늘음.
🎓 강사 노트 — 주니어가 가장 빠지기 쉬운 함정

"이해 없이 머지"는 단기엔 빠르지만 장기엔 본인 실력을 갉아먹습니다. 규칙 하나만 지키세요: "PR에 들어가는 모든 라인을 한 줄씩 설명할 수 있는가?" 답이 "아니"라면 머지 전에 Claude한테 다시 물어 이해부터 채우세요.

또 다른 함정: "Claude가 짠 코드를 본인이 짠 것처럼 말하기". 팀 안에서는 솔직히 "이건 Claude로 만들고 X 부분만 수정했어요"가 결국 신뢰를 더 얻습니다.

닫는 글 — 직군이 바뀌어도 변하지 않는 것

6개 직군의 처방전이 다 달랐지만, 공통점도 명확합니다:

  • Plan mode는 위험 작업에서 모두에게 의무
  • CLAUDE.md는 짧고 구체적으로 (200줄 이내)
  • 본인 직군에 핵심 MCP 1~2개만 신중히 선택
  • 완성 후엔 /code-review → /verify 체인
  • 비용·진척도는 주 1회 확인

도구는 직군마다 다르지만 "신중함 · 검증 · 절제"의 태도는 같습니다. 본인 직군의 처방전을 따르되, 이 태도를 잃지 마세요.
