Chapter 13

프롬프팅 기술 — 같은 도구, 다른 결과

Claude Code가 망가지는 가장 흔한 이유는 도구가 아니라 지시 방식입니다. 9가지 패턴만 손에 익히면 같은 모델로도 결과 품질이 두 배 이상 차이 납니다.

좋은 vs 나쁜 프롬프트 — 전후 비교

나쁜 프롬프트좋은 프롬프트왜 차이가 나는가
"버그 고쳐줘" "POST /users에 빈 email 보내면 500이 떠야 하는데 200이 옴. 어떤 케이스에서 검증이 빠졌는지 찾고 수정해줘" 증상·기대·범위 명시
"리팩터링해줘" "src/auth/login.ts의 함수 1개가 80줄이라 너무 김. 책임 단위로 3개로 쪼개되, 시그니처는 유지하고 테스트는 그린 유지" 대상·기준·제약 명시
"테스트 작성해줘" "src/utils/date.ts의 formatDate 함수에 대해 Vitest로 unit test. 케이스: 정상·null·잘못된 형식·timezone 경계 4가지" 도구·범위·케이스 명시
"이거 어떻게 해?" "이 에러를 해결할 방법 3가지를 짧은 장단점 표로. 코드 수정은 아직 X" 출력 형식·금지 사항
🎓 강사 노트 — 5요소 공식

좋은 프롬프트엔 5가지가 들어 있습니다: ① 무엇을(대상) ② 어떻게(방법·도구) ③ 어디까지(범위) ④ 어떤 형태로(출력) ⑤ 하지 말 것(금지). 한두 개 빠져도 됩니다. 다만 막힐 땐 빠진 항목부터 보충하세요.

패턴 1 · "계획부터 보여줘"

가장 강력하고 가장 적게 쓰이는 패턴. Plan mode를 안 켜도 효과가 비슷합니다.

# 모든 큰 작업 앞에 한 줄
> "... [작업 설명] ... 파일 수정 전에 1) 변경할 파일 목록과 2) 각 파일별로 무엇을 바꿀지 먼저 보여줘. 내 OK 받고 진행."

이 한 줄이 ① 잘못된 방향 차단 ② 본인 머릿속 정리 ③ 토큰 절약(다 한 뒤 되돌리기보다 싸다)을 한꺼번에 줍니다.

패턴 2 · 컨텍스트 주는 3가지 방법

같은 파일을 참조시키는 데 세 가지 방법이 있고, 각각 비용·정확도가 다릅니다.

방법예시특징
경로"src/api.ts를 보고..."가장 가벼움. Claude가 알아서 Read
@-멘션 (IDE)@src/api.ts:42-78특정 줄 범위 정확히 지정
본문 붙여넣기코드 직접 paste가장 비쌈. 외부 코드·로그 출력 같은 비파일성 데이터만
💡 80% 케이스는 경로면 충분

"왜 본문을 다 붙여넣지?" 자문해 보세요. 같은 리포 안 파일이면 경로만으로 99% 충분합니다. 본문 붙여넣기는 토큰 낭비 1위 원인.

패턴 3 · 출력 형식 강제

"표로", "JSON으로", "체크리스트로"를 명시하면 후속 처리가 훨씬 쉬워집니다.

# 표 형식
> "package.json의 dependencies를 [이름 | 현재 버전 | 최신 버전 | breaking?] 4컬럼 표로"

# JSON (자동화에 유용)
> "위 결과를 JSON으로만 출력. 다른 설명 텍스트 X. 키: name, current, latest, breaking"
# → 그대로 파이프로 jq에 넘기기 가능

# 체크리스트 (행동 가능한 형태로)
> "보안 점검 결과를 [심각도] [파일:줄] [수정 제안] 체크박스 마크다운으로"

패턴 4 · 작게 끊기 vs 한 번에 묶기

상황전략예시
각 단계가 독립적 한 번에 묶기 (메시지 수 절약) "5가지 사용성 문제: ① ② ③ ④ ⑤. 순서대로 다 고쳐줘"
이전 결과를 보고 다음 결정 작게 끊기 (방향 보정 가능) 먼저 "원인 분석만" → 결과 보고 → "수정 진행"
실수하면 위험 큼 작게 + Plan mode DB 마이그레이션·결제 로직

패턴 5 · 모델별 다른 프롬프팅

모델잘하는 것프롬프트 스타일
Opus 4.7 복잡한 추론, 설계, 모호한 문제 해석 "이 문제를 어떻게 접근할지 생각해보고..." 같은 여백을 두는 프롬프트
Sonnet 4.6 일반 코딩 작업, 균형 구체적 지시 + 적당한 여백. 본 장의 5요소 공식 적용
Haiku 4.5 명확한 변환·분류·생성 작업 매우 명확한 지시 + 예시 1~2개. 모호함 금물
🎬 시나리오 · "같은 작업, 모델만 바꿔서 비교"
작업: "이 함수에 적절한 한국어 주석을 달아줘" Opus에 위 그대로 — 문맥적·과도하게 친절한 주석 Haiku에 위 그대로 — "적절한"이 모호해서 들쭉날쭉 Haiku용 개선: "각 함수 선언 위에 JSDoc 형식으로 1-2줄. @param·@returns 포함. 함수 본문엔 주석 X" — Opus와 비슷한 품질로 수렴

패턴 6 · 부정문 → 긍정문

"X 하지 마"보다 "Y로 해"가 훨씬 잘 먹힙니다. Claude는 부정문을 가끔 무시합니다.

# ❌ 부정문
> "라이브러리 추가하지 마. 외부 패키지 쓰지 마."

# ✅ 긍정문 (같은 의도)
> "표준 라이브러리만 사용. 의존성 추가 시 먼저 나에게 물어보고 진행."

패턴 7 · 자체 검증 요청

마무리 단계에서 "본인이 확인해" 한 줄 추가:

> "수정 끝나면:
   1) 영향받는 테스트 파일 목록
   2) 그 테스트를 직접 실행해 결과 확인
   3) 빠뜨린 케이스가 없는지 자가 검토"

이 한 단락이 사람이 한 단계 검증한 효과를 냅니다. 시간은 +20초, 사고 차단 효과는 큼.

패턴 8 · "거의 다 됐는데..." 마지막 1마일

90% 완성에서 멈추는 경우, 보통 "이제 끝났네"라고 Claude가 판단합니다. 명시적으로 끝까지 가게 만드는 한 줄:

> "위까지가 80%. 남은 20%(엣지 케이스 · 에러 메시지 · 한국어화 · 접근성)를 끝내야 PR 가능. 모두 마무리해."

"완성 정의(Definition of Done)"를 미리 합의해 두면 더 좋습니다.

패턴 9 · 모호하면 되묻게 만들기

당신이 모호하게 시키면 Claude는 보통 임의로 결정합니다. 이를 막는 한 줄:

> "...작업 설명... 진행 전 확실하지 않은 결정 사항이 있으면 먼저 질문 3개 이내로 물어봐."

특히 디자인·아키텍처·라이브러리 선택처럼 되돌리기 비싼 결정에 유용합니다.

세션 운영·이어가기 전략

한 번 시작한 Claude 세션을 언제 끊고, 어떻게 이어갈지는 프롬프팅만큼 중요한 기술입니다. 잘하면 며칠짜리 작업도 자연스럽게 흘러가고, 못하면 매일 처음부터 다시 설명하는 데 30분씩 씁니다.

세션 길이 결정 — 어디서 끊을까

기준권장 분리 지점
작업 단위기능 1개 / 디버깅 1건 / PR 1개가 끝나면 분리
시간90분 이상이면 잠깐 끊고 정리 권장
컨텍스트 사용률아래 표 기준
주제 변경"이제 다른 일 하자" 싶으면 무조건 새 세션

컨텍스트 사용률별 액션

세션 안에서 /context로 수시 확인. 사용률에 따라 행동을 다르게.

사용률액션이유
~50%그냥 진행여유 있음
50~70%마무리 페이스 빠르게큰 작업 새로 시작 X
70~85%/compact 또는 "결론 요약" 명시다음 작업 위한 공간 확보
85%+handoff 후 새 세션응답 느려지고 품질 저하

이어가기 4가지 패턴

① /compact — 자동 압축 (가장 빠름)

Claude가 알아서 대화를 요약해 컨텍스트를 줄입니다. 즉시 가능, 사람 시간 0초.

> /compact
# 보통 30~50% 절감, 핵심은 보존

한계: 자동 요약이라 무엇이 중요한지 모름. 며칠 단위의 긴 작업엔 부족.

② 명시적 요약 (자동 압축보다 정확)

"무엇이 중요한지" 사람이 알려주면서 요약. /compact보다 효과 큼.

> "지금까지 결론과 남은 액션 3가지만 요약해줘. 시도해본 잘못된 가설은 한 줄로만, 상세 코드 출력은 제외."
> /compact

③ handoff.md — 별도 파일로 인수인계 (가장 강력)

세션 상태를 마크다운 파일로 저장해 다음 세션에서 통째로 읽기. 며칠 ~ 몇 주짜리 작업에 적합.

# 세션 끝낼 때
> "현재 작업 상태를 .claude/handoff/2026-06-20-payment-refactor.md 로 저장해줘.
   형식:
   ## 목표
   ## 현재 진행 상태 (어디까지)
   ## 시도했지만 안 통한 것
   ## 다음 액션 1순위·2순위·3순위
   ## 주의점 / 함정"

# 새 세션 시작할 때
$ claude
> "@.claude/handoff/2026-06-20-payment-refactor.md 를 읽고 이어 작업하자. '다음 액션 1순위'부터 시작."
💡 handoff.md 운영 팁
  • 위치 규칙: .claude/handoff/YYYY-MM-DD-주제.md — 시간순 정렬되고 주제 식별 쉬움
  • 커밋 여부: 팀 작업이면 ✅ 커밋, 혼자면 ❌ .gitignore
  • 완료된 handoff: 분기마다 .claude/handoff/_archive/로 이동 (디렉토리 어수선 방지)
  • 최대 크기: 100줄 넘으면 너무 자세함. 다음 사람(=다음 세션의 나)이 5분 안에 읽을 분량

④ 작업 일지 패턴 — 장기 프로젝트용

며칠 ~ 몇 주 짜리 큰 작업에서 매일 끝낼 때 일지를 적고, 다음날 일지를 먼저 읽고 시작.

# 매일 작업 끝낼 때
> "오늘 작업 일지를 .claude/log/2026-06-20.md 로 작성. 형식:
   ## 오늘 한 것
   ## 발견·결정 사항
   ## 내일 시작점
   ## 미해결 질문"

# 다음날 시작 시
> "@.claude/log/2026-06-20.md 어제 일지. '내일 시작점'부터 이어가자."
🎓 강사 노트 — "사람도 일지에서 회복한다"

이 패턴의 진짜 가치는 Claude가 아니라 당신 본인입니다. 주말 지나고 월요일에 "어디까지 했더라" 헤매는 시간을 0으로 만듭니다. 일지는 Claude뿐 아니라 인간 본인이 다시 읽을 때 가장 유용합니다.

멀티 세션 동시 진행 — worktree 활용

큰 작업과 작은 작업을 한 세션에 섞으면 컨텍스트가 빠르게 오염됩니다. worktree로 분리:

# 메인 작업 — feature 브랜치
$ claude --worktree feature/auth
> "인증 시스템 리팩토링 계속"
# → .claude/handoff/auth-progress.md 도 이 worktree 안에서 관리

# 별도 터미널 — 핫픽스
$ claude --worktree hotfix/email-bug
> "이메일 발송 안 되는 버그 우선 수정"

# 두 세션이 완전 격리 — 파일도 컨텍스트도 안 섞임

자주 쓰는 5가지 세션 운영 프롬프트

  1. 세션 시작 (handoff 받기) 
    > "@.claude/handoff/LATEST.md 읽고 현재 상태 한 문단으로 요약 후, '다음 액션 1순위'부터 시작."
  2. 중간 점검 
    > "지금까지 진행 상황을 5줄로 요약. 다음으로 할 일은? 막힌 지점이 있다면 함께."
  3. 방향 전환 (이전 결정 폐기) 
    > "앞서 X로 가기로 했지만 Y가 더 낫다고 판단됨. 이전 결정 무효화하고, Y 관점에서 영향받는 작업을 재정리."
  4. 마무리·인수인계 
    > "오늘 작업 마무리. .claude/handoff/오늘날짜-주제.md 로 다음 세션의 나에게 인수인계 작성."
  5. 잘못된 방향 빨리 끊기 
    > "잠깐, 지난 3번의 시도가 모두 같은 패턴으로 실패. 멈추고: 1) 공통 실패 원인 가설 2) 다른 접근 2가지 — 작업 전에 먼저 답."

3대 안티패턴 — 절대 피하기

안티패턴왜 나쁜가대신
"잘 만들어줘" "잘"의 기준이 없음 → 평균치 결과 "이 3가지 기준을 만족해야 함: ①... ②... ③..."
긴 대화에 새 주제 끼우기 이전 컨텍스트가 오염 새 세션 또는 /clear로 분리
오류 메시지 통째로 붙이고 "고쳐줘" 맥락 없이 추측 → 엉뚱한 수정 "이 에러가 발생한 상황: [무엇을 시도했는가]. 로그: [stderr]. 가설 3개 제시"

13장 체크리스트

  • ☐ 큰 작업 직전 "계획부터 보여줘" 한 줄
  • ☐ 파일 참조는 본문 paste 대신 경로/멘션
  • ☐ 출력 형식(표·JSON·체크리스트) 명시
  • ☐ Haiku 사용 시 예시 1~2개 포함
  • ☐ "끝나면 본인이 verify" 마무리 한 줄
  • ☐ 부정문은 긍정문으로 변환
  • ☐ "잘 만들어줘" 같은 추상어 추방
