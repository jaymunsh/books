Chapter 03

스킬·서브에이전트 — 재사용 가능한 능력

Claude Code의 진짜 강점은 스킬과 서브에이전트입니다. 반복 작업은 스킬로 박아 넣고, 무거운 작업은 서브에이전트로 분리해 메인 컨텍스트를 보호합니다.

스킬(Skills)이란?

스킬은 재사용 가능한 지시사항 묶음입니다. 마크다운 파일 하나(SKILL.md)에 "이런 식으로 일해줘"라고 적어두면, Claude가 필요할 때 자동으로 가져다 쓰거나 슬래시 명령(/스킬이름)으로 직접 호출할 수 있어요.

스킬 vs 슬래시 명령어 vs CLAUDE.md

구분역할로딩
슬래시 명령(/help·/clear)Claude Code 내장 동작항상
스킬(/code-review·/verify)지시사항/프로세스 묶음. 직접 호출하거나 Claude가 자동 사용필요할 때만 본문 로드
CLAUDE.md매 세션 자동 로드되는 영구 지시항상 (토큰 소모)
🎓 강사 노트 — 셋의 분담

일반적인 규칙: 항상 필요한 것은 CLAUDE.md, 가끔 필요한 것은 스킬, Claude Code 자체 조작은 슬래시 명령. CLAUDE.md가 200줄 넘기 시작하면 절반은 스킬로 옮기세요. 스킬은 본문이 100KB여도 매 세션 토큰을 안 먹습니다.

내장 스킬 — 자주 쓸 것부터

설치 직후부터 바로 쓸 수 있는 내장 스킬 중 실무에서 만나는 빈도순으로 정리했습니다.

스킬한 줄 설명실전 사용
/code-review현재 diff를 리뷰 (correctness · 클린업)커밋/PR 전 자체 리뷰
/simplify/code-review --fix 단축형. 리뷰 결과 자동 적용리뷰 후 정리
/verify앱을 실제로 실행해 동작 확인PR 머지 직전 검증
/run프로젝트 앱 빌드·구동로컬 시연·디버깅
/initCLAUDE.md 자동 생성/갱신새 프로젝트 진입 직후
/reviewGitHub PR 단위 리뷰팀 PR 검토
/security-review보안 관점 변경 검토릴리즈 직전
/deep-research웹 다중 출처 조사 + 검증 + 인용 보고서기술 선택·이슈 조사
/schedule크론 기반 루틴 생성일일 보안 스캔 등
/loop특정 간격으로 명령/스킬 반복"5분마다 CI 결과 확인"
/claude-apiClaude API 코드 작성/디버그 보조SDK 호출부 작업 시
/fewer-permission-prompts자주 승인한 명령을 allow 후보로 자동 추천권한 노이즈 줄이기
/update-configsettings.json 안전 편집(권한·훅·env)설정 변경 시
/keybindings-help단축키 커스터마이징 도우미키 충돌 해결
🎬 시나리오 · "PR 올리기 직전 안전망 (/code-review → /simplify → /verify)"
상황: 큰 기능 작업 끝. 머지하기 전에 한 번 더 점검하고 싶은데 시간은 없음. 이전 방식: 직접 diff 훑기 → 빠뜨림 → PR 받은 리뷰어가 발견 → 왕복 1일. 적용: 3개 스킬을 순서대로. 
> /code-review            # 1) 자체 리뷰 (correctness · 클린업)
> /simplify               # 2) 지적사항 자동 반영
> /verify                 # 3) 앱 실제 실행해 동작 확인
결과: 10분 안에 셀프 QA 완료. 리뷰어가 발견할 거 90%를 미리 차단. PR 라운드트립 1회 단축.

가장 자주 쓰는 한 줄 시연

# 커밋 직전 자체 리뷰
> /code-review

# 리뷰 지적사항 자동 반영
> /simplify

# 실제 앱을 실행해 동작 확인
> /verify

# PR 단위 리뷰 (PR 번호 지정도 가능)
> /review
> /review 1234

# 5분마다 PR 상태 확인
> /loop 5m /review

커스텀 스킬 만들기 — 5분 안에

팀 안에서 반복되는 표준 절차(릴리즈·핫픽스·문서 갱신)는 스킬로 만들어 두면 누구나 똑같이 수행합니다.

디렉토리 구조

.claude/skills/
├── release-check/
│   ├── SKILL.md          # 필수: 본문
│   └── checklist.md      # 선택: 참고 파일
└── apply-style/
    └── SKILL.md

SKILL.md 최소 예시

---
name: release-check
description: "릴리즈 직전 안전 검증 체크리스트를 수행"
---

# Release Check

다음을 순서대로 수행하고 결과를 리포트하라.

1. `npm run lint` 실패 항목 정리
2. `npm test` 결과 (실패한 테스트 파일·이유)
3. `git status` 의 unstaged 파일 목록
4. CHANGELOG.md 의 가장 위 버전과 package.json 버전이 일치하는지
5. 위 4가지를 표로 정리해 출력

저장만 하면 자동으로 인식됩니다. 세션 안에서 /release-check로 호출하세요.

프론트매터 옵션 (자주 쓰는 것만)

필드설명예시
name슬래시 명령으로 노출될 이름 (필수)release-check
description드롭다운에 보일 한 줄 설명 (필수)"릴리즈 전 검증"
autoInvokeClaude가 자동 사용 여부 판단할지true
requiresApproval실행 전 사용자 승인 요구true
💡 팁 — 스킬 본문은 길어도 됩니다

SKILL.md는 필요할 때만 로드되므로 길이에 부담 갖지 마세요. 체크리스트, 가지치기 로직, 예제 명령어 다 넣어도 됩니다. 다만 "언제 이 스킬을 써야 하는지"는 description에 명확히 적어 두는 게 자동 사용을 잘 받게 만드는 비결입니다.

🎬 시나리오 · "팀 공통 릴리즈 절차를 스킬로"
상황: 매주 릴리즈마다 5단계 체크리스트(lint·test·changelog·버전 매칭·git tag)를 사람마다 다르게 빼먹음. 적용: .claude/skills/release-check/SKILL.md를 만들고 커밋 → 팀 전원이 같은 명령으로 검증. 
> /release-check
# → 5단계를 표로 출력 (PASS/FAIL)
# → 실패 항목은 자동 수정 시도
결과: 사람마다 다르던 릴리즈 품질이 일정해짐. 신입도 첫 릴리즈를 사고 없이 수행.

플러그인 스킬 (plugin:skill 형식)

마켓플레이스 플러그인은 스킬을 묶어 배포합니다. 슬래시 자동완성에 플러그인명:스킬명 형태로 노출됩니다.

/claude-hud:setup       # claude-hud 플러그인의 setup
/claude-hud:configure   # 같은 플러그인의 configure

서브에이전트(Subagents) — 컨텍스트 분리

스킬이 "어떻게 할지의 묶음"이라면, 서브에이전트는 "누구에게 시킬지의 분리"입니다. 메인 대화와 별도의 컨텍스트 윈도우를 가지며, 결과만 요약으로 돌려받습니다.

왜 분리하나? — 3가지 이유

  1. 컨텍스트 보호 — 웹 검색·대량 파일 읽기처럼 결과가 큰 작업을 분리
  2. 병렬화 — 여러 모듈을 동시에 분석
  3. 비용 최적화 — 단순 작업은 Haiku 같은 저가 모델로 위임

내장 서브에이전트

이름역할언제
Explore코드베이스 탐색 (파일 검색·grep)"X가 어디 정의돼 있나" 같은 위치 추적
Plan구현 계획 수립큰 변경 전 설계 받기
general-purpose범용 다단계 작업모호한 다단계 작업의 디폴트
claude-code-guideClaude Code 자체에 대한 질문"이 슬래시 명령이 뭐 하지" 등 메타 질의

자동 위임 vs 명시 위임

# [자동 위임] 메인 Claude가 "Explore가 적합" 판단 → 자동으로 분리
> "authMiddleware가 어느 파일에 정의돼 있고 어디서 import 되는지 찾아줘"

# [명시 위임] /agents 명령으로 직접 호출
> /agents

커스텀 서브에이전트 — .claude/agents/

# 디렉토리
.claude/agents/
└── code-reviewer/
    └── agent.md

# agent.md 예시
---
name: code-reviewer
description: "엄격한 코드 리뷰어 — correctness · 성능 · 보안"
model: claude-opus-4-7
tools: [Read, Bash, Grep]
---

당신은 20년차 시니어 리뷰어입니다.
스타일이 아닌 정확성·성능·보안만 본다.
의심스러우면 단정하지 말고 질문하라.
필드설명
name고유 이름
description언제 위임할지 메인 Claude가 판단할 근거
model모델 오버라이드 (Haiku로 두면 비용 절감)
tools허용 도구 (보안 강화 시 Read만 등)
🎬 시나리오 · "코드베이스 처음 만났을 때 (Explore 서브에이전트)"
상황: "authMiddleware가 어디 정의돼 있고 어디서 import되는지" 알고 싶은데, 메인 대화에 grep 결과를 잔뜩 끌어오긴 싫음. 이전 방식: 메인 세션에서 직접 grep → 결과 100줄이 컨텍스트에 박힘 → 다음 작업 토큰 부족. 적용: 그냥 자연어로 시키면 Claude가 알아서 Explore에 위임. 
> "authMiddleware 가 정의된 파일, import 된 파일들을 표로 정리해줘"
# 메인 Claude → Explore 서브에이전트로 자동 위임
# Explore가 격리된 컨텍스트에서 grep · 파일 읽기
# 메인에는 "요약 결과만" 돌아옴
결과: 메인 컨텍스트는 깨끗하게 유지되면서 정답만 받음. 같은 세션에서 후속 작업 가능.

병렬 실행 — 한 번에 여러 에이전트

"검토와 테스트 분석을 동시에" 같은 요청에는 Claude가 한 메시지에서 여러 서브에이전트를 동시에 띄웁니다. 메인 컨텍스트가 깨끗하게 유지되면서 시간도 절약됩니다.

> "PR을 코드리뷰하면서, 동시에 테스트 커버리지도 점검해줘"
# → code-reviewer 와 test-analyzer 가 병렬로 동시 실행
📌 참고 — Subagent · Workflow · Agent Teams 의 차이
  • Subagent: 한 세션 내 컨텍스트 분리 (오늘 다룬 것)
  • Workflow(6장): JS 스크립트로 결정적 오케스트레이션 — 수십~수백 에이전트
  • Agent Teams: 여러 독립 세션 조율 — 더 큰 규모의 장기 작업

실전 레시피 — "스킬 + 서브에이전트" 조합

레시피 1. 커밋 직전 안전 검증

> /code-review
> /simplify        # 지적사항 자동 반영
> /verify           # 앱 실행해서 동작 확인
> /security-review  # 보안 관점 점검

레시피 2. 모르는 코드베이스 빠르게 파악

> /init                                           # CLAUDE.md 초안
> "src/ 전체 구조를 모듈별 한 줄로 요약해줘"     # Explore에 자동 위임
> "엔트리 포인트와 주요 의존성 흐름을 다이어그램 텍스트로"

레시피 3. 이슈 분석 → 패치 → 검증

> "이슈 #234를 읽고 재현 방법과 가능한 원인 3가지 제시"
> "가장 가능성 높은 원인부터 수정 시도해보고, 각 단계마다 테스트 실행"
> /verify

3장 체크리스트

  • ☐ 자주 쓰는 내장 스킬 3개(/code-review, /verify, /security-review) 한 번씩 실행
  • ☐ 본인 팀의 반복 절차 1개를 커스텀 스킬로 만들기
  • ☐ "Explore에게 검색을 위임"하는 패턴 체득 — 메인 컨텍스트가 가벼워짐
  • ☐ (선택) Haiku를 쓰는 비용 절감용 서브에이전트 1개 등록
