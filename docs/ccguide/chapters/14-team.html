Chapter 14

팀·협업 운영 — 혼자가 아니라 같이 쓸 때

한 명이 잘 쓰는 것과 팀이 잘 쓰는 것은 다른 문제입니다. .claude/ 운영, 권한 정책, 비용 정책, AI 코드 리뷰 문화까지 — 팀 도입 직전에 합의해 둬야 할 7가지를 정리합니다.

.claude/ — 무엇을 커밋하고 무엇을 ignore할까

가장 자주 틀리는 첫 결정. 잘못 커밋하면 보안 사고, 잘못 ignore하면 협업 마찰.

파일/폴더커밋?이유
.claude/settings.json✅ 커밋팀 공통 권한·기본값
.claude/settings.local.json❌ ignore개인 모델·테마·API 키
.claude/skills/✅ 커밋팀 표준 작업 절차
.claude/agents/✅ 커밋팀 공통 서브에이전트
.claude/rules/✅ 커밋경로별 코딩 규칙
.claude/worktrees/❌ ignore임시 작업 디렉토리
.mcp.json✅ 커밋 (단, 토큰 ${} 참조)팀 공통 MCP 서버
CLAUDE.md✅ 커밋프로젝트 영구 지시
CLAUDE.local.md❌ ignore개인 작업 노트

권장 .gitignore:

# Claude Code
.claude/settings.local.json
.claude/worktrees/
.claude/cache/
CLAUDE.local.md

팀 권한 정책 — 합의 5단계

"내 PC에선 잘 되는데" 문제의 90%는 권한 차이입니다. 팀 표준 settings.json을 만들 때 합의 순서:

  1. defaultMode 정하기 — 보통 plan이 신입 친화적
  2. 모두가 자주 쓰는 명령(테스트·빌드·git 일부)을 allow로
  3. 위험 명령(prod DB·force push·rm -rf)을 deny로
  4. 편차 큰 명령(deploy·migrate)은 ask로 둬서 의도 확인
  5. 3개월에 1회 회고 — 자주 묻는 것은 allow로 승격, 사고 난 건 deny로

실전 예시 (10명 백엔드 팀):

{
  "defaultMode": "plan",
  "permissions": {
    "allow": [
      "Bash(npm run *)", "Bash(pnpm *)", "Bash(yarn *)",
      "Bash(git log *)", "Bash(git diff *)", "Bash(git status)",
      "Bash(gh pr view *)", "Bash(gh issue view *)",
      "Read(*)"
    ],
    "ask": [
      "Bash(git push *)",
      "Bash(npm publish *)",
      "Bash(*migrate*)",
      "Edit(.env*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Bash(*prod*delete*)",
      "Read(.env.production)",
      "mcp__db__execute_sql"
    ]
  }
}

모델·비용 정책 — 누가 Opus를 쓰는가

"모두 Opus" 정책은 첫 달 청구서를 보면 후회합니다. 역할별·작업별 가이드라인이 현실적.

역할/상황기본 모델업그레이드 조건
일반 기능 개발Sonnet 4.6설계·복잡 디버깅만 Opus
주니어 / 학습 단계Sonnet 4.6막힐 때만 Opus 1회
시니어 / 아키텍처Sonnet 4.6 + Opus 혼용본인 판단
대량 변환·분류 작업Haiku 4.5품질 부족하면 Sonnet
CI 자동 리뷰Sonnet 4.6 (effort medium)대형 PR만 Opus
💡 한 줄 가이드

"Sonnet을 디폴트로, Opus는 의식적 선택"으로 두면 비용·품질 균형이 가장 좋습니다.

CLAUDE.md 충돌 — 누가 쓰고 누가 지키나

여러 명이 CLAUDE.md를 편집하기 시작하면 곧 200줄 → 500줄 → 사실상 폐기로 갑니다. 운영 원칙:

  • 오너 1명을 정한다 (테크리드·플랫폼 담당자)
  • PR 리뷰 의무: CLAUDE.md 변경은 오너 승인 필수
  • 3개월 다이어트: 분기에 한 번 200줄 이하로 자르기 — 안 쓰는 규칙 삭제
  • 경로별 규칙은 .claude/rules/api.md·.claude/rules/ui.md로 분리 (필요할 때만 로드)
  • 개인 노트는 CLAUDE.local.md에 (커밋 안 됨)

신입 온보딩 — 첫 주 체크리스트

신입에게 코드베이스 + Claude Code 둘 다를 동시에 적응시키는 게 핵심. 둘이 서로 가속합니다.

Day코드베이스 적응Claude Code 적응
Day 1리포 clone·실행설치·로그인·/init으로 CLAUDE.md 한 번 훑기
Day 2주요 모듈 코드리뷰 1개Plan mode 켜고 작은 버그픽스 1개
Day 3로컬 환경 완성팀 권한·스킬 익히기 (/agents·/skills)
Day 4첫 PR 작성/code-review로 셀프 리뷰 후 PR
Day 5PR 머지회고 — 무엇이 빠르고 무엇이 어색했나
🎬 시나리오 · "온보딩 메모를 스킬로"
적용: 멘토는 신입에게 매번 같은 설명을 하지 않고 .claude/skills/onboard-day1.md·day2.md를 만들어 둔다. 신입 사용: /onboard-day1을 치면 그날 할 일·관련 코드·체크리스트가 자동 안내. 결과: 멘토 시간 80% 절감. 동시에 신입은 "Claude를 어떻게 쓰는지"를 자연스럽게 습득.

AI가 짠 코드 — 리뷰에서 어떻게 다룰까

가장 많이 도는 논쟁: "Claude가 짠 거니까 어차피 좋겠지" vs "AI 코드는 다 의심부터". 둘 다 틀립니다. 똑같이 다루는 게 정답.

팀 합의 권장 사항

  1. PR 본문에 표시 의무 없음 — "AI로 작성"이라는 라벨이 리뷰어를 편향시킴
  2. 리뷰 기준은 동일 — correctness·테스트·보안·가독성. 출처와 무관
  3. 작성자가 책임자 — Claude가 짰어도 PR을 올린 사람이 모든 라인을 이해해야 함
  4. "안 읽고 머지"는 절대 금지 — 자동 리뷰 봇이 OK 줘도 사람 1명은 봐야 함
  5. 큰 PR 쪼개기 — Claude가 만든 PR이 자주 비대. 리뷰 가능 단위로 쪼개기 요청

비밀 정보 — 5가지 절대 규칙

  • ❌ .claude/settings.json에 API 키·토큰 직접 입력 절대 금지
  • ❌ .mcp.json에 토큰 하드코딩 금지 (반드시 ${ENV_VAR} 참조)
  • ❌ Claude한테 cat .env 시켜서 출력 받은 뒤 그대로 두기 금지 — 토큰이 컨텍스트·로그에 박힘
  • ✅ 환경변수는 OS·shell·1Password CLI 같은 외부 도구로 주입
  • ✅ 정기 git secrets scan (또는 GitHub secret scanning) 활성화
⚠️ 토큰이 transcript에 남는다

Claude 세션 transcript는 ~/.claude/projects/<proj>/sessions/에 저장됩니다. 토큰을 한 번이라도 화면에 띄웠으면 그 파일에 남아 있어요. 의심 시 해당 파일 삭제 + 토큰 즉시 회전.

사례 — 팀이 망친 경우 vs 잘된 경우

🎬 망한 사례 · "권한을 너무 풀어 둔 팀"
상황: 시작 첫 주, 모두가 bypassPermissions 모드로 작업. 사고: 한 개발자의 Claude가 git push --force를 실행 → 메인 브랜치의 동료 커밋 5개 날아감. 교훈: 편함을 위해 안전을 다 끄면 결국 비용을 치름. defaultMode: "plan" + 위험 명령 deny는 최소 안전망.
🎬 잘된 사례 · "리뷰 봇 + 사람 리뷰 이중화"
상황: 30명 풀스택 팀. claude-code-action을 모든 PR에 자동 1차 리뷰. 운영: 봇 리뷰는 5분 안에 와서 명백한 버그·스타일을 잡아 줌 → 사람 리뷰어는 설계·맥락에 집중. 결과: 평균 PR 머지 시간 2일 → 5시간. 사람 리뷰 부담 감소, 품질은 오히려 상승.

14장 체크리스트 (팀 도입 직전)

  • ☐ .gitignore에 Claude 관련 6가지 항목 추가
  • ☐ 팀 settings.json 합의 — defaultMode + allow/ask/deny
  • ☐ 모델·비용 정책 1페이지 문서화
  • ☐ CLAUDE.md 오너 1명 지정
  • ☐ 신입 온보딩 5일 체크리스트 보유
  • ☐ AI 코드 리뷰 가이드라인 합의 (5가지)
  • ☐ 비밀 정보 5가지 절대 규칙 공유
  • ☐ 분기 1회 회고 (권한·스킬·CLAUDE.md 다이어트)
