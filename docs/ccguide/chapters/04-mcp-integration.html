Chapter 04

MCP — Claude를 세상과 연결하기

MCP(Model Context Protocol)는 Claude가 외부 서비스를 도구처럼 다루기 위한 개방 표준입니다. GitHub, Slack, Notion, 데이터베이스, 브라우저 자동화까지 모두 같은 방식으로 연결합니다.

MCP가 무엇이고 왜 필요한가

"Claude한테 우리 GitHub PR 정보를 직접 읽게 하고 싶다", "Slack 메시지에 응답하게 하고 싶다", "PostgreSQL 쿼리 결과를 분석하게 하고 싶다" — 이런 욕구를 매번 따로 통합하지 않아도 되게 만든 게 MCP입니다.

MCP 서버는 특정 서비스를 Claude가 쓸 수 있는 도구로 노출합니다. 서버를 갈아끼우면 Claude 입장에서는 "도구 목록이 바뀐 것"뿐입니다.

📌 참고 — 두 가지 전송 방식
  • HTTP (호스팅) — 서버가 URL로 떠 있음. 보통 인증 토큰 필요.
  • Stdio (로컬) — npx·파이썬 스크립트 등 로컬 실행. 데이터가 내 머신을 떠나지 않음.

claude mcp add — MCP 서버 추가

HTTP 서버 추가

# 기본 형태
claude mcp add --transport http <name> <URL>

# 예: Sentry MCP (인증 필요)
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# 인증은 세션 안에서
> /mcp
# → 서버 선택 → Authenticate → 브라우저에서 OAuth 로그인

Stdio(로컬) 서버 추가

# 기본 형태
claude mcp add <name> -- <실행 명령>

# 예: Playwright 브라우저 자동화
claude mcp add playwright -- npx -y @playwright/mcp@latest

# 예: 내부 파이썬 MCP 서버
claude mcp add my-db -- python3 /opt/mcp/my_db_server.py

설정 파일로 한꺼번에 — .mcp.json

프로젝트 루트에 두면 팀원 모두 같은 MCP 서버를 자동으로 갖게 됩니다.

{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"]
    },
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    },
    "my-db": {
      "type": "stdio",
      "command": "python3",
      "args": ["/opt/mcp/my_db_server.py"],
      "env": { "DB_HOST": "localhost" }
    }
  }
}

관리 명령

claude mcp list              # 연결된 서버 목록
claude mcp get <name>        # 단일 서버 정보
claude mcp remove <name>     # 제거
/mcp                         # 세션 안 GUI (인증 포함)

설정 범위(Scope)

scope의미언제
local (기본)이 프로젝트만, 나만임시·실험
user나, 모든 프로젝트개인 GitHub 토큰 같은 것
project이 프로젝트, 팀 전체.mcp.json 커밋

자주 쓰는 MCP 서버 15선

모든 서버를 다 깔 필요는 없습니다. 당신의 워크플로우에서 자주 손이 가는 도구만 골라 붙이세요.

💡 팁 — Playwright는 정말 쓸모 많다

"이 페이지 스크린샷 찍어줘", "로그인 폼이 제대로 동작하는지 확인해줘"가 가능해집니다. 프론트엔드 일을 한다면 가장 먼저 깔아 보세요.

🎬 시나리오 · "프론트엔드 변경 검증 (Playwright MCP)"
상황: 로그인 페이지 디자인을 바꿨는데, 브라우저에서 확인하려면 dev 서버 띄우고 직접 클릭해야 함. 매번 귀찮음. 적용: Playwright MCP를 깔고 자연어로. 
> "http://localhost:3000/login 에 가서 스크린샷 찍고, 이메일 input의 placeholder가 정상인지 확인"
# → Claude가 Playwright MCP를 호출해 헤드리스 브라우저 띄움
# → 스크린샷 첨부, placeholder 텍스트까지 추출해 표로 보고
결과: "수정 → 스크린샷 확인" 루프가 30초로 단축. 같은 흐름으로 E2E 테스트 시나리오도 작성 가능.

권한·보안 — MCP는 신중하게

MCP 서버는 실제 외부 시스템에 행위를 합니다. 권한을 잘 잠그지 않으면 위험합니다.

3가지 안전 장치

  1. scope: 민감 서버는 project(커밋되는 .mcp.json) 대신 user 로 두기.
  2. permissions: .claude/settings.json으로 MCP 도구 단위 제어.
  3. 토큰: 환경변수 참조. 토큰 자체를 .mcp.json에 적지 말 것.
{
  "permissions": {
    "deny": [
      "mcp__github__*write*",     // GitHub MCP의 쓰기 도구 전부 차단
      "mcp__db__execute_sql"      // 임의 SQL 실행 차단
    ],
    "allow": [
      "mcp__github__list_pulls",
      "mcp__playwright__screenshot"
    ]
  }
}

환경변수로 토큰 주입

{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://mcp.github.com/api",
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    }
  }
}
⚠️ 주의 — DB·결제 등 위험 서버

프로덕션 DB나 결제 API의 쓰기 권한을 가진 MCP는 절대 auto 모드와 함께 두지 마세요. 그런 서버는 항상 defaultMode: "plan"에서만 동작하게 만들고, 도구도 deny로 좁히는 게 안전합니다.

GitHub CLI(gh) — MCP보다 가볍게 쓰기

GitHub은 MCP뿐 아니라 gh CLI도 잘 동작합니다. 권한이 이미 잡혀 있고, Bash로 호출만 하면 끝나서 부담이 적어요.

# 인증 (1회)
gh auth login

# PR 만들기 — Claude가 이 명령을 알아서 호출함
> "현재 변경사항으로 PR 만들어줘. 제목은 변경 내용 요약, 본문은 변경 파일 목록과 테스트 결과"

# 직접 실행 예
gh pr create --title "feat: add auth middleware" --body "..."
gh pr list
gh pr view 1234
gh pr checks 1234
gh api repos/OWNER/REPO/pulls/1234/comments
🎬 시나리오 · "이슈에서 PR까지 한 호흡으로 (gh CLI)"
상황: 이슈 #234 "회원가입 시 이메일 검증 누락"이 들어옴. 보통 이슈 읽기 → 브랜치 → 수정 → push → gh pr create로 5분이 흩어짐. 적용: 한 번에 위임. 
> "gh로 이슈 #234를 읽고, 적절한 브랜치 이름으로 체크아웃, 문제 수정 후 테스트 통과시키면 PR 생성. PR 본문엔 'Fixes #234' 포함"
# Claude가 차례로:
# 1) gh issue view 234 → 본문 파싱
# 2) git checkout -b fix/234-email-validation
# 3) src/auth/signup.ts 의 검증 누락 수정
# 4) npm test 통과 확인
# 5) gh pr create --body "..."
결과: 5분 → 90초. 작은 이슈는 사실상 "지시 → 결과 검토"만 하면 됨.
🎓 강사 노트 — gh vs GitHub MCP

둘 다 됩니다. 짧게 말해: 일반 PR/이슈 작업은 gh CLI가 가볍고 빠릅니다. 긴 컨텍스트 안에서 반복적으로 GitHub을 조회해야 하는 자동화(예: 한 시간 동안 여러 PR을 동시에 다루는 워크플로우)는 MCP가 일관성 있게 동작합니다. 보통은 gh부터 시작하고, 필요할 때 MCP를 추가하세요.

Git Worktree — Claude를 격리해서 안전하게

큰 작업이나 위험한 변경을 시킬 때는 현재 워킹 트리를 건드리지 않고 별도 브랜치 디렉토리에서 작업시키는 게 깔끔합니다. Claude Code는 worktree를 1급으로 지원합니다.

# 새 기능 브랜치를 worktree로 시작
claude --worktree feature/payments
# → .claude/worktrees/feature-payments/ 자동 생성
# → 그 디렉토리에서 Claude 세션이 작동 (메인 작업 영향 X)

# 기존 PR의 브랜치로 진입
claude --worktree "#1234"

# 정리
git worktree list
git worktree remove .claude/worktrees/feature-payments

장점: 메인 작업 디렉토리는 그대로 두고, 여러 브랜치를 병렬로 시킬 수 있고, 실수해도 메인 코드가 안전합니다.

💡 팁 — .gitignore에 worktree 폴더 추가

echo ".claude/worktrees/" >> .gitignore 한 줄로 깨끗하게 운영하세요.

🎬 시나리오 · "위험한 마이그레이션을 worktree에 격리"
상황: ORM을 통째로 갈아엎는 큰 변경. 메인 워킹 트리에서 시도하다 망하면 다른 작업도 영향. 적용: 
# 1) worktree로 격리된 새 세션
claude --worktree refactor/orm-migration

# 2) Claude에게 큰 작업 위임
> "Sequelize에서 Drizzle로 마이그레이션. 모델·쿼리 호출부 전부."

# 3) 메인 디렉토리에서 다른 작업 동시 진행
#    (worktree는 완전 격리됨)

# 4) 마이그레이션이 성공하면 PR, 아니면 worktree 삭제
git worktree remove .claude/worktrees/refactor-orm-migration
결과: 큰 변경의 부담이 사라짐. 망해도 메인은 무사. 성공하면 그대로 PR.

Slack에서 Claude Code 부르기

Anthropic Slack 앱을 설치하면 워크스페이스에서 @Claude 멘션으로 세션을 띄울 수 있습니다.

  1. Slack App Marketplace에서 Claude 검색 → 워크스페이스에 추가
  2. 앱의 App Home → Connect → claude.ai 계정 인증
  3. claude.ai/code에서 GitHub 리포지토리 1개 이상 연결
  4. 채널에서 /invite @Claude
  5. 이슈를 멘션: @Claude 이 PR의 보안 문제 봐줘 → 자동 세션 생성 → Slack에 진행상황·결과·"PR 만들기" 버튼 노출
📌 참고 — 사용량은 개인 계정

각 사용자는 자신의 Claude 계정으로 세션을 돌립니다. Team/Enterprise 플랜이면 세션이 조직에 공개되어 다른 팀원이 이어 작업할 수 있어요.

4장 체크리스트

  • ☐ claude mcp list로 현재 연결된 서버 확인
  • ☐ Playwright 1개라도 깔아 보고 "스크린샷 찍어줘" 시켜 보기
  • ☐ gh auth login → "PR 만들어줘" 시켜 보기
  • ☐ 위험 작업은 --worktree로 격리
  • ☐ MCP 권한은 deny 먼저, allow는 좁게
