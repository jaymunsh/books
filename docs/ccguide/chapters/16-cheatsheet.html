치트시트 & 템플릿 — 인쇄해서 옆에 두기

앞 챕터들의 핵심을 복붙·인쇄 가능한 형태로 모아 둔 부록. 브라우저 인쇄(⌘P)하면 사이드바·복사 버튼이 제거된 책 형태로 출력됩니다.

💡 사용법

각 섹션은 독립적입니다. 필요한 템플릿을 통째로 복사해 본인 프로젝트의 해당 파일에 붙여넣고, 주석을 따라 본인 값으로 교체하세요.

🟡 팀 프로젝트 (50줄 이내)

# [프로젝트명]

## Stack
- Frontend: React 19 + Vite + TailwindCSS in `apps/web/`
- Backend: Hono + Cloudflare Workers in `apps/api/`
- DB: D1 (SQLite). 모든 쿼리는 Drizzle ORM 경유.

## Commands
- Dev: `pnpm dev`
- Test: `pnpm test`
- E2E: `pnpm e2e`
- Deploy: `pnpm deploy` (wrangler)

## Conventions
- TypeScript strict mode, no `any`
- 함수형 컴포넌트 + hooks만 (class 금지)
- API 응답은 `{ ok: boolean, data?, error? }` 형태
- 마이그레이션은 `apps/api/migrations/NNN-...sql` 번호 prefix

## Forbidden
- prod DB에 raw user input 직접 쿼리 금지 (Drizzle만)
- 환경변수는 wrangler.toml의 secret으로만
- `apps/api/core/auth.ts` 수정 시 반드시 integration test 통과

## Common Pitfalls
- Workers는 request body를 한 번만 읽음 (stream 소진)
- D1은 동시 트랜잭션 미지원 — 순차 실행 필수

## Sub-rules
@.claude/rules/api.md
@.claude/rules/ui.md

🔵 오픈소스 프로젝트

# [프로젝트명] — Contributor Guide for Claude Code

## Project
- 한 줄: [무엇을 하는 라이브러리/도구인가]
- License: [MIT/Apache 등]
- Audience: [누가 쓰는가]

## Setup
- Install: `pnpm install`
- Dev: `pnpm dev`
- Test: `pnpm test` (Vitest + 90% coverage 목표)

## Contribution Rules
- 모든 PR: 테스트 추가 + CHANGELOG.md 항목 추가
- Commit: Conventional Commits (feat: / fix: / docs: ...)
- breaking change는 별도 PR + 마이그레이션 가이드 필수
- 라이브러리 의존성 추가 시 메인테이너 사전 동의

## Code Style
- Linter: `pnpm lint` (errors 0)
- Formatter: `pnpm format` (Prettier)

## Architecture
- Public API: `src/index.ts` (이 파일은 SemVer 보호)
- Internal: `src/internal/` (자유롭게 변경 가능)
- Examples: `examples/` (PR마다 갱신 권장)

# Korean Comment Style

새로 작성한 함수/클래스에 한국어 주석을 다음 규칙으로 추가:

- JSDoc 형식, 1~2줄로 간결
- `@param`·`@returns`는 영어 + 한국어 병기
- 함수 본문 내 주석은 **왜** (의도)만, **무엇** (코드 그대로) 금지
- 예시:
  /**
   * 사용자 정보를 외부 API에 동기화 (배치용)
   * @param userId - 동기화할 사용자 ID
   * @returns 동기화 성공 여부
   */

# Deploy to Production

1. main 브랜치인지 확인, 아니면 중단
2. 모든 테스트 통과 확인 (실패 시 중단)
3. CHANGELOG의 최신 버전 한 줄을 사용자에게 보여주고 확인 받기
4. `wrangler deploy --env production`
5. 배포 후 health-check 엔드포인트 호출, 200 확인
6. 실패 시 즉시 `wrangler rollback` 시도
7. 결과를 Slack #releases 채널에 알림

🪝 훅(hooks) 스니펫 10선

모두 .claude/settings.json의 "hooks" 배열에 들어갑니다.

1. 저장 시 자동 포맷

2. 저장 시 자동 린트

3. 커밋 전 자동 테스트

{ "event": "PreToolUse", "tool": "Bash",
  "command": "echo \"$INPUT_command\" | grep -q 'git commit' && npm test || exit 0" }

4. 위험 명령 차단 (스크립트 분리)

{ "event": "PreToolUse", "tool": "Bash",
  "command": "~/.claude/hooks/danger-guard.sh" }
#!/bin/bash
# ~/.claude/hooks/danger-guard.sh (chmod +x)
case "$INPUT_command" in
  *"rm -rf /"*|*"git push --force"*|*"DROP TABLE"*)
    echo "BLOCKED: $INPUT_command"; exit 1 ;;
esac
exit 0

💬 자주 쓰는 프롬프트 5종

1. 안전한 큰 작업 시작

> "[작업 설명]. 파일 수정 전에 1) 변경할 파일 목록 2) 각 파일에 무엇을 바꿀지 먼저 보여줘. 내 OK 받고 진행."

2. 막힌 곳 가설 받기

> "이 증상이 발생: [증상]. 시도한 것: [목록]. 가능한 원인 3가지를 [가능성 높음→낮음]으로 제시, 각각 검증 방법 한 줄씩."

3. 다 끝낸 척 방지

> "수정 끝나면: 1) 영향받는 테스트 목록 2) 직접 실행해서 결과 확인 3) 빠뜨린 케이스 자가 검토"

4. 모호한 결정 되묻기

> "[작업]. 진행 전 확실하지 않은 결정 사항이 있으면 먼저 질문 3개 이내로 물어봐."

5. 인수인계 (새 세션으로 가기 전)

> "새 세션으로 이어가려고 한다. 다음 세션의 나(=너)가 5분 안에 이어 작업할 수 있도록 현재 상태·다음 액션·주의점을 마크다운으로 요약. 파일 수정 X, 출력만."
