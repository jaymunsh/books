기초 개념 — LLM과 Claude Code의 작동 원리

Claude Code의 명령어를 외우기 전에 알아두면 좋은 "왜 이렇게 동작하는가"의 개념들. 깊은 ML 지식 없이도 이해 가능한 수준으로, 그러나 실무에서 결정을 내릴 때 직접 도움이 되는 정도까지 정리합니다.

💡 이 장은 안 봐도 1장으로 갈 수 있습니다

당장 손을 움직이고 싶다면 1장 시작하기로 바로 가세요. 다만 "왜 Claude가 가끔 이상해지지?", "왜 비용이 이렇게 나오지?" 같은 의문이 들면 언제든 여기로 돌아오세요. 대부분의 답이 여기 있습니다.

LLM 동작 원리 — 60초 압축

Claude는 LLM(Large Language Model)의 한 종류입니다. 핵심만 보면 의외로 단순한 구조예요.

① 텍스트는 토큰으로 쪼개진다

입력 텍스트는 모델이 보기 전에 토큰이라는 작은 조각으로 분해됩니다. 영어는 보통 단어 ≈ 토큰, 한국어는 글자 단위에 가깝습니다.

"안녕하세요 Claude"  →  ["안", "녕", "하", "세", "요", " Cla", "ude"]   // 약 7토큰
"Hello Claude"        →  ["Hello", " Cla", "ude"]                // 약 3토큰

실용적 시사점: 한국어가 영어보다 같은 의미를 표현하는 데 약 2~3배의 토큰이 듭니다. 비용·컨텍스트 한도 모두 한국어가 불리. 다만 차이는 줄어드는 추세입니다.

② 모델은 "다음 토큰의 확률"만 예측한다

놀랍게도, LLM이 하는 일은 매 순간 "다음에 올 가장 그럴듯한 토큰"을 확률로 계산하는 것뿐입니다. 이 과정을 한 토큰씩 반복해 긴 답변이 만들어집니다.

입력: "파이썬의 print 함수는"
모델이 본 직후의 다음 토큰 확률:
  " "       (32%)
  " 가장"   (18%)
  " 표준"   (12%)
  ...
→ " "을 골라 출력 → 다시 다음 토큰 예측 → ...

실용적 시사점:

  • "환각"이 일어나는 이유: 모델은 "사실"이 아니라 "그럴듯함"을 예측. 자신이 모르는 영역에서도 자신감 있는 답을 만듦.
  • 같은 질문에 다른 답이 나오는 이유: 확률에서 샘플링하므로 항상 똑같지 않음.
  • 긴 답일수록 누적 오류: 한 토큰 실수가 다음 토큰 예측에 영향.

③ 컨텍스트 윈도우 = 모델이 한 번에 볼 수 있는 토큰

Claude는 최대 200,000 토큰(약 150,000 단어)을 한 번에 볼 수 있고, 1M 모델은 그 5배. 이 안에 시스템 프롬프트·CLAUDE.md·대화 히스토리·첨부 파일이 모두 포함됩니다.

④ 추론(Thinking) — 답 전에 "생각"하는 단계

최신 Claude는 답을 내기 전에 별도의 생각 단계를 거칠 수 있습니다(8장 §Extended Thinking). 마치 사람이 어려운 문제에 "잠깐, 정리해 보자" 하는 것과 비슷.

없을 때있을 때
입력 → 즉시 답입력 → 생각(N 토큰) → 답
빠르고 저렴느리지만 복잡 추론 강함
분류·요약·변환에 적합수학·디버깅·설계에 적합

🔤 꼭 알아야 할 LLM 용어 20선

가이드 본문이나 다른 자료에서 자주 마주치지만 막상 설명은 잘 안 되어 있는 용어들을 카테고리별로 정리. 이 한 섹션만 익혀도 LLM 관련 문서·블로그·논문이 훨씬 잘 읽힙니다.

📦 컨텍스트 · 메모리

Context Window 컨텍스트 윈도우

모델이 한 번에 볼 수 있는 입력 토큰의 최대 크기. Claude는 200k(기본) ~ 1M(확장). 시스템 프롬프트·메모리·대화·첨부가 모두 여기 들어감.

KV Cache Key-Value 캐시

트랜스포머가 매 토큰 생성 시 이전 토큰들의 Key·Value 행렬을 다시 계산하지 않도록 저장한 캐시. 이게 있어서 긴 응답이 빨라짐. Prompt Caching의 기반.

Prompt Caching 프롬프트 캐싱

같은 시스템 프롬프트·문서를 반복 사용할 때 입력 토큰을 캐시해 90% 비용 절감 + 응답 가속. Anthropic은 cache_control로 명시 활성화 (8·15장).

Context Pollution 컨텍스트 오염

긴 대화에서 누적된 잘못된 추측·삭제된 결정이 다음 응답을 흐리는 현상. 해결: /compact·새 세션·handoff.md (12·13장).

🧩 토큰 · 출력

Token 토큰

모델이 텍스트를 처리하는 최소 단위. 영어는 보통 단어 ≈ 토큰, 한국어는 영어의 2~3배 토큰이 필요해 같은 의미라도 비용 증가.

Streaming 스트리밍

완성된 답을 한 번에 받지 않고 토큰이 생성되는 대로 점진 출력. UX 개선용. Claude API는 stream: true로 활성화 (8장).

Temperature 온도

출력의 무작위성. 0이면 항상 가장 확률 높은 토큰만(결정적), 1에 가까울수록 다양·창의적. 코드 작성은 0~0.3, 창작은 0.7~1.0이 일반적.

Stop Token / EOS 종료 토큰

모델이 "응답 끝"을 알리는 특수 토큰. 자체적으로 EOS를 내거나, stop_sequences로 특정 문자열에서 멈추게 강제 가능.

🎓 모델 · 학습

Pretraining 사전 학습

웹·책 등 거대 텍스트로 다음 토큰 예측만 학습하는 단계. 모델의 "지식"이 여기서 형성. 모든 LLM의 기반.

Fine-tuning 파인 튜닝

사전 학습된 모델을 특정 도메인 데이터로 추가 학습해 특화. Claude는 직접 fine-tuning 미지원, 대신 시스템 프롬프트로 적응.

RLHF 인간 피드백 강화학습

"이 답이 더 좋다"는 인간 평가로 모델을 정렬(align). GPT의 핵심 정렬 방법. Claude는 이걸 변형한 Constitutional AI 사용.

Constitutional AI 헌법적 AI

Anthropic 고유 방식. 사람이 매번 평가하는 대신 원칙(헌법)을 모델이 스스로 적용하도록 학습. Claude가 지시를 잘 따르는 이유.

💡 추론 · 패턴

Zero-shot / Few-shot 예시 없음 / 몇 개

Zero-shot: 예시 없이 지시만. Few-shot: 입출력 예시 2~5개를 보여주고 같은 패턴 요구. Haiku 사용 시 few-shot이 품질 향상에 큰 효과.

Chain of Thought (CoT) 사고의 사슬

답 전에 "단계별로 생각해 보자" 식으로 추론 과정을 유도. 수학·논리 문제 정답률 크게 향상. Extended Thinking의 이론적 기반.

Reasoning Tokens 추론 토큰

최신 모델이 답 전에 "생각"하는 데 쓰는 별도 토큰 영역. Claude의 Extended Thinking은 budget_tokens로 양 조절 (8장).

Tool Use / Function Calling 도구 호출

모델이 외부 함수·API를 호출할 수 있게 하는 메커니즘. Claude Code의 Bash·Read·MCP 모두 이 위에서 동작. 응답 대신 tool_use 객체 반환.

🌐 확장 · 통합

RAG Retrieval-Augmented Generation

질문 시점에 외부 DB·문서에서 관련 내용을 검색해 컨텍스트에 함께 넣음. 모델 지식 한계·환각을 줄이는 표준 패턴. Embedding이 기반.

Embedding 임베딩

텍스트를 고차원 숫자 벡터로 변환. 의미가 비슷한 텍스트는 벡터도 가까움. 유사도 검색·RAG의 핵심.

Multi-modal 다중 모달

텍스트 외에 이미지·오디오·PDF도 입력으로 받음. Claude는 vision 지원, Sonnet 4.6 이상은 이미지 분석 우수.

Quantization 양자화

모델 가중치의 정밀도를 낮춰(예: 32→8 bit) 크기·메모리·속도 개선. 로컬 모델(Ollama 등)에서 자주 사용. Claude API는 무관.

🎓 강사 노트 — 외울 게 아니라 만났을 때 다시 찾기

이 20개를 다 외울 필요 없습니다. 다른 챕터·블로그·논문에서 만났을 때 여기로 돌아와 확인하는 정도면 충분. 실무에서 자주 등장하는 것은 Token · Context Window · Prompt Caching · Tool Use · Temperature · RAG 정도. 나머지는 깊이 들어갈 때 만나게 됩니다.

GPT·Gemini·LLaMA 같은 다른 LLM과 비교했을 때 Claude가 잘하는 영역이 분명히 있습니다.

📌 Constitutional AI란?

Anthropic이 Claude를 학습시킬 때 적용한 접근. 사람이 일일이 답마다 평가하는 대신, "도움이 되되 무해하라" 같은 원칙(헌법)을 모델이 스스로 적용하도록 학습. 결과적으로 시스템 프롬프트·CLAUDE.md의 지시를 더 잘 따르고, 안전 측면에서 일관적인 행동을 보입니다.

Claude Code는 어떻게 동작하는가

당신이 claude를 입력하고 메시지를 보내는 순간부터의 흐름을 그려 봅시다.

단계별 자세히

  1. 당신이 메시지 입력 → 터미널에서 엔터
  2. 컨텍스트 조립 — Claude Code가 자동으로 다음을 합칩니다:
    • 시스템 프롬프트 (Claude Code의 기본 지시)
    • 당신의 CLAUDE.md (프로젝트 영구 지시)
    • auto memory (이전에 학습한 내용)
    • 이번 세션의 이전 대화
    • 당신의 새 메시지
  3. 모델 호출 — 전부 토큰화해서 Anthropic API로 전송
  4. 모델이 응답 생성 — 텍스트 답변이거나 도구 호출(tool_use) 요청
  5. 분기:
    • 텍스트면 화면에 표시하고 끝
    • 도구 호출이면 → 권한 검사 → 실행 → 결과를 모델에 다시 보냄 → 다음 응답
  6. 반복 — 모델이 "끝"이라고 판단하거나 텍스트만 반환할 때까지
🎓 강사 노트 — "한 번의 응답 = 여러 도구 호출"

한 메시지에 대한 응답이 5초가 아니라 30초 걸린다면, 보통 그 사이에 Claude가 여러 번 도구를 호출하고 결과를 받고 다음 도구를 호출하는 중입니다. "왜 안 끝나지?" 싶으면 Ctrl+C로 끊고 /context로 무엇이 일어났는지 확인해 보세요.

알아두면 도움 되는 4가지 메탈 모델

Claude를 잘 다루는 사람들은 다음 직관을 공유합니다.

① "Claude는 매번 처음 보는 사람"

새 세션이 시작되면 Claude는 당신을 모릅니다. 어제 했던 작업도, 본인이 만든 약속도 기억 못 합니다. 그래서 CLAUDE.md·handoff.md·auto memory가 필요합니다 — "매번 새로 만나는 동료에게 자기소개를 미리 준비해 두는 것"입니다.

② "Claude는 본인이 한 일을 같은 세션 안에선 잘 기억"

한 세션 안에선 모든 도구 호출·파일 수정·결정이 컨텍스트에 누적됩니다. 그래서 "30분 전에 우리가 결정한 A를 다시 적용해줘"가 잘 동작합니다. 반대로 다른 세션·다른 worktree에서 한 일은 알 수 없습니다.

③ "도구는 Claude의 손, 권한은 안전장치"

Claude 자체는 텍스트만 생성합니다. 파일을 읽거나 명령을 실행하는 실제 행위는 도구가 합니다. 도구 호출 직전에 권한 시스템(2장)이 검사합니다. 그래서 권한 설계가 곧 안전 설계입니다.

④ "토큰 = 비용 = 시간"

모든 입력·출력이 토큰으로 측정되고, 토큰이 곧 청구액과 응답 시간이 됩니다. "이 정보를 매번 보낼 가치가 있나?"를 자문하는 습관이 비용·속도 둘 다 개선합니다(15·17장).

"환각"을 줄이는 4가지 습관

LLM은 본질적으로 환각할 수 있습니다. Claude도 예외가 아닙니다. 다만 줄일 수 있어요.

  1. 맥락 충분히 주기 — 추측 여지를 줄이면 환각이 줄어듭니다. "함수 X 고쳐줘"보다 "src/api.ts:42의 X 함수를 ..."
  2. 검증 가능한 형식으로 요청 — "표로", "JSON으로" 요청하면 모호한 자신감이 사라집니다.
  3. 모르면 모른다고 답하라고 지시 — "확신 없으면 '모릅니다' 또는 추가 질문" 한 줄 추가.
  4. 자체 검증 시키기 — "끝나면 본인이 verify"로 마무리 (13장 패턴 7).

0장 체크리스트

  • ☐ LLM은 "다음 토큰 확률"을 예측한다는 것을 이해
  • ☐ 한국어가 영어보다 약 2~3배 토큰 든다는 점 인지
  • ☐ 컨텍스트 = 시스템 + CLAUDE.md + 메모리 + 대화 임을 안다
  • ☐ "도구 호출 → 권한 검사 → 실행 → 다음 응답" 흐름 이해
  • ☐ 4가지 메탈 모델 중 하나라도 직관적으로 잡힘
  • ☐ "환각 줄이는 4가지 습관" 중 1개 본인 패턴에 도입
  • ☐ 용어 20선 중 자주 만날 6개(Token · Context · Prompt Caching · Tool Use · Temperature · RAG)는 정의를 말할 수 있다
  • ☐ KV Cache와 Prompt Caching의 관계 한 줄로 설명 가능
