Chapter 05

자동화 · CI/CD — 사람 없이 돌게 만들기

Claude Code의 마지막 도약은 "사람이 옆에 없을 때도 돌게 만드는 것"입니다. /loop로 반복, /schedule로 예약, 헤드리스 모드로 CI에 박고, 훅으로 자동 트리거합니다.

/loop — 일정 간격 반복

"5분마다 CI 상태 봐줘", "10초마다 테스트 실행 후 결과 알려줘" 같은 폴링·반복 작업에 적합합니다.

# 명시적 간격 — 5분마다 코드 리뷰
> /loop 5m /code-review

# 10초마다 테스트
> /loop 10s npm test

# Claude가 알아서 페이스 조절 (간격 미지정)
> /loop /deep-research

유용한 조합

# PR 상태가 변할 때까지 폴링
> /loop 60s sh -c "gh pr checks 1234 || echo waiting..."

# 빌드 실패하면 자동으로 분석 → 수정 시도
> /loop 2m "npm run build || /code-review --fix"
💡 팁 — 폴링은 70초 이상으로

너무 짧은 간격(5초·10초)은 외부 API rate limit을 건드릴 수 있고, 토큰을 빠르게 소모합니다. "사람이 결과를 읽고 반응할 만한 최소 간격"을 기준 삼으세요 — 보통 30초 ~ 5분이 무난합니다.

🎬 시나리오 · "긴 CI 끝날 때까지 폴링 (/loop)"
상황: 큰 PR을 올렸는데 CI가 30분짜리. 끝날 때까지 다른 일 하면서 결과를 알고 싶음. 이전 방식: 5분마다 GitHub 새로고침. 집중 끊김 + 완료 시점 놓침. 적용: 
> /loop 90s sh -c 'gh pr checks 1234 --watch || true; gh pr checks 1234 | head -20'
# → 90초마다 상태 확인
# → 실패 발견 시 즉시 멈추고 로그 보여달라고 추가 지시
결과: 다른 작업하다 CI 결과만 적시에 통보받음. 실패 시 바로 원인 분석 흐름으로 진입.

/schedule — 크론 기반 원격 에이전트

/loop가 "지금 세션에서 반복"이라면, /schedule은 "원격 서버에서 정해진 시간에" 작업을 돌립니다. Claude Code를 끄고 자도 됩니다.

# 매일 오전 9시에 일일 리포트 생성
> /schedule "0 9 * * *" /generate-report

# 매주 월요일 오전 10시 코드베이스 스캔
> /schedule "0 10 * * 1" /security-review

# 일회성 예약 — 자연어도 가능
> /schedule "내일 오후 3시" /code-review

스케줄 관리

> /schedule list                # 등록된 루틴 목록
> /schedule run <id>           # 즉시 실행
> /schedule delete <id>        # 삭제

실전 — 매일 보안 점검

> /schedule "0 6 * * *" sh -c '/security-review && gh pr create --title "보안 점검 결과" --body "..."'
🎬 시나리오 · "월요일 아침, 주말 머지본 한 줄 요약 받기 (/schedule)"
상황: 주말에 다른 시간대 팀원들이 머지한 코드가 쌓여 있음. 월요일에 일일이 들여다보기 힘듦. 적용: 매주 월요일 새벽 7시 자동 실행. 
> /schedule "0 7 * * 1" sh -c '
  gh pr list --search "merged:>=$(date -v-3d +%Y-%m-%d)" --json title,author,url \
    | claude -p "이 PR 목록을 한 줄씩 요약: 영향 범위·리스크" \
    | slack-cli post --channel "#monday-digest"
'
결과: 월요일 9시 출근하면 슬랙에 주말 변경사항 한눈 요약이 있음. 5분 만에 컨텍스트 따라잡음.
🎓 강사 노트 — 크론 표현식 안 외워도 됩니다

실수 줄이려면 crontab.guru에서 확인하거나, 자연어("매일 오전 9시")로 지정해 보세요. /schedule list로 한 번 확인하는 습관을 들이면 "왜 안 돌지?"가 줄어듭니다.

헤드리스 모드 — CI/CD에서 Claude 부르기

claude -p는 상호작용 없이 stdout으로만 결과를 뱉습니다. 이게 곧 헤드리스 모드입니다. CI 파이프라인, 크론, 쉘 스크립트 어디에서나 호출할 수 있어요.

# 가장 단순한 형태 — 질문 → 답변만 출력
claude -p "package.json의 dependencies를 알파벳 순서로 정렬한 JSON만 출력"

# 스킬 호출 — PR 자체에서 리뷰 코멘트 생성
claude -p /code-review --comment --effort high

# 출력 형식 지정
claude -p /security-review --output json > security.json

GitHub Actions 예제 — PR 자동 리뷰

# .github/workflows/claude-review.yml
name: Claude Code Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # diff 전체 필요

      - name: Setup Claude Code
        run: |
          curl -fsSL https://claude.ai/install.sh | bash
          echo "$HOME/.local/bin" >> $GITHUB_PATH

      - name: Run review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p /code-review --comment --effort high

GitLab CI 예제

# .gitlab-ci.yml
code-review:
  stage: review
  image: node:20
  before_script:
    - curl -fsSL https://claude.ai/install.sh | bash
    - export PATH="$HOME/.local/bin:$PATH"
  script:
    - claude -p /code-review --comment --effort medium
  only:
    - merge_requests
  variables:
    ANTHROPIC_API_KEY: $CI_ANTHROPIC_KEY
🎬 시나리오 · "외부 컨트리뷰터 PR 자동 1차 리뷰 (헤드리스 CI)"
상황: 오픈소스 리포에 외부 PR이 매주 10개씩 들어옴. 메인테이너가 일일이 처음부터 보긴 부담. 적용: GitHub Actions에서 PR마다 Claude가 1차 리뷰 → 메인테이너는 머지 직전 최종 점검만. 
# .github/workflows/claude-first-pass.yml
on:
  pull_request_target:
    types: [opened, synchronize]
jobs:
  first-pass:
    if: github.event.pull_request.author_association != 'MEMBER'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { ref: '${{ github.event.pull_request.head.sha }}' }
      - run: curl -fsSL https://claude.ai/install.sh | bash
      - env: { ANTHROPIC_API_KEY: '${{ secrets.ANTHROPIC_API_KEY }}' }
        run: claude -p /code-review --comment --effort medium
결과: 메인테이너 리뷰 시간 60% 절감. 컨트리뷰터도 즉시 피드백 받음.
⚠️ 주의 — CI에서 권한·비용
  • CI 환경은 비대화식이므로 권한 프롬프트가 멈춥니다. --permission-mode acceptEdits 같은 옵션 또는 .claude/settings.json의 allow 규칙을 미리 갖춰야 합니다.
  • 큰 리포지토리에서 매 PR마다 Opus로 돌리면 비용이 큽니다. --effort medium 또는 Sonnet 4.6 정도가 무난합니다.

훅으로 "내 환경 자동화"

5장 자동화의 마지막 조각은 2장에서 잠깐 본 훅입니다. 자주 만들어두면 좋은 4가지를 소개합니다.

1) 저장 후 자동 포맷 + 린트

{
  "hooks": [
    {
      "event": "FilesModified",
      "command": "npx prettier --write $INPUT_paths && npx eslint --fix $INPUT_paths"
    }
  ]
}

2) 커밋 전 자동 테스트

{
  "hooks": [
    {
      "event": "PreToolUse",
      "tool": "Bash",
      "command": "~/.claude/hooks/pre-commit-tests.sh"
    }
  ]
}
#!/bin/bash
# ~/.claude/hooks/pre-commit-tests.sh
# git commit 명령에 한해 테스트 자동 실행
if [[ "$INPUT_command" == *"git commit"* ]]; then
  npm test || exit 1
fi
exit 0

3) 세션 시작 시 환경 점검

{
  "hooks": [
    {
      "event": "SessionStart",
      "command": "node -v && npm -v && gh auth status 2>&1 | head -3"
    }
  ]
}

4) 위험 명령 막기 (안전망)

#!/bin/bash
# ~/.claude/hooks/danger-guard.sh
case "$INPUT_command" in
  *"rm -rf /"*|*"git push --force"*|*"DROP TABLE"*)
    echo "BLOCKED: dangerous command"
    exit 1
    ;;
esac
exit 0
🎓 강사 노트 — 훅은 "안 보이는 안전벨트"

훅의 가장 큰 가치는 실수가 일어나지 않게 만드는 것입니다. 위 4개를 깔아 두면, Claude가 잘못된 명령을 시도해도 자동으로 막히거나, 자동으로 정리됩니다. 한 번 깔면 평생 갑니다.

실전 레시피 — "Claude를 야간에 출근시키기"

전 회사에서 잘 썼던 패턴 하나 소개합니다. 매일 새벽 5시에 다음 4가지를 자동 실행:

  1. 밤사이 머지된 PR 목록을 정리
  2. 실패한 야간 빌드를 분석해 원인 후보 작성
  3. 의존성 보안 취약점 점검
  4. 위 셋을 묶어 Slack 채널에 아침 리포트 게시
# 단일 스킬로 묶기 — .claude/skills/morning-report/SKILL.md
---
name: morning-report
description: "야간 PR/빌드/보안을 정리한 아침 리포트 생성"
---

1. `gh pr list --search "merged:>=$(date -v-1d +%Y-%m-%d)"` 결과를 표로
2. 가장 최근 실패한 CI 빌드 로그를 가져와 원인 후보 3개
3. `npm audit` 결과 중 high·critical만
4. 위를 묶어 마크다운으로 출력
# 스케줄 등록 — 매일 새벽 5시
> /schedule "0 5 * * *" sh -c 'claude -p /morning-report | slack-cli post --channel "#dev-morning"'

출근하면 Slack에 어제의 요약이 와 있어요. 사람이 30분 들이던 일이 0분이 됩니다.

자동화 패턴 한눈에

상황도구예
지금 세션에서 반복/loop5분마다 빌드 상태 확인
정해진 시간에 자동/schedule매일 오전 9시 일일 리포트
CI/CD에서 호출claude -p (헤드리스)PR마다 자동 리뷰
이벤트 트리거훅 (.claude/settings.json)저장 시 자동 포맷, 커밋 전 테스트
외부에서 멘션Slack 통합 (4장)@Claude PR 만들어줘

5장 체크리스트

  • ☐ /loop 30s npm test 한 번 실행해 보기
  • ☐ /schedule로 의미 있는 일일 자동화 1개 등록
  • ☐ claude -p /code-review를 CI에 추가 (GitHub Actions or GitLab CI)
  • ☐ danger-guard.sh 같은 안전 훅 1개
  • ☐ FilesModified 훅으로 저장 시 자동 포맷
