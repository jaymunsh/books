Chapter 11 · 스토리텔링

실무 빌딩 스토리 — 서비스 0 → 1까지

앞서 9장까지의 기능들이 실제 서비스 구축에서 어떻게 조합되는지, 세 가지 다른 색깔의 프로젝트를 날짜별 일지처럼 풀어 봅니다. 같은 도구라도 어디까지 쓰느냐에 따라 결과물의 결이 완전히 달라집니다.

📌 이 장의 사용법

각 스토리는 독립적입니다. 본인 상황에 가까운 것부터 읽으세요. 스토리 안의 프롬프트와 출력은 실제 작업 흐름을 정리한 것이며, 그대로 따라 해도 비슷한 결과가 나옵니다.

세 스토리 한눈에 비교

스토리만든 것스킬 사용기간비용*
A · 미니멀 결혼식 안내 정적 사이트 ❌ 스킬 없음 (그냥 claude) 주말 하루 ~$1
B · 풀 자동화 URL 단축 SaaS (인증·관리자·과금) ⭐ 적극 활용 (superpowers · MCP · CI Action · ccusage) 2주 ~$45
C · 균형 가족 공유 소비 트래커 PWA 🎯 핵심 2~3개만 (/init·/code-review·gh) 3~4일 ~$8

*비용은 API 크레딧 환산. Pro/Max 구독 사용자는 무료 한도 내에서 동일 작업 가능한 범위입니다.

🎓 강사 노트 — "스킬을 더 많이 쓰는 게 항상 좋은가?"

아니요. 작은 작업에 무거운 셋업은 오히려 방해입니다. 스토리 B처럼 장기 유지보수 + 팀 협업이 들어가야 비로소 도구를 다 끄집어내는 게 이득입니다. 스토리 A·C처럼 가벼운 작업은 "그냥 Claude에게 시키기"가 가장 빠릅니다. 도구는 무게가 정당화될 때만 추가하세요.

🎓 강사 노트 — "스킬 없이도 충분한 경우"

이런 프로젝트는 스킬·MCP·CI를 깔면 오히려 손해입니다. 셋업 시간이 작업 시간보다 길어지거든요. 조건은 분명합니다: ① 일회성 ② 혼자 ③ 짧음 ④ 위험 작업 없음. 이 네 가지가 다 맞으면 그냥 claude만 켜고 대화하세요.

한 가지 J가 잘한 것: "파일 만들기 전에 계획부터 보여줘"라고 지시해 첫 골격이 잘못된 방향으로 가지 않게 잡은 것. Plan mode를 안 켰지만 효과는 비슷합니다.

📖 스토리 B · URL 단축 SaaS — 스킬을 적극 활용

주인공스타트업 3년차 풀스택 K. 사이드 프로젝트로 URL 단축 서비스를 만들어 유료화 시도.
목표실제 사용자 받을 production 서비스. 인증·관리자·과금·통계 완비.
스택Cloudflare Workers + D1(SQLite) + Hono · React 대시보드. 단일 리포 모노레포.
전략혼자라서 더더욱 자동화·자체 QA·안전망이 필요. 스킬·MCP·CI Action·모니터링까지 풀로.
사용 모델일반 작업 Sonnet 4.6, 설계·복잡 디버깅 Opus 4.7. 자동화는 Haiku 4.5도 일부.

🛠 Day 0 (월요일 저녁) — 환경 셋업이 곧 자산

K는 코드 한 줄도 짜기 전에 2시간을 셋업에 투자했다. "이게 다 길게 보면 회수된다"는 신념.

1) 권한·모드 기본값

// .claude/settings.json (커밋 대상)
{
  "defaultMode": "plan",
  "model": "claude-sonnet-4-6",
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(pnpm *)",
      "Bash(wrangler *)",
      "Bash(gh pr *)",
      "Bash(gh issue *)",
      "Bash(git log *)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(wrangler d1 execute *delete*)",
      "Bash(wrangler d1 execute *drop*)"
    ]
  }
}

2) 커뮤니티 도구 4종 도입

  • superpowers — TDD 사이클 강제. /plugin install superpowers@claude-plugins-official
  • Playwright MCP — 대시보드 E2E 검증. claude mcp add playwright -- npx -y @playwright/mcp@latest
  • anthropics/claude-code-action — PR 자동 1차 리뷰
  • ccusage — 비용 모니터링 (별도 설치 불필요, npx)

3) CLAUDE.md 초안

# Linkly — URL 단축 SaaS

## Stack
- Cloudflare Workers + D1 + Hono (apps/api)
- React + Vite + TailwindCSS (apps/web)
- pnpm workspaces, Turbo

## Commands
- Dev: `pnpm dev` (api + web 동시)
- Test: `pnpm test` (Vitest)
- E2E: `pnpm e2e` (Playwright)
- Deploy: `pnpm deploy` (wrangler)

## Conventions
- TypeScript strict mode
- API 응답은 항상 `{ ok: boolean, data?, error? }` 형태
- 모든 DB 마이그레이션은 `apps/api/migrations/` 에 번호 prefix

## Forbidden
- D1 prod 쿼리에 raw user input 직결 금지 (Drizzle ORM만)
- 환경변수는 wrangler.toml의 secret으로만
💡 셋업 시간 = 신뢰 시간

이 2시간 덕분에 이후 K는 모든 세션에서 "잠깐, 그건 하지 마"라고 끊을 일이 거의 없었다. CLAUDE.md의 Forbidden 섹션이 특히 효자.

🚀 Day 1~2 (화·수) — 핵심 기능을 TDD로

K는 첫 기능 "단축 URL 생성/리다이렉트"부터 superpowers의 사이클을 그대로 탔다.

> /superpowers:plan-feature "POST /links 단축 URL 생성, GET /:slug 리다이렉트"

# Claude의 응답 (요약):
# 1. Spec: 입력/출력 스키마, 에러 케이스 6개
# 2. Test plan: unit 4개, integration 3개
# 3. Implementation outline: 5개 파일 신규
# 4. Risk: slug 충돌, race condition, custom slug 검증

K는 spec을 보고 "expire_at 필드도 넣자"고 추가 요청. Claude가 spec을 업데이트했다.

이어서 RED → GREEN → REFACTOR.

> "이제 RED phase. 실패하는 테스트를 먼저 작성. 구현은 절대 건드리지 마."
> "GREEN: 가장 단순한 형태로 테스트만 통과시켜. 최적화 X."
> "REFACTOR: 중복 제거, 함수 분리. 단, 테스트는 계속 그린이어야 함."

각 phase 사이에 K는 git commit을 끊었다. "RED 커밋이 남아 있으면 나중에 회귀 테스트가 더 잘 잡힌다."

🔍 Day 3 (목) — 셀프 코드 리뷰 + 동적 검증

핵심 기능이 완성된 시점에서 K는 PR을 올리기 전 두 단계를 거쳤다.

> /code-review --effort high
# → 11개 findings (real bug 2, refactor 5, nit 4)

> /simplify
# → real bug 2개 + refactor 5개 자동 패치
# → nit는 K가 직접 수동 검토 (취향 영역)

> /verify
# → pnpm dev 띄움
# → Playwright MCP로 단축 URL 생성·리다이렉트 시나리오 자동 실행
# → 스크린샷 첨부, "redirect 302 OK" 확인

"리뷰어 한 사람을 옆에 둔 느낌"이라고 K는 회상했다. 발견된 진짜 버그 2개 중 하나는 슬러그 충돌 시 race condition으로 같은 슬러그가 두 번 생성될 수 있음이었다. 사람이라면 놓쳤을 수 있는 케이스.

🔐 Day 4 (금) — 보안 점검

> /security-review --effort high

# Claude가 발견:
# 1. /admin 라우트에 RBAC 미적용 — CRITICAL
# 2. custom slug에 javascript: 스킴 허용 가능성 — HIGH
# 3. rate limit이 IP 단위만 — Cloudflare 환경에선 부족
# 4. 비밀번호 hashing이 PBKDF2 100k iter — 살짝 약함

K는 1·2번을 즉시 수정, 3번은 Cloudflare Turnstile로 보강, 4번은 Argon2id 마이그레이션으로 다음 스프린트 백로그에 등록.

🤖 Day 5 (월) — PR 자동 리뷰 봇 활성화

혼자 작업이지만 K는 매 PR마다 "또 하나의 리뷰어"를 두기로 결정. .github/workflows/claude.yml에 공식 액션 설정.

name: Claude Review
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  review:
    runs-on: ubuntu-latest
    permissions: { contents: read, pull-requests: write, issues: write }
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: "/code-review --comment --effort medium"

이후 PR을 올릴 때마다 봇이 5~10분 안에 인라인 코멘트로 리뷰. K는 자기 자신의 코드를 더 객관적으로 보게 됐다.

📡 Day 6~10 — 대시보드 · 과금 · 통계

두 번째 주는 React 대시보드와 Stripe 과금. 패턴은 같다: /superpowers:plan-feature → TDD → /code-review → /verify. 특히 결제 같은 민감 영역은 K가 Plan mode를 절대 안 끄고 작업했다.

🎬 막힌 곳 — "Stripe webhook 서명 검증 실패"
증상: 로컬 테스트는 통과, 배포 후 Stripe webhook이 401. K의 행동: 새 세션에서 모델을 Opus 4.7로 올리고 /effort xhigh. 
> "이 webhook 서명 검증이 로컬에선 OK, 프로덕션에서만 401. wrangler 로그 첨부:"
> [wrangler tail 로그 50줄 붙여넣기]
Claude의 추론: Cloudflare Workers는 request body를 한 번 읽으면 stream이 소진됨 → 서명 검증 전에 라우터가 JSON parse를 먼저 함 → raw bytes가 깨짐. 해결: webhook 라우트에 한해 body를 raw로 보존하는 미들웨어 추가. 20분 만에 해결.

#  Wk 2 (Mon-Fri)   2.1M / 190k       $16.90
#  Total            6.3M / 570k       $45.30

K는 "Opus 사용 비중이 너무 높다"는 결론을 내고, 다음 스프린트부터는 설계·디버깅만 Opus, 일반 작업은 Sonnet으로 명확히 분리.

📊 결과 정리

총 작업 시간10 영업일 (하루 평균 4~5시간)
사용 토큰6.3M 입력 / 570k 출력
실제 비용$45 (API 직접) — Pro/Max였다면 한도 내
발견·차단된 버그코드리뷰 13건, 보안 4건 (그 중 CRITICAL 1건)
설치한 도구superpowers · Playwright MCP · claude-code-action · ccusage
출시 후 1주 운영장애 0건. 가장 비싸 보였던 셋업이 가장 큰 보험이었음.
🎓 강사 노트 — "혼자일수록 자동화가 사람 몫"

흔히 자동화는 "팀 규모가 클 때나 필요한 것"이라 생각합니다. 사실 정반대입니다. 혼자 작업할수록 사람 리뷰어가 없으니 자동 리뷰가 절실하고, 사람 QA가 없으니 /verify가 절실합니다. K의 선택은 "혼자라서 적게 쓴다"가 아니라 "혼자라서 더 많이 쓴다"였고, 결과는 옳았습니다.

또 한 가지: Plan mode는 결제·인증·DB 마이그레이션에서만큼은 절대 끄지 마세요. Claude가 한 줄 잘못 쓰면 돈이 사라집니다.

📖 스토리 C · 가족 공유 소비 트래커 PWA — 핵심만 골라서

주인공5년차 모바일 개발자 S. 부부 둘이 쓸 소비 트래커가 필요. 시판 앱들이 다 무겁고 광고 많음.
목표"우리 부부 둘만 쓸" 단순 PWA. 카테고리 5개, 월별 합계, 그래프 하나. 끝.
스택React + Vite + Dexie(IndexedDB). 백엔드 없음. GitHub Pages 호스팅.
전략스토리 A처럼 "맨손"은 아니고, B처럼 "풀 셋업"도 아닌 중간. /init과 /code-review 둘만 의도적으로 선택.
사용 모델Sonnet 4.6 일관

🌱 Day 1 (수요일) — 형식 잡기

S는 새 Vite 프로젝트를 만들자마자 /init으로 CLAUDE.md 초안을 받았다. 잘 쓰진 않지만 "신호용"으로 가치 있다고 본다.

pnpm create vite cashbook -- --template react-ts
cd cashbook && claude
> /init

그 다음, S는 자기가 신경 쓰는 부분만 CLAUDE.md에 덧붙였다. 길어지면 의미 없으니 5줄만.

## My rules (custom)
- 모든 컴포넌트는 함수형, hooks만 사용
- 상태관리는 Zustand만 (Redux 금지)
- IndexedDB 직접 호출 X, src/db/queries.ts 만 경유
- Tailwind class 우선, styled-components 금지

이게 전부. 그리고 바로 본격 작업.

> 이 프로젝트의 핵심 화면 3개를 짜자.
  1) 거래 입력 (+/- · 금액 · 카테고리 5종 · 메모)
  2) 월별 리스트 (날짜 그룹)
  3) 월별 합계 + 카테고리별 도넛 차트 1개
  각 화면을 한 파일씩, 디자인은 미니멀하게. CRUD 로직은 src/db/queries.ts 에 다 모아.

30분 만에 3개 화면이 동작. S는 핸드폰에서 PWA로 설치하고 실제 영수증 몇 개를 입력해 봤다. "어색한 부분만 메모해 두자."

🔧 Day 2 (목요일) — 사용성 다듬기

S는 핸드폰에서 발견한 짜증 5개를 한 번에 정리해서 던졌다. 한 메시지에 묶어 보내는 게 답변이 일관적이다.

> 핸드폰에서 써보니 5가지가 어색하다.
  1) 입력 후 키보드가 안 닫힘
  2) 금액 input이 그냥 text라 숫자 키패드 안 뜸 (iOS Safari)
  3) "오늘" 버튼이 없어서 매번 날짜 픽커 만짐
  4) 월 이동 화살표가 너무 작아서 잘 안 눌림 (44px 미만)
  5) 다크모드 안 됨
  순서대로 다 고쳐줘. 각 수정마다 한 줄 commit message도 같이 제안.

Claude는 5개를 차례로 수정하고, 각 수정마다 한 줄 commit 메시지를 함께 제공. S는 그대로 5개의 작은 커밋을 만들었다. git log가 깔끔하게 떨어진다.

🧪 Day 3 (금요일) — 셀프 리뷰 한 번

출시 전에 S는 /code-review를 한 번만 돌렸다. 다른 스킬은 굳이 안 쓴다.

> /code-review --effort medium

# 7개 findings:
#   real bug: 1 (월 이동 시 음수 month 시 NaN)
#   refactor: 3
#   nit: 3

real bug는 즉시 수정. refactor는 "당장 안 해도 됨" 1개와 "확실히 더 깔끔" 2개로 분류해 2개만 반영. nit는 무시. 완벽함보다 "납득 가능한 품질"에 멈추는 게 사이드 프로젝트의 미덕.

🚢 Day 4 (토요일) — 배포 + 아내에게 공유

> GitHub Pages로 배포하고 싶다. Vite 설정과 gh-pages 액션까지 한 번에.

# Claude가 차례로:
# 1) vite.config.ts 에 base 경로 설정
# 2) .github/workflows/deploy.yml 생성
# 3) gh repo create + push
# 4) Pages 활성화 명령 안내

> gh pr create --title "초기 배포 셋업" --body "GitHub Pages 자동 배포 워크플로우 추가"

30분 뒤 https://USERNAME.github.io/cashbook에서 동작. 아내가 핸드폰에서 PWA로 설치하고 첫 거래를 입력했다. "광고 없는 게 진짜 좋다"는 평.

📊 결과 정리

총 작업 시간4일 · 누적 약 12시간
사용 토큰약 800k
실제 비용$8 안쪽 (Sonnet 4.6)
설치한 도구0개 추가 (Claude Code 기본)
쓴 스킬/init, /code-review — 단 2종
기타gh CLI 1회 사용 (PR 생성)
🎓 강사 노트 — "절제의 미학"

S는 도구를 더 쓸 줄 알면서도 일부러 안 썼습니다. 이유는 분명합니다: "이 프로젝트는 단 두 명을 위한 것이고, 절대 SaaS로 키울 생각 없다." 그러면 superpowers·CI·MCP는 오버킬이고, 셋업 시간이 작업 만족도를 갉아먹습니다.

대신 "꼭 필요한 두 가지"는 살려뒀습니다: /init으로 시작점을 명확히, /code-review로 출시 직전 안전망 한 번. 이게 "균형" 패턴의 핵심입니다.

본인이 진행하는 프로젝트에 대해 "이건 어느 스토리에 가까운가"를 물어보세요. 답이 나오면 도구 선택이 자동으로 결정됩니다.

세 스토리에서 추리는 5가지 교훈

  1. 도구의 무게는 프로젝트의 무게에 맞춰라. 무거운 셋업은 무거운 작업에서만 회수됩니다.
  2. "파일 만들기 전에 계획부터"는 Plan mode를 안 켜도 효과를 냅니다. 명시적으로 지시하세요.
  3. 혼자일수록 자동 리뷰·자동 검증의 가치가 큽니다. 사람 리뷰어가 없으니 도구가 그 역할.
  4. CLAUDE.md는 짧을수록 좋다. 스토리 B의 K도 30줄, 스토리 C의 S는 10줄. 길수록 무시당합니다.
  5. 비용은 매주 한 번 확인. ccusage 한 줄이면 됩니다. 비싼 패턴은 발견하자마자 교정.
💡 이 장 다 읽었으면 다음 할 일

본인 프로젝트 1개를 떠올리고, 스토리 A·B·C 중 가장 가까운 것을 골라 그 셋업을 그대로 따라 해 보세요. 한 번 손에 익으면 다음 프로젝트에서 망설임이 사라집니다.
