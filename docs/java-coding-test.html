05.입출력 — 모든 문제의 출발점

"문제 풀이는 잘했는데 시간 초과가 나요." 십중팔구 입출력이 범인입니다. 이 장만 외워도 합격률이 올라갑니다.

왜 Java의 입출력은 느린가

Java의 기본 입출력 클래스인 Scanner와 System.out.println은 입문자에게 친절하지만, 코딩 테스트에서는 치명적으로 느립니다. 입력이 10만 줄만 넘어가도 시간 초과가 납니다. 이유는 단순합니다.

  • Scanner는 내부적으로 정규식과 동기화 처리를 거쳐 한 번 읽을 때마다 오버헤드가 큽니다.
  • System.out.println은 호출할 때마다 버퍼를 즉시 비웁니다(flush). 100만 번 호출하면 100만 번 flush합니다.
!한 줄 규칙
Scanner는 잊으세요. 입력은 BufferedReader, 출력은 StringBuilder + BufferedWriter. 이 패턴 하나로 시간 초과의 90%가 사라집니다.

입출력 보일러플레이트 — 외워서 시작하세요

모든 문제의 시작점입니다. 처음 30번은 손으로 직접 쓰고, 그 다음부터는 머릿속에서 자동으로 나와야 합니다.

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringBuilder sb = new StringBuilder();

        // 1. 한 줄 읽기
        int n = Integer.parseInt(br.readLine().trim());

        // 2. 공백으로 구분된 숫자 여러 개 읽기
        StringTokenizer st = new StringTokenizer(br.readLine());
        int a = Integer.parseInt(st.nextToken());
        int b = Integer.parseInt(st.nextToken());

        // 3. 출력은 sb에 모았다가 마지막에 한 번에
        sb.append(a + b).append('\n');

        System.out.print(sb);
        br.close();
    }
}

상황별 입력 패턴 7가지

패턴 1 — 정수 한 개

int n = Integer.parseInt(br.readLine().trim());

패턴 2 — 한 줄에 정수 여러 개 (공백 구분)

StringTokenizer st = new StringTokenizer(br.readLine());
int a = Integer.parseInt(st.nextToken());
int b = Integer.parseInt(st.nextToken());
int c = Integer.parseInt(st.nextToken());

패턴 3 — 한 줄에 N개의 정수를 배열로

int n = Integer.parseInt(br.readLine().trim());
int[] arr = new int[n];
StringTokenizer st = new StringTokenizer(br.readLine());
for (int i = 0; i < n; i++) {
    arr[i] = Integer.parseInt(st.nextToken());
}

패턴 4 — N줄에 걸쳐 정수 한 개씩

int n = Integer.parseInt(br.readLine().trim());
int[] arr = new int[n];
for (int i = 0; i < n; i++) {
    arr[i] = Integer.parseInt(br.readLine().trim());
}

패턴 5 — 2차원 배열 (격자/맵)

StringTokenizer st = new StringTokenizer(br.readLine());
int n = Integer.parseInt(st.nextToken());
int m = Integer.parseInt(st.nextToken());
int[][] map = new int[n][m];
for (int i = 0; i < n; i++) {
    st = new StringTokenizer(br.readLine());
    for (int j = 0; j < m; j++) {
        map[i][j] = Integer.parseInt(st.nextToken());
    }
}

패턴 6 — 공백 없는 숫자 문자열 ("11100")

String line = br.readLine();
int[][] map = new int[n][m];
for (int j = 0; j < m; j++) {
    map[i][j] = line.charAt(j) - '0';  // char를 int로
}

패턴 7 — 입력 끝까지 (EOF)

String line;
while ((line = br.readLine()) != null && !line.isEmpty()) {
    // 처리
}

출력 최적화 — StringBuilder의 위력

출력이 많을 때 System.out.println을 반복 호출하면 사망 플래그입니다. 결과를 StringBuilder에 모았다가 마지막에 한 번만 출력하세요.

BAD — 10만 줄 출력 시 약 4~6초
for (int x : results) {
    System.out.println(x);
}
GOOD — 같은 작업 0.1초 이내
StringBuilder sb = new StringBuilder();
for (int x : results) {
    sb.append(x).append('\n');
}
System.out.print(sb);
✓실전 팁
sb.append("\n")보다 sb.append('\n')이 미세하게 빠릅니다. char가 String보다 가볍기 때문입니다. 또한 줄바꿈은 System.lineSeparator()보다 그냥 '\n'을 쓰세요. 채점 서버는 LF 기반입니다.

자주 만나는 입출력 함정 5가지

  1. readLine 뒤에 trim()을 빼먹는 경우 — 줄 끝에 보이지 않는 공백/캐리지리턴이 붙어 NumberFormatException이 납니다.
  2. StringTokenizer를 재선언하지 않는 경우 — 다른 줄을 읽을 때마다 st = new StringTokenizer(...)로 새로 만들어야 합니다.
  3. sb를 마지막에 print하지 않는 경우 — 답은 맞는데 "출력 형식이 잘못되었습니다"가 뜬다면 거의 이 경우입니다.
  4. 줄바꿈 vs 공백 혼동 — 문제가 "한 줄에 출력"이라고 했는데 '\n'으로 구분하면 오답.
  5. 마지막 줄의 줄바꿈 — 어지간하면 마지막에 '\n'을 붙여도 정답으로 처리되지만, 일부 채점기는 깐깐합니다. 기본은 붙이는 것이 안전합니다.

첫 줄에 두 정수 A, B가 공백으로 구분되어 들어온다. A+B를 출력.

패턴 B — 테스트 케이스 여러 개 [백준 10953 A+B - 6]

첫 줄에 테스트 케이스 수 T. 그 다음 T줄에 각각 A,B(쉼표 구분). 각 줄에 대해 A+B 출력.

INPUT5 1,1 2,3 3,4 9,8 5,2
OUTPUT2 5 7 17 7
import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringBuilder sb = new StringBuilder();   // 출력 누적용

        // 1) 테스트 케이스 개수
        int T = Integer.parseInt(br.readLine().trim());

        // 2) T번 반복
        for (int i = 0; i < T; i++) {
            // 구분자가 쉼표(,)이므로 StringTokenizer에 구분자 지정
            StringTokenizer st = new StringTokenizer(br.readLine(), ",");
            int a = Integer.parseInt(st.nextToken());
            int b = Integer.parseInt(st.nextToken());

            // 결과를 출력 버퍼에 누적
            sb.append(a + b).append('\n');
        }

        // 3) 한 번에 출력 (println 반복 절대 금지)
        System.out.print(sb);
    }
}

이 패턴이 적용되는 문제: 테스트 케이스가 여러 개인 모든 문제 — 입력 형태만 바꿔서 풀이 로직을 동일하게 적용.

✓딱 하나만 외운다면
BufferedReader + StringTokenizer + StringBuilder 세 가지를 한 세트로 외우세요. 코딩 테스트의 거의 모든 입출력은 이 세 클래스의 조합으로 해결됩니다. 시험장에서 막힘 없이 손이 가는 게 목표.
CHAPTER 06

06.시간복잡도 직관 잡기

"왜 시간 초과지?"는 결국 시간복잡도를 못 읽었다는 뜻입니다. 외울 건 딱 하나. "10⁸ ≒ 1초".

Big-O가 뭔지, 한 문장으로

Big-O는 "입력 크기 N이 커질 때, 연산 횟수가 N에 비례해 어떻게 늘어나는가"의 표기법입니다. 상수와 작은 항은 무시하고 가장 빠르게 커지는 항만 남깁니다.

  • O(1) — N과 무관하게 항상 같은 시간. 배열 인덱스 접근.
  • O(log N) — N이 두 배 되어도 1회만 추가. 이분 탐색.
  • O(N) — N에 정비례. 한 번 훑기.
  • O(N log N) — 정렬의 표준 속도.
  • O(N²) — 이중 for문. N=1만 넘으면 위험.
  • O(2ᴺ), O(N!) — 백트래킹/완전탐색. N=20 근처가 한계.

"10⁸ ≒ 1초" 룰

Java는 1초에 대략 1억(10⁸) 회의 단순 연산을 수행합니다. 문제에 시간 제한이 1초라면 "내 알고리즘의 총 연산 횟수가 10⁸을 넘는가?"를 먼저 따지세요.

i왜 Java는 이렇게 '안전마진'을 잡아야 하나
C++ 기준은 보통 10⁸~10⁹입니다. Java는 JIT 컴파일 + GC가 있어 보통 C++의 2~3배 느리다고 가정합니다. 그래서 "Java는 10⁷ 안전, 10⁸은 빡빡, 10⁹은 거의 불가능"이 실전 감각입니다.

N에 따른 알고리즘 선택표

지문에서 N을 보자마자 머리에 떠올라야 합니다.

N의 크기허용 복잡도가능한 접근
N ≤ 10O(N!)순열/조합 완전탐색, 백트래킹
N ≤ 20O(2ᴺ)부분집합 탐색, 비트마스킹
N ≤ 500O(N³)플로이드-워셜, 3중 for문 DP
N ≤ 5,000O(N²)이중 for문, 기본 DP
N ≤ 100,000O(N log N)정렬, 이분탐색, 우선순위 큐
N ≤ 1,000,000O(N) ~ O(N log N)한 번 훑기, 투포인터, 슬라이딩 윈도우
N ≤ 100,000,000O(log N) or O(1)수학적 풀이, 이분탐색
✓실전 사고법
"N=100,000인데 이중 for문 쓸까?" → 10¹⁰. 100초 걸립니다. 100% 시간 초과. 이런 식으로 코딩 전에 1초만 계산하면 잘못된 길로 가는 30분을 아낄 수 있습니다.

Java 특유의 성능 함정

1. String 결합은 + 가 아니라 StringBuilder

// 절대 금지 — O(N²)
String s = "";
for (int i = 0; i < n; i++) s += i;

// 정답 — O(N)
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) sb.append(i);

2. ArrayList보다 배열이 빠르다

크기가 고정이라면 int[]을 쓰세요. ArrayList<Integer>는 박싱/언박싱 비용이 추가됩니다.

3. HashMap은 빠르지만 "상수가 큽니다"

이론상 O(1)이지만 실제로는 배열 접근의 5~10배. 키가 0~N 범위 정수라면 그냥 배열을 쓰세요.

4. 재귀는 스택을 폭파시킬 수 있다

Java 기본 스택 깊이는 약 1~2만. 깊이 10만짜리 DFS는 StackOverflowError가 납니다. 깊은 그래프는 반복문 + Stack 자료구조로 풀거나, 별도 스레드에 큰 스택을 할당하세요.

실전 시간복잡도 계산 — 시나리오 4개

"이 문제는 어떻게 풀면 안전한가?"를 빠르게 답하는 훈련. 시험장에서 코드를 짜기 전 1초 만에 판단할 수 있어야 합니다.

시나리오 ① — N = 100,000, 제한 시간 1초

가능한 풀이

  • O(N log N) — 정렬, 이분탐색, 우선순위 큐 → 약 170만 연산. 안전.
  • O(N) — 한 번 훑기, 투포인터 → 매우 안전.

불가능한 풀이

  • O(N²) — 이중 for문. 100억 연산. 시간 초과 확정.

결론 — "이중 for문이 떠올랐다면, 그건 답이 아닙니다." 정렬 후 투포인터 또는 이분탐색을 고민하세요.

시나리오 ② — N = 1,000, 제한 시간 1초

가능한 풀이

  • O(N²) — 이중 for문. 100만 연산. 안전.
  • O(N² log N) — 약 1천만 연산. 빡빡하지만 OK.

위험한 풀이

  • O(N³) — 10억 연산. 1초 안 들어옴.

결론 — DP 표를 채우는 N² 풀이가 표준. 플로이드-워셜은 안 됨.

시나리오 ③ — N = 20, 제한 시간 1초

가능한 풀이

  • O(2ᴺ) — 부분집합 완전탐색. 약 100만. 안전.
  • O(N × 2ᴺ) — 비트마스킹 DP. 약 2천만. OK.
  • O(N!) — N=10까지만. N=12부터는 위험.

결론 — N ≤ 20을 보자마자 비트마스킹 / 완전탐색 / 백트래킹을 떠올리세요.

시나리오 ④ — N = 10⁹, 제한 시간 1초

가능한 풀이

  • O(log N) — 이분탐색, 빠른 거듭제곱.
  • O(1) — 수학적 공식 (등차/등비, 조합 공식).

불가능한 풀이

  • O(N)조차 안 됩니다. 10억 번 루프 → 10초 이상.

결론 — N이 10⁹이면 수학 또는 이분탐색으로 풀라는 강한 힌트. 답을 직접 추측하는 매개변수 탐색이 단골.

✓1초 룰의 활용법
문제를 받자마자 "N 보기 → 가능한 시간복잡도 추정 → 그 안에서 풀이 후보 좁히기"를 30초 안에 끝내세요. 이것만 잘 해도 잘못된 길로 가는 30분을 매번 절약합니다.

1. 배열 & ArrayList

가장 기본. 크기를 알면 배열, 모르면 ArrayList가 원칙입니다. 둘은 비슷해 보이지만 성능 차이가 큽니다 — 정수 1억 개를 다룬다면 int[]가 ArrayList<Integer>의 2~3배 빠릅니다 (오토박싱 비용).

핵심 사용법

// ─── 배열 (크기 고정, 빠름) ───
int[] arr = new int[100];
arr[0] = 42;
Arrays.fill(arr, -1);                 // 전체 -1로 초기화
int len = arr.length;                // 길이는 .length (메서드 아님)

// ─── ArrayList (크기 가변) ───
List<Integer> list = new ArrayList<>();
list.add(42);                         // 추가 O(1) 분할상환
int v = list.get(0);                 // 접근 O(1)
list.remove(0);                       // 중간 삭제 O(N) → 비추
int size = list.size();              // 길이는 .size() (메서드)

// ─── 정렬 ───
Arrays.sort(arr);                       // 배열 오름차순
Collections.sort(list);                 // 리스트 오름차순
list.sort(Comparator.reverseOrder());   // 리스트 내림차순

// 2차원 배열 정렬 (커스텀 비교자)
int[][] arr2D = new int[N][2];
Arrays.sort(arr2D, (a, b) -> a[0] - b[0]);  // 첫 번째 원소 기준

미니 예제 — 수 정렬하기 3 [백준 10989]

N개의 수를 오름차순 정렬 후 한 줄에 하나씩 출력. (N ≤ 10,000,000)

주의 — N이 천만이라 ArrayList나 박싱은 메모리 폭발. int[] 사용 + Arrays.sort. 단, 수의 범위가 작다면(0~10000) 카운팅 정렬이 더 빠릅니다.

int[] count = new int[10001];          // 값별 등장 횟수
for (int i = 0; i < N; i++) {
    int v = Integer.parseInt(br.readLine().trim());
    count[v]++;                       // 그 값의 카운트 +1
}
StringBuilder sb = new StringBuilder();
for (int v = 0; v <= 10000; v++) {
    for (int c = 0; c < count[v]; c++) {
        sb.append(v).append('\n');
    }
}
System.out.print(sb);

핵심: 자료구조 선택이 곧 풀이. 수의 범위를 알면 정렬도 O(N)에 끝낼 수 있습니다.

!주의 — int[]의 sort 함정
int[]를 Arrays.sort()로 정렬하면 듀얼 피벗 퀵소트로 평균 O(N log N)이지만, 최악 O(N²)가 됩니다. 백준의 일부 문제는 이 약점을 노려 적대적 데이터를 깔아두므로 의심되면 Integer[]로 박싱 후 정렬하세요 — 머지소트라 O(N log N) 보장.

2. HashMap

"X가 몇 번 등장했나", "이름→점수", "단어→번역" 같은 매핑의 90%는 HashMap. 해시 충돌이 없으면 O(1)로 매우 빠릅니다.

핵심 사용법

Map<String, Integer> count = new HashMap<>();

// 1) getOrDefault — 카운팅의 정석 (외워두세요)
count.put(word, count.getOrDefault(word, 0) + 1);

// 2) merge — getOrDefault의 더 깔끔한 대안 (Java 8+)
count.merge(word, 1, Integer::sum);

// 3) 존재 확인
if (count.containsKey(word)) { ... }

// 4) 순회 — entrySet이 keySet보다 빠름
for (Map.Entry<String, Integer> e : count.entrySet()) {
    System.out.println(e.getKey() + " → " + e.getValue());
}

미니 예제 — 단어 빈도수 [최빈 단어 찾기]

N개의 단어가 입력될 때 가장 많이 등장한 단어 1개를 출력하라 (동률이면 사전순).

Map<String, Integer> freq = new HashMap<>();
for (int i = 0; i < N; i++) {
    String w = br.readLine().trim();
    freq.merge(w, 1, Integer::sum);    // 빈도 +1
}

// 정렬: 빈도 내림차순, 같으면 사전 오름차순
String answer = freq.entrySet().stream()
    .sorted((a, b) -> {
        if (!a.getValue().equals(b.getValue()))
            return b.getValue() - a.getValue();   // 빈도 내림
        return a.getKey().compareTo(b.getKey());   // 사전 오름
    })
    .findFirst().get().getKey();

System.out.println(answer);

핵심: merge(key, 1, Integer::sum)는 getOrDefault의 가장 깔끔한 대안. 정렬 기준 2단계는 그리디·해시 문제의 단골입니다.

!키가 정수라면 배열을 쓰세요
키가 0~N 범위 정수라면 HashMap<Integer, Integer>보다 int[]가 압도적으로 빠릅니다(5~10배). HashMap은 박싱 + 해시 계산 + 충돌 처리의 상수가 큽니다.

3. HashSet

"이미 본 적이 있나?"를 O(1)에 답합니다. 중복 제거의 대표 도구. HashMap의 키 부분만 빌려쓴다고 보면 됩니다.

핵심 사용법

Set<Integer> seen = new HashSet<>();
seen.add(42);
boolean exists = seen.contains(42);   // true
seen.remove(42);
int size = seen.size();

// 컬렉션에서 중복 제거
List<Integer> unique = new ArrayList<>(new HashSet<>(list));

미니 예제 — 서로 다른 부분 문자열의 개수

문자열 S가 주어진다. S의 모든 서로 다른 부분 문자열의 개수를 구하라.

String S = br.readLine().trim();
Set<String> substrings = new HashSet<>();

for (int i = 0; i < S.length(); i++) {
    for (int j = i + 1; j <= S.length(); j++) {
        substrings.add(S.substring(i, j));   // 중복은 자동으로 안 들어감
    }
}

System.out.println(substrings.size());

핵심: "서로 다른 ~의 개수"는 HashSet의 단골 패턴. add가 중복을 자동 거르므로 별도 처리 불필요. (참고: 큰 입력은 Suffix Array가 필요)

4. Stack — LIFO (ArrayDeque로 대체)

"마지막에 넣은 것을 가장 먼저 꺼낸다(Last-In First-Out)". Java의 Stack 클래스는 레거시이고 동기화 오버헤드가 있어 실전에서는 ArrayDeque를 Stack처럼 씁니다.

핵심 사용법

Deque<Integer> stack = new ArrayDeque<>();
stack.push(1);           // 위에 쌓기
stack.push(2);
int top  = stack.pop();    // 꺼내기: 2
int peek = stack.peek();   // 안 꺼내고 확인: 1
boolean empty = stack.isEmpty();

미니 예제 — 괄호 짝 검사 [백준 9012 유형]

괄호 문자열이 주어진다. 모든 여는 괄호가 같은 종류의 닫는 괄호와 짝지어지면 VPS, 아니면 NO를 출력하라.

입력: (())() → YES / (() → NO

Deque<Character> stack = new ArrayDeque<>();
String s = br.readLine().trim();

boolean valid = true;
for (int i = 0; i < s.length(); i++) {
    char c = s.charAt(i);
    if (c == '(') {
        stack.push(c);                       // 여는 괄호는 쌓고
    } else {
        // 닫는 괄호인데 스택이 비었다면 짝이 없음
        if (stack.isEmpty()) {
            valid = false;
            break;
        }
        stack.pop();                         // 짝 맞음 → 여는 괄호 제거
    }
}
// 끝났는데 스택에 남아있다면 닫는 짝이 부족했던 것
if (!stack.isEmpty()) valid = false;

System.out.println(valid ? "YES" : "NO");

핵심: 스택의 가장 대표적 사용. "여는 건 쌓고, 닫는 건 짝 맞춰서 꺼낸다"는 패턴은 괄호, HTML 태그, 후위표기식, 히스토그램에 모두 동일하게 적용됩니다.

5. Queue — FIFO (ArrayDeque)

"먼저 넣은 것을 먼저 꺼낸다(First-In First-Out)". BFS의 심장. LinkedList로 구현해도 되지만 ArrayDeque가 2~3배 빠르고 메모리도 적게 씁니다.

핵심 사용법

Queue<Integer> q = new ArrayDeque<>();
q.offer(1);              // 뒤에 추가 (add도 됨)
q.offer(2);
int head = q.poll();    // 앞에서 꺼냄: 1
int peek = q.peek();    // 안 꺼내고 앞 확인: 2

// 좌표를 큐에 넣을 때는 int[] 형태가 객체 생성 비용이 적음
Queue<int[]> coords = new ArrayDeque<>();
coords.offer(new int[]{x, y, dist});

미니 예제 — 카드 게임 [백준 2164]

1~N까지 적힌 카드가 위에서 1, 2, ..., N 순으로 쌓여있다. ① 맨 위 카드를 버린다 ② 맨 위 카드를 맨 아래로 옮긴다. ①, ②를 반복해 마지막에 남는 카드는?

입력: 6 → 출력: 4

Queue<Integer> q = new ArrayDeque<>();
for (int i = 1; i <= N; i++) q.offer(i);

while (q.size() > 1) {
    q.poll();                // ① 맨 위 버림
    q.offer(q.poll());       // ② 맨 위를 빼서 맨 뒤로
}
System.out.println(q.poll());

핵심: 큐는 "줄 서기"의 자연스러운 모델. BFS, 시뮬레이션, 카드/번호표 문제에 자주 등장합니다.

6. PriorityQueue (힙)

"항상 가장 작은(또는 큰) 원소부터 꺼내고 싶다"의 답. 내부는 이진 힙. offer/poll이 O(log N)이고 peek은 O(1).

핵심 사용법

// 최소 힙 — 기본
PriorityQueue<Integer> minHeap = new PriorityQueue<>();
minHeap.offer(3);
minHeap.offer(1);
minHeap.offer(2);
minHeap.poll();   // 1 (가장 작은 값)

// 최대 힙
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());

// 커스텀 비교자 — int[] 형태에서 두 번째 원소(거리) 기준 오름차순
PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[1] - b[1]);
pq.offer(new int[]{0, 5});   // {노드, 거리}

// 다중 키 정렬 — 첫 번째 같으면 두 번째
PriorityQueue<int[]> pq2 = new PriorityQueue<>((a, b) -> {
    if (a[0] != b[0]) return a[0] - b[0];
    return a[1] - b[1];
});

미니 예제 — K번째 큰 수 [실시간 스트림]

스트림으로 숫자가 하나씩 들어온다. 매 입력 후 지금까지 본 숫자 중 K번째로 큰 값을 출력하라.

아이디어 — 크기 K인 최소 힙을 유지. 새 숫자가 힙의 최솟값(top)보다 크면 top을 빼고 새 숫자를 넣음. 항상 K번째 큰 수가 top에 위치.

int K = 3;
PriorityQueue<Integer> pq = new PriorityQueue<>();   // 최소 힙

for (int x : stream) {
    pq.offer(x);
    if (pq.size() > K) {
        pq.poll();          // 크기를 K로 유지 → 가장 작은 거 버림
    }
    // pq의 크기가 K이면 그 top이 K번째 큰 수
    int kthLargest = pq.peek();
    System.out.println(kthLargest);
}

핵심: "K번째 큰 수"는 크기 K짜리 최소 힙으로 풀립니다. 직관적으로는 최대 힙이지만, 반대를 쓰는 게 트릭. 다익스트라, 부록 PROBLEM 07의 작업 스케줄링도 같은 패턴.

7. Deque (양방향 큐)

앞뒤 양쪽에서 넣고 빼기가 자유로움. Stack도 되고 Queue도 됨. 슬라이딩 윈도우 최댓값 문제의 단골 도구입니다.

핵심 사용법

Deque<Integer> dq = new ArrayDeque<>();
dq.offerFirst(1);       // 앞에 추가
dq.offerLast(2);        // 뒤에 추가
dq.pollFirst();          // 앞에서 꺼냄
dq.pollLast();           // 뒤에서 꺼냄
dq.peekFirst();          // 앞 확인
dq.peekLast();           // 뒤 확인

미니 예제 — 슬라이딩 윈도우 최댓값 [어려움]

크기 N의 배열과 윈도우 크기 K가 주어진다. 윈도우가 한 칸씩 이동할 때 각 윈도우의 최댓값을 모두 출력. (N ≤ 1,000,000)

아이디어 — Deque에 "윈도우 안의 후보 인덱스를 큰 값 순서대로" 유지. 새 원소가 들어올 때 그보다 작은 뒤쪽 후보를 다 제거.

Deque<Integer> dq = new ArrayDeque<>();  // 인덱스를 저장
StringBuilder sb = new StringBuilder();

for (int i = 0; i < N; i++) {
    // 1) 윈도우 밖으로 나간 인덱스 제거 (앞쪽)
    while (!dq.isEmpty() && dq.peekFirst() <= i - K) {
        dq.pollFirst();
    }
    // 2) 새 값보다 작은 뒤쪽 후보는 영원히 최댓값이 될 수 없음 → 제거
    while (!dq.isEmpty() && arr[dq.peekLast()] < arr[i]) {
        dq.pollLast();
    }
    // 3) 현재 인덱스 추가
    dq.offerLast(i);

    // 4) 윈도우가 완성됐다면 최댓값(앞쪽)을 기록
    if (i >= K - 1) {
        sb.append(arr[dq.peekFirst()]).append(' ');
    }
}
System.out.println(sb);

핵심: Deque로 "단조 감소하는 후보"를 유지하는 패턴. 각 원소가 최대 한 번씩만 추가/제거되어 전체 O(N). 슬라이딩 윈도우 최댓값/최솟값의 정석 풀이입니다.

8. TreeMap / TreeSet

키를 정렬된 상태로 유지하면서 모든 연산이 O(log N). HashMap이 "있나 없나"라면 TreeMap은 "x보다 큰 것 중 가장 작은 것" 같은 범위 검색에 강합니다.

핵심 사용법

TreeMap<Integer, String> tm = new TreeMap<>();
tm.put(3, "c");
tm.put(1, "a");
tm.put(5, "e");

// 정렬된 키 접근
tm.firstKey();         // 1 (최솟값)
tm.lastKey();          // 5 (최댓값)

// 범위 검색 — 코딩 테스트의 핵심 기능
tm.ceilingKey(4);     // 5  — 4 이상 중 가장 작은 키
tm.floorKey(4);       // 3  — 4 이하 중 가장 큰 키
tm.higherKey(3);      // 5  — 3 초과 중 가장 작은 키 (자기 제외)
tm.lowerKey(3);       // 1  — 3 미만 중 가장 큰 키

// 정렬 순서대로 순회
for (Map.Entry<Integer, String> e : tm.entrySet()) { ... }

미니 예제 — 가까운 수 찾기

정수의 집합 S가 동적으로 변한다(추가/삭제). 매번 어떤 값 x가 주어지면 S 안의 값 중 x에 가장 가까운 값을 찾아라.

TreeSet<Integer> set = new TreeSet<>();

// 추가/삭제
set.add(10); set.add(25); set.add(40);

// x에 가장 가까운 값 찾기
int x = 22;
Integer up   = set.ceiling(x);   // 25 (x 이상 중 가장 작은)
Integer down = set.floor(x);     // 10 (x 이하 중 가장 큰)

Integer nearest;
if (up == null) nearest = down;
else if (down == null) nearest = up;
else nearest = (up - x <= x - down) ? up : down;

System.out.println(nearest);     // 25 (3 차이, 12 차이보다 가까움)

핵심: HashSet은 못 하는 일. ceiling과 floor로 양쪽 후보를 본 뒤 더 가까운 쪽 선택. "동적 + 정렬 + 범위"가 들어오면 TreeMap/TreeSet 떠올리세요.

01

완전탐색 (Brute Force)

N ≤ 10 O(N!) / O(2ᴺ) 재귀 + 다중 for문

핵심 개념

"가능한 모든 경우의 수를 만들어보고, 그 중 조건을 만족하는 것을 골라낸다." 가장 무식한 방법이지만 가장 확실한 출발점입니다. 더 빠른 풀이가 안 보일 때는 일단 완전탐색으로 정답을 만들고, 거기서 최적화를 고민하는 게 정석입니다.

언제 쓰는가

  • N이 매우 작다 — 보통 N ≤ 10. N! = 약 360만 이하라 1초 안에 끝납니다.
  • "모든 경우", "전부", "어떤 순서로든" 같은 표현이 보이는 문제
  • 순열, 조합, 부분집합 개념이 등장하는 문제

동작 원리

재귀로 "한 자리씩 결정 → 다음 자리로 들어감 → 끝까지 갔으면 결과 처리 → 돌아와서 다음 선택지 시도"의 패턴을 반복합니다. 트리 모양으로 모든 가능성이 펼쳐진다고 상상하세요. 가지 끝(리프)에 도달하면 한 케이스가 완성됩니다.

연습 예제 — "N과 M (1)" 유형 [백준 15649]

자연수 N, M이 주어질 때, 1부터 N까지의 자연수 중 중복 없이 M개를 고른 수열을 사전 순으로 모두 출력하라. (1 ≤ M ≤ N ≤ 8)

INPUT3 1
OUTPUT1 2 3
INPUT4 2
OUTPUT1 2 1 3 1 4 2 1 2 3 2 4 3 1 ...

풀이 사고 과정

  1. 유형 파악 — "1~N에서 M개를 골라 모든 수열을 만들어라" → 완전탐색 + 백트래킹의 전형
  2. N의 크기 확인 — N ≤ 8 → 최대 8! = 40,320개의 케이스. 완전탐색으로 충분히 가능
  3. 상태 설계 — picked[depth]: 현재 depth번째 자리에 놓인 숫자 / used[i]: 숫자 i를 이미 썼는지
  4. 사전 순 보장 — 1부터 N까지 차례대로 시도하면 별도 정렬 없이도 자동으로 사전 순
  5. 출력 처리 — M개 수열을 한 줄로 출력 후 줄바꿈. 출력이 많을 수 있어 StringBuilder 필수

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    // ─── 전역 상태 ────────────────────────────────────────
    static int N, M;
    static int[] picked;          // 현재까지 고른 수열 (자리별 숫자)
    static boolean[] used;       // used[i]=true → 숫자 i를 이미 사용
    static StringBuilder sb = new StringBuilder();

    public static void main(String[] args) throws IOException {
        // 1) 입력: 한 줄에 N과 M이 공백으로 구분되어 들어옴
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringTokenizer st = new StringTokenizer(br.readLine());
        N = Integer.parseInt(st.nextToken());
        M = Integer.parseInt(st.nextToken());

        // 2) 자료구조 준비
        //    - picked: M자리짜리 수열
        //    - used:   숫자 i(1~N)의 사용 여부 → 1-indexed라 N+1 크기
        picked = new int[M];
        used   = new boolean[N + 1];

        // 3) 0번째 자리부터 결정 시작
        dfs(0);

        // 4) 모아둔 출력을 한 번에 (System.out.println 반복은 매우 느림)
        System.out.print(sb);
    }

    /**
     * depth: 지금 결정할 자리 (0부터 M-1까지)
     */
    static void dfs(int depth) {
        // ─── 종료 조건 ───
        // M개를 모두 골랐다 → 한 줄로 출력하고 돌아감
        if (depth == M) {
            for (int x : picked) sb.append(x).append(' ');
            sb.append('\n');
            return;
        }

        // ─── 선택지 펼치기 ───
        // 1~N을 차례대로 시도 → 사전 순 자동 보장
        for (int i = 1; i <= N; i++) {
            if (used[i]) continue;          // 이미 사용한 숫자는 건너뜀

            // (a) i를 depth 자리에 놓고
            picked[depth] = i;
            used[i] = true;

            // (b) 다음 자리(depth+1)로 들어감
            dfs(depth + 1);

            // (c) 돌아오면 i를 도로 빼고 다음 후보로
            //     ★ 이 한 줄이 백트래킹의 정수. 빼먹으면 답이 망가짐
            used[i] = false;
            // picked[depth]는 다음 반복에서 덮어쓰여지므로 초기화 불필요
        }
    }
}

이 풀이의 핵심 포인트

  • "선택 → 재귀 → 원복"이 백트래킹의 본질. used[i] = false를 빼먹으면 한 번 쓴 숫자가 영영 못 쓰이게 됩니다.
  • 1부터 차례대로 시도 → 사전 순 정렬을 따로 안 해도 자동으로 됩니다.
  • picked[depth]는 다음 반복에서 덮어쓰여지므로 굳이 초기화할 필요 없음. (불필요한 코드를 안 쓰는 습관)
  • 출력이 수만 줄 → println 반복하면 시간 초과. StringBuilder는 선택이 아닌 필수.
외울 한 줄 — 완전탐색은 "자리를 정해놓고 한 자리씩 채워나가는" 재귀. 채운 후 원복하는 것까지가 한 세트입니다.
02

정렬 + 그리디 (Greedy)

"최소 / 최대 개수" O(N log N) 정렬 후 1-pass

핵심 개념

그리디는 "매 순간 가장 좋아 보이는 선택"을 누적해서 답을 만드는 방법입니다. 다만 그 "현재 최선의 선택"이 정말 전체 최적으로 이어진다는 직관(또는 증명)이 있어야 통합니다. 그리디의 90%는 정렬 기준을 어떻게 잡느냐에 달려 있습니다.

언제 쓰는가

  • "최대 개수", "최소 횟수", "최소 비용" — 그러나 DP까지 갈 필요는 없어 보일 때
  • 정렬해놓고 보면 답이 보이는 문제
  • "앞에서부터 욕심껏 골라도 손해 안 본다"는 직관이 서는 경우

주의 — 그리디가 안 통할 때

그리디 풀이가 잘 안 되면 DP를 의심하세요. 동전 거스름돈 문제에서 동전 종류가 일반적이면 그리디로 안 풀리고 DP로 풀어야 합니다. "그리디로 안 되는 반례가 떠오르는가?"가 판별 기준입니다.

연습 예제 — 회의실 배정 [백준 1931]

회의실 1개에 N개의 회의 신청이 들어왔다. 각 회의는 시작 시간과 끝 시간이 있다. 회의들이 겹치지 않도록 했을 때 최대 몇 개의 회의를 배정할 수 있는가? (N ≤ 100,000)

INPUT11 1 4 3 5 0 6 5 7 3 8 5 9 6 10 8 11 8 12 2 13 12 14
OUTPUT4 (1-4, 5-7, 8-11, 12-14)

풀이 사고 과정

  1. 유형 파악 — "겹치지 않는 회의의 최대 개수" → 활동 선택 문제 (Activity Selection). 정렬+그리디의 정석.
  2. 왜 시작 시간으로 정렬하면 안 될까? — 0시 시작 24시 끝나는 회의가 1번으로 채택되면 다른 회의를 다 못 함
  3. 왜 끝나는 시간 정렬이 최적일까? — 빨리 끝낼수록 "이후 남는 시간"이 최대화 → 뒤에 더 많은 회의를 끼울 자리가 생김
  4. 끝 시간이 같으면? — 시작 시간 빠른 것 우선. ("시작=끝"인 0초짜리 회의 처리)
  5. 구현 — 정렬 후 한 번 훑으며 "직전 회의 끝 시간 이후에 시작 가능하면" 채택

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        int N = Integer.parseInt(br.readLine().trim());

        // 회의: [시작, 끝] 형태로 2차원 배열에 저장
        int[][] meetings = new int[N][2];
        for (int i = 0; i < N; i++) {
            StringTokenizer st = new StringTokenizer(br.readLine());
            meetings[i][0] = Integer.parseInt(st.nextToken());  // 시작
            meetings[i][1] = Integer.parseInt(st.nextToken());  // 끝
        }

        // ★ 정렬 기준 2단계 ★
        //   1순위: 끝나는 시간 오름차순 → 빨리 끝나는 회의 우선
        //   2순위: 시작 시간 오름차순 → 끝=시작인 0초 회의 처리용
        Arrays.sort(meetings, (a, b) -> {
            if (a[1] != b[1]) return a[1] - b[1];
            return a[0] - b[0];
        });

        // 한 번 훑으며 욕심껏 선택
        int count = 0;
        int currentEnd = 0;             // 마지막으로 채택한 회의의 끝 시간

        for (int[] m : meetings) {
            // 직전 회의 끝 시간 이후(또는 같은 시각)부터 시작 가능 → 채택
            if (m[0] >= currentEnd) {
                count++;
                currentEnd = m[1];       // 끝 시간 갱신
            }
        }

        System.out.println(count);
    }
}

이 풀이의 핵심 포인트

  • "끝 시간 정렬" 한 줄이 풀이의 90%. 왜 이게 최적인지 한마디로 설명할 수 있어야 합니다 — "빨리 끝낼수록 뒤에 회의 끼울 자리가 더 많이 남는다".
  • "끝나자마자 다음 회의 시작" 케이스도 허용(m[0] >= currentEnd). 부등호 방향이 >가 아니라 >=임에 주의.
  • 2단계 정렬을 람다로 짜는 패턴은 그리디 전반에서 매우 자주 등장합니다 — 외워두세요.
외울 한 줄 — 그리디 = "올바른 정렬 기준 + 1번 훑기". 정렬 기준이 정해지면 코드는 5줄로 끝납니다.
03

이분탐색 (Binary Search)

정렬된 데이터 O(log N) 매개변수 탐색 응용

핵심 개념

정렬된 데이터에서 "가운데를 보고 답이 왼쪽에 있는지 오른쪽에 있는지 판단" → 매번 범위를 절반으로 줄임. 100만 개에서 단 20번 비교로 답을 찾습니다(2²⁰ ≈ 100만).

언제 쓰는가

  • "정렬된" 데이터에서 특정 값을 빠르게 찾는다
  • 매개변수 탐색 — "답이 X일 때 조건을 만족하는가?"를 이분탐색으로 찾는 응용
  • "최댓값의 최소값", "최솟값의 최대값" 같은 패턴이 보일 때

동작 원리

lo(왼쪽 경계)와 hi(오른쪽 경계) 사이의 가운데 mid를 본 뒤, 답이 어느 쪽에 있는지 판단해 한쪽을 버립니다. "이 가운데를 보고 어느 쪽이 답인가?"의 판단 기준이 명확해야 통합니다.

!흔한 오버플로 함정
int mid = (lo + hi) / 2는 lo + hi가 int 최대값을 넘으면 음수가 됩니다. 안전하게 int mid = lo + (hi - lo) / 2 또는 int mid = (lo + hi) >>> 1 (부호 없는 시프트)를 쓰세요.

연습 예제 — 수 찾기 [백준 1920]

N개의 정수 A[1..N]이 주어진다. 그 다음 M개의 정수가 주어질 때, 각 정수가 A 안에 존재하는지 0 또는 1로 출력하라. (N, M ≤ 100,000)

INPUT5 4 1 5 2 3 5 1 3 7 9 5
OUTPUT1 1 0 0 1

풀이 사고 과정

  1. 유형 파악 — "특정 값이 있는가?" × M번 → 빠른 검색이 필요
  2. 시간복잡도 계산 — 단순 선형탐색은 O(N×M) = 10¹⁰ → 시간 초과. O(log N)으로 줄여야 함
  3. 전략 선택 — A를 정렬 O(N log N) 후, 각 질의를 O(log N)에 처리 → 총 O((N+M) log N) ≈ 200만
  4. 구현 — Arrays.sort로 정렬, 표준 이분탐색 루프로 검색
  5. 출력 처리 — M=10만 줄 출력 → StringBuilder 필수

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringBuilder sb = new StringBuilder();

        // 1) A 배열 입력 및 정렬 (이분탐색의 전제 조건)
        int N = Integer.parseInt(br.readLine().trim());
        int[] A = new int[N];
        StringTokenizer st = new StringTokenizer(br.readLine());
        for (int i = 0; i < N; i++) {
            A[i] = Integer.parseInt(st.nextToken());
        }
        Arrays.sort(A);                              // ★ 정렬해야 이분탐색 가능

        // 2) 질의 처리
        int M = Integer.parseInt(br.readLine().trim());
        st = new StringTokenizer(br.readLine());
        for (int q = 0; q < M; q++) {
            int target = Integer.parseInt(st.nextToken());
            sb.append(contains(A, target) ? 1 : 0).append('\n');
        }

        System.out.print(sb);
    }

    /**
     * 정렬된 배열 A에서 target이 존재하면 true.
     * 가운데를 보고 절반씩 잘라가며 O(log N)에 검색.
     */
    static boolean contains(int[] A, int target) {
        int lo = 0, hi = A.length - 1;

        while (lo <= hi) {
            // ★ 오버플로 방지: (lo+hi)>>>1 또는 lo + (hi-lo)/2
            int mid = (lo + hi) >>> 1;

            if (A[mid] == target) {
                return true;                  // 찾음
            } else if (A[mid] < target) {
                lo = mid + 1;                   // 답은 오른쪽 절반
            } else {
                hi = mid - 1;                   // 답은 왼쪽 절반
            }
        }
        return false;
    }
}

매개변수 탐색 — 한 단계 위의 응용

"답"이 무엇인지 모를 때, 답을 직접 추측하고 그 추측이 가능한지 검사하는 패턴입니다. 부록 PROBLEM 04(나무 자르기)가 정확히 이 응용. 핵심 사고는:

  • "답 X가 가능한지"를 O(N)에 검사할 수 있다 → 답을 이분탐색
  • "가능하면 더 큰 값을 시도", "불가능하면 더 작은 값을 시도"
  • 답의 범위(예: 0 ~ 최대 나무 길이)를 lo, hi로 설정

이 풀이의 핵심 포인트

  • Arrays.sort() 한 줄을 깜빡하면 이분탐색의 전제가 깨져 답이 완전히 망가집니다.
  • Java 내장 Arrays.binarySearch()는 정확히 일치만 찾고 위치는 보장하지 않습니다 — 매개변수 탐색에는 직접 구현이 필요합니다.
  • 10만 줄 출력에는 반드시 StringBuilder.
외울 한 줄 — 이분탐색은 "찾기" 그 자체보다 "매개변수 탐색의 도구"로 더 자주 쓰입니다. "답을 추측 → 검사"의 사고 전환이 핵심.
04

BFS (너비 우선 탐색)

최단 거리 O(V + E) Queue + 거리 배열

핵심 개념

시작점에서 가까운 곳부터 한 겹씩 퍼져나가는 탐색. 큐를 사용하기 때문에 먼저 들어간 칸이 먼저 처리되고, 그 결과 "가중치가 모두 같은 그래프에서의 최단 거리"를 자연스럽게 보장합니다.

언제 쓰는가

  • "최단 거리", "최소 횟수", "최소 이동" + 격자 또는 그래프
  • 가중치가 없거나 모두 같은 그래프 (가중치가 다르면 다익스트라)
  • "몇 번 만에 도달?", "퍼지는 데 며칠?" 같은 표현

동작 원리 — 한 마디로

"시작점부터 1칸 거리의 모든 칸을 먼저 / 그 다음 2칸 거리의 모든 칸을 / …" 식으로 동심원처럼 퍼져 나갑니다. 큐에 넣는 순간 거리가 확정되므로, 최초로 도착한 거리가 최단 거리입니다.

!가장 흔한 실수
방문 체크를 "큐에서 꺼낼 때" 하면 같은 칸이 여러 번 큐에 들어가 시간 초과나 오답이 발생합니다. 반드시 "큐에 넣는 그 순간" 방문 표시하세요.

연습 예제 — 미로 탐색 [백준 2178]

N×M 미로. 1은 이동 가능, 0은 벽. (1,1)에서 출발해 (N,M)까지 갈 때 지나야 하는 최소 칸 수(출발+도착 포함)를 구하라. 4방향 이동. (N, M ≤ 100)

INPUT4 6 101111 101010 101011 111011
OUTPUT15

풀이 사고 과정

  1. 유형 파악 — "최소 이동" + 격자 + 1칸씩 → BFS의 정석
  2. 좌표계 결정 — 0-indexed로 변환. (0,0)에서 (N-1, M-1)까지
  3. 거리 저장 — dist[N][M]에 시작점부터의 거리. 미방문은 0 또는 -1로
  4. 입력 파싱 — 공백 없는 숫자 문자열 → charAt(j) - '0'
  5. 4방향 처리 — dx, dy 배열로 깔끔하게
  6. "칸 수"의 정의 — 출발 칸도 1로 카운트. 시작 dist를 1로 두는 게 깔끔

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    static int N, M;
    static int[][] map;
    static int[][] dist;

    // 4방향 이동: 상, 하, 좌, 우
    static int[] dx = {-1, 1,  0, 0};
    static int[] dy = { 0, 0, -1, 1};

    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));

        // 1) 첫 줄: N M
        StringTokenizer st = new StringTokenizer(br.readLine());
        N = Integer.parseInt(st.nextToken());
        M = Integer.parseInt(st.nextToken());

        // 2) 미로 입력 — 공백 없이 붙어 있는 숫자 문자열
        map  = new int[N][M];
        dist = new int[N][M];                // 0이면 미방문
        for (int i = 0; i < N; i++) {
            String line = br.readLine();
            for (int j = 0; j < M; j++) {
                // '1'(아스키 49) - '0'(아스키 48) = 정수 1
                map[i][j] = line.charAt(j) - '0';
            }
        }

        // 3) BFS
        bfs(0, 0);

        // 4) 출력 — 도착점까지의 거리
        System.out.println(dist[N - 1][M - 1]);
    }

    static void bfs(int sx, int sy) {
        Queue<int[]> q = new ArrayDeque<>();
        q.offer(new int[]{sx, sy});
        dist[sx][sy] = 1;                    // 출발 칸도 1로 카운트

        while (!q.isEmpty()) {
            int[] cur = q.poll();
            int x = cur[0], y = cur[1];

            // 4방향 시도
            for (int d = 0; d < 4; d++) {
                int nx = x + dx[d];
                int ny = y + dy[d];

                // (a) 격자 밖이면 패스
                if (nx < 0 || ny < 0 || nx >= N || ny >= M) continue;
                // (b) 벽(0)이면 패스
                if (map[nx][ny] == 0) continue;
                // (c) 이미 방문(dist>0)했으면 패스
                if (dist[nx][ny] != 0) continue;

                // (d) 거리 = 이전 칸 + 1 / 큐에 넣을 때 방문 표시
                dist[nx][ny] = dist[x][y] + 1;
                q.offer(new int[]{nx, ny});
            }
        }
    }
}

이 풀이의 핵심 포인트

  • dx, dy 배열로 4방향을 깔끔하게 처리. 8방향은 8개로 늘리면 끝.
  • 방문 체크는 큐에 넣는 순간. 꺼낼 때 하면 중복 방문으로 시간 초과.
  • 거리 배열을 dist[][] 하나로 통합 — visited와 distance를 따로 안 만들고 "0이면 미방문" 약속.
  • 큐 구현은 ArrayDeque가 표준. LinkedList는 2~3배 느립니다.
  • 이 템플릿은 약 70% 이상의 BFS 문제에 그대로 적용됩니다 — 손에 익히세요.
외울 한 줄 — BFS는 "큐 + 거리 배열 + 4방향"의 삼총사. "넣을 때 방문 표시" 하나만 안 잊으면 됩니다.
05

DFS (깊이 우선 탐색)

연결성 / 구조 O(V + E) 재귀 또는 Stack

핵심 개념

한 방향으로 최대한 깊이 들어갔다가, 더 못 가면 되돌아와서 다른 방향을 시도하는 탐색. 큐 대신 재귀(또는 스택)를 씁니다. "모든 경로", "연결된 영역의 크기" 같이 "구조"를 살펴야 할 때 강합니다.

언제 쓰는가

  • 연결된 영역의 개수 / 크기 (섬 개수, 단지 인구 등)
  • "모든 경로 출력", "특정 조건을 만족하는 경로 존재 여부"
  • 트리의 부모-자식 탐색, 사이클 검출

BFS vs DFS — 언제 뭘 쓸까

상황선택
최단 거리, 최소 횟수BFS
연결 영역의 개수 / 크기 / 모양DFS or BFS (편한 것)
모든 경로, 백트래킹DFS
그래프 깊이가 매우 깊을 수 있음 (10⁵ 이상)BFS (재귀 스택 오버플로 위험)

연습 예제 — 단지번호붙이기 [백준 2667]

N×N 격자에 1(집)과 0(빈 땅). 상하좌우로 연결된 1의 덩어리가 "단지". 단지의 총 개수와 각 단지에 속한 집의 수를 오름차순으로 출력하라. (N ≤ 25)

INPUT7 0110100 0110101 1110101 0000111 0100000 0111110 0111000
OUTPUT3 7 8 9

풀이 사고 과정

  1. 유형 파악 — "연결된 1 덩어리 + 각 덩어리 크기" → DFS/BFS로 영역 탐색
  2. 전체 격자를 순회하면서 "방문 안 한 1을 만나면 DFS 시작"
  3. DFS 중 카운트 — 방문하는 칸마다 +1해 단지 크기 계산
  4. 각 단지 크기를 리스트에 저장 후 정렬 → 출력
  5. 입력 처리 — 공백 없는 숫자 문자열, charAt - '0'

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    static int N;
    static int[][] map;
    static boolean[][] visited;
    static int houseCount;          // 현재 DFS 중인 단지의 집 수

    static int[] dx = {-1, 1,  0, 0};
    static int[] dy = { 0, 0, -1, 1};

    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        N = Integer.parseInt(br.readLine().trim());

        map     = new int[N][N];
        visited = new boolean[N][N];
        for (int i = 0; i < N; i++) {
            String line = br.readLine();
            for (int j = 0; j < N; j++) {
                map[i][j] = line.charAt(j) - '0';
            }
        }

        // 단지마다의 집 개수를 모으는 리스트
        List<Integer> sizes = new ArrayList<>();

        // 격자 전체를 훑으며 시작점 발굴
        for (int i = 0; i < N; i++) {
            for (int j = 0; j < N; j++) {
                // 1이고 방문 안 했다면 새 단지의 시작점
                if (map[i][j] == 1 && !visited[i][j]) {
                    houseCount = 0;       // 새 단지마다 카운트 초기화
                    dfs(i, j);
                    sizes.add(houseCount);
                }
            }
        }

        // 출력
        Collections.sort(sizes);
        StringBuilder sb = new StringBuilder();
        sb.append(sizes.size()).append('\n');
        for (int s : sizes) sb.append(s).append('\n');
        System.out.print(sb);
    }

    static void dfs(int x, int y) {
        visited[x][y] = true;
        houseCount++;                            // 현재 단지 집 +1

        // 4방향으로 깊이 들어감
        for (int d = 0; d < 4; d++) {
            int nx = x + dx[d];
            int ny = y + dy[d];

            if (nx < 0 || ny < 0 || nx >= N || ny >= N) continue;
            if (visited[nx][ny] || map[nx][ny] == 0) continue;

            dfs(nx, ny);                          // 같은 단지이므로 더 깊이
        }
    }
}

이 풀이의 핵심 포인트

  • houseCount를 새 단지마다 0으로 초기화하는 것을 잊지 마세요. 전역 변수의 함정.
  • "이중 for로 시작점 발굴 → 만나면 DFS" 패턴은 연결 영역 문제의 정석.
  • 단지 크기를 리스트에 모았다가 정렬 → 출력은 "정렬해서 출력" 같은 요구사항을 깔끔하게 처리하는 패턴.
  • N=25라 재귀 깊이 걱정 없음. N이 1000 이상이면 재귀 대신 BFS로 풀어야 안전합니다(스택 오버플로).
외울 한 줄 — DFS는 "끝까지 가보고 돌아오는" 탐색. BFS와 차이는 큐냐 재귀냐일 뿐 골격은 거의 같습니다.
06

동적 계획법 (Dynamic Programming)

중복 부분 문제 최적 부분 구조 점화식

핵심 개념

큰 문제를 작은 부분 문제들로 쪼개고, 작은 답을 표(dp)에 저장해두었다가 큰 답을 만들 때 재활용하는 기법. 부분 문제가 여러 번 등장하는 경우(중복 부분 문제) 폭발적인 속도 향상을 가져옵니다.

언제 쓰는가

  • "경우의 수", "최소/최대값"인데 완전탐색이 너무 느린 경우
  • "같은 부분 문제가 여러 번 반복"되는 구조
  • "앞 단계의 답을 알면 다음 단계의 답을 만들 수 있다"는 직관

DP 3단계 사고법

  1. 상태 정의: dp[i](또는 dp[i][j])가 정확히 무엇을 의미하는지 한 문장으로 쓸 수 있어야 합니다.
  2. 점화식: dp[i]를 더 작은 dp[?]들로 표현. "내가 i에 도달하려면 어디서 왔는가?"
  3. 초기값 / 기저 사례: 가장 작은 경우의 답을 직접 채워넣음.
!DP가 막힐 때 체크리스트
① 상태 정의가 모호 — "dp[i]가 무엇인가"를 한 줄로 못 적으면 식이 안 나옵니다. ② 차원이 부족 — 1차원 안 되면 2차원으로 확장. ③ 방향이 안 보임 — N=3, 4 같은 작은 경우를 손으로 표 그려보세요.

연습 예제 — 정수 삼각형 [백준 1932]

n층 정수 삼각형이 주어진다. 맨 위에서 출발해 아래로 내려가면서 거치는 수의 합이 최대가 되는 경로의 합을 구하라. 각 단계에서 바로 아래 또는 오른쪽 아래로만 이동 가능. (n ≤ 500)

INPUT5 7 3 8 8 1 0 2 7 4 4 4 5 2 6 5
OUTPUT30 (경로: 7→3→8→7→5)

풀이 사고 과정

  1. 완전탐색의 한계 확인 — n=500이면 경로 수 2⁴⁹⁹, 절대 불가능. DP 필수
  2. 상태 정의 — dp[i][j] = "(i, j) 칸에 도달했을 때까지의 최대 합"
  3. 점화식 — (i, j)에 도달하는 경로는 (i-1, j-1)에서 오거나 (i-1, j)에서 오거나 둘 중 하나
    → dp[i][j] = max(dp[i-1][j-1], dp[i-1][j]) + triangle[i][j]
  4. 경계 처리 — j=0은 왼쪽 위가 없고, j=i는 오른쪽 위가 없음. 인덱스 음수 조심
  5. 초기값 — dp[0][0] = triangle[0][0]
  6. 정답 — 마지막 줄 dp[n-1][0..n-1] 중 최댓값

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        int n = Integer.parseInt(br.readLine().trim());

        // 1) 삼각형 입력 (가변 길이 배열)
        int[][] tri = new int[n][];
        for (int i = 0; i < n; i++) {
            tri[i] = new int[i + 1];          // i층은 i+1개의 수
            StringTokenizer st = new StringTokenizer(br.readLine());
            for (int j = 0; j <= i; j++) {
                tri[i][j] = Integer.parseInt(st.nextToken());
            }
        }

        // 2) DP 테이블 초기화
        // dp[i][j] = (i, j)에 도달했을 때까지의 최대 누적 합
        int[][] dp = new int[n][];
        for (int i = 0; i < n; i++) dp[i] = new int[i + 1];
        dp[0][0] = tri[0][0];                  // 기저

        // 3) 위에서 아래로 점화식 적용
        for (int i = 1; i < n; i++) {
            for (int j = 0; j <= i; j++) {
                // 후보 1: 왼쪽 위 (i-1, j-1) — j=0이면 없음
                // 후보 2: 바로 위 (i-1, j)   — j=i이면 없음
                int fromLeft  = (j == 0) ? 0 : dp[i - 1][j - 1];
                int fromAbove = (j == i) ? 0 : dp[i - 1][j];

                // (i, j)에 오는 두 경로 중 더 큰 누적 + 현재 칸 값
                dp[i][j] = Math.max(fromLeft, fromAbove) + tri[i][j];
            }
        }

        // 4) 마지막 줄 중 최댓값이 정답
        int answer = 0;
        for (int v : dp[n - 1]) answer = Math.max(answer, v);

        System.out.println(answer);
    }
}

이 풀이의 핵심 포인트

  • "i, j에 도달했을 때의 최대 누적"이라는 명확한 상태 정의가 풀이의 절반. 모호하면 식이 안 나옵니다.
  • 경계 처리(j == 0, j == i)를 깔끔하게 — 인덱스 음수가 가장 흔한 버그.
  • 가변 길이 2차원 배열 int[i + 1]로 메모리 절약. 정사각형 배열로 짜도 답은 같지만 N=10⁴ 정도 되면 차이가 큽니다.
  • "위에서 아래로"는 bottom-up. 재귀+메모이제이션의 top-down 풀이도 가능하지만 bottom-up이 상수가 작아 코딩 테스트에서 더 선호됩니다.
외울 한 줄 — DP는 "상태 정의 → 점화식 → 초기값" 3종 세트. "dp[i]가 정확히 뭔지 한 줄로 못 적으면 식이 안 나온다"는 것이 가장 큰 진실입니다.
07

백트래킹 (Backtracking)

N ≤ 15 정도 가지치기 DFS + 조건 검사

핵심 개념

완전탐색과 거의 같지만, "이 가지를 더 탐색해도 답이 안 나옴"이 명백할 때 즉시 포기(가지치기)하는 게 차이점. 모든 경우를 다 보긴 하되, 무의미한 곳에 시간을 안 쓰는 영리한 완전탐색입니다.

언제 쓰는가

  • "조건을 만족하는 경우의 수"가 정답인 문제
  • N이 작다 (보통 N ≤ 15) — 완전탐색 시간 초과인데 가지치기로 줄일 수 있을 때
  • "~배치 가능한가", "~를 만족하는 가능한 모든 ~" 같은 표현

가지치기의 효과

N=15짜리 N-Queen은 완전탐색이면 15! = 1.3조 시도지만, 가지치기를 잘 하면 몇 만 번 시도로 끝납니다. 가지치기는 옵션이 아니라 필수.

연습 예제 — N-Queen [백준 9663]

N×N 체스판에 N개의 퀸을 서로 공격할 수 없게 놓는 경우의 수를 구하라. 퀸은 같은 행/열/대각선에 있는 다른 말을 공격한다. (N ≤ 15)

INPUT8
OUTPUT92

풀이 사고 과정

  1. 유형 파악 — "조건을 만족하는 배치 수" → 백트래킹
  2. 핵심 관찰 — 행마다 퀸 하나씩만 놓을 수 있음. "row번째 행의 퀸을 어느 열에 놓을까"의 문제로 환원
  3. 상태 — cols[row] = row번째 행의 퀸이 놓인 열
  4. 충돌 검사 — 이전 행들에 놓인 퀸과 (a) 같은 열인가, (b) 대각선상에 있는가
  5. 가지치기 — 충돌하면 이 열에 안 놓고 다음 열 시도. 모든 열이 안 되면 이전 행으로 돌아감
  6. 종료 조건 — row == N이면 N개 다 놓는 데 성공 → 카운트 +1

완성된 풀이 코드

import java.io.*;

public class Main {
    static int N;
    static int[] cols;            // cols[row] = 그 행에서 퀸이 놓인 열
    static int count = 0;

    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        N = Integer.parseInt(br.readLine().trim());
        cols = new int[N];

        backtrack(0);                // 0번째 행부터 시작

        System.out.println(count);
    }

    /**
     * row번째 행의 퀸을 어느 열에 놓을지 결정.
     */
    static void backtrack(int row) {
        // [종료 조건] N개 행을 모두 놓는 데 성공
        if (row == N) {
            count++;
            return;
        }

        // row번째 행의 0~N-1열을 차례로 시도
        for (int col = 0; col < N; col++) {
            // ★ 가지치기: 이 열이 가능한가?
            if (!isSafe(row, col)) continue;

            cols[row] = col;             // (row, col)에 퀸 놓음
            backtrack(row + 1);          // 다음 행으로
            // cols[row]는 다음 col에서 덮어쓰여지므로 원복 불필요
        }
    }

    /**
     * (row, col)에 퀸을 놓는 게 안전한가?
     * 이전 행들에 놓인 퀸과 비교:
     *  1) 같은 열인가
     *  2) 대각선 상에 있는가 (행 차이 == 열 차이)
     */
    static boolean isSafe(int row, int col) {
        for (int r = 0; r < row; r++) {
            // 같은 열
            if (cols[r] == col) return false;
            // 대각선: 행 차이의 절댓값 == 열 차이의 절댓값
            if (Math.abs(cols[r] - col) == row - r) return false;
        }
        return true;
    }
}

이 풀이의 핵심 포인트

  • "행마다 하나씩 놓는다"는 모델링이 핵심. 그러면 "같은 행" 충돌은 자동으로 사라집니다.
  • 대각선 검사는 "행 차이 == 열 차이"로 단순화. Math.abs로 양쪽 대각선 한 번에 처리.
  • isSafe는 O(row). 전체 가지치기로 실제 호출 수는 극적으로 줄어듭니다.
  • 같은 패턴 응용: 부분집합 합, 순열 생성, 스도쿠, 단어 퍼즐.
외울 한 줄 — 백트래킹 = "완전탐색 + 가지치기". "이 가지 더 가도 답 없다"를 빨리 알아채는 것이 통과의 비결입니다.
08

다익스트라 (Dijkstra)

가중치 그래프 O((V + E) log V) PriorityQueue

핵심 개념

한 시작점에서 모든 정점까지의 최단 거리를 구하는 알고리즘. BFS와 비슷하지만 큐 대신 우선순위 큐를 사용해 "현재까지 발견된 가장 가까운 노드부터 처리"합니다. 음수 간선이 없는 그래프에서만 동작합니다.

언제 쓰는가

  • "A에서 B로 가는 최단 거리"인데 간선마다 가중치가 다른 경우
  • "한 노드에서 모든 노드까지의 최단 거리"
  • (가중치가 모두 같으면 BFS면 충분 — 다익스트라는 오버킬)

핵심 아이디어 — "가장 가까운 것부터 확정"

아직 확정 안 된 노드 중 가장 거리가 짧은 노드를 꺼내 그 거리를 확정합니다. 이 노드를 거쳐 갈 수 있는 다른 노드들의 거리를 업데이트(relax). 이 과정을 반복.

연습 예제 — 최단경로 [백준 1753]

방향 그래프와 시작 정점 K가 주어진다. K에서 다른 모든 정점까지의 최단 거리를 구하라. 도달 불가능하면 "INF" 출력. (V ≤ 20,000, E ≤ 300,000, 가중치 ≤ 10)

INPUT5 6 1 5 1 1 1 2 2 1 3 3 2 3 4 2 4 5 3 4 6
OUTPUT0 2 3 7 INF

풀이 사고 과정

  1. 유형 파악 — "한 점에서 모든 점" + 양수 가중치 → 다익스트라
  2. 그래프 표현 — 정점 V=20,000개, 간선 E=300,000개. 인접 행렬은 4억 메모리로 불가능. 인접 리스트 사용
  3. 거리 배열 — dist[i]를 Integer.MAX_VALUE로 초기화, 시작점은 0
  4. PQ에 (정점, 누적거리) 형태로 저장, 누적거리 짧은 순으로 꺼냄
  5. 핵심 최적화 — 같은 정점이 PQ에 여러 번 들어가지만, "꺼낼 때의 거리가 이미 알고 있는 dist보다 크면 무시"
  6. 출력 — 도달 못한 정점은 "INF" 문자열로

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));

        // 1) V (정점 수), E (간선 수)
        StringTokenizer st = new StringTokenizer(br.readLine());
        int V = Integer.parseInt(st.nextToken());
        int E = Integer.parseInt(st.nextToken());
        int K = Integer.parseInt(br.readLine().trim());  // 시작 정점

        // 2) 인접 리스트 (1-indexed)
        //    graph[u] = u에서 출발하는 간선들의 리스트. 각 간선은 {도착, 가중치}
        List<int[]>[] graph = new List[V + 1];
        for (int i = 1; i <= V; i++) graph[i] = new ArrayList<>();

        for (int i = 0; i < E; i++) {
            st = new StringTokenizer(br.readLine());
            int u = Integer.parseInt(st.nextToken());
            int v = Integer.parseInt(st.nextToken());
            int w = Integer.parseInt(st.nextToken());
            graph[u].add(new int[]{v, w});
        }

        // 3) 다익스트라
        int[] dist = new int[V + 1];
        Arrays.fill(dist, Integer.MAX_VALUE);
        dist[K] = 0;

        // PQ: {정점, 시작점부터의 거리}, 거리 짧은 순
        PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[1] - b[1]);
        pq.offer(new int[]{K, 0});

        while (!pq.isEmpty()) {
            int[] cur = pq.poll();
            int u = cur[0], d = cur[1];

            // ★ 핵심 최적화 ★
            // 같은 노드가 PQ에 여러 번 들어가는데,
            // 이미 더 짧은 경로로 처리된 적이 있다면 이 항목은 stale → 무시
            if (d > dist[u]) continue;

            // u에서 이웃 v로 가는 간선들 완화(relax)
            for (int[] e : graph[u]) {
                int v = e[0], w = e[1];
                int next = dist[u] + w;     // u 거쳐 v 가는 거리
                if (next < dist[v]) {
                    dist[v] = next;
                    pq.offer(new int[]{v, next});
                }
            }
        }

        // 4) 출력
        StringBuilder sb = new StringBuilder();
        for (int i = 1; i <= V; i++) {
            sb.append(dist[i] == Integer.MAX_VALUE ? "INF" : dist[i]).append('\n');
        }
        System.out.print(sb);
    }
}

이 풀이의 핵심 포인트

  • if (d > dist[u]) continue; 한 줄이 안 쓰면 시간 초과 — PQ에 같은 노드가 중복으로 들어가는데, 옛 항목은 무시해야 합니다.
  • 인접 리스트가 V·E가 큰 그래프의 표준. 인접 행렬은 메모리 폭발.
  • 음수 간선이 있으면 벨만-포드, 모든 쌍 최단경로면 플로이드-워셜로 갈아탑니다.
  • 오버플로 주의: dist[u] + w가 int를 넘을 수 있으면 long 사용.
외울 한 줄 — 다익스트라 = "BFS + 우선순위 큐". 옛 항목을 무시하는 if (d > dist[u]) continue;가 통과의 비밀입니다.
09

유니온 파인드 (Disjoint Set / DSU)

집합 관리 거의 O(1) parent 배열

핵심 개념

여러 원소들을 "같은 집합인지 다른 집합인지" 빠르게 관리하는 자료구조. 각 원소는 자신이 속한 집합의 대표(루트)를 가리키는 트리 구조. 두 연산만 있습니다.

  • find(x): x가 속한 집합의 대표(루트)를 찾기
  • union(a, b): a와 b가 속한 두 집합을 하나로 합치기

언제 쓰는가

  • "두 노드가 연결되어 있는가?"를 빠르게 판정해야 할 때
  • 그래프 사이클 검출 (간선을 union하다 같은 집합이면 사이클)
  • 최소 신장 트리 (MST) — 크루스칼 알고리즘의 핵심 부품
  • "친구의 친구는 친구" 같은 동치 관계 묶기

두 가지 최적화 — 안 쓰면 O(N²)

  • 경로 압축 (Path Compression): find하는 동안 거쳐온 노드를 모두 루트에 직접 연결
  • 랭크/사이즈 기반 합치기: 작은 트리를 큰 트리 아래에 붙임

두 최적화를 모두 쓰면 한 연산의 평균 시간이 거의 O(1)이 됩니다 (정확히는 아커만 역함수).

연습 예제 — 집합의 표현 [백준 1717]

초기에 {0}, {1}, ..., {n} 의 단원소 집합들이 있다. 두 가지 연산 처리:
① 0 a b : a가 포함된 집합과 b가 포함된 집합 합치기
② 1 a b : a와 b가 같은 집합에 있나? YES/NO 출력
(n ≤ 10⁶, 연산 ≤ 10⁵)

INPUT7 8 0 1 3 1 1 7 0 7 6 1 7 1 0 3 7 0 4 2 0 1 1 1 1 1
OUTPUTNO NO YES

풀이 사고 과정

  1. 유형 파악 — "같은 집합인가" 판정 → 유니온 파인드의 정석
  2. 초기화 — 모든 i에 대해 parent[i] = i (자기 자신이 루트)
  3. find 구현 — 재귀로 루트까지 올라가며 경로 압축
  4. union 구현 — 두 루트를 비교, 한쪽을 다른 쪽 밑에 붙임
  5. 같은 집합 판정 — find(a) == find(b)
  6. 출력 — StringBuilder에 모으기

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    static int[] parent;

    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringBuilder sb = new StringBuilder();

        StringTokenizer st = new StringTokenizer(br.readLine());
        int n = Integer.parseInt(st.nextToken());
        int m = Integer.parseInt(st.nextToken());

        // 0~n까지 사용 (총 n+1개 원소)
        parent = new int[n + 1];
        for (int i = 0; i <= n; i++) {
            parent[i] = i;       // 자기 자신이 루트인 단원소 집합
        }

        for (int i = 0; i < m; i++) {
            st = new StringTokenizer(br.readLine());
            int op = Integer.parseInt(st.nextToken());
            int a  = Integer.parseInt(st.nextToken());
            int b  = Integer.parseInt(st.nextToken());

            if (op == 0) {
                union(a, b);
            } else {
                // 같은 집합이면 두 루트가 같음
                sb.append(find(a) == find(b) ? "YES" : "NO").append('\n');
            }
        }
        System.out.print(sb);
    }

    /**
     * x의 루트를 찾는다.
     * ★ 경로 압축: 거쳐온 모든 노드를 루트에 직접 연결 → 다음 호출은 O(1)
     */
    static int find(int x) {
        if (parent[x] == x) return x;
        return parent[x] = find(parent[x]);
    }

    /**
     * a와 b가 속한 집합을 합친다.
     * 이미 같은 집합이면 아무 일도 안 함.
     */
    static void union(int a, int b) {
        int ra = find(a);
        int rb = find(b);
        if (ra == rb) return;

        // 더 작은 번호가 루트가 되도록 (선택사항 — 일관성 유지용)
        if (ra < rb) parent[rb] = ra;
        else          parent[ra] = rb;
    }
}

이 풀이의 핵심 포인트

  • 경로 압축은 return parent[x] = find(parent[x]) 한 줄에 다 들어있습니다. 안 쓰면 트리가 길어져 시간 초과.
  • parent[i] = i 초기화를 빼먹는 게 흔한 버그.
  • "노드 a와 b를 union" → "어디서 노드 번호를 받았는지 추적" → 그래프 사이클 검출, 크루스칼 MST 등 응용이 다양합니다.
  • 여러 집합의 크기를 알고 싶다면 size[] 배열을 추가해 union 시 합산.
외울 한 줄 — DSU = "parent 배열 + find + union". 경로 압축 한 줄이 핵심. 외워두면 친구 관계, 사이클, MST를 다 풀 수 있습니다.
10

누적합 (Prefix Sum)

구간 합 질의 전처리 O(N) + 질의 O(1)

핵심 개념

배열의 부분 합을 미리 한 번 계산해두면, 그 다음부터 임의의 구간 [l, r]의 합을 단 O(1)에 답할 수 있습니다. 매 질의를 O(N)으로 풀던 것을 O(1)로 줄이는 마법.

왜 동작하나 — 핵심 식 한 줄

prefix[i] = "원본 배열의 0부터 i-1까지의 합" 으로 정의하면,

arr[l..r] 합 = prefix[r+1] - prefix[l]

이게 전부입니다. "끝까지의 합 − 시작 직전까지의 합 = 구간 합" — 적분의 직관과 똑같습니다.

언제 쓰는가

  • "구간 [l, r]의 합을 여러 번 질의" — 단순 반복은 O(N×M), 누적합은 O(N+M)
  • 슬라이딩 윈도우 합, 부분합 비교
  • 2차원 응용 — 직사각형 영역 합 (2D 누적합)

연습 예제 — 구간 합 구하기 4 [백준 11659]

N개의 수와 M개의 질의(i, j)가 주어진다. 각 질의마다 i번째부터 j번째까지의 수의 합을 출력하라. (N, M ≤ 100,000)

INPUT5 3 5 4 3 2 1 1 3 2 4 5 5
OUTPUT12 9 1

풀이 사고 과정

  1. 유형 파악 — "구간 합 × M번" → 누적합
  2. 왜 단순 반복은 안 되나 — 매 질의가 O(N) → 총 O(N×M) = 10¹⁰ → 시간 초과
  3. 1-indexed 누적합 설계 — prefix[0] = 0, prefix[i] = arr[1] + ... + arr[i]
  4. 질의 처리 — prefix[j] - prefix[i-1]
  5. 오버플로 — N=10⁵개의 정수 합은 int 안전하지만, 만약을 위해 long 권장
  6. 출력 — M=10⁵줄, StringBuilder 필수

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringBuilder sb = new StringBuilder();

        StringTokenizer st = new StringTokenizer(br.readLine());
        int N = Integer.parseInt(st.nextToken());
        int M = Integer.parseInt(st.nextToken());

        // 1) 누적합 배열 — 1-indexed로 만들면 계산이 깔끔
        //    prefix[0] = 0
        //    prefix[i] = arr[1] + arr[2] + ... + arr[i]
        long[] prefix = new long[N + 1];
        st = new StringTokenizer(br.readLine());
        for (int i = 1; i <= N; i++) {
            int v = Integer.parseInt(st.nextToken());
            prefix[i] = prefix[i - 1] + v;
        }

        // 2) M개의 질의를 각각 O(1)에 처리
        for (int q = 0; q < M; q++) {
            st = new StringTokenizer(br.readLine());
            int i = Integer.parseInt(st.nextToken());
            int j = Integer.parseInt(st.nextToken());

            // 구간 [i..j]의 합 = prefix[j] - prefix[i-1]
            long sum = prefix[j] - prefix[i - 1];
            sb.append(sum).append('\n');
        }
        System.out.print(sb);
    }
}

이 풀이의 핵심 포인트

  • 1-indexed로 설계하면 prefix[i-1]의 i=1 케이스가 자연스럽게 처리됩니다(0이 됨). 0-indexed면 매번 경계 검사 필요.
  • 전처리 O(N), 질의 O(1) — 시간복잡도가 극적으로 줄어듭니다.
  • 응용 — 2차원 누적합은 prefix[i][j] = prefix[i-1][j] + prefix[i][j-1] - prefix[i-1][j-1] + arr[i][j]로 확장. 직사각형 영역 합을 O(1)에 답합니다.
외울 한 줄 — 누적합 = "끝까지의 합 - 시작 직전까지의 합". 이 한 줄을 안 잊으면 1차원, 2차원, 차분 다 통합니다.

투포인터 / 슬라이딩 윈도우

연속 부분 구간 O(N) left / right 두 인덱스

핵심 개념

배열을 두 개의 인덱스(left, right)로 훑으며 "창문(window)을 늘리고 줄이는" 패턴. 전체 배열을 한 번만 훑어 O(N)에 답을 냅니다.

두 가지 변형

  • 슬라이딩 윈도우 — 창문 크기를 조건에 맞게 늘이거나 줄이며 최적값 갱신
  • 투포인터 — 두 인덱스가 다른 방향에서 만나는 경우(정렬된 배열에서 합이 X인 두 수 찾기 등)

언제 쓰는가

  • "연속된 부분 수열 중 조건을 만족하는 ~"
  • "합이 S 이상/이하인 가장 긴/짧은 구간"
  • "중복 없는 가장 긴 부분 수열"
  • 정렬된 배열에서 "두 수의 합이 X인 쌍" 찾기

왜 O(N)일까 — 직관

two for문이라면 O(N²)이지만, 슬라이딩 윈도우는 left와 right가 각각 최대 N번씩만 이동합니다. 둘 다 합쳐도 2N번 → O(N). "right는 항상 전진, left는 조건에 따라 전진"이 핵심.

연습 예제 — 부분합 [백준 1806]

N개의 양의 정수 배열에서 연속된 수의 합이 S 이상이 되는 가장 짧은 부분 수열의 길이를 구하라. 없다면 0. (N ≤ 100,000)

INPUT10 15 5 1 3 5 10 7 4 9 2 8
OUTPUT2 (10 + 7 = 17 ≥ 15)

풀이 사고 과정

  1. 유형 파악 — "연속 부분 수열" + "조건 만족하는 최소/최대 길이" → 슬라이딩 윈도우의 정석
  2. 윈도우 정의 — [left..right] 구간이 현재 윈도우. sum은 그 구간의 합
  3. 전략 — right를 1칸씩 늘리면서 sum 누적. sum ≥ S가 되면 left를 최대한 당겨서 윈도우를 줄임
  4. 답 갱신 — sum ≥ S일 때마다 윈도우 길이를 최솟값으로 갱신
  5. 도달 불가 — 한 번도 갱신 안 됐으면 0 출력

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));

        StringTokenizer st = new StringTokenizer(br.readLine());
        int N = Integer.parseInt(st.nextToken());
        int S = Integer.parseInt(st.nextToken());

        int[] arr = new int[N];
        st = new StringTokenizer(br.readLine());
        for (int i = 0; i < N; i++) {
            arr[i] = Integer.parseInt(st.nextToken());
        }

        // 슬라이딩 윈도우 변수
        int left = 0;
        int sum = 0;
        int answer = Integer.MAX_VALUE;       // 최솟값을 갱신할 거니 큰 값으로 시작

        // right를 한 칸씩 전진
        for (int right = 0; right < N; right++) {
            sum += arr[right];                  // 윈도우 오른쪽 확장

            // 조건(sum ≥ S)을 만족하는 동안 왼쪽을 최대한 당겨서 윈도우 축소
            while (sum >= S) {
                // 현재 윈도우 길이 (right - left + 1)로 답 갱신
                answer = Math.min(answer, right - left + 1);

                // left를 1칸 오른쪽으로 → 윈도우에서 arr[left] 제거
                sum -= arr[left];
                left++;
            }
        }

        // 한 번도 조건을 만족 못 했다면 0
        System.out.println(answer == Integer.MAX_VALUE ? 0 : answer);
    }
}

이 풀이의 핵심 포인트

  • "right는 항상 전진, left는 조건에 따라 전진" — 각 인덱스가 최대 N번 → 총 O(N).
  • 윈도우 길이는 right - left + 1 (인덱스 차이 + 1) — 자주 헷갈리는 부분.
  • 유의 — "양의 정수"라는 조건이 슬라이딩 윈도우 가능 이유. 음수가 섞이면 윈도우를 줄였다고 sum이 줄어든다는 보장이 없어 누적합 + 이분탐색 등 다른 접근이 필요.
  • "가장 긴" 문제면 Math.max로, 조건 부등호 반대 방향으로 같은 패턴 적용.
외울 한 줄 — 슬라이딩 윈도우는 "right는 무조건 전진, left는 조건 만나면 추격". 한 번 훑기로 O(N²)을 O(N)으로 줄이는 마법입니다.

비트마스킹 (Bitmasking)

N ≤ 20 O(2ᴺ × N) 집합 상태 압축

핵심 개념

N개의 원소 각각의 "포함 여부"를 한 정수의 비트로 표현합니다. N=10이라면 0~1023(2¹⁰)의 정수 하나로 모든 부분집합을 표현할 수 있습니다. 메모리도 적고 비트 연산이 빠릅니다.

왜 N=20까지인가

2²⁰ ≈ 100만. 이 정도면 모든 부분집합을 순회해도 1초 안에 끝납니다. 2²⁵부터는 시간 초과 위험.

핵심 비트 연산 4가지

연산의미
1 << ii번 비트만 1인 마스크 (예: i=2 → 0b100)
mask | (1 << i)i번 비트 켜기 (원소 추가)
mask & ~(1 << i)i번 비트 끄기 (원소 제거)
(mask & (1 << i)) != 0i번 원소가 포함됐는지 확인

언제 쓰는가

  • "부분집합 전체를 시도" + N ≤ 20
  • "방문 상태가 작은 정수로 표현 가능" — TSP(외판원), 동전 도시 방문 등
  • DP의 상태로 "지금까지 방문한 도시 집합"을 비트마스크로 표현

연습 예제 — 부분수열의 합 [백준 1182]

N개의 정수로 이루어진 수열이 있다. 그 부분수열(공집합 제외) 중 합이 정확히 S가 되는 것의 개수를 구하라. (N ≤ 20)

INPUT5 0 -7 -3 -2 5 8
OUTPUT1 (-3, -2, 5)

풀이 사고 과정

  1. 유형 파악 — "모든 부분수열" + N ≤ 20 → 비트마스킹 완전탐색의 정석
  2. 부분집합 표현 — N비트 정수 mask로 표현. i번 비트가 1이면 arr[i]를 포함
  3. 전체 순회 — mask를 0부터 (1<<N)-1까지 (총 2ᴺ개)
  4. 합 계산 — 각 mask마다 N개 비트를 확인해 포함된 원소를 더함 → O(N)
  5. 전체 O(2ᴺ × N) — N=20이면 약 2천만, 1초 내 충분
  6. 공집합 제외 — mask=0은 건너뜀

완성된 풀이 코드

import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));

        StringTokenizer st = new StringTokenizer(br.readLine());
        int N = Integer.parseInt(st.nextToken());
        int S = Integer.parseInt(st.nextToken());

        int[] arr = new int[N];
        st = new StringTokenizer(br.readLine());
        for (int i = 0; i < N; i++) {
            arr[i] = Integer.parseInt(st.nextToken());
        }

        int count = 0;

        // mask = 0 : 공집합 (제외)
        // mask = 1 : {arr[0]}
        // mask = 2 : {arr[1]}
        // mask = 3 : {arr[0], arr[1]}
        // ...
        // mask = (1<<N)-1 : 전체 집합
        for (int mask = 1; mask < (1 << N); mask++) {
            int sum = 0;

            // mask의 i번째 비트가 1이면 arr[i] 더하기
            for (int i = 0; i < N; i++) {
                if ((mask & (1 << i)) != 0) {
                    sum += arr[i];
                }
            }

            if (sum == S) count++;
        }

        System.out.println(count);
    }
}

이 풀이의 핵심 포인트

  • mask < (1 << N)이 끝까지 도는 조건 — N비트 정수 전체 순회.
  • 공집합(mask=0) 제외는 문제 조건. 시작을 mask = 1로 두면 자연스럽게 처리.
  • (mask & (1 << i)) != 0으로 비트 확인. 괄호 위치 주의 — &는 !=보다 우선순위가 낮습니다.
  • 응용 — "지금까지 방문한 도시 집합"을 비트마스크 + DP로 표현하면 TSP가 O(N² · 2ᴺ)에 풀립니다.
외울 한 줄 — 비트마스킹 = "N개의 boolean을 한 정수로 압축". N ≤ 20이면 모든 부분집합을 정수 한 변수로 순회 가능합니다.
i유형 인식 훈련법
문제를 풀기 전에 지문만 보고 "이건 OO 유형"을 5초 안에 떠올려보고 시작하세요. 풀이 후 정답이 어떤 유형이었는지 비교하면, 30문제만 풀어도 직관이 폭발적으로 자랍니다. 이 책의 12개 패턴 각각의 식별 신호를 외워두는 게 합격의 절반입니다.

09.실전 풀이 전략과 디버깅

"같은 알고리즘을 알아도 누구는 풀고 누구는 못 풉니다." 그 차이의 절반은 알고리즘이 아니라 풀이 습관에서 나옵니다. 한 문제를 30~40분 안에 끝내는 루틴과, 코딩 외적 메타 스킬을 같이 익히세요.

문제 풀이의 "메타 스킬" 9가지

알고리즘 외적인 합격 비결입니다. 잘하는 사람의 손버릇이라 봐도 됩니다. 작은 차이가 모이면 30분짜리 문제와 2시간짜리 문제가 갈립니다.

① 지문을 주석으로 옮겨 적기 — 가장 강력한 한 가지

문제를 풀기 전에 코드 맨 위에 주석으로 지문의 핵심을 옮겨 적으세요. "이해 = 정리"입니다. 지문을 적는 그 과정이 곧 이해의 과정입니다.

// ════════════════════════════════════════════
// [문제] 회의실 배정
// N개의 회의가 있다. 각 회의는 (시작, 끝).
// 회의실 1개로 겹치지 않게 최대 몇 개 배정?
// ─────────────────────────────────────────
// [입력]
//   첫 줄: N  (1 ≤ N ≤ 100,000)
//   N줄  : 시작 끝
//
// [출력] 최대 회의 수 (정수 1개)
//
// [복잡도 한계] N log N 정도까지 OK
//
// [풀이 아이디어]
//   - 끝나는 시간으로 정렬
//   - 욕심껏 선택 (그리디)
//   - 끝 시간이 같으면 시작 빠른 것 우선 (0초 회의 처리)
//
// [코너 케이스]
//   - 시작 == 끝 (0초짜리)
//   - N == 1
//   - 모두 겹치는 경우 → 답 1
// ════════════════════════════════════════════

왜 강력한가

  • 지문 위로 자꾸 돌아가지 않게 되어 시야가 코드에 집중됩니다.
  • 적는 동안 빠뜨린 조건을 발견합니다.
  • 막혔을 때 본인이 정리한 노트를 다시 보면 됩니다.
  • 라이브 코딩 면접에서는 "사고가 정리된 사람" 인상을 줍니다.

② 변수명은 "짧게"가 아니라 "정확하게"

변수명을 잘 짓는 데 든 10초가 디버깅 30분을 아낍니다. a, b, t, x 같은 이름은 코드 외계어로 만듭니다.

BAD
int[] a = new int[n];
int t = 0;
for (int i = 0; i < n; i++) {
    if (a[i] > b) t++;
}
GOOD
int[] scores = new int[n];
int passCount = 0;
for (int i = 0; i < n; i++) {
    if (scores[i] > threshold) passCount++;
}

문제 요약

식당 영수증을 정리하는 프로그램. 메뉴 이름, 가격, 주문 횟수가 주어질 때 "가장 많이 팔린 메뉴의 매출"을 구합니다. 단, 같은 매출이 여러 개면 메뉴명 사전순으로 가장 빠른 것을 답합니다.

핵심 신호

  • "가장 많이 팔린" → 카운팅 → HashMap
  • "동률이면 사전순" → 정렬 기준 2개 (매출 내림차순, 이름 오름차순)

접근법

  1. 메뉴별 매출을 HashMap<String, Long>에 누적
  2. entrySet을 List로 만든 후 비교자 2단계로 정렬
  3. 맨 앞 원소가 정답

Java 풀이

import java.util.*;

public class Main {
    public static String solution(String[] menus, int[] prices, int[] counts) {
        Map<String, Long> sales = new HashMap<>();
        for (int i = 0; i < menus.length; i++) {
            long revenue = (long) prices[i] * counts[i];
            sales.merge(menus[i], revenue, Long::sum);
        }
        List<Map.Entry<String, Long>> list = new ArrayList<>(sales.entrySet());
        list.sort((a, b) -> {
            if (!a.getValue().equals(b.getValue()))
                return Long.compare(b.getValue(), a.getValue()); // 매출 내림차순
            return a.getKey().compareTo(b.getKey()); // 이름 오름차순
        });
        return list.get(0).getKey();
    }
}

이 문제에서 배울 점

매출 계산은 반드시 long으로 (가격×횟수가 int 범위를 쉽게 초과). 동률 처리는 비교자 안에서 if-else로 2단계 정렬을 작성하면 깔끔합니다.

"new_id" 같은 아이디 후보를 받아 주어진 7단계 규칙(소문자화 · 허용문자만 · 연속 마침표 축소 · 처음/끝 마침표 제거 · 빈 문자열이면 "a" · 16자 이상이면 자르고 마침표 제거 · 3자 미만이면 마지막 글자 반복)을 거쳐 최종 아이디로 변환합니다.

핵심 신호

  • "이런 규칙으로 변환하라" → 그냥 순서대로 구현하는 시뮬레이션
  • 정규식 또는 StringBuilder로 단계별 처리

Java 풀이 (정규식 활용)

public String solution(String s) {
    // 1단계: 소문자화
    s = s.toLowerCase();
    // 2단계: 허용 문자만 (소문자, 숫자, -, _, .)
    s = s.replaceAll("[^a-z0-9\\-_.]", "");
    // 3단계: 연속 마침표 → 1개
    s = s.replaceAll("\\.+", ".");
    // 4단계: 처음/끝 마침표 제거
    s = s.replaceAll("^\\.|\\.$", "");
    // 5단계: 빈 문자열이면 "a"
    if (s.isEmpty()) s = "a";
    // 6단계: 16자 이상이면 자르고 끝 마침표 제거
    if (s.length() >= 16) {
        s = s.substring(0, 15);
        s = s.replaceAll("\\.$", "");
    }
    // 7단계: 2자 이하이면 마지막 글자 반복
    while (s.length() < 3) {
        s += s.charAt(s.length() - 1);
    }
    return s;
}

이 문제에서 배울 점

카카오는 "긴 지문을 정확히 옮기는 능력"을 봅니다. 단계마다 변수에 결과를 저장하고, 각 단계를 끝낸 직후 출력해보며 디버깅하세요. 정규식이 익숙하지 않다면 StringBuilder로 한 글자씩 처리해도 됩니다.

PROBLEM 03 백준 단골 / 네이버 (BFS)
난이도 중
BFS그래프격자

문제 요약

N×M 격자에서 1은 땅, 0은 바다. 상하좌우로 연결된 1의 덩어리(섬)의 개수를 구합니다.

핵심 신호

  • "연결된 덩어리 개수" → BFS/DFS 전형
  • 격자 + 4방향 → dx, dy 배열

Java 풀이 (BFS)

static int[][] map;
static boolean[][] visited;
static int N, M;
static int[] dx = {-1, 1, 0, 0};
static int[] dy = {0, 0, -1, 1};

static void bfs(int sx, int sy) {
    Queue<int[]> q = new ArrayDeque<>();
    q.offer(new int[]{sx, sy});
    visited[sx][sy] = true;
    while (!q.isEmpty()) {
        int[] cur = q.poll();
        for (int d = 0; d < 4; d++) {
            int nx = cur[0] + dx[d];
            int ny = cur[1] + dy[d];
            if (nx<0||ny<0||nx>=N||ny>=M) continue;
            if (visited[nx][ny] || map[nx][ny] == 0) continue;
            visited[nx][ny] = true;
            q.offer(new int[]{nx, ny});
        }
    }
}

public static void main(String[] args) throws IOException {
    // 입력 생략
    int count = 0;
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < M; j++) {
            if (map[i][j] == 1 && !visited[i][j]) {
                bfs(i, j);
                count++;
            }
        }
    }
    System.out.println(count);
}

이 문제에서 배울 점

"방문 체크는 큐에 넣을 때" 합니다. 꺼낼 때 하면 중복 방문이 발생합니다. 8방향 문제로 변형되어도 dx, dy 배열만 8개로 늘리면 됩니다.

PROBLEM 04 백준 단골 (이분탐색/매개변수)
난이도 중
이분탐색매개변수탐색

문제 요약

길이가 제각각인 나무 N개를 잘라 적어도 M미터의 나무를 가져가야 합니다. 절단기 높이 H를 정수로 설정할 때, 가져갈 수 있는 H의 최댓값은? (나무 길이 ≤ 10⁹, N ≤ 10⁶)

핵심 신호

  • "최댓값/최솟값을 구하라" + 답의 범위가 명확
  • 특정 H가 가능한지 O(N)에 검사 가능 → 매개변수 탐색

Java 풀이

static long total(int[] trees, int h) {
    long sum = 0;
    for (int t : trees) {
        if (t > h) sum += t - h;
    }
    return sum;
}

public static int solve(int[] trees, long M) {
    int lo = 0, hi = 0;
    for (int t : trees) hi = Math.max(hi, t);

    int ans = 0;
    while (lo <= hi) {
        int mid = (lo + hi) >>> 1;
        if (total(trees, mid) >= M) {
            ans = mid;       // 더 높게도 가능할 수 있으니 갱신 후 위로
            lo = mid + 1;
        } else {
            hi = mid - 1;
        }
    }
    return ans;
}

이 문제에서 배울 점

"H를 찾는다"가 아니라 "H를 추측하고 가능한지 검사한다"는 사고 전환이 매개변수 탐색의 본질. 합이 int 범위 넘으니 반드시 long. 최댓값을 찾을 땐 "가능하면 갱신 후 더 큰 쪽으로"가 패턴입니다.

PROBLEM 05 삼성 SW 역량 (시뮬레이션 + 완전탐색)
난이도 상
시뮬레이션완전탐색백트래킹

문제 요약

N×N 격자에 연구소가 있고 일부는 벽, 일부는 바이러스. 벽을 정확히 3개 더 세웠을 때, 바이러스가 퍼지지 않는 "안전 영역"의 최댓값을 구합니다. (N ≤ 8)

핵심 신호

  • N이 매우 작다 → 완전탐색 가능
  • "정확히 3개" → 조합 선택
  • "퍼지는 영역" → BFS

접근법

  1. 빈 칸 좌표를 모아둠
  2. 그 중 3개를 고르는 모든 조합 (C(빈칸, 3))
  3. 각 조합마다 임시로 벽 세우고 BFS로 바이러스 확산
  4. 남은 0의 개수가 안전 영역 → 최댓값 갱신

Java 풀이 (핵심부)

static int N, M, ans;
static int[][] map;
static List<int[]> empties = new ArrayList<>();

static void pickWalls(int start, int depth) {
    if (depth == 3) {
        ans = Math.max(ans, spreadAndCount());
        return;
    }
    for (int i = start; i < empties.size(); i++) {
        int[] p = empties.get(i);
        map[p[0]][p[1]] = 1;        // 임시 벽
        pickWalls(i + 1, depth + 1);
        map[p[0]][p[1]] = 0;        // 원복
    }
}

static int spreadAndCount() {
    int[][] copy = new int[N][M];
    Queue<int[]> q = new ArrayDeque<>();
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < M; j++) {
            copy[i][j] = map[i][j];
            if (copy[i][j] == 2) q.offer(new int[]{i, j});
        }
    }
    // BFS로 2 확산 (생략)
    // 0의 개수 반환
    int cnt = 0;
    for (int[] r : copy) for (int v : r) if (v == 0) cnt++;
    return cnt;
}

이 문제에서 배울 점

삼성 SW 역량은 "완전탐색 + 시뮬레이션 + 상태 복원" 콤보가 단골입니다. 시뮬레이션 전 원본을 깊은 복사하거나, 처리 후 원상 복구하는 패턴을 익혀두세요.

PROBLEM 06 백준 / 네이버 라인 (DP)
난이도 중
DP배낭문제

문제 요약

무게와 가치가 있는 N개의 물건, 최대 K까지 담을 수 있는 가방. 가치 합을 최대로 만드는 가치 합은? (각 물건 1개씩)

핵심 신호

  • "넣을지 말지의 선택을 N번" → 0/1 배낭 DP
  • 상태: dp[i][w] = i번째까지 보았고 무게 w 이하일 때 최대 가치

Java 풀이

// 2차원 DP
int[][] dp = new int[N + 1][K + 1];
for (int i = 1; i <= N; i++) {
    for (int w = 0; w <= K; w++) {
        dp[i][w] = dp[i - 1][w];  // 안 넣음
        if (w >= weight[i]) {
            dp[i][w] = Math.max(dp[i][w],
                                dp[i - 1][w - weight[i]] + value[i]); // 넣음
        }
    }
}
System.out.println(dp[N][K]);

// 1차원 최적화 (메모리 절약): 무게는 반드시 내림차순으로!
int[] dp1 = new int[K + 1];
for (int i = 1; i <= N; i++) {
    for (int w = K; w >= weight[i]; w--) {
        dp1[w] = Math.max(dp1[w], dp1[w - weight[i]] + value[i]);
    }
}

이 문제에서 배울 점

1차원 최적화에서 무게 루프를 내림차순으로 도는 이유는 "같은 물건을 두 번 쓰지 않기 위해서"입니다. 오름차순으로 돌면 무한 배낭(같은 물건 여러 개)이 됩니다.

PROBLEM 07 쿠팡 / LeetCode 스타일 (자료구조)
난이도 중
PriorityQueue힙스케줄링

문제 요약

N개의 작업이 있고 각 작업은 마감일과 보상을 가집니다. 한 번에 한 작업만 처리 가능하며 각 작업은 1일 소요됩니다. 얻을 수 있는 보상의 최댓값은?

핵심 신호

  • "가능한 한 큰 보상을 챙겨야" → 그리디
  • "마감일까지만 가능" + "취소도 가능" → 최소 힙으로 "현재 채택한 작업들"을 관리

접근법

  1. 마감일 오름차순 정렬
  2. 최소 힙에 보상을 넣고 채택. 단, 마감일을 넘으면 가장 작은 보상 작업을 꺼내고 새것 채택
  3. 마지막에 힙에 남은 보상의 합이 정답

Java 풀이

public int maxProfit(int[][] jobs) { // jobs[i] = {deadline, reward}
    Arrays.sort(jobs, (a, b) -> a[0] - b[0]);
    PriorityQueue<Integer> pq = new PriorityQueue<>(); // 최소 힙

    for (int[] job : jobs) {
        pq.offer(job[1]);
        if (pq.size() > job[0]) {
            pq.poll(); // 가장 작은 보상 버림
        }
    }
    int sum = 0;
    while (!pq.isEmpty()) sum += pq.poll();
    return sum;
}

이 문제에서 배울 점

그리디인데 "잘못 골랐으면 더 좋은 것이 왔을 때 바꿔치기"가 가능한 구조에서는 최소 힙으로 후회 없는 선택을 만들 수 있습니다. 이 패턴은 회의실, 작업 스케줄링, 발사대 문제 등으로 변형됩니다.

PROBLEM 08 백준 단골 (그래프 - 다익스트라)
난이도 상
다익스트라그래프최단경로

문제 요약

방향 그래프와 시작점이 주어집니다. 시작점에서 모든 정점까지의 최단 거리를 구합니다. (간선 가중치 양수, V ≤ 20,000, E ≤ 300,000)

핵심 신호

  • "한 점에서 모든 점까지 최단" + 양수 가중치 → 다익스트라
  • V, E의 크기 → 인접 리스트 + 우선순위 큐 (O(E log V))

Java 풀이

static List<int[]>[] graph;  // graph[u] = list of {v, w}

static int[] dijkstra(int V, int start) {
    int[] dist = new int[V + 1];
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[start] = 0;

    PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[1] - b[1]);
    pq.offer(new int[]{start, 0});

    while (!pq.isEmpty()) {
        int[] cur = pq.poll();
        int u = cur[0], d = cur[1];
        if (d > dist[u]) continue;
        for (int[] e : graph[u]) {
            int v = e[0], w = e[1];
            if (dist[u] + w < dist[v]) {
                dist[v] = dist[u] + w;
                pq.offer(new int[]{v, dist[v]});
            }
        }
    }
    return dist;
}

이 문제에서 배울 점

if (d > dist[u]) continue; 한 줄이 핵심입니다. 같은 노드가 우선순위 큐에 여러 번 들어가는데, 더 짧은 거리로 이미 처리된 노드의 옛 항목을 무시해야 시간 초과를 막을 수 있습니다.
