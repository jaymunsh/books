Beginner's field guide

AI로 만드는 나만의 블로그 운영 시스템

글쓰기와 파일 작업을 돕는 AI 작업 도구인 Codex를 중심에 두고, 자료 조사, 이미지, 노코드 자동화와 성과 분석 도구를 필요한 만큼 연결하는 방법을 처음부터 안내합니다.

Codex는 여러 자료와 원고를 다루는 중심 작업실로 사용합니다. 다만 검색, 이미지 제작, 발행과 성과 분석처럼 다른 도구가 더 적합한 일은 억지로 Codex 안에서 해결하지 않습니다.

01

블로그 자동화, 비전공자도 할 수 있을까?

코딩을 몰라도 시작할 수 있습니다. 먼저 글 한 편을 만드는 과정을 작게 나누고, 반복되는 부분만 AI와 도구에 맡기면 됩니다.

블로그를 운영하다 보면 주제 정하기, 자료 찾기, 목차 만들기, 문장 다듬기처럼 비슷한 일이 계속 생깁니다. 블로그 자동화는 이 모든 일을 한 번에 기계에 넘기는 것이 아니라, 매번 반복하는 일을 정리해 더 빠르고 일정하게 처리하는 방법입니다.

예를 들어 “초보자를 위한 무선 청소기 관리법”을 쓴다고 해보겠습니다. 사람은 독자와 주제를 정하고 실제 경험, 제품 상태와 중요한 사실을 확인합니다. 글쓰기와 파일 정리를 돕는 Codex 같은 AI는 목차 후보를 만들고, 빠진 질문을 찾고, 초안을 읽기 쉽게 고치는 일을 도울 수 있습니다. 마지막으로 사람은 사실과 표현을 검수한 뒤 네이버 블로그나 Tistory에 발행합니다.

1.1

블로그 자동화란 무엇인가

오늘은 메모장, 내일은 채팅창처럼 작업 장소가 계속 바뀌면 결과도 흔들립니다. 주제 → 자료 → 목차 → 초안 → 검수 → 발행 순서를 먼저 정하는 것이 자동화의 출발점입니다.

AI에게 글 한 편을 써 달라고 요청하는 것은 ‘생성’에 가깝습니다. 자동화는 그보다 넓습니다. 어떤 자료를 사용할지, 결과를 어디에 저장할지, 누가 검토한 뒤 다음 단계로 넘길지까지 정해져 있어야 같은 과정을 다음 글에서도 반복할 수 있습니다.

초보자는 먼저 종이에 현재 순서를 적어보면 됩니다. 순서를 설명할 수 없다면 도구를 연결해도 혼란만 빨라집니다. 반대로 순서가 명확하면 코드를 몰라도 메모 양식과 체크리스트만으로 첫 자동화를 시작할 수 있습니다.

1.2

블로그 자동화를 시작하는 이유

AI는 아이디어를 넓히고 초안을 빠르게 만드는 데 유용하지만, 실제 경험이나 최신 사실을 스스로 보증하지는 못합니다. 무엇을 맡기고 무엇을 직접 확인할지 정해야 합니다.

자동화의 가장 현실적인 효과는 글을 무한히 만드는 것이 아닙니다. 빈 화면 앞에서 고민하는 시간을 줄이고, 발행 주기를 일정하게 유지하며, 링크 누락이나 파일명 혼동 같은 반복 실수를 줄이는 데 있습니다. 여러 글의 진행 상태도 한눈에 볼 수 있어 오래된 원고를 잊는 일이 줄어듭니다.

  • 제목과 목차의 초안을 빠르게 준비한다.
  • 매번 확인할 항목을 체크리스트로 고정한다.
  • 원고와 자료를 같은 규칙으로 저장한다.
  • 사람은 경험과 최종 판단에 시간을 더 쓴다.
1.3

완전 자동화보다 반자동화가 좋은 이유

버튼 한 번으로 글을 만들어 바로 공개하는 방식은 편해 보이지만, 틀린 정보와 어색한 표현도 함께 발행될 수 있습니다. 초보자는 AI가 제안하고 사람이 승인하는 방식으로 시작하는 편이 안전합니다.

특히 날짜, 가격, 정책과 제품 기능은 작성 시점과 발행 시점 사이에도 바뀔 수 있습니다. AI가 출처처럼 보이는 주소를 만들어내거나 다른 글과 비슷한 표현을 반복할 수도 있습니다. 자동 발행까지 연결하면 이런 오류를 발견할 마지막 기회가 사라집니다.

따라서 초안 생성 뒤에는 반드시 ‘검토 대기’ 단계를 둡니다. 사람이 사실, 출처, 문체와 이미지를 확인한 뒤 승인해야 발행 단계로 넘어가는 구조가 초보자에게 가장 현실적입니다. 익숙해져도 책임이 큰 판단은 자동화 대상에서 제외합니다.

특히 수익화를 생각한다면 글의 개수보다 독자가 실제로 얻는 정보와 신뢰가 먼저입니다. 비슷한 글을 대량으로 만들거나 확인하지 않은 내용을 반복 발행하는 것은 장기적인 운영에 도움이 되지 않습니다. 네이버 블로그, Tistory, AdSense처럼 사용하는 플랫폼이 달라도 이 원칙은 같습니다.

1장 정리

이번 장에서는 도구를 설치하지 않아도 됩니다. 내가 반복하는 일을 한 줄씩 적고, AI에게 맡길 수 있는 부분 하나만 고르면 충분합니다. 다음 장에서는 이 역할에 맞춰 Codex와 다른 도구를 어떻게 조합할지 살펴봅니다.

02

블로그 자동화에는 어떤 도구가 필요한가?

도구 이름부터 외우지 말고, 작성·조사·관리·디자인·연결·분석이라는 역할로 구분합니다.

블로그 자동화를 검색하면 수십 가지 서비스가 등장하지만, 처음부터 전부 쓸 필요는 없습니다. 도구는 이름보다 어떤 일을 맡길지로 구분하면 훨씬 단순해집니다. 글을 만드는 도구, 근거를 찾는 도구, 진행 상황을 기록하는 도구, 반복 작업을 연결하는 도구가 각각 한 자리씩 있다고 생각하면 됩니다.

Codex는 이 가운데 글쓰기와 파일 정리를 담당하는 중심 작업대로 쓸 수 있습니다. 한 폴더에 주제 메모, 목차, 초안과 수정본을 모아두고 “이 자료로 목차를 만들어줘” 또는 “초보자가 이해하기 어려운 표현을 찾아줘”처럼 요청하는 방식입니다. 다만 최신 정보와 실제 경험은 웹의 공식 자료와 운영자가 직접 확인해야 합니다.

한 번의 질문으로 완성본을 받기보다 독자와 목적을 알려주고 목차부터 검토하세요. 도구마다 답의 길이와 표현은 다르지만 어느 하나가 항상 정답인 것은 아닙니다. 결과가 내 기준에 맞는지가 선택 기준입니다.

Codex를 중심 도구로 사용하는 이유

여러 글과 자료를 파일로 관리하고, 같은 작성 기준을 프로젝트 단위로 반복 적용합니다.

대화형 AI가 한 번의 대화를 돕는다면 Codex는 실제 작업 폴더를 다루는 작업실에 가깝습니다. 자료, 목차, 초안과 검수 기록을 파일로 남기고 여러 문서에 같은 형식과 규칙을 적용할 때 장점이 커집니다.

다만 최신 사실 확인이나 대표 이미지 디자인까지 모두 Codex에 맡길 필요는 없습니다. Codex는 전체 흐름과 원고를 관리하고, 검색과 디자인은 전용 도구를 연결하는 방식이 효율적입니다.

Codex에 일을 맡길 때는 네 가지를 함께 적으면 결과를 검토하기 쉽습니다. 목표는 무엇을 만들지, 맥락은 어떤 파일과 독자를 참고할지, 제약은 지켜야 할 문체·출처·금지 사항, 완료 기준은 어떤 파일이 생기고 어떤 검사를 통과해야 하는지입니다.

자료 조사와 정보 정리 도구

공식 자료를 우선하고 AI 검색은 탐색 보조로 사용합니다.

검색엔진은 최신 공식 페이지를 직접 찾는 기본 도구입니다. Perplexity 같은 AI 검색은 여러 자료의 방향을 빠르게 훑을 때 편리하고, NotebookLM은 내가 모은 문서를 바탕으로 요약하거나 질문할 때 유용합니다.

어떤 도구를 쓰더라도 날짜, 가격, 정책과 수치는 원문을 열어 확인해야 합니다. AI가 제시한 요약과 출처가 실제 원문의 의미와 같은지도 사람이 판단합니다.

자동화 연결 도구

반복 작업이 늘어난 뒤 노코드 자동화 도구를 추가합니다.

n8n, Make와 Zapier는 앱 사이에서 정보를 옮기고 정해진 조건에 따라 다음 작업을 실행합니다. 예를 들어 주제표에 새 항목이 생기면 원고 폴더를 만들고, 초안 상태가 ‘검토 요청’으로 바뀌면 알림을 보낼 수 있습니다.

주 1편을 혼자 작성하는 단계라면 아직 없어도 됩니다. 같은 복사와 알림을 매주 반복하고, 입력과 결과를 한 문장으로 설명할 수 있을 때 도입하세요. 처음에는 발행보다 알림과 자료 이동을 연결하는 편이 안전합니다.

03

내게 맞는 블로그 플랫폼 선택하기

검색 유입, 수익화, 비용과 관리 난이도에 따라 출발점이 달라집니다.

플랫폼은 “어디가 가장 좋은가”보다 내가 어떤 글을 누구에게 보여주고 싶은가로 골라야 합니다. 국내 후기와 지역 정보를 빠르게 시작하려는 사람, 검색형 정보 글과 광고 수익을 함께 고려하는 사람, 주소와 기능을 직접 관리하려는 사람의 답은 서로 다릅니다.

선택할 때는 네 가지만 적어보세요. 예상 독자, 주로 쓸 글의 종류, 원하는 수익 방식, 매달 감당할 수 있는 관리 시간입니다. 플랫폼을 옮기는 일은 새로 시작하는 것보다 번거로우므로, 기능이 많다는 이유만 보고 고르기보다 몇 달 동안 꾸준히 운영할 수 있는지를 먼저 봅니다. 여기서 호스팅은 블로그 파일이 머무는 서버 서비스, 스킨은 화면 모양, 플러그인은 기능을 추가하는 부품을 뜻합니다.

A

네이버 블로그

국내 이용자가 네이버 안에서 후기, 장소와 생활 정보를 찾는 주제에 잘 맞습니다. 별도 서버를 관리하지 않아도 되고 편집과 이웃 소통을 바로 시작할 수 있어 첫 블로그의 부담이 작습니다.

대신 사이트 구조와 외부 자동화의 자유도는 제한적입니다. 광고 수익을 고려한다면 네이버의 수익화 방식을 확인하고, 외부 광고를 자유롭게 배치하는 사이트와는 운영 목적이 다르다는 점을 이해해야 합니다.

  • 시작: 별도 호스팅 없이 비교적 간단
  • 독자: 국내 후기·장소·생활 정보
  • 수익: 네이버의 운영·수익화 안내 확인
  • 자동화: 외부 연결의 선택 폭은 제한적
  • 관리: 서버와 업데이트 부담이 적음
B

Tistory

검색을 통해 오래 읽히는 정보형 글과 Google AdSense를 함께 검토하는 사람에게 익숙한 선택지입니다. 스킨과 HTML을 일부 조정할 수 있어 네이버보다 화면과 광고 구성을 직접 다룰 여지가 있습니다.

그만큼 검색 노출, 스킨 오류와 서비스 정책 변화를 스스로 확인해야 합니다. 개발을 몰라도 시작할 수 있지만 변경한 설정과 원고 원본은 따로 보관하는 습관이 필요합니다.

  • 시작: 계정과 기본 스킨 설정 필요
  • 독자: 검색형 정보 글 중심
  • 수익: AdSense 같은 외부 광고를 검토 가능
  • 자동화: 네이버보다 외부 연결 선택 폭이 넓음
  • 관리: 검색 노출·스킨·운영 정책 확인 필요
C

WordPress

자신의 도메인과 사이트를 장기적인 콘텐츠 자산으로 운영하고 싶은 경우에 적합합니다. 디자인, 분석, 회원 기능과 자동화 연결의 선택 폭이 넓어 블로그를 넘어 전문 사이트로 확장하기 좋습니다.

WordPress.com 같은 관리형 서비스와 직접 호스팅하는 방식을 구분해야 합니다. 직접 운영하면 도메인과 호스팅 비용뿐 아니라 보안, 백업과 업데이트 책임도 생기므로 ‘자유도’와 ‘관리 부담’을 함께 선택하는 셈입니다.

  • 시작: WordPress.com과 직접 호스팅을 구분
  • 독자: 검색·브랜드·전문 콘텐츠에 폭넓게 대응
  • 수익: 요금제와 설치 방식에 따라 선택지가 다름
  • 자동화: 플러그인과 외부 연결의 확장성이 큼
  • 관리: 비용·업데이트·보안 책임이 늘 수 있음

블로그 자동화의 전체 흐름 이해하기

처음부터 모든 단계를 자동화하지 않고, 사람의 판단이 필요한 지점을 표시합니다.

  1. 주제와 독자 정하기
  2. 자료 조사와 목차
  3. 초안과 사실 검수
  4. 이미지와 임시 발행
  5. 성과 분석과 개선
4.1

가장 기본적인 작업 순서

주제와 독자를 정한 뒤 자료를 모으고, 목차와 초안을 만들고, 사실 검수와 이미지 준비를 거쳐 임시글로 옮깁니다. 사람이 최종 화면을 확인한 뒤 발행하고, 결과를 다음 글에 반영하면 한 바퀴가 완성됩니다.

각 단계에는 입력과 결과가 있어야 합니다. 예를 들어 자료 조사의 입력은 ‘확인할 질문’이고 결과는 ‘출처가 표시된 메모’입니다. 결과가 분명해야 다음 작업을 도구에 맡길 수 있습니다.

  1. 독자와 해결할 문제를 한 문장으로 적습니다.
  2. 확인할 질문을 만들고 공식 자료를 수집합니다.
  3. 질문 순서대로 목차를 정하고 소제목별 초안을 씁니다.
  4. 수치·정책·경험 표현을 검수하고 필요한 이미지를 준비합니다.
  5. 임시글에서 모바일 화면까지 확인한 후 발행합니다.
  6. URL과 발행일을 기록하고 2~4주 뒤 성과를 확인합니다.
완료 결과

한 글의 폴더 안에 자료.md, 초안.md, 이미지 폴더, 발행 URL이 남으면 다음 글에서도 같은 순서를 재사용할 수 있습니다.

4.2

처음 자동화하기 좋은 작업

제목 후보, 목차, 맞춤법 검사, 파일명과 발행 체크리스트처럼 오류를 쉽게 발견하고 되돌릴 수 있는 작업부터 시작합니다. 결과를 사람이 바로 읽어볼 수 있다는 공통점이 있습니다.

  • 제목·목차: 잘못돼도 발행 전에 바로 고칠 수 있습니다.
  • 문장 검사: 중복 표현과 긴 문장을 표시하게 하고 수정 여부는 사람이 결정합니다.
  • 파일 정리: 날짜와 글 번호를 이용한 이름 규칙을 반복 적용합니다.
  • 체크리스트: 링크, 출처, 이미지, 공개 설정 누락을 같은 순서로 확인합니다.

첫 자동화는 10분 이상 반복되는 일 하나만 고르세요. 여러 작업을 동시에 바꾸면 무엇이 시간을 줄였고 무엇이 오류를 만들었는지 알 수 없습니다.

4.3

나중에 자동화할 작업

외부 서비스에서 데이터를 가져오거나 예약 실행, 알림, 임시글 등록과 통계 보고서를 연결하는 일은 기본 흐름이 안정된 뒤 진행합니다. 계정과 공개 상태를 바꾸는 작업에는 항상 승인 단계를 둡니다.

‘매주 월요일 오전 9시에 승인된 원고의 링크를 검토자에게 보낸다’처럼 조건·입력·결과를 한 문장으로 설명할 수 있을 때 연결합니다. 실패하면 무엇을 되돌릴지도 먼저 정합니다.

중단 기준

중복 글이 생기거나, 승인 전 원고가 넘어가거나, 오류 알림이 없는 경우에는 자동화를 즉시 끄고 마지막 정상 기록부터 수동으로 확인합니다.

4장 정리

자동화할 단계마다 ‘무엇을 받아 무엇을 만들고, 누가 확인하는가’를 한 문장으로 적으세요. 설명할 수 없는 작업은 아직 연결하지 않습니다.

4장 따라 하기: 내 작업 흐름 한 장으로 그리기

최근 발행한 글 하나를 고르고 메모장에 단계별로 시작 시각, 사용한 도구, 만들어진 결과와 확인한 사람을 적습니다. 각 단계 옆에는 ‘반복’, ‘판단’, ‘위험’ 중 하나를 표시하세요.

  1. 반복이면서 위험이 낮은 작업은 첫 자동화 후보입니다.
  2. 판단이 필요한 작업은 Codex가 제안하고 사람이 선택하게 합니다.
  3. 계정·결제·공개 상태를 바꾸는 위험 작업에는 승인과 복구 절차를 둡니다.

완성된 흐름에서 입력이나 결과가 비어 있는 단계가 있다면 자동화보다 작업 정의를 먼저 보완합니다.

05

첫 번째 콘텐츠 제작 과정 따라 하기

독자 정의부터 목차, 초안과 다른 AI를 이용한 반대 검토까지 한 편을 완성합니다.

5.1

블로그의 독자와 주제 정하기

‘누가, 어떤 상황에서, 무엇을 해결하려고 읽는 글인가’를 한 문장으로 정합니다. 주제가 넓다면 한 사람이 한 번에 해결할 질문 하나로 줄입니다.

예를 들어 ‘블로그 자동화’보다 ‘개발을 모르는 Tistory 초보자가 첫 정보 글의 목차를 만드는 방법’이 좋습니다. 독자, 플랫폼, 경험 수준과 원하는 결과가 들어 있기 때문입니다.

  1. 예상 독자의 경험 수준을 적습니다.
  2. 검색 직전에 겪은 문제를 적습니다.
  3. 글을 읽고 할 수 있어야 하는 행동을 적습니다.
  4. 이번 글에서 다루지 않을 범위를 정합니다.
5.2

키워드와 글감 찾기

검색 자동완성, 연관 질문과 기존 글의 댓글에서 실제 표현을 모읍니다. 검색량보다 내가 직접 확인하고 구체적으로 설명할 수 있는지를 먼저 봅니다.

후보마다 ‘독자의 질문’, ‘내가 줄 수 있는 답’, ‘확인할 출처’를 한 줄씩 적으세요. 세 칸 중 하나가 비면 아직 글감이 아니라 추가 조사가 필요한 아이디어입니다.

글감 카드 예시

질문: n8n 없이도 시작할 수 있나? · 답: 주 1편이면 Codex와 Sheets만으로 충분함 · 출처/근거: 실제 작업 순서와 도구 공식 안내

5.3

AI로 목차 만들기

독자, 검색 목적, 반드시 포함할 내용과 제외할 표현을 알려주고 본문보다 목차를 먼저 요청합니다. 각 소제목이 서로 다른 질문에 답하는지 사람이 검토합니다.

좋은 목차는 배경 설명만 길게 이어지지 않습니다. 독자가 가장 먼저 원하는 짧은 답, 준비 사항, 단계별 방법, 실패 대응, 다음 행동 순서로 이어집니다.

  • 서로 겹치는 소제목은 합칩니다.
  • 제목만 다르고 답이 같은 절은 제거합니다.
  • 공식 확인이 필요한 절에는 출처 메모를 연결합니다.
  • 경험을 넣을 위치를 미리 표시합니다.
5.4

Codex를 중심으로 초안 만들기

소제목 하나씩 작성하고 확인이 필요한 문장에는 `[확인 필요]` 표시를 남깁니다. 원고 파일과 출처 메모를 함께 두면 수정 이유를 추적하기 쉽습니다.

  1. 폴더에 자료와 확정 목차를 저장합니다.
  2. 한 절의 목적과 사용할 자료를 지정해 초안을 요청합니다.
  3. 사실 주장 옆에 출처 또는 `[확인 필요]`를 남깁니다.
  4. 운영자의 사례와 판단을 직접 추가합니다.
  5. 전체 초안에서 반복과 연결 문장을 마지막에 다듬습니다.

Codex가 파일을 수정하기 전에는 원본을 보존하고, 변경 후에는 제목·링크·누락된 문단을 비교합니다.

5.5

다른 AI의 결과도 활용하는 방법

두 번째 AI에는 같은 글을 다시 쓰게 하기보다 반대 관점, 빠진 질문과 이해하기 어려운 표현을 찾게 합니다. 여러 초안을 무작정 합치지 말고 최종 문체는 운영자가 결정합니다.

검토 요청은 ‘틀린 부분을 찾아줘’보다 구체적이어야 합니다. 초보자가 실행할 수 없는 단계, 근거 없는 단정, 앞뒤 모순, 광고처럼 느껴지는 표현을 각각 표시하게 하세요. 지적을 반영할지는 원문과 운영 목적을 보고 결정합니다.

첫 글 완성 기준

독자의 질문에 답하고, 출처를 확인했으며, 운영자의 경험이나 판단이 들어갔다면 첫 원고로 충분합니다. 완벽한 자동화보다 완성된 한 편이 먼저입니다.

5장 따라 하기: 60분 초안 세션

  1. 10분: 독자와 질문을 한 문장으로 확정합니다.
  2. 15분: 공식 자료 두 개와 실제 사례 하나를 모읍니다.
  3. 10분: Codex로 목차를 만들고 중복 절을 지웁니다.
  4. 20분: 가장 중요한 두 절만 먼저 작성합니다.
  5. 5분: 확인할 사실에 표시하고 다음 작업을 기록합니다.

60분 안에 발행본을 만들 필요는 없습니다. 자료·목차·초안이 각각 파일로 남고 다음 시작점이 분명하면 정상 결과입니다.

첫 목차를 만들 때 사용할 질문

본문 전체보다 독자와 목차를 먼저 점검합니다.

[주제]에 대해 [예상 독자]가 읽을 블로그 글을 작성하려고 해.
독자가 검색한 이유는 [해결하려는 문제]라고 생각해.

본문을 바로 쓰지 말고 먼저
1. 독자가 가장 궁금해할 질문 5개
2. 질문에 답하는 순서로 구성한 목차
3. 빠지기 쉬운 내용과 확인이 필요한 사실
4. 직접 경험이나 의견을 추가하면 좋은 위치
를 제안해줘. 어려운 전문용어는 풀어서 설명해줘.

이후 “2번 소제목만 초안으로 작성해줘”처럼 나누어 요청하면 검수하기 쉽습니다.

06

AI가 쓴 글을 믿을 수 있는 글로 바꾸기

날짜, 가격, 수치, 정책과 실제 경험처럼 틀리면 신뢰를 잃는 내용을 우선 확인합니다.

검수

AI가 제시한 링크가 실제로 해당 주장을 뒷받침하는지 직접 확인합니다.

6.1

반드시 사람이 확인할 내용

날짜, 가격, 수치, 정책, 법률·세금·건강 정보와 제품 기능은 원문을 대조합니다. 직접 사용하지 않은 제품을 사용한 것처럼 쓰거나 방문하지 않은 장소의 경험을 만들어내지 않습니다.

검수할 때 문장을 ‘사실’, ‘해석’, ‘경험’으로 표시해보세요. 사실은 출처가 필요하고, 해석은 근거와 반대 가능성을 설명해야 하며, 경험은 실제로 겪은 사람만 쓸 수 있습니다.

  • 가격에는 확인 날짜와 통화·세금 포함 여부를 적습니다.
  • 통계에는 조사 기관, 대상, 기간을 함께 기록합니다.
  • 서비스 기능은 공식 도움말의 현재 화면과 비교합니다.
  • 건강·법률·금융 정보는 일반 안내와 개인 조언을 구분합니다.
6.2

믿을 수 있는 참고자료 찾기

정부·공공기관, 서비스 공식 도움말, 기업 공식 발표를 우선합니다. 작성일과 업데이트 날짜를 확인하고, AI 검색과 NotebookLM은 원문을 찾고 정리하는 보조 수단으로 사용합니다.

  1. 주장의 핵심 단어로 공식 출처를 찾습니다.
  2. 페이지 작성자와 최근 수정일을 확인합니다.
  3. 문장 앞뒤 문맥까지 읽어 실제 주장을 확인합니다.
  4. 출처 URL, 확인일과 사용할 내용을 자료 메모에 저장합니다.
  5. 서로 충돌하는 자료가 있으면 최신성·권한·조사 방법을 비교합니다.

블로그나 커뮤니티 글은 실제 문제 사례를 찾는 데 유용하지만, 정책과 수치의 최종 근거로 쓰지 않습니다.

6.3

AI 특유의 문장 줄이기

내용 없는 서론, 과장된 표현과 같은 결론의 반복을 지웁니다. 구체적인 상황, 실제 선택 이유와 실패 경험을 넣으면 글의 목소리가 생깁니다.

‘현대 사회에서 매우 중요합니다’처럼 어느 글에도 붙일 수 있는 문장은 삭제합니다. ‘n8n은 강력합니다’ 대신 어떤 조건에서 무엇이 편해지고 무엇을 관리해야 하는지 씁니다.

수정 예시

수정 전: “효율적인 블로그 운영을 위해 자동화는 필수입니다.”
수정 후: “주 1편을 혼자 발행한다면 연결 자동화 없이도 충분합니다. 같은 복사 작업이 매주 반복될 때 알림부터 연결하세요.”

6.4

발행 전 품질 체크리스트

  • 제목에서 약속한 답이 첫 화면과 본문에 있는가
  • 출처 없는 수치·정책·단정이 없는가
  • 인용과 이미지의 사용 권리를 확인했는가
  • 실제 경험처럼 꾸민 문장이 없는가
  • 링크가 열리고 최신 페이지를 가리키는가
  • 광고와 제휴 관계가 정보보다 앞서지 않는가
  • 모바일에서 표·코드·이미지가 잘리지 않는가

검수자는 가능하면 작성 직후가 아니라 시간을 두고 읽습니다. 같은 사람이 검수하더라도 소리 내어 읽고 링크를 실제로 눌러보면 누락을 더 잘 발견할 수 있습니다.

6장 따라 하기: 주장 검수표 만들기

초안에서 숫자, 날짜, ‘반드시’, ‘가장’, ‘가능하다’ 같은 단정 표현을 찾아 표로 옮깁니다.

문장유형출처확인일처리
월 1만 원이면 충분하다가격·판단공식 요금 안내2026-07-16적용 조건 추가

출처가 없는 문장은 삭제, 완화, 직접 검증 중 하나로 처리합니다. 출처가 있어도 문장이 원문의 범위를 과장하면 다시 씁니다.

07

검색 노출을 고려해 글 다듬기

제목에서 약속한 답을 앞부분에 주고, 각 소제목이 하나의 질문에 답하도록 정리합니다.

Intent

검색하는 사람의 목적 이해하기

정보를 알고 싶은지, 방법을 따라 하려는지, 상품을 비교하거나 구매 직전 확인하는지 구분합니다. 같은 키워드라도 목적에 따라 글의 순서와 필요한 근거가 달라집니다.

  • 정보형: 용어와 차이를 먼저 설명합니다.
  • 방법형: 준비물, 순서, 정상 결과와 오류 해결이 필요합니다.
  • 비교형: 동일한 기준과 추천 조건을 제시합니다.
  • 구매형: 비용, 제한, 대안과 광고 관계를 분명히 밝힙니다.

검색 결과 상위 글의 제목만 베끼지 말고, 독자가 댓글과 연관 질문에서 계속 묻는 내용을 확인하세요.

Read

읽기 쉬운 글의 기본 구조

제목에서 약속한 답을 앞부분에 제시하고 소제목마다 질문 하나에 답합니다. 짧은 문단, 목록과 표를 사용하되 장식 때문에 글이 끊기지 않게 합니다.

  1. 첫 두 문단에서 결론과 적용 대상을 밝힙니다.
  2. 배경 설명은 독자가 행동하는 데 필요한 만큼만 씁니다.
  3. 단계는 번호 목록, 항목 비교는 표를 사용합니다.
  4. 주의사항은 해당 단계 가까이에 둡니다.
  5. 마지막에는 독자가 지금 할 한 가지 행동을 제시합니다.
7.3

플랫폼별 글 구성 차이

네이버는 경험과 사진의 흐름, Tistory와 WordPress는 검색 질문과 해결 순서가 상대적으로 중요합니다. 같은 원고를 그대로 복사하지 말고 플랫폼 독자에 맞게 다시 편집합니다.

네이버 후기라면 방문 동선과 실제 사진이 핵심이고, 검색형 가이드라면 정의·준비·단계·문제 해결이 핵심입니다. 두 플랫폼에 모두 올릴 때는 제목, 도입, 이미지 순서와 내부 링크를 각각 다시 구성합니다.

7.4

검색 노출만 노린 글을 피해야 하는 이유

키워드 반복과 비슷한 글의 대량 생성은 독자의 문제를 해결하지 못합니다. 직접 확인한 사례, 비교 기준과 최신 출처처럼 다른 글에서 얻기 어려운 내용을 넣습니다.

발행 전 “이 글에서만 얻을 수 있는 것은 무엇인가?”를 물으세요. 실제 화면, 실패 사례, 운영 비용, 선택을 바꾼 이유처럼 고유한 답이 없다면 자료와 경험을 더 보강합니다.

7장 따라 하기: 검색 의도 점검

목표 검색어로 실제 검색한 뒤 상위 결과의 제목을 베끼지 말고 공통으로 답하는 질문을 기록합니다. 내 글의 각 소제목 옆에 ‘정보·방법·비교·구매’ 중 하나를 표시하세요.

서로 다른 의도가 한 글 안에서 경쟁하면 핵심 의도 하나를 앞에 두고 나머지는 관련 글로 분리합니다. 제목, 첫 문단과 첫 번째 소제목이 같은 약속을 하고 있으면 정상입니다.

9.1

가장 안전한 초보자 방식

Codex에서 원고와 이미지 목록을 준비하고 블로그에는 임시글로 저장합니다. 모바일과 데스크톱 미리보기에서 사람이 확인한 글만 공개합니다.

  1. 원고 상태를 ‘검토 완료’로 바꿉니다.
  2. 편집기에 붙여넣고 제목·소제목·목록을 확인합니다.
  3. 이미지와 링크를 실제로 열어봅니다.
  4. 카테고리, 태그, 공개 시간과 대표 이미지를 확인합니다.
  5. 모바일 미리보기 후 사람이 공개 버튼을 누릅니다.
  6. 발행 URL과 시간을 상태표에 기록합니다.

9.4

화면이 달라도 알아야 할 발행 개념

서비스마다 버튼 이름과 위치는 달라도 발행 과정은 소유권·권한·상태·주소·기록이라는 다섯 개념으로 이해할 수 있습니다. 이 다섯 가지를 먼저 구분하면 화면이 바뀌어도 무엇을 확인해야 하는지 찾을 수 있습니다.

  • 소유권과 계정: 누가 블로그를 소유하고 복구 수단을 관리하는지 뜻합니다. 개인 비밀번호를 자동화에 넘기지 않고, 가능하면 서비스가 제공하는 별도 연결 권한을 사용합니다.
  • 권한: 글 읽기, 임시 저장, 수정, 공개와 삭제 중 연결 도구가 할 수 있는 일을 뜻합니다. 처음에는 임시 저장에 필요한 최소 권한만 허용합니다.
  • 발행 상태: 아이디어, 작성 중, 검수 완료, 임시 저장, 예약, 공개, 비공개처럼 글의 현재 단계를 나타냅니다. 자동화의 입력 조건과 사람이 승인할 지점을 이 상태로 정합니다.
  • 주소와 분류: 글의 고유 URL, 카테고리, 태그와 공개 범위입니다. 제목을 바꿔도 관리 번호와 발행 URL을 함께 보관해야 같은 글을 다시 만들지 않습니다.
  • 실행 기록과 되돌리기: 누가 언제 무엇을 공개했는지, 실패하면 임시글이나 비공개로 어떻게 되돌릴지를 남기는 일입니다. 자동화에는 중단 방법과 마지막 정상 실행 기록이 있어야 합니다.
화면 없는 확인 시나리오

“검수 완료인 001번 원고를 임시글로 한 번 저장하고, 운영자가 미리보기에서 승인하면 공개한다. 실패하면 공개하지 않고 오류와 원고 위치를 기록한다.”라고 설명할 수 있으면 어떤 플랫폼에서도 필요한 설정을 역으로 찾을 수 있습니다.

9.6

네이버·Tistory 반자동 운영을 실제로 해보기

공개 API가 없어도 준비 과정을 체계화하면 발행 직전의 복사와 확인 시간을 크게 줄일 수 있습니다. 예를 들어 매주 수요일에 한 편을 발행한다면 월요일에는 자료와 초안을 준비하고, 화요일에는 검수와 이미지를 끝내며, 수요일에는 편집기에서 사람이 최종 등록합니다. 자동화의 목표를 ‘사람을 없애기’가 아니라 ‘사람이 확인할 상태까지 정확히 준비하기’로 바꾸는 것입니다.

공개 API가 없는 플랫폼의 안전한 흐름

  1. 준비Codex에서 발행 묶음 생성

    제목, 본문, 이미지 위치, 태그 후보와 출처 목록을 한 폴더에 저장합니다.

  2. 승인상태표를 검수 완료로 변경

    사실·링크·이미지·공개 범위를 사람이 확인하고 확인자와 날짜를 기록합니다.

  3. 발행공식 편집기에서 직접 등록

    미리보기 후 공개하고 실제 URL과 수정 예정일을 다시 상태표에 적습니다.

네이버 예시: 원고에 [사진 01: 주차장 입구]처럼 사진 위치를 표시하고, 사진 폴더의 번호와 맞춥니다. 편집기에 붙여넣은 뒤 장소명, 운영 시간과 지도 정보가 실제 방문·공식 정보와 일치하는지 확인합니다. 발행 URL을 기록하면 다음 주 관련 글에서 내부 연결 후보로 사용할 수 있습니다.

Tistory 예시: Markdown 원고를 복사할 때 기본 모드와 HTML 모드 중 최종 편집 방식을 먼저 정합니다. 표와 코드가 있다면 미리보기에서 모바일 폭을 확인하고, 공개 후 RSS에 새 글이 나타나는지 봅니다. RSS를 n8n·Make·Zapier에 연결하면 발행 이후의 알림과 기록은 자동화할 수 있습니다.

9.7

발행 과정에서 자주 발생하는 문제

중복 등록을 막기 위해 글마다 고유한 관리 번호와 발행 URL을 기록합니다. 이미지 누락, 깨진 링크, 표와 문단 오류가 있으면 공개 대신 임시 저장 상태로 되돌립니다.

  • 중복 발행: id와 기존 URL을 먼저 조회합니다.
  • 이미지 누락: 원고의 이미지 목록과 업로드 수를 비교합니다.
  • 문단 깨짐: 외부 서식을 제거하고 기본 스타일로 다시 붙입니다.
  • 잘못된 공개: 즉시 비공개 처리하고 자동화 실행 기록을 보존합니다.

10

n8n·Make·Zapier로 자동화 범위 넓히기

같은 작업을 반복하거나 여러 앱 사이에서 자료를 옮길 때 연결 자동화를 검토합니다.

10.1

연결 자동화 도구는 언제 필요한가

매주 같은 자료를 앱 사이에서 옮기거나, 정해진 시간에 실행하고, 승인과 알림을 반복할 때 필요합니다. 아직 글 형식이 자주 바뀌거나 한 앱 안에서 끝나는 작업이라면 수동 운영이 빠릅니다.

10.2

n8n, Make, Zapier의 차이 이해하기

n8n은 데이터·분기·코드를 세밀하게 다루기 좋고, Make는 시나리오의 흐름을 화면에서 이해하기 쉬우며, Zapier는 지원 앱을 빠르게 연결하기 편합니다. 연결할 앱, 실행 횟수, 오류 확인 방식과 관리 난이도를 비교합니다.

10.3

처음부터 끝까지 따라 하는 자동화 시나리오

이 장에서는 “승인된 원고가 생기면 담당자에게 최종 검토 메일을 보내는 흐름” 하나를 완성합니다. Google Sheets가 입력표, n8n이 조건 검사와 이동 경로, 이메일이 결과입니다. 블로그 공개 버튼은 건드리지 않습니다.

성공 기준은 단순합니다. ready + yes인 새 원고는 메일이 한 번 오고, 승인되지 않은 원고와 필수값이 빠진 원고는 보류되며, 같은 ID를 다시 실행해도 메일이 두 번 오지 않아야 합니다.

10.4

Codex와 연결 도구의 역할 나누기

Codex는 자료와 원고 파일을 만들고 검수 목록을 작성합니다. 연결 도구는 상태에 따라 데이터를 옮깁니다. 공개 여부는 사람이 결정하고 승인 결과만 자동화에 전달합니다.

10.5

연결 자동화 없이 시작해도 되는 경우

블로그가 하나이고 주 1~2편을 쓰거나 형식이 아직 바뀌는 중이라면 필요하지 않습니다. Codex와 상태표로 열 편을 운영한 뒤 같은 수작업이 세 번 이상 반복되는지 확인하세요.

10.6

API라는 말을 어렵게 생각하지 않기

API는 서비스끼리 정해진 형식으로 요청과 결과를 주고받는 창구입니다. 공식 API를 우선하고, API 키는 코드나 원고에 적지 말고 자동화 도구의 Credential 저장소에 보관합니다.

10.7

WordPress 임시글 자동 등록 구조 이해하기

WordPress에서는 REST API가 공식 글쓰기 창구 역할을 합니다. 자동화 도구는 사이트 주소와 별도 인증 정보를 이용해 제목, 본문과 발행 상태를 전송하고 WordPress는 생성된 글의 번호와 주소를 돌려줍니다. 처음에는 공개 상태인 publish가 아니라 임시글 상태인 draft만 사용합니다.

WordPress 연결에서 데이터가 이동하는 순서

  1. 입력검수 완료 원고 읽기

    관리 번호, 제목, HTML 본문, 이미지와 카테고리 값을 준비합니다.

  2. 요청REST API로 임시글 전송

    자동화 도구의 Credential에 저장한 인증 정보로 Posts 주소에 요청합니다.

  3. 결과글 번호와 미리보기 기록

    응답의 글 번호와 링크를 상태표에 적고 운영자가 화면을 검수합니다.

연결 전에 준비할 것

  1. 전용 사용자: 개인 관리자 계정을 그대로 연결하지 말고 필요한 권한만 가진 사용자를 별도로 만듭니다.
  2. 애플리케이션 비밀번호: WordPress 사용자 프로필에서 자동화 전용 비밀번호를 발급합니다. 화면에 한 번 표시된 값을 원고에 붙이지 말고 n8n·Make·Zapier의 Credential 저장소에 넣습니다.
  3. HTTPS 주소: 실제 사이트가 https://내주소/wp-json/wp/v2/posts로 접근 가능한지 확인합니다.
  4. 테스트 원고: 제목 앞에 [TEST]를 붙이고 상태를 draft로 고정한 짧은 원고를 사용합니다.

HTTP 요청에서 전달할 최소 값

POST https://example.com/wp-json/wp/v2/posts
인증: WordPress 전용 사용자 + 애플리케이션 비밀번호

{
  "title": "[TEST] 초보자 블로그 자동화 시작하기",
  "content": "<h2>먼저 알려줄 답</h2><p>본문...</p>",
  "status": "draft"
}

정상이라면 응답에 새 글의 id, status: draft와 링크 정보가 포함됩니다. 401 오류는 인증 정보, 403 오류는 사용자 권한, 404 오류는 사이트 주소와 고유주소 설정을 먼저 확인합니다. 테스트 글이 관리자 화면의 임시글 목록에 한 번만 생기고 공개 주소에서는 노출되지 않으면 첫 연결이 성공한 것입니다.

공개 전 승인선

자동 생성된 임시글에서 제목, 본문 구조, 이미지, 카테고리와 모바일 미리보기를 사람이 확인한 뒤에만 공개합니다. 자동화가 공개 상태를 직접 바꾸게 하는 것은 임시글 생성과 중복 방지가 충분히 안정된 이후에도 별도로 판단할 일입니다.

따라 하기: 승인 원고 알림 자동화

실습의 전체 지도

  → 값 검사·정리

approved=yes

이 흐름은 “자동 발행”이 아니라 “발행 직전까지 준비하는 자동화”입니다. 초보자가 결과를 눈으로 확인할 수 있고, 잘못된 글이 공개될 위험도 낮습니다.

1단계: Google Sheets에 예시 데이터 입력하기

새 스프레드시트의 시트 이름을 posts로 바꾸고 첫 행에 아래 영문 열 이름을 그대로 입력합니다. 대소문자와 밑줄도 코드가 읽는 이름의 일부이므로 한글 제목으로 바꾸지 않습니다. 화면에서 뜻을 알아보기 어려우면 셀 메모에 한글 설명을 덧붙이세요.

idtopicplatformstatusapproveddraft_path
001초보자 블로그 도구tistoryreadyyesdrafts/001.md
002네이버 글쓰기 순서naverreviewnodrafts/002.md
003비워 둠tistoryreadyyesdrafts/003.md
  • 001은 정상 통과를 확인하는 행입니다.
  • 002는 아직 검토 중이어서 보류되어야 합니다.
  • 003은 실제 셀의 topic 값을 비워 필수값 누락을 시험합니다. 표의 “비워 둠”이라는 글자를 입력하지 않습니다.

id는 글마다 바뀌지 않는 고유 번호입니다. status는 idea, drafting, review, ready, published 중 하나만 사용하고, approved는 사람이 원고를 확인한 뒤에만 yes로 바꿉니다.

2단계: n8n에서 노드 아홉 개 연결하기

  1. 새 Workflow를 만들고 이름을 블로그 원고 최종 검토 알림으로 지정합니다.
  2. Manual Trigger를 추가합니다. 시험이 끝나기 전에는 Schedule Trigger를 사용하지 않습니다.
  3. Google Sheets 노드를 연결하고 Credential에서 자신의 Google 계정을 연결합니다. Resource는 Sheet Within Document, Operation은 Get Row(s), 문서는 방금 만든 파일, 시트는 posts를 선택합니다.
  4. Code 노드를 연결하고 이름을 입력값 검사로 바꾼 뒤 다음 단계의 코드를 붙여넣습니다.
  5. If 노드를 연결합니다. Boolean 조건의 왼쪽 값은 {{ $json.readyToPublish }}, 비교는 is true로 설정합니다.
  6. true 출력에는 Remove Duplicates를 연결합니다. Operation은 Remove Items Processed in Previous Executions, Keep Items Where는 Value Is New, Value to Dedupe On은 id, Scope는 기본값 Node로 둡니다.
  7. true 쪽 Remove Duplicates 뒤에 Gmail·Send Email·Slack 중 자신이 실제로 받을 수 있는 알림 노드를 연결하고 제목을 최종 검토 요청으로 설정합니다.
  8. false 출력에도 Remove Duplicates를 연결해 같은 보류 알림의 반복을 막습니다. Operation과 Keep Items Where는 6번과 같고, Value to Dedupe On은 holdKey로 지정합니다. 그 뒤에 제목이 자동화 보류인 알림 노드를 연결합니다.

연결선은 Manual Trigger → Google Sheets → 입력값 검사 → If 순서입니다. If에서 두 갈래로 나뉘며, 각 갈래는 Remove Duplicates와 알림 노드로 이어집니다. 화면에는 트리거를 포함해 모두 아홉 개 노드가 보여야 합니다. 노드 하나를 설정할 때마다 Execute step으로 오른쪽 출력에 예상 필드가 보이는지 확인하세요.

3단계: Code 노드에 검사 코드 붙여넣기

Code 노드의 실행 모드는 기본값인 Run Once for All Items로 둡니다. 아래 코드는 각 행의 공백과 대소문자를 정리하고, 통과 여부와 보류 이유를 새 필드로 만듭니다. 누락된 행 때문에 전체 실행을 중단하지 않고 해당 행만 false 경로로 보내도록 작성했습니다.

return $input.all().map((item, index) => {
  const row = item.json;
  const required = ["id", "topic", "platform", "status", "approved", "draft_path"];
  const missing = required.filter(
    (key) => !String(row[key] ?? "").trim(),
  );

  const status = String(row.status ?? "").trim().toLowerCase();
  const approved = String(row.approved ?? "").trim().toLowerCase() === "yes";
  const holdReasons = [];

  if (missing.length > 0) holdReasons.push(`필수값 누락: ${missing.join(", ")}`);
  if (status !== "ready") holdReasons.push(`현재 상태: ${status || "없음"}`);
  if (!approved) holdReasons.push("사람의 승인 필요");

  const holdReason = holdReasons.join(" / ");

  const safeTopic = String(row.topic ?? "")
    .trim()
    .replace(/[^0-9A-Za-z가-힣]+/g, "-")
    .replace(/^-|-$/g, "")
    .toLowerCase();

  return {
    json: {
      ...row,
      status,
      approved,
      readyToPublish: holdReasons.length === 0,
      holdReason,
      holdKey: `${row.id || "no-id"}|${holdReason}`,
      folderName: `${row.id || "no-id"}-${safeTopic || "no-topic"}`,
      checkedAt: new Date().toISOString(),
    },
    pairedItem: { item: index },
  };
});

required는 비어 있으면 안 되는 열이며, 알림에서 원고를 열 수 있도록 draft_path도 포함합니다. holdReasons에는 통과하지 못한 이유가 쌓입니다. holdKey는 같은 글의 같은 보류 사유를 다시 알리지 않기 위한 비교값입니다. 이유가 하나도 없을 때만 readyToPublish가 Boolean 값 true가 됩니다. 마지막의 pairedItem은 출력 행이 어느 입력 행에서 왔는지 n8n이 계속 추적하게 합니다.

Code 노드에서 확인할 값

실행 결과의 001 행에는 readyToPublish: true와 빈 holdReason이 보여야 합니다. 002와 003에는 readyToPublish: false가 보이고, 각각 “현재 상태·사람의 승인 필요”와 “필수값 누락: topic”이 표시되어야 합니다.

현재 상태: {{ $json.status }}
승인 여부: {{ $json.approved }}

Sheets에서 값을 고친 뒤 수동으로 다시 실행하세요.

Gmail, Send Email과 Slack은 계정 연결 방식이 다르지만, 앞 노드에서 만든 $json 값을 제목과 본문에 넣는 원리는 같습니다. 비밀번호나 API 키를 본문에 쓰지 말고 n8n의 Credential에 저장합니다.

5단계: 세 번 실행해서 성공·보류·중복 확인하기

시험바꿀 값예상 결과통과 기준
정상·보류예시 3행 그대로001은 검토 요청, 002·003은 보류메일 3개의 ID와 사유가 표와 일치
중복아무것도 바꾸지 않고 다시 실행두 갈래 모두 Remove Duplicates에서 제거001·002·003 알림이 모두 다시 오지 않음
수정 후 재시도002를 ready/yes, ID는 그대로002가 처음으로 검토 요청 경로 통과002 검토 요청 메일이 한 번 도착

실행이 다르면 n8n의 Executions에서 마지막 실행을 열고, 데이터가 처음 예상과 달라진 노드부터 봅니다. Sheets 출력에 영문 키가 없으면 첫 행의 열 이름을, If가 모두 false라면 Code 출력의 holdReason을, 메일이 반복되면 각 갈래 Remove Duplicates의 Operation과 id 또는 holdKey 설정을 확인합니다.

6단계: 수동 실행을 예약 실행으로 바꾸기

세 시험을 모두 통과한 뒤에만 Manual Trigger를 Schedule Trigger로 교체합니다. 예를 들어 평일 오전 9시에 실행하되 Workflow Settings의 시간대를 Asia/Seoul로 먼저 확인합니다. Schedule Trigger는 워크플로를 저장하고 게시해야 정해진 시간에 동작합니다.

  1. Manual Trigger는 삭제하지 말고 먼저 복사본 워크플로를 만들어 보관합니다.
  2. Schedule Trigger의 간격을 Days, 시각을 오전 9시로 지정합니다.
  3. Workflow Settings에서 Timezone을 Asia/Seoul로 지정합니다.
  4. Publish 전에 001 같은 기존 통과 행이 중복 제거되는지 다시 확인합니다.
  5. Executions에서 실패 기록을 확인할 주기와 자동화를 끌 사람을 정합니다.

Webhook은 이 Sheets 시나리오에 동시에 추가하지 않습니다. 외부 서비스가 즉시 원고 데이터를 보낼 때 Google Sheets와 Manual Trigger를 Webhook 하나로 교체하는 별도 방식입니다. n8n은 테스트 URL과 운영 URL을 따로 제공하므로, Webhook을 배울 때도 새 복사본에서 시험하세요.

11

AdSense를 고려한 블로그 운영

네이버 애드포스트와 Google AdSense의 차이를 이해하고, 광고보다 유용한 콘텐츠와 기본 페이지를 먼저 준비합니다.

11.1

네이버 애드포스트와 Google AdSense의 차이

사용할 수 있는 광고 서비스는 플랫폼에 따라 다릅니다. 네이버 블로그는 네이버 안의 수익화 방식을, Tistory와 WordPress는 AdSense를 포함한 외부 광고 가능 여부를 검토합니다. 광고 외에도 제휴 링크, 협찬과 자체 상품이 있지만 표시 의무와 플랫폼 정책을 확인해야 합니다.

광고 서비스는 단순히 코드를 붙이는 도구가 아닙니다. 신청 주체, 지급 정보, 사이트 소유 확인, 콘텐츠 및 트래픽 정책을 충족해야 하고 이후에도 위반 여부를 관리해야 합니다. 승인 조건과 지원 플랫폼은 바뀔 수 있으므로 신청 직전에 공식 안내를 다시 확인합니다.

  • 플랫폼이 해당 광고 방식을 허용하는지 확인합니다.
  • 본인 확인과 지급 정보를 정확히 준비합니다.
  • 광고·제휴 관계를 독자가 알아볼 수 있게 표시합니다.
  • 자신의 광고를 클릭하거나 클릭을 유도하지 않습니다.
11.2

수익화 신청에서 운영까지의 개념

광고 서비스의 화면이 달라져도 과정은 대체로 자격 확인 → 사이트 연결 → 검토 → 광고 노출 → 정산과 사후 관리로 이해할 수 있습니다. 승인은 마지막이 아니라 운영 책임이 시작되는 지점입니다.

  1. 자격 확인: 내 플랫폼이 광고 서비스를 지원하는지, 신청자와 지급 정보를 준비할 수 있는지 확인합니다.
  2. 사이트 연결: 광고 서비스가 내가 관리하는 사이트인지 확인할 수 있도록 코드, 메타 태그 또는 플랫폼 연결 방식으로 소유권을 증명합니다. 인증용 값과 계정 비밀번호는 원고나 공개 코드에 적지 않습니다.
  3. 검토: 콘텐츠의 독창성만이 아니라 메뉴, 소개, 문의, 개인정보 안내, 깨진 링크와 모바일 사용성을 함께 확인합니다. 검토는 보통 며칠이지만 더 오래 걸릴 수 있으며 승인 결과는 자동화로 만들 수 없습니다.
  4. 광고 노출: 사이트가 Ready 상태가 된 뒤 자동 광고 또는 직접 정한 위치가 본문을 가리거나 실수 클릭을 유도하지 않는지 실제 글에서 봅니다.
  5. 정산과 사후 관리: 본인 확인, 지급 기준, 세금 관련 정보와 정책 알림을 관리하고 이상 트래픽이나 경고를 정기적으로 확인합니다.

AdSense Management API는 보고서와 일부 광고 자원을 다룰 수 있지만 사이트 추가, 심사 요청과 승인을 대신하지는 않습니다. WordPress의 Site Kit이나 Tistory의 수익 메뉴도 연결을 돕는 기능이지 승인을 보장하는 기능이 아닙니다.

공식 근거: Google의 새 사이트 추가·검토 안내와 AdSense Management API 목록을 확인합니다.

화면보다 중요한 질문

지금 하는 작업이 사이트 소유 확인인지, 심사 준비인지, 광고 위치 설정인지, 지급 준비인지 먼저 구분하세요. 목적을 알면 메뉴 이름이 바뀌어도 공식 도움말에서 현재 절차를 찾을 수 있습니다.

11.3

AdSense보다 먼저 준비할 것

주제가 분명한 원본 콘텐츠, 독자가 이동할 메뉴와 카테고리, 운영자 소개, 문의와 개인정보 관련 페이지를 준비합니다. 승인만을 목표로 얕은 글을 채우기보다 사람이 직접 검수하고 관리하는 사이트라는 신뢰를 보여주는 것이 먼저입니다.

  1. 빈 카테고리와 준비 중 페이지를 정리합니다.
  2. 소개, 연락 방법, 개인정보 처리 관련 안내를 찾기 쉽게 둡니다.
  3. 모바일에서 메뉴·본문·광고 자리가 겹치지 않는지 봅니다.
  4. 출처 없는 재가공 글과 중복 글을 수정하거나 제거합니다.
  5. 사이트가 HTTPS로 정상 열리고 깨진 링크가 없는지 확인합니다.

‘몇 편이면 승인된다’는 고정 숫자보다 각 글이 독립적인 질문에 충분히 답하는지가 중요합니다.

11.4

수익화를 위한 콘텐츠 구성

검색 유입을 만드는 정보형 글, 선택을 돕는 비교형 글, 실제 경험을 담은 신뢰형 글과 여러 글을 연결하는 핵심 가이드를 함께 운영합니다. 광고 수가 아니라 독자가 다음 글까지 읽을 이유를 만듭니다.

콘텐츠 묶음 예시

핵심 가이드 ‘초보자 블로그 자동화 시작하기’를 중심으로 도구 선택, 첫 글 작성, 검수, 이미지, 발행과 분석 글을 연결합니다. 개별 글은 한 질문에 답하고 핵심 가이드는 전체 순서를 안내합니다.

비교 글에는 추천 대상뿐 아니라 추천하지 않는 대상, 비용 외의 관리 부담과 선택을 바꿀 조건을 함께 씁니다.

11.5

광고만을 목적으로 자동화하면 안 되는 이유

비슷하고 얕은 글을 대량으로 만들면 검색과 광고 정책 위험뿐 아니라 독자 신뢰를 잃습니다. 자동화는 검수 시간을 확보하는 데 쓰고, 수익 조건과 정책은 신청 시점의 공식 문서에서 확인합니다.

발행량이 늘었는데 재방문, 관련 글 이동과 문의가 줄었다면 자동화를 멈추고 품질을 점검합니다. 광고 수익만 보고 독자를 오도하는 제목이나 불필요하게 긴 글을 만들지 않습니다.

11장 따라 하기: 수익화 준비 점검표

  • 사이트 주제와 주요 독자를 한 문장으로 설명할 수 있습니다.
  • 소개·문의·개인정보 안내와 주요 카테고리로 이동할 수 있습니다.
  • 각 글에는 운영자가 확인한 정보와 고유한 설명이 있습니다.
  • 광고가 없어도 글이 독자의 문제를 해결합니다.
  • 신청일에 공식 정책과 플랫폼 지원 여부를 다시 확인했습니다.

하나라도 준비되지 않았다면 광고 코드를 붙이는 대신 콘텐츠와 사이트 구조를 먼저 보완합니다.

4단계: 필요한 부분만 자동화 도구로 연결하기

  3. 테스트 데이터로 수동 실행합니다.
  4. 거부·누락·중복 사례를 시험합니다.
  5. 실행 기록과 중단 방법을 확인한 뒤 예약합니다.

한 달마다 ‘절약한 시간’, ‘사람이 수정한 오류’, ‘실패한 실행’, ‘발행 후 수정 건수’를 기록합니다. 자동화가 글 수만 늘리고 수정과 관리 시간을 늘렸다면 범위를 줄입니다.

13개 장을 세 구간으로 묶어 보기

  1. 1–4장기준과 작업 흐름 만들기

    독자, 플랫폼, 도구와 상태를 정해 무엇을 자동화할지 결정합니다.

  2. 5–8장검수 가능한 원고 완성하기

    자료, 목차, 초안, 사실 확인, 검색 구조와 이미지를 한 폴더에 모읍니다.

  3. 9–13장발행하고 다음 글에 반영하기

    임시글, 승인, 알림, 수익화 준비와 성과 기록을 운영 루프로 연결합니다.

  1. 1장 · 자동화 범위를 정합니다

    ‘AI가 초안을 돕고 사람이 사실과 공개 여부를 승인한다’는 원칙을 운영원칙.md에 적습니다. 자동 공개는 범위에서 제외합니다.

  2. 2장 · 도구의 역할을 배정합니다

    Codex는 파일과 초안, 웹 검색은 공식 자료 확인, Google Sheets는 상태표, WordPress는 임시글과 발행본을 담당합니다.

  3. 3장 · 플랫폼 선택 이유를 기록합니다

    검색형 가이드를 장기간 소유하고 REST API로 임시글을 만들기 위해 WordPress를 선택합니다. 서버 관리 부담도 함께 기록합니다.

  4. 4장 · 입력과 결과를 정의합니다

    입력은 주제 카드와 공식 출처, 결과는 검수 완료 원고·이미지 명세·발행 URL입니다. 상태는 아이디어 → 작성 중 → 검수 대기 → 검수 완료 → 임시글 → 발행 완료로 고정합니다.

  5. 5장 · 자료와 초안을 파일로 만듭니다

    자료.md에는 공식 출처와 확인일, 원고.md에는 독자 질문 순서의 목차와 초안을 저장합니다. 경험하지 않은 사례에는 ‘예시’라고 표시합니다.

  6. 6장 · 주장을 검수합니다

    가격·정책·제품 기능 문장을 표로 옮겨 출처와 대조합니다. 확인하지 못한 문장은 삭제하거나 조건부 표현으로 고칩니다.

  7. 7장 · 검색 의도와 구조를 맞춥니다

    핵심 의도를 ‘방법형’으로 정하고 첫 문단에 최소 도구와 전체 순서를 먼저 제시합니다. 각 소제목은 준비·단계·오류·다음 행동 중 하나에 답하게 합니다.

  8. 8장 · 이미지 명세를 완성합니다

    대표 이미지 1장과 실제 WordPress 임시글 화면 1장을 준비합니다. 파일명, 출처·권리, 대체 텍스트와 원고 속 삽입 위치를 이미지.md에 적습니다.

  9. 9장 · 임시글과 승인 기록을 남깁니다

    WordPress에는 draft로 한 번만 등록하고 데스크톱·모바일 미리보기를 확인합니다. 네이버·Tistory라면 같은 발행 묶음을 공식 편집기에 사람이 옮깁니다.

  10. 10장 · 반복 알림만 연결합니다

    상태표가 ‘검수 완료’가 되면 최종 확인 메일을 한 번 보내도록 연결합니다. 중복 ID, 누락된 URL과 승인되지 않은 원고가 통과하지 않는지 시험합니다.

  11. 11장 · 수익화보다 사이트 기본을 확인합니다

    소개·문의·개인정보 안내, 메뉴와 모바일 사용성을 점검합니다. 이 글 하나를 이유로 AdSense 승인을 기대하지 않고 유용한 콘텐츠 묶음을 계속 만듭니다.

  12. 12장 · 28일 뒤 수정 가설을 만듭니다

    노출·클릭·검색어·첫 화면 이탈을 같은 기간으로 기록합니다. 제목 또는 첫 문단 중 하나만 바꾸고 변경 전 수치를 남깁니다.

  13. 13장 · 다음 원고에서 재사용합니다

    POST-001에서 만든 폴더 구조, 상태값, 검수표와 실패 기록을 POST-002에 복사합니다. 필요 없었던 항목은 줄이고 누락된 항목은 양식에 추가합니다.

실전 예제가 끝났을 때 남아야 할 폴더

blog-project/
├─ 운영원칙.md
├─ 상태표.csv
├─ templates/
│  ├─ 원고양식.md
│  └─ 발행체크.md
└─ posts/POST-001/
   ├─ 자료.md
   ├─ 원고.md
   ├─ 이미지.md
   ├─ 발행체크.md
   └─ 성과-28일.csv

완료 기준은 파일 개수가 아닙니다. 상태표에서 POST-001의 현재 상태, 원고 위치, 확인한 출처, 임시글 또는 발행 URL, 다음 점검일을 한 줄로 찾을 수 있어야 합니다. 이 구조를 복사해 POST-002를 시작할 수 있으면 한 번의 작업이 반복 가능한 운영 방식으로 바뀐 것입니다.

처음 실행할 때

13단계를 하루에 끝내지 않아도 됩니다. 첫 주에는 1–8단계로 검수 완료 원고를 만들고, 둘째 주에는 9–11단계로 발행과 사이트 점검을 끝낸 뒤, 발행 28일 후 12–13단계로 개선 기록을 남기면 됩니다.

14

블로그 자동화를 시작하기 전 꼭 기억할 것

AI가 쓴 글의 책임은 운영자에게 있으며, 독자에게 실제로 도움이 되는지가 최종 기준입니다.

마지막 원칙

Codex로 기본 과정을 만든 뒤 검색, 디자인, 자동화와 분석 도구를 필요한 순간에 하나씩 추가합니다.

14.1

자동화는 발행량보다 품질을 위한 도구다

절약한 시간을 자료 확인, 경험 추가와 오래된 글 수정에 사용합니다. 글 수가 늘어도 독자가 얻는 답이 얕아지면 좋은 자동화가 아닙니다.

운영 목표를 ‘주 7편’이 아니라 ‘독자의 질문에 답하는 검수된 글을 정해진 주기로 발행’처럼 정의하세요. 속도는 품질 기준을 지킨 뒤에만 의미가 있습니다.

14.2

AI가 쓴 글의 책임은 운영자에게 있다

AI가 만든 오류, 저작권 침해와 오해를 부르는 표현도 발행한 운영자의 책임입니다. 출처와 공개 여부의 최종 판단을 도구에 넘기지 않습니다.

  • 개인정보와 비밀 자료를 입력하지 않습니다.
  • 타인의 글과 이미지를 허가 없이 재사용하지 않습니다.
  • 협찬·제휴·생성 이미지 관계를 필요한 경우 밝힙니다.
  • 오류를 발견하면 수정일과 변경 내용을 남깁니다.
14.3

처음부터 모든 과정을 연결할 필요는 없다

한 편을 직접 완성하고 반복 기준이 생긴 뒤 작은 작업부터 연결합니다. 단순한 흐름은 오류를 찾고 되돌리기 쉽습니다.

도구가 하나 늘 때마다 계정, 비용, 권한, 오류와 정책 변경을 관리해야 합니다. 새 도구는 해결할 문제가 분명하고 기존 도구로 해결할 비용보다 작을 때만 추가합니다.

14.4

Codex로 기본 과정을 만든 뒤 도구를 추가한다

Codex에서 원고와 파일 규칙을 안정시킨 뒤 검색, 디자인, 연결 자동화와 분석 도구를 실제 필요에 따라 하나씩 추가합니다.

프로젝트 폴더에는 글쓰기 기준, 출처 기록법, 상태 정의와 발행 체크리스트를 남깁니다. 도구를 바꾸더라도 이 기준 문서와 원고 원본이 있으면 운영 방식을 이어갈 수 있습니다.

14.5

독자에게 실제로 도움이 되는지가 최종 기준이다

도구 수, 자동화 단계와 발행 횟수보다 독자가 문제를 해결하고 다시 찾을 이유가 있는지를 봅니다. 이 질문에 답할 수 없다면 발행을 멈추고 글을 고칩니다.

발행 전 마지막 질문

이 글을 읽은 초보자가 무엇을 이해하고, 무엇을 직접 해볼 수 있으며, 실패했을 때 어디를 확인해야 하는가? 세 가지가 본문에 보이면 발행하고, 하나라도 없으면 그 부분을 보강합니다.

마지막 운영 원칙 카드

  1. 원본 자료와 원고를 내가 통제할 수 있는 곳에 보관합니다.
  2. AI의 결과는 제안으로 받고 사실과 공개 여부는 직접 판단합니다.
  3. 자동화에는 승인, 실행 기록, 오류 알림과 중단 방법을 둡니다.
  4. 정책과 가격처럼 변하는 정보에는 확인일을 남깁니다.
  5. 독자에게 도움이 되지 않는 자동화는 제거합니다.

이 다섯 가지를 지킬 수 있는 가장 단순한 구성부터 시작하는 것이 이 참고서의 결론입니다.

부록 A. WordPress 자동화를 위한 설치와 준비

WordPress를 아직 설치하지 않은 초보자가 관리형 호스팅을 선택하고, 10장의 임시글 자동화 실습을 시작할 수 있는 상태까지 준비합니다.

부록의 범위

관리형 WordPress 호스팅을 기본 경로로 설명합니다. VPS 직접 설치는 전체 구조와 책임 범위만 다루며, 운영체제·웹 서버·데이터베이스 명령어는 별도의 서버 구축 주제로 남깁니다.

A.1

어떤 WordPress 방식을 선택해야 하는가

WordPress라는 이름은 같아도 서비스가 관리해 주는 범위가 다릅니다. 초보자가 블로그 자동화를 먼저 경험하려면 서버 업데이트와 장애 대응을 직접 맡지 않아도 되는 관리형 호스팅이 가장 단순합니다. 직접 설치형은 플러그인과 서버 구성을 넓게 통제할 수 있지만 보안, 백업과 복구 책임도 함께 생깁니다.

방식내가 관리할 것자동화 관점추천 대상
WordPress.com글, 디자인과 서비스 설정요금제와 현재 기능에서 REST API·플러그인 사용 범위를 확인서버 관리를 피하고 서비스 안에서 운영할 사람
관리형 WordPress 호스팅콘텐츠, 사용자, 플러그인과 백업 확인독립 사이트와 REST API를 쓰면서 서버 관리 부담을 줄이기 쉬움이 부록을 따라 하는 대부분의 초보자
일반 웹호스팅설치, 업데이트와 일부 장애 대응비용은 낮을 수 있으나 지원 범위와 제한을 직접 확인호스팅 관리 화면에 익숙한 사람
VPS 직접 설치운영체제부터 보안·백업·복구까지 전체제어 범위는 크지만 콘텐츠 자동화보다 서버 운영이 먼저 필요장애와 보안을 직접 책임질 수 있는 사람

상품 이름만 보고 결정하지 말고 REST API, HTTPS, 백업·복원, 플러그인과 사용자 권한을 현재 상품에서 지원하는지 가입 직전에 확인합니다. WordPress.com과 각 호스팅의 기능·가격은 바뀔 수 있습니다.

A.2

호스팅 상품을 고를 때 확인할 기준

특정 업체의 광고 문구보다 설치 후 내가 해야 할 일을 기준으로 비교합니다. 아래 항목 중 모르는 것이 있다면 가입 전에 고객 지원에 질문하고 답변을 보관하세요.

  • 자동 설치: WordPress와 데이터베이스를 관리 화면에서 함께 만들 수 있는가
  • HTTPS: 인증서를 무료로 발급하고 자동 갱신하는가
  • 백업과 복원: 파일과 데이터베이스를 함께 백업하며 내가 복원 버튼을 실행할 수 있는가
  • 지원 범위: WordPress 로그인, PHP 오류와 데이터베이스 문제 중 어디까지 도와주는가
  • REST API: /wp-json/ 접근이나 외부 자동화 연결을 차단하지 않는가
  • 사용자·플러그인: 별도 사용자와 필요한 플러그인을 만들고 설치할 수 있는가
  • 비용: 첫 결제뿐 아니라 갱신 가격, 백업·복원과 트래픽 추가 비용은 얼마인가
가입 전 문의 예시

“WordPress REST API와 애플리케이션 비밀번호를 이용해 외부 자동화 도구에서 임시글을 만들 예정입니다. HTTPS, 자동 백업과 직접 복원, 별도 사용자 생성이 현재 상품에서 가능한가요?”라고 묻습니다.

A.3

설치 화면에서 만나는 용어 이해하기

호스팅 화면의 이름은 달라도 아래 다섯 개념의 관계는 같습니다. 도메인은 주소이고 호스팅은 사이트가 놓이는 공간이며, DNS가 둘을 연결합니다. WordPress는 호스팅 안에서 실행되고 글과 설정은 데이터베이스에도 저장됩니다.

A.4

관리형 호스팅에서 설치하기

업체마다 버튼 이름은 다르지만 입력과 정상 결과는 같습니다. 자동 설치 과정에서 사이트 이름, 관리자 이메일과 비밀번호를 입력하고 설치 위치는 특별한 이유가 없다면 기본 도메인의 루트로 둡니다.

  1. 호스팅을 준비합니다. WordPress 자동 설치와 HTTPS가 포함된 상품을 선택합니다.
  2. 도메인을 연결합니다. 호스팅이 안내한 DNS 값을 도메인 관리 화면에 입력합니다. 반영 중에는 주소가 열렸다 닫힐 수 있으므로 완료 안내를 기다립니다.
  3. WordPress를 자동 설치합니다. 관리자 이름을 admin처럼 단순하게 정하지 않고, 다른 사이트에서 쓰지 않은 긴 비밀번호를 사용합니다.
  4. HTTPS를 켭니다. 브라우저 주소가 https://로 열리고 관리자 화면에서도 보안 경고가 없는지 확인합니다.
  5. 관리자 화면에 로그인합니다. https://내주소/wp-admin/에서 대시보드가 열리는지 봅니다.
  6. 테스트 임시글을 저장합니다. 제목이 [TEST] 설치 확인인 글을 공개하지 않고 임시글로 저장합니다.
이 단계의 정상 결과

시크릿 창과 휴대전화에서 HTTPS 사이트가 열리고, 관리자 화면에서 테스트 글이 ‘임시글’로 보이며, 공개 방문자는 그 글을 볼 수 없어야 합니다.

공식 참고: WordPress의 시작 안내와 HTTPS 설명에서 현재 절차와 의미를 확인합니다.

A.5

설치 후 기본 설정하기

자동화를 연결하기 전에 사람이 직접 글을 만들고 수정할 수 있는 기본 사이트부터 완성합니다. 설정 이름은 버전에 따라 바뀔 수 있으므로 아래의 목적과 정상 결과를 기준으로 찾습니다.

설정권장 작업정상 결과
일반사이트 제목, 관리자 이메일, 한국어와 서울 시간대 확인예약 시간과 화면 날짜가 한국 시간으로 표시됨
고유주소글 주소가 제목 또는 정한 규칙으로 생성되는 방식 선택새 글 주소가 열리고 REST API 주소도 정상 응답
읽기·공개검색엔진 차단 여부와 홈페이지 표시 방식 확인준비 중에는 의도한 공개 범위, 운영 시작 후에는 정상 색인 가능 상태
토론·회원댓글 승인과 회원가입 필요 여부 결정스팸과 불필요한 계정 생성을 관리할 기준이 있음
기본 콘텐츠예제 글·페이지와 사용하지 않는 플러그인·테마 정리관리 목록에 실제 운영 항목만 남음

소개, 문의와 개인정보 안내 페이지를 만들고 상단이나 하단 메뉴에서 이동할 수 있게 합니다. 테마를 바꾸기 전에는 모바일 메뉴, 본문 폭, 표와 코드 블록이 현재 글에서 정상인지 확인합니다.

A.6

보안·업데이트·백업 기준 만들기

사이트가 열리는 것과 운영 준비가 끝난 것은 다릅니다. 자동화는 계정과 콘텐츠를 더 자주 변경하므로 누가 어떤 권한으로 연결되는지, 실패하면 어디서 복원할지 먼저 정합니다.

  • 계정 분리: 사이트 관리용 관리자와 글 작성 자동화용 사용자를 분리합니다.
  • 로그인 보호: 긴 고유 비밀번호와 가능한 경우 2단계 인증을 사용합니다.
  • 최소 설치: 사용하지 않는 테마와 플러그인은 비활성화만 하지 말고 필요 여부를 확인해 제거합니다.
  • 업데이트 전 백업: WordPress, 테마와 플러그인을 업데이트하기 전에 파일과 데이터베이스를 함께 백업합니다.
  • 복원 시험: ‘백업 성공’ 표시만 믿지 말고 테스트 환경이나 호스팅 복원 기능으로 되돌리는 절차를 확인합니다.
  • 변경 기록: 업데이트 날짜, 바꾼 항목과 문제가 생겼을 때 되돌릴 지점을 기록합니다.

백업 주기는 글과 주문 등 변경 빈도에 맞춥니다. 새 글을 자주 올리는데 한 달 전 백업만 있다면 복원해도 최근 콘텐츠를 잃습니다.

공식 참고: WordPress는 업데이트 전에 백업할 것을 안내하며, 유지관리 문서에서도 사이트와 데이터베이스 백업을 정기 작업으로 다룹니다. 업데이트 안내와 사이트 유지관리 안내를 확인합니다.

A.7

자동화를 위한 전용 계정과 REST API 준비

개인 관리자 비밀번호를 n8n·Make·Zapier에 저장하지 않습니다. 자동화에 필요한 글 작성 권한만 가진 별도 사용자를 만들고, 외부 애플리케이션 인증용 비밀번호를 따로 발급합니다. 애플리케이션 비밀번호는 관리자 화면 로그인용이 아니며 개별적으로 폐기할 수 있습니다.

  1. 자동화 전용 사용자를 만들고 글 작성·수정에 필요한 최소 역할을 부여합니다.
  2. 해당 사용자 프로필에서 연결 이름을 blog-draft-automation처럼 알아볼 수 있게 적습니다.
  3. 애플리케이션 비밀번호를 발급하고 표시된 값을 즉시 Credential 저장소로 옮깁니다.
  4. 원고, 스프레드시트, 채팅과 화면 캡처에는 비밀번호를 남기지 않습니다.
  5. https://내주소/wp-json/와 Posts API 주소가 열리는지 확인합니다.
  6. 10.7의 요청을 status: draft로 실행하고 응답의 글 번호를 기록합니다.
  7. 연결을 중단할 때는 자동화 도구의 Credential과 WordPress의 애플리케이션 비밀번호를 모두 폐기합니다.
인증 실패를 구분하는 순서

401이면 사용자 이름과 애플리케이션 비밀번호, 403이면 사용자 역할과 글 작성 권한, 404이면 사이트·API 주소와 서버 라우팅을 확인합니다. 오류를 해결하려고 개인 관리자 비밀번호를 대신 넣지 않습니다.

공식 참고: WordPress REST API의 인증 안내와 애플리케이션 비밀번호 안내를 기준으로 합니다.

A.8

설치 완료와 10장 연결 준비 점검표

아래 항목을 모두 확인하면 WordPress 준비가 끝난 것입니다. 하나라도 실패하면 자동화 노드를 더 만들기보다 해당 설치·권한·복원 문제부터 해결합니다.

  • HTTPS 주소로 사이트와 관리자 화면이 열린다.
  • 시크릿 창과 휴대전화에서도 보안 경고 없이 열린다.
  • 사이트 언어, 서울 시간대와 관리자 이메일이 정확하다.
  • 글을 직접 임시 저장하고 미리볼 수 있다.
  • 소개·문의·개인정보 안내와 기본 메뉴가 있다.
  • 파일과 데이터베이스가 함께 백업된다.
  • 백업을 실제로 복원하는 위치와 절차를 알고 있다.
  • 관리자와 분리된 자동화 전용 사용자가 있다.
  • 애플리케이션 비밀번호가 Credential 저장소에만 있다.
  • /wp-json/wp/v2/posts 주소가 응답한다.
  • API 테스트 글이 draft로 한 번만 생성된다.
  • 테스트 후 글 번호, 미리보기 주소와 실행 시각이 상태표에 기록된다.

부록에서 10장으로 이어지는 마지막 확인

  1. 설치 완료사람이 직접 임시글 작성

    사이트, HTTPS, 기본 설정과 모바일 화면을 확인합니다.

  2. 연결 완료API로 테스트 임시글 작성

    전용 계정과 애플리케이션 비밀번호로 한 번만 생성합니다.

  3. 10장 이동승인·중복 방지 자동화 연결

    검수 완료 상태와 글 번호를 이용해 안전장치를 추가합니다.

부록의 최종 결과

내 WordPress 주소, 관리자 복구 방법, 백업·복원 위치, 자동화 사용자 이름, Credential 이름, 테스트 글 번호와 폐기 방법을 한 장의 비공개 운영 메모에서 찾을 수 있으면 준비가 끝났습니다. 실제 비밀번호 값은 메모에 적지 않습니다.

VPS 직접 설치 경로는 왜 별도 주제인가

VPS에서는 서버 준비 → 방화벽 → 웹 서버 → PHP → 데이터베이스 → WordPress → HTTPS → 백업·모니터링 순서를 모두 직접 관리합니다. 운영체제와 웹 서버 조합에 따라 명령과 보안 설정이 달라지며, 잘못된 설정은 사이트뿐 아니라 계정과 데이터에도 영향을 줍니다.

이 경로를 선택하려면 업데이트 실패와 서버 중단을 직접 복구할 수 있어야 합니다. 처음 콘텐츠 자동화를 구축하는 독자는 관리형 호스팅으로 10장의 draft 흐름을 먼저 완성하고, 서버 운영이 별도의 목표가 되었을 때 VPS 구축 가이드를 분리해 학습하는 편이 안전합니다.
