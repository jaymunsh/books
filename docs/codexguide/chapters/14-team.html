팀 profile 정책 — 합의 5단계

  1. 기본 profile 정하기 — 보통 dev (workspace-write + on-request + network false)
  2. review · ci · danger profile 정의
  3. 위험 명령은 deny 또는 ask
  4. 편차 큰 명령은 ask로 의도 확인
  5. 3개월에 1회 회고

실전 예시: 11장 스토리 B의 config.toml 참조.

AGENTS.md 충돌 관리

  • 오너 1명 정함 (테크리드)
  • PR 리뷰 의무 (CLAUDE.md 변경은 오너 승인)
  • 분기 다이어트 (200줄 이하)
  • 경로별 규칙은 하위 AGENTS.md로 분리
  • 개인 노트는 AGENTS.override.md (커밋 X)

AI가 짠 코드 — 5가지 합의

  1. PR 본문에 "AI 작성" 표시 의무 없음 — 편향 방지
  2. 리뷰 기준은 동일 — correctness·테스트·보안·가독성
  3. 작성자가 책임자 — 모든 라인 이해 필수
  4. "안 읽고 머지" 절대 금지
  5. 큰 PR은 쪼개기

비밀 정보 — 5가지 절대 규칙

  • ❌ .codex/config.toml(커밋)에 토큰 직접 입력
  • ❌ MCP env에 토큰 하드코딩 (반드시 ${ENV_VAR})
  • ❌ Codex한테 cat .env 시킨 결과 두기 — 세션 로그에 박힘
  • ✅ 환경변수는 OS·shell·1Password CLI
  • ✅ git secrets scan·GitHub secret scanning 활성화
⚠️ 세션 로그에 토큰

Codex 세션 로그는 ~/.codex/sessions/에 저장. 토큰을 한 번 화면에 띄웠으면 그 파일에 남음. 토큰 즉시 회전.

사례 — 망한 / 잘된

🎬 망한 사례 · "danger profile 디폴트"
시작 첫 주, 편하다고 모두 danger profile 사용. 한 개발자가 prod DB에 실수 실행 → 데이터 손상. 교훈: dev profile + ask로 위험 명령 가드.
🎬 잘된 사례 · "Cloud + Codex Action 이중화"
30명 팀. Codex Action으로 모든 PR 자동 리뷰. 사람은 설계 리뷰만 집중. 결과: 평균 머지 2일 → 5시간.
