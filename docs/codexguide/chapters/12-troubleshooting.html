Chapter 12

트러블슈팅 — Codex가 이상하게 행동할 때

강의 현장에서 가장 자주 받는 SOS 8가지와 해결법.

1) approval이 무한 반복

증상: 같은 명령(pnpm test)인데 매번 승인 요청.

원인: approval_policy = "untrusted"이거나 granular 정책으로 모든 도구 호출에 묻는 상태.

해결:

# 일시 변경
> /approvals on-request

# 영구 변경 ~/.codex/config.toml
approval_policy = "on-request"

2) sandbox가 네트워크·파일을 차단

증상: npm install 시도 → "network access denied".

원인: sandbox_workspace_write.network_access = false가 기본.

# 임시 (이 세션만)
> /sandbox workspace-write
# 후 approval 1회 받고 진행 가능

# 또는 ci profile 같은 별도 프로파일로 전환
> /profile ci

3) 컨텍스트 폭발 — /compact도 효과 없음

# 1단계: 명시적 요약
> "결론·다음 액션 3가지만 요약. 분석 과정은 버려도 됨."
> /compact

# 2단계: 새 세션으로
> "위 요약을 새 세션 시작 프롬프트로 마크다운 정리"
> /quit
$ codex
> [요약 붙여넣기]

# 또는 codex resume으로 깔끔히 분리

4) 잘못 수정한 파일 복구

상황방법
방금 작업만 되돌리기세션 안에서 /undo 또는 "마지막 변경 되돌려"
여러 단계 전git log + revert
git에 한 번도 커밋 안 됨이미 늦음. Codex 응답의 diff 역적용

예방: 큰 작업 직전 git commit -m "WIP" 30초 투자.

5) 토큰 폭주

# 진단
# Platform → Usage 대시보드
# 또는 ~/.codex/sessions/ 의 jsonl 직접 분석

흔한 범인 5가지: 큰 로그 grep · 스크린샷 다수 · node_modules 검색 · 큰 JSON cat · 반복 재시도.

6) codex resume가 세션을 못 찾음

원인: ~/.codex/sessions/ 경로 변경, 또는 세션 expire.

ls ~/.codex/sessions/  # 세션 파일 직접 확인
codex resume <session-id>  # ID 직접 지정

7) 자주 만나는 에러

메시지의미대처
sandbox_mode deniedsandbox가 차단/sandbox로 변경 또는 다른 profile
approval rejected사용자가 거절의도면 OK, 아니면 profile·정책 재검토
Rate limit (429)OpenAI 한도대기 또는 모델 다운그레이드
Context window exceeded컨텍스트 초과위 3번 + 모델 변경
MCP server not respondingMCP 서버 다운codex mcp list로 점검
insufficient_quotaAPI 잔액 부족Platform → Billing

/status — 막힐 땐 첫 명령

현재 모델·profile·sandbox·approval·MCP 상태를 한 번에 점검. "뭐가 문제인지 모르겠음"이면 가장 먼저.

최후의 수단 — 새 세션

  • 같은 질문을 3번 했는데 답이 모순
  • 응답 첫 토큰까지 30초+
  • 본인부터 "뭘 시키고 있었지?" 헷갈림

12장 체크리스트

  • ☐ profile 4종(dev·review·ci·danger) 정의로 approval 노이즈 줄임
  • ☐ 큰 작업 전 git commit -m "WIP" 습관
  • ☐ Usage 대시보드 주 1회 확인
  • ☐ 문제 생기면 /status가 첫 명령
  • ☐ 막히면 새 세션 또는 codex resume <id>
