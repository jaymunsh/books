Chapter 02

슬래시 명령·3축 권한·AGENTS.md·config.toml

Codex CLI의 "조종 장치" 네 가지를 한 번에 익힙니다. 슬래시 명령으로 세션을 통제하고, 3축 권한(approval·sandbox·network)으로 안전을 보장하며, AGENTS.md로 프로젝트 컨텍스트를 영구 학습시키고, config.toml로 전역 설정을 관리합니다.

자주 쓰는 슬래시 명령

전부 외울 필요는 없습니다. /help로 언제든 목록을 볼 수 있어요.

세션 관리

명령동작
/help사용 가능한 모든 명령 목록
/clear현재 대화 히스토리 비우기
/compact대화 컨텍스트 압축
/new새 세션 시작 (현재 세션 보존)
/quit 또는 Ctrl+D세션 종료

설정·메모리·권한

명령동작
/initAGENTS.md 자동 생성 / 갱신
/model모델 변경 (gpt-5.5 / 5.4-mini / spark)

💡 명령 자동완성

슬래시(/)만 치면 사용 가능한 명령이 드롭다운으로 떠요. 실제로 외울 명령은 위 표 정도면 충분.

3축 권한 시스템 — Codex만의 차별점

Codex의 권한은 3개의 독립된 축으로 동작합니다. Claude Code의 단일 allow/ask/deny보다 세밀합니다.

① approval_policy — 언제 사용자에게 물을까

값동작사용 시기
untrusted모든 도구 호출에 승인 요청의심스러운 코드베이스·첫 작업
on-request (기본)위험·민감 작업만 승인 요청일반 개발
never승인 요청 없이 모두 진행격리된 컨테이너·CI
granular도구 카테고리별 세밀 제어특정 도구만 차단·자동화

② sandbox_mode — 기술적 경계

값동작사용 시기
read-only파일 읽기·grep만, 수정 X코드 리뷰·탐색
workspace-write (기본)워크스페이스 내 읽기·수정·로컬 명령일반 개발 (디폴트)
danger-full-accesssandbox 해제, 전 시스템 접근VM·컨테이너 안에서만

③ network_access — 외부 통신

workspace-write의 하위 옵션으로 기본 false. npm install 같은 네트워크 명령은 그때마다 approval. 자동화에선 true.

config.toml로 묶어서 설정

# ~/.codex/config.toml (전역) 또는 프로젝트별
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[sandbox_workspace_write]
network_access = false

model = "gpt-5.5"

실시간 전환

> /approvals            # GUI로 변경
> /sandbox read-only    # 즉시 변경
> /status               # 현재 상태 확인
🎓 강사 노트 — "안전은 sandbox로, 편함은 approval로"

두 축의 역할을 헷갈리기 쉽습니다. sandbox는 "물리적 경계"(이 폴더 밖은 못 만짐), approval은 "확인 시점"(이 작업은 묻고 가자). 가장 흔한 실무 조합: sandbox_mode = "workspace-write" + approval_policy = "on-request" — 워크스페이스 안에서만 + 위험 작업만 확인.

AGENTS.md — 프로젝트 영구 컨텍스트

AGENTS.md는 매 세션 자동으로 컨텍스트 윈도우에 주입되는 지시사항 파일입니다. Claude Code의 CLAUDE.md에 대응하지만, Codex만의 차별점은 AGENTS.override.md 우선순위입니다.

위치와 우선순위

경로범위우선순위
~/.codex/AGENTS.override.md개인 전역 override1 (가장 강함)
~/.codex/AGENTS.md개인 전역 (override 없을 때만)2
./AGENTS.md (프로젝트 루트)이 프로젝트 (팀 공유)3
./하위경로/AGENTS.md해당 경로만4 (가장 좁음)

전부 발견되면 컨텍스트에 합쳐 주입됩니다. Override는 글로벌 AGENTS.md를 완전 대체합니다.

잘 쓴 AGENTS.md 예시

# Backend API Project

## Quick Start
- Install: `pnpm install`
- Dev: `pnpm dev` (http://localhost:3000)
- Test: `pnpm test`
- Build: `pnpm build`

## Architecture
- Entry: `src/index.ts`
- Routes: `src/routes/`
- Models: `src/models/` (Drizzle schemas)

## Code Standards
- Indentation: 2 spaces
- camelCase for functions, PascalCase for classes
- TypeScript strict mode required
- Run `pnpm lint` before commit

## Forbidden
- raw SQL — Drizzle ORM only
- Direct DB queries in route handlers — service layer 경유
- console.log — pino logger only

작성 원칙 5가지

  1. 200줄 이하 목표 — 매 세션 토큰 소모. 길수록 무시될 가능성 증가.
  2. 구체성 — "좋은 포맷" 대신 "2-space 인덴트".
  3. 해야 할 것 + 하지 말 것을 같이 명시.
  4. 큰 규칙은 하위 경로 AGENTS.md로 분리 (자동으로 해당 경로 작업 시만 로드).
  5. /init으로 초안 생성 후 다듬는 게 가장 빠릅니다.
🎬 시나리오 · "전사 글로벌 규칙 + 개인 override"
상황: 회사가 ~/.codex/AGENTS.md에 전사 규칙(코딩 컨벤션·금지 라이브러리)을 깔아 둠. 본인은 추가로 본인 취향(이모지 사용·언어)을 더하고 싶음. 적용: ~/.codex/AGENTS.override.md에 본인 규칙 + 전사 규칙 인용. Override는 글로벌을 완전 대체하므로, 회사 규칙도 여기에 복사 + 본인 규칙 추가. 결과: 본인만의 워크플로우 유지, 회사 규칙도 깨지지 않음.

config.toml — 어디에 무엇이 들어가나

Codex의 모든 설정은 TOML 형식입니다(Claude Code의 JSON과 다름). 4계층으로 적용:

레이어경로공유 여부
전역 (사용자)~/.codex/config.tomlX (개인)
프로젝트 (공유)./.codex/config.toml✅ (커밋)
프로젝트 (개인)./.codex/config.local.tomlX (gitignore)
CLI 인자codex --model gpt-5.4-mini임시

아래로 갈수록 우선순위가 높습니다.

자주 쓰는 키

# 모델 · 추론
model = "gpt-5.5"
reasoning_effort = "medium"     # low / medium / high

# 권한 3축
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[sandbox_workspace_write]
network_access = false

# 환경변수 (도구 실행에 주입)
[env]
NODE_ENV = "development"

# UI
theme = "dark"

# 메모리·컨텍스트
auto_compact = true
⚠️ 비밀은 절대 config.toml에

API 키·토큰은 config.toml(커밋되는 경우 위험)에 넣으면 큰 일. 반드시 config.local.toml 또는 OS 환경변수로.

2장 체크리스트

  • ☐ /help로 명령 목록 한 번 훑기
  • ☐ 3축(approval·sandbox·network)의 역할 분리 이해
  • ☐ ~/.codex/config.toml에 본인 기본 설정
  • ☐ AGENTS.md를 200줄 이내로 정리
  • ☐ (선택) AGENTS.override.md로 본인 취향 분리
  • ☐ /status로 현재 권한·sandbox 확인 습관
