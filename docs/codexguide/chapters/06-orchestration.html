Chapter 06

멀티 에이전트 오케스트레이션

Codex는 Claude Code의 "Workflow 도구" 같은 결정적 스크립트 시스템은 없지만, Codex Cloud 병렬 작업 + 모델 분리(gpt-5.4-mini 서브에이전트) + Agents SDK(7장)로 같은 효과를 낼 수 있습니다.

3가지 접근 방식

방식적합 상황도구
Cloud 병렬독립적 작업 여러 개 동시 진행Codex Cloud (3장)
모델 분리비용 격리 — 단순 작업은 mini로config.toml + /model
Agents SDK코드로 결정적 오케스트레이션OpenAI Agents SDK (7장)

방식 1 · Codex Cloud 병렬 작업

가장 직관적이고 강력. 3장에서 다룬 Cloud의 병렬 기능을 멀티 에이전트로 활용.

전형적 시나리오 — 큰 PR을 모듈별 분할

> "이 마이그레이션 작업을 Cloud로 분할 위임:
   - Task 1: src/auth/ 의 OAuth 마이그레이션
   - Task 2: src/api/ 의 응답 형식 변환
   - Task 3: src/models/ 의 schema 업데이트
   - Task 4: tests/ 의 fixture 갱신
   각각 별도 PR로."

# Cloud가 4개 태스크 병렬 실행
# 30분~1시간 후 4개 PR 준비됨

방식 2 · 모델 분리 — 비용 격리 패턴

같은 세션 안에서 작업 유형별로 다른 모델 사용. 단순 분류·변환은 gpt-5.4-mini, 추론은 gpt-5.5.

# 세션 안에서 모델 전환
> /model gpt-5.4-mini
> "이 1000개 로그 라인을 [INFO/WARN/ERROR] 분류해줘"

> /model gpt-5.5
> "위 결과에서 ERROR 패턴 3가지 가설 제시 (복잡 추론 필요)"

전용 config — 단순 작업 자동화 스크립트

# config.toml — 정형 작업용 mini 프로파일
[profiles.bulk]
model = "gpt-5.4-mini"
reasoning_effort = "low"
approval_policy = "never"
sandbox_mode = "workspace-write"

# 대량 분류 자동화
codex exec --profile bulk "data/feedback.jsonl 의 5000건을 [Bug/Feature/Other] 분류"
🎓 강사 노트 — gpt-5.4-mini의 진가

정형 작업에선 gpt-5.5와 차이가 거의 없으면서 비용은 1/5 수준. 분류·요약·변환·라벨링·간단한 코드 생성에 mini를 적극 활용하세요. 복잡 추론·설계만 gpt-5.5로 올리면 청구액이 크게 줄어듭니다.

방식 3 · OpenAI Agents SDK (7장에서 자세히)

코드로 결정적 오케스트레이션을 짜고 싶다면 Agents SDK를 씁니다. Codex CLI를 임베드하는 게 아니라, OpenAI 모델을 직접 호출해 본인만의 멀티 에이전트 시스템 구축.

# Python — 5개 finder를 병렬로
from openai import AsyncOpenAI
import asyncio

client = AsyncOpenAI()

async def finder(area):
    r = await client.responses.create(
        model="gpt-5.5",
        input=f"src/{area}/ 의 보안 취약점 분석"
    )
    return { "area": area, "findings": r.output_text }

async def main():
    results = await asyncio.gather(*[
        finder(a) for a in ["auth", "api", "db", "middleware", "config"]
    ])
    return results

asyncio.run(main())

5가지 패턴 (Codex 컨텍스트로 재해석)

패턴구현예
Pipeline한 세션에서 순차 위임"X 분석 → Y 검증 → Z 보고서"
ParallelCodex Cloud 또는 SDK asyncio.gather5개 영역 보안 감사 병렬
Judge panelSDK로 N명 독립 호출 + 다수결중요 결정에 다중 검증
Loop-until-drySDK 코드로 발견 없을 때까지버그 탐색
모델 분리config.toml 프로파일mini로 분류 → 5.5로 분석

실전 예시 — 보안 감사 (3축 결합)

# 1) Cloud 병렬 — 5개 영역 동시 스캔
> "Cloud로 5개 영역 동시 보안 감사:
   - auth/ 의 인증·세션 취약점
   - api/ 의 입력 검증 누락·SQLi
   - 전체 비밀 정보 하드코딩
   - 의존성 취약점
   - docker·CI 설정 미스컨피그
   각각 별도 보고서로."

# 2) 결과 모이면 mini로 dedup·정리
> /model gpt-5.4-mini
> "위 5개 보고서를 dedup + 심각도 정렬 표로"

# 3) gpt-5.5 + reasoning high로 최종 추천 액션
> /model gpt-5.5
> "위 정리된 발견 중 CRITICAL/HIGH 3개에 대한 패치 방안 제시"

모범 사례 & 안티패턴

✅ 모범❌ 안티패턴
독립 작업 → Cloud 병렬한 세션에 순차 위임 (시간 낭비)
단순 작업은 mini로 분리모든 작업에 gpt-5.5 (비용 폭주)
중요 결정엔 Judge panel한 번의 응답으로 결론 (검증 부족)
Loop-until-dry로 자동 종료임의 "N번까지만" cap (놓침)
중간에 진행 상황 보고10분 침묵 → 끊고 싶음

6장 체크리스트

  • ☐ Cloud 병렬 1번 시도 — 작업 3개 동시 위임
  • ☐ profile.bulk 정의 (gpt-5.4-mini + low effort)
  • ☐ 같은 세션에서 모델 전환 패턴 체득
  • ☐ (선택) Agents SDK로 본인만의 오케스트레이션 1개
  • ☐ "단순 작업 = mini" 원칙 본인 워크플로우에 도입
