Chapter 04

MCP — Codex를 세상과 연결하기

MCP(Model Context Protocol)는 Codex가 외부 서비스를 도구처럼 다루기 위한 개방 표준입니다. GitHub·Slack·DB·브라우저 자동화까지 같은 방식으로 연결합니다.

MCP가 무엇이고 왜 필요한가

"Codex한테 GitHub PR 정보를 직접 읽게 하고 싶다", "Sentry 에러를 분석하게 하고 싶다" — 이런 욕구를 매번 따로 통합하지 않아도 되게 만든 게 MCP입니다. Anthropic이 처음 표준화했지만 OpenAI/Codex도 같은 프로토콜을 지원합니다.

codex mcp add — MCP 서버 추가

STDIO 서버 (로컬)

# 기본 형태
codex mcp add <name> -- <실행 명령>

# 예: Playwright 브라우저 자동화
codex mcp add playwright -- npx -y @playwright/mcp@latest

# 예: 내부 파이썬 MCP 서버
codex mcp add my-db -- python3 /opt/mcp/my_db_server.py

HTTP 서버 (호스팅)

# 기본 형태
codex mcp add --transport http <name> <URL>

# 예: Sentry MCP (인증 필요)
codex mcp add --transport http sentry https://mcp.sentry.dev/mcp

# 인증은 세션 안에서
> /mcp
# → 서버 선택 → Authenticate → 브라우저 OAuth

설정 파일로 한꺼번에 — config.toml

# ~/.codex/config.toml 또는 ./.codex/config.toml

[mcp_servers.playwright]
type = "stdio"
command = "npx"
args = ["-y", "@playwright/mcp@latest"]

[mcp_servers.sentry]
type = "http"
url = "https://mcp.sentry.dev/mcp"

[mcp_servers.my-db]
type = "stdio"
command = "python3"
args = ["/opt/mcp/my_db_server.py"]
env = { DB_HOST = "localhost" }

프로젝트 루트 ./.codex/config.toml에 두면 팀원 모두 같은 MCP 서버를 자동으로 갖게 됩니다.

관리 명령

codex mcp list               # 연결된 서버 목록
codex mcp get <name>        # 단일 서버 정보
codex mcp remove <name>     # 제거
/mcp                         # 세션 안 GUI (인증 포함)

MCP 권한·보안

MCP 서버는 실제 외부 시스템에 행위를 합니다. 권한을 잘 잠그지 않으면 위험.

3가지 안전 장치

  1. scope 분리: 민감 서버는 ~/.codex/config.toml(개인 전역)에, 공통 서버는 프로젝트 루트에.
  2. 3축 권한: MCP 도구도 3축 권한(2장) 적용 받음. 위험 MCP는 approval 강제.
  3. 토큰 환경변수: 토큰을 config.toml에 직접 박지 말고 ${ENV_VAR} 참조.
[mcp_servers.github]
type = "http"
url = "https://mcp.github.com/api"
env = { GITHUB_TOKEN = "${GITHUB_TOKEN}" }
⚠️ prod DB·결제 등 위험 서버

프로덕션 DB나 결제 API의 쓰기 권한 MCP는 절대 approval_policy = "never"와 조합하지 마세요. 그런 서버는 approval_policy = "untrusted" + 읽기 전용 토큰 + 도구 단위 deny가 안전.

GitHub CLI(gh) — MCP보다 가볍게 쓰기

GitHub은 MCP뿐 아니라 gh CLI도 잘 동작합니다. 권한이 이미 잡혀 있고, Shell로 호출만 하면 끝나서 부담이 적어요.

# 인증 (1회)
gh auth login

# Codex가 이 명령을 알아서 호출함
> "현재 변경사항으로 PR 만들어줘. 제목은 변경 요약, 본문은 변경 파일 목록과 테스트 결과"

# 직접 예
gh pr create --title "feat: add auth" --body "..."
gh pr list
gh pr view 1234
gh pr checks 1234
🎓 강사 노트 — gh vs GitHub MCP

일반 PR/이슈 작업은 gh CLI가 가볍고 빠릅니다. 반복 자동화(예: 한 시간 동안 여러 PR을 다루는 워크플로우)는 MCP가 일관성 있게 동작합니다. 보통 gh부터 시작하고, 필요할 때 MCP 추가.

Git Worktree — Codex를 격리해서 안전하게

큰 작업이나 위험한 변경을 시킬 때는 별도 worktree에서 작업시키는 게 깔끔합니다.

# 새 기능 브랜치를 worktree로
git worktree add ../my-proj-feature feature/payments
cd ../my-proj-feature
codex
# → 메인 작업 디렉토리는 영향 없음

# 정리
git worktree list
git worktree remove ../my-proj-feature

Codex Cloud(3장)와 조합하면 더 강력: 큰 작업은 Cloud로, 작은 격리 작업은 worktree로.

4장 체크리스트

  • ☐ codex mcp list로 현재 연결 확인
  • ☐ Playwright 1개라도 깔아 보고 "스크린샷 찍어줘" 시험
  • ☐ gh auth login → Codex로 PR 자동 생성
  • ☐ MCP 토큰은 환경변수로만
  • ☐ 위험 MCP는 approval 강제 (3축 활용)
  • ☐ 위험 작업은 worktree로 격리
