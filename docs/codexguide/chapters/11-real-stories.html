📖 스토리 B · URL 단축 SaaS — 풀 자동화

주인공스타트업 3년차 풀스택 K. 사이드 프로젝트로 URL 단축 서비스.
스택Cloudflare Workers + D1 + Hono · React 대시보드
전략permission profile 3종 + Codex Cloud + MCP + GitHub Action

🛠 Day 0 — 셋업 (2시간)

K는 코드 한 줄 전에 셋업에 투자.

# .codex/config.toml (커밋 대상)
[profiles.dev]
model = "gpt-5.5"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
[profiles.dev.sandbox_workspace_write]
network_access = false

[profiles.review]
model = "gpt-5.5"
approval_policy = "never"
sandbox_mode = "read-only"

[profiles.ci]
model = "gpt-5.4-mini"
approval_policy = "never"
sandbox_mode = "workspace-write"
[profiles.ci.sandbox_workspace_write]
network_access = true

[mcp_servers.playwright]
type = "stdio"
command = "npx"
args = ["-y", "@playwright/mcp@latest"]

🎓 강사 노트 — 절제의 미학

S는 도구를 더 쓸 줄 알면서도 안 썼습니다. "두 명만 위한 것, SaaS 안 키움"이라는 명확한 범위. 그러면 Cloud·MCP·Action·CI는 오버킬. 본인 프로젝트가 어느 스토리에 가까운지 자문해 보세요.

세 스토리에서 추리는 5가지 교훈

  1. 도구의 무게 = 프로젝트의 무게에 맞춰라
  2. "파일 만들기 전에 계획부터"는 모든 스토리 공통
  3. 혼자일수록 자동화 가치가 큼 (B 스토리)
  4. AGENTS.md는 짧을수록 좋다 (B도 30줄, C는 10줄)
  5. 3축 권한 기본값이 의외로 잘 보호 — 작은 프로젝트는 기본만으로 OK

11장 체크리스트
