Chapter 03 · Codex 차별점

Codex Cloud · Permission Profiles

Codex만의 가장 큰 차별점 두 가지를 한 장에서. Codex Cloud는 클라우드에서 백그라운드 작업을 돌리고 IDE에서 결과만 받는 워크플로우를, Permission Profiles(v0.129+)는 3축 권한을 묶어 팀과 공유하는 방법을 제공합니다.

Codex Cloud — 클라우드 백그라운드 에이전트

Codex CLI가 "로컬에서 돌리는 에이전트"라면, Codex Cloud는 "OpenAI의 클라우드 환경에서 돌고 결과만 받는 에이전트"입니다. IDE·웹·GitHub 어디서든 작업을 던져두고 다른 일 하다가 결과를 받아옵니다.

핵심 기능 5가지

기능설명
백그라운드 작업위임 후 본인은 다른 일. 완료 시 알림.
병렬 실행여러 작업 동시 진행 가능. 큰 PR을 모듈별로 분할 위임.
IDE 통합VS Code·JetBrains에서 우클릭 → "Send to Codex Cloud". 결과를 IDE에서 diff로 검토.
GitHub 통합이슈·PR에 멘션으로 작업 위임. 클라우드가 브랜치 만들고 PR 생성.
UI 스크린샷프론트엔드 작업 시 GPT-5-Codex가 브라우저 스크린샷 첨부해 결과 보고.

Cloud vs Local — 언제 무엇

상황Local CLICodex Cloud
빠른 인터랙티브 작업✅✗ (대기 시간)

전형적 사용 흐름

# 1) IDE에서 큰 작업 우클릭 → Send to Codex Cloud
#    또는 web UI에서 "New Task"

# 2) Codex Cloud가 격리된 환경에서 작업 진행
#    - 본인 리포 clone
#    - 격리된 컨테이너에서 빌드·테스트
#    - 변경 사항 diff 생성

# 3) 본인은 다른 일
#    - 작은 버그픽스
#    - 리뷰

# 4) 완료 알림 → IDE에서 결과 확인
#    - "Apply locally" 버튼으로 로컬에 적용
#    - 또는 "Create PR"로 바로 PR
🎬 시나리오 · "PR 5개를 동시에 — Codex Cloud 병렬"
상황: 백로그에 작은 작업 5개. 하나씩 처리하면 하루 종일 걸림. 적용: 
# 5개를 동시에 Codex Cloud로 위임
> "이슈 #234·#235·#236·#237·#238을 각각 별도 PR로 만들어 Codex Cloud로 보내줘"

# 30분 뒤: 5개 PR 모두 준비됨 → 리뷰만 하면 끝
결과: 하루 → 1시간. 검토만 사람이 하면 됨.

UI 검증 (GPT-5-Codex 스크린샷)

프론트엔드 작업 시 Codex Cloud는 실제 브라우저로 렌더링한 스크린샷을 결과와 함께 첨부합니다. 디자인 변경의 시각 회귀 확인에 매우 유용.

> "로그인 폼의 입력 검증 에러 메시지를 한국어화하고,
   변경 후 모바일(360px)·데스크탑(1280px)에서 스크린샷 찍어줘"

# Codex Cloud:
# 1) 코드 수정
# 2) dev 서버 자동 띄움
# 3) Playwright로 두 viewport 캡처
# 4) 결과에 스크린샷 2장 첨부

한계

  • 로컬 환경 의존 빌드(예: 특수 빌드 도구, 개인 인증) 작업은 어려움
  • 비밀 정보 접근 작업은 별도 secret 설정 필요
  • 실시간 인터랙티브 대화는 Local CLI보다 답답
  • 네트워크 의존성·외부 API 호출은 별도 설정

Permission Profiles (v0.129+) — 권한 묶음 공유

2장의 3축(approval·sandbox·network)을 매번 따로 설정하면 번거롭습니다. Permission Profile은 "이름 붙인 권한 묶음"으로, 한 번 정의해 두고 프로젝트·팀 단위로 공유합니다.

정의 — config.toml

# ~/.codex/config.toml 또는 ./.codex/config.toml

[profiles.dev]
approval_policy = "on-request"
sandbox_mode = "workspace-write"
[profiles.dev.sandbox_workspace_write]
network_access = false

[profiles.review]
approval_policy = "never"
sandbox_mode = "read-only"
[profiles.review.sandbox_workspace_write]
network_access = false

[profiles.ci]
approval_policy = "never"
sandbox_mode = "workspace-write"
[profiles.ci.sandbox_workspace_write]
network_access = true  # 의존성 설치 필요

[profiles.danger]
approval_policy = "untrusted"
sandbox_mode = "danger-full-access"

사용

# CLI 옵션으로
codex --profile review
codex --profile ci

# 세션 안에서 전환
> /profile review
> /profile dev
💡 추천 프로파일 4종
  • dev: 일반 개발 — 워크스페이스 쓰기 + 위험 작업만 승인 + 네트워크 차단
  • review: 코드 리뷰 전용 — 읽기 전용 + 승인 없음 (안전)
  • ci: CI/CD — 자동 승인 + 네트워크 허용 (의존성 설치)
  • danger: VM/컨테이너 격리 환경 — 풀 액세스 + 모든 승인

팀 공유 — 프로젝트 ./.codex/config.toml에 커밋

# 프로젝트 루트 .codex/config.toml — 팀 전체가 같은 프로파일
[profiles.team-default]
approval_policy = "on-request"
sandbox_mode = "workspace-write"
[profiles.team-default.sandbox_workspace_write]
network_access = false
allowed_write_paths = ["src/**", "tests/**", "docs/**"]

[profiles.team-strict]
approval_policy = "untrusted"
sandbox_mode = "read-only"

팀원 모두 이 프로파일로 작업하므로 "내 PC에선 잘 됐는데" 문제가 사라집니다.

🎓 강사 노트 — Two-layer Security Model

Codex의 공식 표현: "two-layer security model — sandbox enforcement controls what the agent can technically do, while approval policies control when it must ask permission, and both layers must agree before an action proceeds."

두 층이 동시에 통과해야 실행됩니다. 한쪽 풀어도 다른 쪽이 막아주는 보완 구조.

Cloud + Profile 조합 — 가장 강력한 패턴

두 가지를 합치면 "팀 단위 정책 + 클라우드 자동화"가 완성됩니다.

전형적 운영

  1. 팀 프로파일 정의 (.codex/config.toml)
  2. 이슈 자동 처리: GitHub에 이슈 → @codex /profile team-default 멘션 → Cloud가 자동 작업
  3. 결과 PR이 같은 프로파일로 만들어짐 (일관성)
  4. 본인 리뷰: IDE에서 review 프로파일로 안전하게 확인

3장 체크리스트

  • ☐ Codex Cloud 한 번 시도 — 작은 작업 하나 위임
  • ☐ Local CLI vs Cloud 적합한 상황 구분 표 이해
  • ☐ ~/.codex/config.toml에 프로파일 4종 (dev·review·ci·danger) 정의
  • ☐ /profile로 세션 안에서 전환 시도
  • ☐ (팀) .codex/config.toml에 팀 공통 프로파일 커밋
  • ☐ "two-layer security" 직관 체득
